03.技术架构

# 一、技术架构设计

# 1.技术架构概述

技术架构是从物理层面定义对系统进行定义，主要解决系统的非功能性问题。技术架构从软硬件的处理能力出发，提供系统的高可用、高性能和可伸缩等能力。

业务架构解决的是系统功能性问题，主要是从人出发，让程序维护者去更好地理解系统。而技术架构解决的是系统非功能性问题，从软硬件节点的处理能力出发。

目前软件架构中，大多数都是应用层面的开发场景，开发人员主要针对业务逻辑开发，最终将代码打包运行在服务器。不过，代码成功运行需要许多其他系统的支持，例如下面的系统物理模型：

上面图中可以看到，应用系统的运行不单单需要只把业务代码开发好即可，还需要计算机硬件、网络、操作系统、JVM、数据库、外部中间件、运维监控等等的支持。因此，这些依赖到的系统或组件都可能会是技术架构中的瓶颈问题。

# 2.技术架构的问题

技术架构的职责：负责系统所有组件的技术选型，并确保组件可以正常运行。

技术架构问题产生的来源：

  • 硬件问题：

    • 硬件的处理能力有限。
    • 硬件不是 100% 的可靠，它本身也会出问题。

  • 软件问题

硬件的问题可以从水平扩展和垂直扩展的角度处理：

  1. 垂直扩展：就是通过升级硬件来提升处理能力。如升级服务器的CPU、内存、带宽等硬件上的提升。
  2. 水平角度：即通过增加硬件数量来提升处理能力。如一台机器的处理能力有限，就增加机器，可以通道大量廉价的机器设备叠加，增强整个系统的处理能力。

如果碰到性能问题，垂直扩展是我们的首选，垂直扩展也是最简单的方式，但是它有性能的瓶颈，并且硬件超过主流的成本，成本会成指数级增加。此时，就要使用水平扩展弥补性能问题，水平扩展理论上可以应对所有服务器处理能力不足的情况，并且硬件成本和系统处理能力是线性增长的关系。

不过，水平扩展的组件增多，管理大量的机器也是一个问题，我们需要通过复杂的技术架构设计来保障系统中多个组件有效衔接，稳定运行。如负载均衡等方式。

机器数量变多，也放大了系统出故障的概率，导致系统整体的可用性变差。如服务器断电，网络问题、自然灾害导致机房不可用等。因此，技术架构设计，需要充分考虑硬件故障的可能性。

软件的问题

软件弥补了硬件的缺陷。如 Redis 集群的数据分片解决单机服务器内存和带宽的瓶颈问题，多副本和故障节点增加系统的可用性。

软件代码的封装，可以高效地访问系统资源。如数据库使数据的存取更高效，缓存使热点数据的访问更高效。

不过，软件也会带来各种维护的问题。如 Redis 集群的多节点会有内部的网络问题，Redis 数据的多副本也会带来数据的一致性问题。当然这些问题需要根据实际的系统，去选择是否使用。这一点就对应 CAP 理论。

CAP 理论：

  • Consistency 一致性：所有节点在同一时间访问到的数据是完全相同的。一旦数据更新，所有用户都能立即看到最新结果，不存在数据不一致的情况。
  • Availability 可用性：系统在任何时候都能正常响应请求。即使部分节点出现故障，用户的读写操作也能在合理时间内得到成功反馈。
  • Partition Tolerance 分区容错：系统能够容忍网络分区故障。当网络发生中断，导致部分节点之间无法通信时，系统依然能够正常运行，不会因为网络问题而整体瘫痪。

分布式系统的必须满足 P，架构师的核心选择就简化为在 CP 和 AP 之间做权衡。

# 3.技术架构的目标

  • 系统的高可用
  • 系统的高性能
  • 系统的可伸缩
  • 系统的低成本
  • 其他

系统的高可用

可用性的衡量标准是，系统正常工作的时间除以总体时间。

导致系统可用性出问题有如下情况：

  1. 软硬件本身有故障。如机器断电，网络不通。
    • 对于这种情况，我们要么及时解决当前节点的故障问题，要么做故障转移，让备份系统快速顶上。
  2. 高并发引起的系统处理能力的不足。如 CPU 100% 的时候，整个系统完全不工作。
    • 对于这种情况，我们要么提升系统的处理能力，比如采取水平扩展、缓存等措施；要么把流量控制在系统能处理的水平，比如采取限流、降级等措施。

系统的高性能

高性能指的是系统要提供合理的性能，也不是非要绝对性能要多高。

保证系统性能可以采用如下方案：

  1. 对于常规流量，但是处理复杂的情况，我们可以运用技术手段进行优化。如针对海量商品的检索，可以使用搜索系统（ElasticSearch）。
  2. 对于高并发的流量，流量的提升是很难准确预计的，这就要求我们在架构设计时，让系统能够整体上做水平扩展。

系统的可伸缩和低成本

系统的业务量在不同的时间点，有高峰有低谷。如餐饮行业、外卖有用餐高峰，电商的大促场景。因此，在架构设计时要保证系统在业务高峰时，能快速地增加资源来提升系统处理能力；当业务低谷时，可以快速地减少系统资源，保证系统的低成本。

高可用、高性能、可伸缩和低成本，这些技术架构的目标不是孤立的，相互之间有关联。如高峰流量时，系统通过水平扩展，保证系统有高性能，同时也实现了系统的高可用。如果系统的处理能力无法快速提升，无法保证高性能，那我们还是可以通过限流、降级等措施，保证核心系统的高可用。

当然，这些目标很多时候会冲突，这时就要求我们在做技术架构设计时，要根据业务特点，选择最关键的目标实现。

除了以上的目标，还有易部署维护，好管理等目标，架构设计也要从人出发，充分考虑团队成员对技术的熟悉程度等，毕竟是人来维护。很多时候技术架构要求的场景比较复杂，这就要求架构师充分结合实际场景进行架构设计。
