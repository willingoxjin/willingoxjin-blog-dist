01.架构基础

# 一、关于架构设计

# 1.什么是架构？

所谓架构，就是在宏观视角下，通过整体规划，使得各个系统、模块、组件、框架等按照预定模式进行运作。

实际上，软件架构和建筑设计是一样的道理：

  1. 如果我们在家乡建一间小房子，只需要大概丈量一下土地，告诉泥瓦匠大概情况，师傅就会按照他丰富的经验直接动工。
  2. 但是我们要盖一个自建的二层小房子，就需要考量一下楼层高度，楼梯位置，厕所位置等等，即使有了经验丰富的老师傅，但至少也要对房屋整体进行粗略规划。
  3. 如果盖一座高楼大厦，需要考虑的点更多，比如地基深度，钢筋结构、楼层、消防、电梯、水电管道、房间户型等等。因此需要建筑设计师做好整体规划，通过不断修改，最终敲定才能动工。

一个小型的系统，是不需要过于复杂的设计的。而当软件系统达到一定规模时，导致业务或技术的复杂性问题暴露，这时就需要对整个系统重新梳理，规划系统。

相关问题如下：

  • 系统规模庞大，内部耦合严重，开发效率低。
  • 模块或系统之间耦合严重，牵一发动全身，后续修改和扩展困难。
  • 业务逻辑复杂，容易出问题，并且出问题后很难排查和修复。

# 2.系统、模块、组件、框架、架构

下面我们来解释一下系统、模块、组件、框架、架构这些名词的意思以及关系。

1.系统与子系统

在软件领域，系统指的是一组相互关联、相互作用的软件组件和资源，它们协同工作以实现特定的功能或提供一系列相关的服务。

子系统的定义和系统相同，只不过一个系统可能是由多个子系统组成的。例如一个电商系统，由用户、商品、购物车、订单、支付、物流、评价等子系统组成。当然有时候子系统之间也会存在依赖关系。

子系统也可能由其他的子系统组成，例如上面的评价系统，由审核、评论、点赞子系统组成。

2.模块与组件

模块和组件的定义很容易混淆。实际上两者有不同之处：

  • 模块：
    • 模块是从逻辑的角度来拆分系统得到的单元。
    • 模块划分用于职责分离，模块内的内容是紧密相连的，
  • 组件：
    • 组件是从物理的角度来拆分系统得到的单元。
    • 组件划分用于复用，也可以被替换。

例如一个网课系统，按照业务逻辑划分，可分为登录注册、用户信息、我的课程、课程学习记录等模块，模块中的内容是紧密相连的。而按照物理层面划分，可分为 Nginx、Tomcat、MySQL、Redis 等，我们可以把 Tomcat 换成 Jetty，MySQL 换成 PostgreSQL。

3.框架与架构

框架和架构其实相对比较好区分。

框架关注的是规范，并且提供了基础的功能。比如我们使用 Spring MVC 框架，提供了基础的 MVC 功能来帮助我们开发 Web 应用，并且我们必须使用 Spring MVC 提供的规范进行开发（@Controller 和 @RequestMapping 等注解定义接口）。

而架构的关注点则是结构，例如如何根据业务划分模块的边界，前后端分离架构更利于解耦等，数据存储使用关系型数据库还是键值数据库。

# 3.架构设计的目的

其实，我们上面已经讲到为什么要做架构设计了，架构设计的目的是为了解决软件系统复杂度带来的问题。

我们在做架构的时候，一定是要围绕着解决问题来进行设计，不要一味的追求新技术，新架构模式，而不考虑项目是否适合，以及相关的成本。

架构设计的关键思维是判断和取舍，程序设计的关键思维是逻辑和实现。要记住，没有最好的架构，只有最合适的架构。

# 二、架构设计的复杂度

# 1.高性能

软件系统中高性能带来的复杂度主要体现在两方面：

  1. 单台计算机内部为了高性能带来的复杂度。如线程、进程问题。
  2. 多台计算机集群为了高性能带来的复杂度。

单台计算机中，为了构建一个高性能的软件系统，需要考虑如多进程、多线程、进程间通信、多线程并发等技术点，而且这些技术并不是最新的就是最好的，也不是非此即彼的选择。我们在做架构设计的时候，需要花费很大的精力来结合业务进行分析、判断、选择、组合，以解决高性能的问题，这是一个很复杂的过程。

单台计算机的性能有限，随着业务的发展，需要采用多台计算机集群模式才能达到高性能。不过集群并不是增加几台计算机那么简单，还需要考虑集群中计算机的任务分配、任务分解（业务拆分）的问题。

# 2.高可用

高可用的实现一般使用“冗余”的方式，你会发现上面提到的多台计算机集群也是使用了冗余的方式实现的，不过它们有本质的区别：高性能增加机器的目的是为了扩展处理任务的性能，高可用增加机器的目的是为了冗余，避免在其他机器发生故障时，系统无法使用。

通过冗余增加机器，会带来系统架构的复杂性，大概可分为下面两种场景：

  1. 计算（任务处理）高可用

    • “计算”是指对业务逻辑处理，计算无论在哪个机器上执行任务，对于同样的输入，产生的结果都是相同的。

  2. 存储高可用

    • 如果将数据从一台机器与另一台机器同步，同一机房内理论上为几毫秒，若是不同机房就会达到几十毫秒到几百毫秒。
    • 如果不同机器的数据不一致，并发场景读写数据就会造成业务逻辑出现问题。因此，存储高可用的难点在于如何减少或者规避数据不一致对业务造成的影响。
    • 分布式领域的 CAP 定理中，存储高可用不可能同时满足“一致性、可用性、分区容错性”，最多满足其中两个，这就要求我们在做架构设计时结合业务进行取舍。

高可用的实现，需要判断机器的状态是正常还是异常，即状态决策。

不过，通过冗余来实现的高可用系统，本质上不可能做到完全正确判断机器是否正常。

判断机器是否正常有以下方式：

  1. 独裁式：由一台专门的机器（决策者）负责收集信息，其他所有机器（上报者）向该机器上报信息，并根据信息判断进行决策。
  2. 协商式：两个独立的个体通过交流信息，然后根据规则进行决策，主备决策是协商式最常用的决策。
    • 主备决策中，两台机器启动时都是备机，启动后建立连接，互相交换状态信息。然后，其中一台机器作为主机，进行决策，另外一台作为备机。
    • 当两台机器的信息交换出现问题时，决策处理就会出现问题：
      • 备机认为主机故障，但实际主机没有故障，造成两台主机的情况。
      • 备机认为主机没有故障，但实际主机有故障，造成没有主机的情况。
  3. 民主式：多个独立的个体通过投票的方式来进行状态决策。如 ZooKeeper 集群在选举 leader 时就是采用民主式。
    • 民主式决策中，机器之间也是相互交换信息，每个个体做出决策，按照“多数取胜”的规则决定主机。
    • 不过民主决策的选举算法非常复杂，并且会出现“脑裂”的情况（脑裂：原来的集群连接中断，造成两个独立的子集群，每个子集群单独选举，造成两个主机的情况）。
    • 为了解决脑裂的问题，民主式决策的系统一般都采用“投票节点数必须超过系统总节点数一半”规则。不过这种方式也会降低了系统整体的可用性。

不管是使用何种方式，状态决策都不可能做到任何场景下都没有问题，如何选取适合系统的高可用方案，也是一个复杂的过程。

# 3.可扩展

可扩展性要求在系统设计时，能够“正确预测变化、完美封装变化”。当有新的需求出现时，系统不需要或者仅进行少量修改就可以支持。

软件在发布后还会不断地修改和升级，这就意味着会不断地有新需求需要实现，因此在设计之初，就需要“预测变化”。不过预测并不都是准确的，预测变化的复杂性如下：

  1. 不能每个设计点都考虑可扩展性。
  2. 不能完全不考虑可扩展性。
  3. 所有的预测有可能与实际不符。

除了预测变化，还要有应对变化的方案。应对变化有以下几种方案：

  1. 将变化与不变的代码按照模块分离。分为变化层和稳定层。两者独立，通过接口调用。
  2. 拆出一个“抽象层”和一个“实现层”。抽象层作为稳定的代码，实现层作为定制开发的内容。

# 4.低成本

如果我们的系统只有几台或十几台机器时，成本就不是重点关注的问题。但是当我们的系统涉及几百几千甚至上万台机器时，成本就是架构设计中一个必须考虑的问题了。

架构设计中，如果两种方案的机器差异存在几千或几万台机器，增加的成本是非常大的。不过一般为了降低成本，往往需求引入新技术才能达到低成本的目标。

新技术，可以是在架构引入新技术，也可以自己创新新技术。相比来说，创造新技术复杂度更高，也要求企业提供成本、技术、时间等资源的支持。

引入新技术的例子：

  • 为了解决关系型数据库无法应对高并发访问带来的访问压力，于是有了 NoSQL（Memcache、Redis 等）的出现。
  • 为了解决关系型数据库 like 搜索的低效的问题 ，于是有了全文搜索引擎（Sphinx、Elasticsearch、Solr）的出现。

引入新技术，要求架构师和开发人员对新技术非常熟悉，并且还要有新技术引入的架构的方案。

# 5.安全

安全一旦出现问题，对企业的影响是十分巨大的。安全按照技术类型可以分为功能安全和架构安全。

功能安全一般指的是各种漏洞被黑客利用，一般的例如 XSS 攻击、CSRF 攻击、SQL 注入、Windows 漏洞、密码破解等。功能安全一般和具体的编码先关，与架构关系不大。目前很多安全框架都内置了预防常见漏洞的处理，不过，很多框架也被报出重大的安全漏洞。例如，Log4j2 日志框架的 JNDI 远程执行漏洞。

这些功能安全的漏洞往往只能在出现问题之后，才能有针对性的去解决，不能依赖架构去一劳永逸。

架构安全一般指的是黑客直接攻击系统，故意搞破坏，如 DDoS 攻击就是短时间的大规模请求，使目标系统瘫痪不可访问。传统的架构安全使用防火墙控制网络请求，不过这对 DDoS 攻击无效，并且防火墙的价格也很高。

目前，架构安全并没有太好的设计手段来实现，更多地是依靠运营商或者云服务商强大的带宽和流量清洗的能力。

# 6.规模

很多公司的系统没有高性能、高可用的要求，只是一个单体系统。不过系统的功能很多，逻辑多，再加上维护时间长，后面接手的开发同学对系统不熟悉，不知道代码对应的具体功能是什么，无法掌握细节，看不懂、改不动、不敢改、修不了，于是常常抱怨系统很复杂。

有时候，复杂的数量超过一定的阈值后，复杂度就会发生质的变化。

与规模相关的复杂度如下：

  1. 功能越来越多，导致系统复杂度指数级上升。
  2. 数据越来越多，导致系统复杂度发生质变。

# 三、架构设计的原则

合适原则：

合适原则，即合适的才是最好的。

  • 在架构设计的时，要根据公司业务规模、开发人员配比来选择合适的架构。
  • 在技术选型，不要一味选择新技术，需要考虑开发人员对技术的熟悉程度。

简单原则

简单原则就是避免复杂。

软件领域中的复杂性可分为

  1. 结构的复杂性带来的问题如下：
    • 使用的组件多，当某个组件出现故障，导致系统出现故障的概率越高。
    • 某个组件改动，会影响关联的所有组件
    • 复杂系统中的问题定位总是比简单系统更加困难。
  2. 逻辑的复杂性的典型特征如下：
    • 单个组件承担了太多的功能。这会导致系统定位问题、修改逻辑、代码影响范围评估都十分困难。
    • 采用复杂的算法。这会导致逻辑难以理解，后续难以维护。

演化原则

演化优于一步到位。

  • 优秀的架构设计，都是根据业务的积累，遇到了瓶颈或问题，一步一步解决和优化。
  • 业界领先的架构设计或系统，都是不断演化，更类似于自然演化，将优秀的设计、逻辑等留下，并修复有缺陷的设计。
