02.业务架构

一个项目能否成功落地，首先需要的是把业务分析做到位，最后使用什么技术实现是我们第二位才去考虑的因素。因此，我们这里着重看一下业务架构的知识。

# 一、什么是业务架构设计？

# 1.业务架构师的职责

很多时候，大家会觉得产品经理和业务架构师的工作内容似乎差不多，都是对需求的处理。实际上，他们的职责目标是完全不一致的。产品经理是站在用户的角度进行需求分析，其职责是对外部使用系统的用户；业务架构师是站在开发者的角度定义系统内部结构，保证架构在业务上的可扩展性、可复用性等，其职责是对内部开发者。

# 2.业务的可扩展

在实际业务中，需求是不断变化的，但是对于系统来说，它的要求却是相对稳定，尽量避免大的调整。

因此，业务架构设计不但要提供良好的业务扩展性，还要允许业务不断调整和快速生长。

如何做到业务的快速变化和系统的相对稳定呢？其实，我们可以把业务平台和业务线剥离开，让业务平台封装基础通用的功能，业务线则根据个性化需求开发。使业务线只依赖业务平台，业务线彼此之间互相独立，可以互不影响。这样既可以保证系统的相对稳定，又能支持业务的创新和个性化需求。

# 3.业务的可复用

很多时候，我们发现不同项目，它们存在类似的业务需求，如果能对这些相同的业务复用，能够大大增加业务开发效率和质量。

如果在业务架构中，我们只是简单按照业务流程划分系统模块，这个系统模块就要和业务流程严格对应。不过，一旦业务场景发生变化或出现定制化需求，就会导致系统模块跟着变化，并出现定制化功能。这样，系统模块就很难在不同业务场景中复用。

如何解决这个问题呢？我们可以通过业务域来划分业务，将业务流程中的节点拆分到业务域中，按照业务域构造系统模块。

不过，在系统模块的定义时，还要求我们对模块的职责定位要非常清晰，并且模块的数据模型和接口设计要保证通用。

另外。在对模块的复用时，还要对业务的层次划分明确。有时候，我们可以把一个业务域按照层次拆分得更细。例如，一个订单业务域，对于不同类型的订单，底层的增删改查功能是相同的，而上层的订单生命周期管理，外卖订单和堂食订单可能就不一样。我们就可以把订单模块拆分为多个上层订单模块和一个基础订单模块，这样，基础订单模块对于所有类型的订单，都能够提供复用。

在目前流行的微服务架构中，很多公司在设计的时候，在微服务的基础上，通过服务分层，进一步落地了共享服务体系和中台架构，这些都是业务架构可复用的体现。

# 二、可扩展的架构

扩展性的本质是为了使系统更加容易维护，降低复杂度，当一个新需求开发时，可以更小范围的修改代码，以减少修改的影响范围。

而通过构建合理的模块体系，有效地控制系统复杂度，减少业务变化带来的系统调整。

# 1.系统的组成？

系统我们并不陌生，每天工作都在与系统打交道。简单来说，系统就是模块与关系的组合。

  • 模块是系统的基本组成部分，其泛指子系统、应用、服务或功能模块。
  • 关系指模块之间的依赖关系，简单地讲，就是模块之间有调用

# 2.模块

从业务的角度看，每个模块都代表了某个业务概念（或业务领域）。

模块内部由数据和业务逻辑组成，其中数据是核心，业务逻辑围绕着数据，对数据做进一步加工，方便外部使用。

在定义模块的时候，需要考虑其扩展性，对模块的划分要”定位明确，概念完整。降低依赖，粒度适中“。

  1. 定位明确：所谓定位明确，就是说模块拥有哪些职责，只有模块的职责明确时，某个功能的所属才会明确。

  2. 概念完整：

    • 数据上，模块要覆盖对应业务领域的全部数据。比如一个订单模块，它要覆盖所有渠道的订单，如三方平台的订单、自有商城的订单、线下门店的订单等。

    • 功能上，模块要包含业务领域的全部功能。比如订单模块包含所有订单相关的功能，如包括订单数据的增删改查、订单业务规则校验、订单的状态和生命周期管理等。

  3. 降低依赖：模块的业务逻辑尽量围绕自身内部数据进行处理，尽量减少对外部的依赖，这样就不会随着外部模块的调整而调整。

  4. 粒度适中：不能为了追求定位清晰，把粒度划分得很小，导致系统的碎片化。如早期系统开发将用户积分放到用户模块，随着积分的概念在系统中越来越明确，就应该把积分单独设置一个模块。

# 3.模块的依赖关系

业务逻辑处理时，需要不同模块之间协作，从而形成依赖关系。而依赖关系也是需要进行设计，尽量的使其简化有条理。否则，依赖关系就会是一个网状的结构，后期关系维护就会变得十分复杂。

简化依赖关系，可以从依赖的方向和依赖的数量入手。

这里着重说一下依赖的方向：

  • 第一，模块之间的依赖需要时单向的，避免相互引用。这样可以更加直观的展现业务执行的顺序，更容易理解。

  • 第二，模块的依赖最好有一定的层次结构，可以按照模块定位的不同，分为不同层次。如按照按照 MVC 架构分层：

# 4.模块的拆分

对系统进行模块化拆分有两种方式：水平拆分和垂直拆分。

例如打车系统，可以根据业务进行水平拆分和垂直拆分。如下：

  1. 水平拆分：

    • 水平拆分是指从上到下把系统分为多层，按照系统处理的先后顺序，把业务拆分为几个步骤。如上面的打车业务，可以分为 UI 展现、地图搜索、运力调度和订单支付层。

    • 水平拆分更利于业务的深度扩展，当业务发生变化时，只需要在特定的层级模块中修改即可，减少代码修改范围。

  2. 垂直拆分：

    • 垂直拆分指的是按照不同的业务线拆分。例如上面的打车业务，可以分为出租车业务、快车业务和顺风车业务。
    • 垂直拆分可以很好地满足业务广度上的扩展。如增加一条专车业务线，就可以按照这个思路落地系统。

在做业务架构时，一般可以先考虑垂直拆分，从大方向上，把不同业务给区分清楚，然后再针对具体业务，按照业务处理流程进行水平拆分。

# 5.模块整合

系统按照模块拆分完后，就要对模块整合。模块整合有两种方式：通用化整合和平台化整合。

  1. 通用化整合：
  • 通用化是指通过抽象设计，让一个模块具备通用的能力，能够替代多个类似功能的模块。
  • 如上面的打车业务，三条垂直业务线都有地图搜索、运力调度、订单支付模块，而不同的业务线之间，这些模块逻辑高度类似，只是细节方面有差别。我们可以将这些类似的模块整合成一个通用模块，并且在模块对外的接口中，通过输入参数标识调用来自哪个业务。在模块内部，对不同业务线的差异化部分做针对性处理即可。如果要增加新的业务线，就可以使用这个现成的通用模块。
  • 通过模块通用化，使模块的数量减少，模块的定位更清晰，概念更完整，职责更聚焦。
  1. 平台化整合：
  • 平台化整合是指把定位相同的模块组织在一起对外提供服务，一起对业务场景提供全面的支撑。
  • 如下图，地图搜索、运力调度、订单支付模块都是业务线需要的基础能力服务，如果需要增加新的业务线，仍然需要这些基础的模块服务，因此，可以把这些基础模块放在同一层，构成一个基础业务平台。
  • 将一个个离散的服务构建成一个业务平台，对多个业务模块进行包装，形成更大粒度的抽象，并且系统依赖关系也更清晰了。

不过，业务平台化是模块依赖关系层次化的一个特例，它偏向于基础能力。在实际业务中，如果业务线很多，业务规则很复杂时，就可以把底层业务能力抽取出来，进行平台化处理。

# 三、可复用的架构

# 1.复用的分类

复用可以分为技术服用和业务复用。

  • 技术复用包括代码复用和技术组件复用；
  • 业务复用包括业务实体复用、业务流程复用和产品复用

从复用的程度来看，从高到低，可以分为：产品复用 > 业务流程复用 > 业务实体复用 > 组件复用 > 代码复用。

  1. 技术复用：

    • 代码复用：包括自己封装的类库，第三方提供的 SDK，还有各种算法封装等。

    • 组件复用：包括自己封装的组件，开源的中间件，比如 Redis、MQ、Dubbo 等

  2. 业务复用：

    • 业务实体复用：根据业务领域划分，比如订单、商品、用户等领域。它对各个业务领域的数据和业务规则进行封装，将它变成上层应用系统可以直接使用的业务组件。
    • 业务流程复用：根据业务场景划分，把多个业务实体串起来，完成一个端到端的任务。比如下单流程需要访问会员、商品、订单、库存等多个业务，我们就可以这些调用逻辑封装为一个下单流程服务。
    • 产品复用：如 SaaS 系统，PaaS 平台等。

从技术复用到业务复用，越往上，复用程度越高，复用产生的价值也越大，但实现起来也越复杂，它能复用的场景就越有限。

在实际的应用中，业务流程的复用是最常见的，也最考验对业务的理解。我们通过打造业务中间件，并组合业务中间件形成业务平台。

# 2.基础服务边界划分

业务组件化和平台化需要对基础服务有着清晰的边界划分，即对服务的粒度适中和职责明确。

服务内部包含了业务数据和业务规则，并且需要确定哪些数据属于这个服务；服务提供接口给外部访问，需要确定哪些功能暴露给外部。

服务边界划分的原则如下：

  1. 完整性原则：在划分服务边界时，要保证服务数据完整、功能全面，这样才能支撑一个完整的业务领域。
  • 服务内部数据的完整性：例如商品服务的数据模型，不仅要有商品基本信息（名称、价格、分类、图片等），还需要包含商品的扩展信息（标签等），还要包含各种复杂商品类型的定义（组合商品、套餐商品等）。
  • 服务功能的完整性：对于服务调用者来说，他们是以业务的角度看服务，而不是纯粹的数据角度。如一个套餐商品，在服务内部，它是多个单品的复杂组合，但从服务调用者的角度来看，它就是一个商品。

如果一个套餐商品，商品服务是给出一个最终价格呢？还是给出各个单品的价格，然后让调用方自己算最终价格呢？

  • 套餐的价格不是各个单品价格累加的结果，它包含了一定的优惠，如果它的价格由服务调用方来算，这会导致商品的部分业务规则游离于服务外面，破坏了商品服务的功能完整性。

如果服务只是存储基础数据，然后提供简单的增删改查功能，这样一来，服务只是数据访问通道，服务价值就很有限，因此要尽可能在服务内部封装完整的业务规则，对外提供完整的业务功能接口。

  1. 一致性原则：服务的数据和职责要一致，谁拥有信息，谁就负责提供相应的功能。
  • 服务内部的业务逻辑要尽量依赖内部数据，而不是接口输入的数据。使数据和业务规则一致，降低对外部的依赖性。
  • 在面临功能划分到哪个服务的时候，可以结合这个功能所依赖的数据来判断。判断功能所需要的大部分数据存储在哪个服务中，就把功能放到该服务，降低服务对外部的依赖。

例如，订单的优惠计算功能，对于商品原价，满减优惠金额，商品特价的优惠金额数据需要保存在订单服务里。不过，订单的优惠计算过程不该归属于订单服务，而是应该由促销服务负责，因为优惠计算所需要的优惠规则是在促销服务里定义的，促销服务应该在内部拿到所有的优惠规则，然后完成优惠的计算。所以，这里促销服务负责促销规则的维护，以及对应的优惠计算功能；订单服务负责优惠结果数据落地，以及后续的查询功能

  1. 正交原则：基础服务之间不允许有任何的调用关系。
  • 基础服务相互之间是正交的。如说会员服务和商品服务，它们是不同维度的基础业务领域，彼此之间不应该有调用关系。
  • 不过基础服务之间有数据的依赖关系。

例如，订单明细里包含商品 ID 信息，但订单服务内部不应该调用商品服务来获取商品详情。如果页面需要展示订单的商品详情，针对这个具体的业务场景，我们应该在上层的聚合服务里，通过聚合订单服务和商品服务来实现该功能。
