# 一、概览

# 1.MySQL 性能瓶颈

MySQL 性能的瓶颈与多个方面有关，如硬件、操作系统、MySQL 数据本身、业务架构等等。

下图展示了 MySQL 调优维度金字塔：

数据库调优的七个维度：

  1. 业务需求：不合理的需求可能造成很多问题

如某个需求要求：商品列表页的所有数据（包括销售数据，涨幅比，退款率、利润率等等）都展示出来。

像涨幅比，退款率、利润率这种需要数据都需要我们对多张表联合查询，并且还是对于分页列表中。虽然可以使用存储过程、sql调优、索引调优等方式改进。

但是这样的需求本身就不合理，对于列表数据来说，最好将这样的需求改为：分两个模式来展示

  • 模式一（一般模式）：点击商品详情再显示这种商品的比例、涨幅信息等数据。
  • 模式二（统计模式）：保持之前的请求，提示用户此种模式下访问会慢些，减少分页大小。

两个模式可以根据由用户自己的喜好来选择模式，这样一来即保证了需求，用户在使用时体验更好。

程序员要对不合理的需求说不，并且要考虑相应的替代方案。

  1. 系统架构

我们在做架构设计的时候，应该充分考虑业务，针对业务来对数据库做规划。

如

  • 数据库选择及版本：某些场景下可能不适合关系型数据库，根据业务选择数据库
  • 读写分离：根据数据库的压力（用户数量或访问并发数），考虑是否做读写分离
  • 实例的个数：实例过多浪费资源，过少满足不了业务需求
  • 分库分表：根据数据量水平分表，根据业务垂直分表
  1. sql及索引

根据需求编写良好的SQL，并创建高效的索引

  1. 表结构

设计良好的表结构能让查询 sql 编写更加高效

  1. 数据库参数设置

设置合理的数据库参数，如 join buffer、sort buffer...

  1. 系统配置

操作系统提供了各种资源使用策略，合理的配合能够便于数据库充分利用资源。

如 linux 中的 swap（交换区，可以理解为“虚拟内存”。当物理内存不足时，拿出部分硬盘空间当 swap分区（虚拟成内存）使用，从而解决内存容量不足的情况。）。 swappiness 参数是用来设置 swap 的

  • swappiness = 0，操作系统会在没有内存的情况下才会使用 swap
  • swappiness = 100，则数据库会很快使用 swap 来作为内存存储运行时的数据

应该尽量的将 swappiness 设置的小，这样数据库会优先使用物理内存。swappiness 参数默认为60

# 查看 swappiness 参数
cat /proc/sys/vm/swappiness

# 临时修改 swappiness 参数
sudo sysctl vm.swappiness=10

# 永久修改 swappiness 参数
sudo gedit /etc/sysctl.conf
# 最后加上这样一行:
vm.swappiness=10

# 二、查询发现与分析

# 1.开启慢查询日志

MySQL 慢查询配置参数与默认值：

参数 作用 默认值
log_output 日志输出到哪儿，默认FILE，表示文件；设置成TABLE，则将日志记录到mysql.slow_log中。也可设置多种格式，比如 FILE,TABLE FILE
long_query_time 执行时间超过这么久才记录到慢查询日志，单位秒，可使用小数表示小于秒的时间 10
log_queries_not_using_indexes 是否要将未使用索引的SQL记录到慢查询日志中，此配置会无视long_query_time的的配置。生产环境建议关闭；开发环境建议开启。 OFF
log_throttle_queries_not_using_indexes 和log_queries_not_using_indexes配合使用，如果log_queries_not_using_indexes打开，则该参数将限制每分钟写入的、未使用索引的SQL数量。 0
min_examined_row_limit 扫描行数至少达到这么多才记录到慢查询日志 0
log_slow_admin_statements 是否要记录管理语句，默认关闭。管理语句包括ALTER TABLE, ANALYZE TABLE, CHECK TABLE, CREATE INDEX, DROP INDEX, OPTIMIZE TABLE, and REPAIR TABLE。 OFF
slow_query_log_file 指定慢查询日志文件路径 /var路径
log_slow_slave_statements 该参数在从库上设置，决定是否记录在复制过程中超过long_query_time的SQL。如果binlog格式是row，则该参数无效 OFF
log_slow_extra 当log_output=FILE时，是否要记录额外信息（MySQL 8.0.14开始提供），对log_output=TABLE的结果无影响。 OFF

该语句的唯一标识。如果explain的结果包括多个id值，则数字越大越先执行；而对于相同id的行，则表示从上往下依次执行。

2.select_type

查询类型，有如下几种取值：

查询类型 作用
SIMPLE 简单查询（未使用UNION或子查询）
PRIMARY 最外层的查询
UNION 在UNION中的第二个和随后的SELECT被标记为UNION。如果UNION被FROM子句中的子查询包含，那么它的第一个SELECT会被标记为DERIVED。
DEPENDENT UNION UNION中的第二个或后面的查询，依赖了外面的查询
UNION RESULT UNION的结果
SUBQUERY 子查询中的第一个 SELECT
DEPENDENT SUBQUERY 子查询中的第一个 SELECT，依赖了外面的查询
DERIVED 用来表示包含在FROM子句的子查询中的SELECT，MySQL会递归执行并将结果放到一个临时表中。MySQL内部将其称为是Derived table（派生表），因为该临时表是从子查询派生出来的
DEPENDENT DERIVED 派生表，依赖了其他的表
MATERIALIZED 物化子查询
UNCACHEABLE SUBQUERY 子查询，结果无法缓存，必须针对外部查询的每一行重新评估
UNCACHEABLE UNION UNION属于UNCACHEABLE SUBQUERY的第二个或后面的查询

3.table

表示当前这一行正在访问哪张表，如果SQL定义了别名，则展示表的别名

4.partitions

当前查询匹配记录的分区。对于未分区的表，返回null

# 5.估计查询性能

多数情况下，你可以通过计算磁盘的搜索次数来估算查询性能。对于比较小的表，通常可以在一次磁盘搜索中找到行（因为索引可能已经被缓存了），而对于更大的表，你可以使用 B-tree 索引进行估算：你需要进行多少次查找才能找到行：

​ log(row_count) / log(index_block_length / 3 * 2 / (index_length + data_pointer_length)) + 1

在MySQL中，index_block_length 通常是1024字节，数据指针一般是4字节。比方说，有一个500,000的表，key是3字节，那么根据计算公式 log(500,000)/log(1024/3*2/(3+4)) + 1 = 4 次搜索。

该索引将需要 500,000 * 7 * 3/2 = 5.2MB 的存储空间（假设典型的索引缓存的填充率是2/3），因此你可以在内存中存放更多索引，可能只要一到两个调用就可以找到想要的行了。

但是，对于写操作，你需要四个搜索请求来查找在何处放置新的索引值，然后通常需要2次搜索来更新索引并写入行。

前面的讨论并不意味着你的应用性能会因为 log N 而缓慢下降。只要内容被 OS 或 MySQL 服务器缓存，随着表的变大，只会稍微变慢。在数据量变得太大而无法缓存后，将会变慢很多，直到你的应用程序受到磁盘搜索约束（按照log N增长）。为了避免这种情况，可以根据数据的增长而增加key的。对于MyISAM表，key的缓存大小由名为 key_buffer_size 的系统变量控制，详见 Section 5.1.1, “Configuring the Server” (opens new window)

# SHOW PROCESSLIST

SHOW [FULL] PROCESSLIST 用于查看当前正在运行的线程。如果执行此命令的用户拥有 PROCESS (opens new window) 权限，则可看到所有线程；否则只能看到自己的线程（即与当前登录用户关联的线程）。如果不使用FULL关键字，只在Info字段中展示前100个字符。

当遇到“too many connections”错误信息时，想要了解发生了什么，SHOW PROCESSLIST就非常有用。MySQL保留了一个额外的连接，用于让拥有 CONNECTION_ADMIN (opens new window) （或已废弃的 SUPER (opens new window) ）权限的账户使用，从而确保管理员始终能够连接并检查系统。

可使用 KILL (opens new window) 语句杀死线程。

由结果可知，结果包含如下几列：

  • Id：连接的唯一标识，是CONNECTION_ID()函数的返回。

  • User：发出该语句的MySQL用户。

    • system_user表示服务器产生的非客户端线程，用于处理内部任务。这可能是用来在从库复制或延迟行处理器的IO/SQL线程。对于system_user，Host字段将会为空
    • unauthenticated user是指与客户端连接，但尚未完成客户端用户身份认证的线程。

    TIPS

    User字段的值是system_user和 SYSTEM_USER 权限不是一回事，前者指内部线程，后者用来区分系统账户和普通账户的类别，不要搞混了。

  • Host：发出该语句的客户端的主机名（当User是system_user时，Host为空）。TCP/IP链接的主机名以 host_name:client_port 格式上报，以便更轻松地了解哪个客户端在干什么。

  • db：当前执行的命令是在哪一个数据库上。如果没有指定数据库，则值为NULL

  • Command：当前线程正在执行的命令。有关线程命令的描述，可详见 “Examining Thread Information” (opens new window) 。

  • Time：线程处于当前状态的时间（单位秒）。对于从库的SQL线程，该字段的值表示上次复制事件的时间和从库机器的实际时间之间经过了多少秒。详见 “Replication Implementation Details” (opens new window)

  • State：指示线程正在执行的操作、事件或状态。大多数State对应于非常快速的操作。如果线程在给定状态下很久，则需要排查下。

  • Info：当前线程正在执行的语句，如果未执行任何语句则值为NULL。该语句可能是发送到服务器的那条语句，也可能是内部的语句（如果某个语句执行了其他语句）。例如一条CALL语句执行了一条正在执行SELECT语句的存储过程，则Info字段会展示SELECT语句。

Command取值：

  • Binlog Dump：主库上的线程，用于将binlog内容发送到从库
  • Change user：线程正在执行更改用户操作
  • Close stmt：线程正在关闭一个prepared statement
  • Connect：一个复制从库已连接到其主库
  • Connect Out：一个复制从库正在连接到其主库
  • Create DB：线程正在执行create-database操作
  • Daemon：服务器内部线程，而非为客户端连接提供服务的线程
  • Debug：该线程正在生成调试信息
  • Delayed insert：该线程是延迟插入处理程序
  • Drop DB：线程正在执行drop-database操作。
  • Error：你懂的
  • Execute：线程正在执行一个prepared statement
  • Fetch：正在从Prepared Statement 中获取执行结果
  • Field List：该线程正在获取表的字段信息
  • Init DB：线程正在选择默认数据库。
  • Kill：该线程正在杀死另一个线程
  • Long Data：正在从prepared statement中检索long data
  • Ping：线程正在处理server-ping请求。
  • Prepare：该线程正在准备一个prepared statement
  • Processlist：该线程正在生成服务器线程相关信息
  • Query：线程正在执行一条语句
  • Quit：线程正在终止
  • Refresh：该线程是刷新表，日志或缓存；或者正在重置状态变量或在复制服务器信息。
  • Register Slave：该线程正在注册一个从库
  • Reset stmt：线程正在重置prepared statement
  • Set option：线程正在设置或重置client statement-execution选项
  • Shutdown：线程正在关闭服务器
  • Sleep：线程正在等待客户端向其发送statement
  • Statistics：该线程正在生成服务器状态信息
  • Table Dump：线程正在将表内容发送到从属服务器。
  • Time：Unused

# 四、数据库索引

索引通俗的讲就是已经排好序的数据结构，目的是为了方便快速查询、更新数据库表中的数据。

# 1.索引的数据结构

# 1.1 二叉树

二叉树的每个节点最多只能有两个“叉”（两个子节点），但二叉树并不要求每个节点都必须要有两个子节点。

若使用二叉树存储数据。二叉树在查找时，若左侧节点和右侧节点出现严重不一致情况，会导致查找的复杂度增加，如查找8，左边的树只需要三次，右边的树需要5次，查找时非常影响性能

# 1.2 平衡二叉树（AVL树）

  • 平衡二叉树要求每个节点的左子树和右子数的高度差不超过1
  • 对于 n 个节点，树的高度为 log2n，查询的时间复杂度是 O(log2n)

下图左边则为平衡二叉搜索树，右侧则不是平衡二叉搜索树

平衡二叉树解决的问题

  • 使得树平衡，不会出现树一边重

平衡二叉树的问题

  • 树越高，则其查询的复杂度越高

# 1.3 B-Tree （Balance Tree）

B-Tree 是平衡多路搜索树的一种具体实现，如下图：

  • 指针中存储的是：子节点的地址
  • 关键字存储的是：索引列的数据
  • 数据存储的是：数据表中行数据的内存地址

上图中，如果要查找数据17：根节点中就有数据17，直接读取17的数据即可。

如果要查找数据 9 ：

  • 磁盘块1节点（根节点）中：查找根节点中的关键字，由于我们要查找 9，9是小于17，所以要通过 P1 指针定位到磁盘块2节点。
  • 在磁盘块2节点中（中间节点）：由于9大于8，且小于12，所以要通过 P2 指定定位到磁盘6节点。
  • 在磁盘块6节点中（叶子节点）：查找到第一条顺序就是 9。就直接读取9中的数据即可

备注：磁盘块2 中的 P1,P3 所指向的子节点是没有画出来。

特性：

  • 根节点的子节点个数 2<= x <= m ; m 是树的阶（树高）
    • 若 m = 3，则根节点可以有 2~3 个子节点
  • 中间节点的子节点个数为 m/2<=y<=m
    • 若 m = 3，则中间节点至少有两个子节点，最多有三个子节点
  • 每个中间节点包含 n 个关键字，n = 子节点个数 - 1，且按升序排序
    • 若中间节点有 3 个子节点，则里面会有两个关键字，且按升序排序
  • 指针指向
    • P1, P2, P3 为指向子树根节点的指针。P1 指向关键字小于 Key1 的树；P2 指向 Key1 - Key2 之间的子树；P1 指向关键字大于 Key2 的树

B-Tree 解决的问题（相对于平衡二叉树）：

  • 有效的降低了树的高度
  • 每个节点存储更多关键字

B-Tree 的问题：

  • 磁盘写代价高：由于B-Tree的平衡性和有序性要求，插入和删除操作可能需要进行节点的分裂和合并，导致数据的重新平衡。这些操作可能涉及多次磁盘 IO 操作，导致更新代价较高。
  • 查询速度不稳定：
  • 顺序查询效率更低：

# 1.4 B+Tree

B+Tree 是在 B -Tree 基础上做的优化。Mysql 中的 InnerDB 存储引擎就是基于 B+Tree

B+Tree 和 B-Tree 不同：

  • 关键字
    • B+Tree 有 n 个子节点的节点中含有 n 个关键字。
    • B-Tree 有 n 个子节点的节点中含有 n-1 个关键字。
  • 数据的保存
    • B+Tree 中的节点仅用于索引，不保存数据，行记录存储在叶子节点中。
    • B-Tree 中的节点既保存索引，也保存数据记录。
  • 叶子节点（没有子节点的节点，即最下面那一层节点）
    • B+Tree 中，所有的叶子节点中包含全部关键字信息，且叶子节点按照关键字的大小自小而大按照顺序链接，构成一个有序链表。
    • B-Tree 中，叶子节点不包括所有的关键字。

上图中，如果要查找数据 28 ：根节点中能直接找到关键字 28，如果关键字中存在我们需要查询的字段，则直接使用，就不继续找了；若不存在需要查询的字段，则查找磁盘块3 中的P1——>最终找到28对应的数据读取即可

如果要查找数据 9 ：

  • 磁盘块1节点（根节点）中：查找根节点中的关键字，由于我们要查找 9，9 是大于5，且小于28的，所以要通过 P1 指针定位到磁盘块2节点。
  • 在磁盘块2节点中（中间节点）：由于9大于5，且小于10，所以要通过 P1 指定定位到磁盘块2节点的第一个子节点。
  • 磁盘块2节点的第一个子节点中（叶子节点）：按顺序查找到第三数据就是 9。就直接读取9中的数据即可

B+Tree 解决的问题

  • 减少磁盘 I/O，树比较矮胖
  • 对于连续的数据（范围查询），只需要连续的遍历查找
  • 查询速度稳定

# 1.5 Hash 索引

Hash 索引的存储方式：通过索引对应的列字段计算出 hash 值，根据hash值存储到hash表，并将数据的内存地址一起存储起来。

当出现hash冲突时，会对hash值对应的数组循环查找，性能会降低。

  • keys 为索引字段

  • buckets 为通过索引字段计算的hash值，根据hash值组成hash表

  • entities 为数据内容

hash索引的优点：

  • 时间复杂度为O(1)，性能优越

hash索引的缺点/限制：

  • 不支持范围/模糊查询，只支持等值查询（= 或 in）。
  • 不支持部分索引列匹配查找，因为hash索引是全局索引列的hash取值的。
  • Hash索引suoyin不是按照索引值来排序的，因此排序不会走索引。
  • Hash冲突越严重，性能就会越低。

Hash 索引的支持情况

  • Memory 引擎支持显式Hash索引
  • InberDB 引擎支持 “自适应Hash索引”

# 2.3 物理存储角度（聚簇索引、非聚簇索引）

聚集索引

聚集索引在存储的时候，可以按照主键（不是必须，看情况）来排序存储数据，B+Tree 的叶子结点就是完整的数据行，查找的时候，找到了主键也就找到了完整的数据行。

如下图，在聚集索引中，叶子结点保存了每一行的数据。

在聚集索引里，表中数据行按索引的排序方式进行存储，对查找行很有效。只有当表包含聚集索引时，表内的数据行才会按找索引列的值在磁盘上进行物理排序和存储。每张表只能有一个聚集索引，原因很简单，因为数据行本身只能按一个顺序存储。

当我们基于 InnoDB 引擎创建一张表的时候，都会创建一个聚集索引，每张表都有唯一的聚集索引：

  1. 如果这张表定义了主键索引，那么这个主键索引就作为聚集索引。
  2. 如果这张表没有定义主键索引，那么该表的第一个唯一非空索引作为聚集索引。
  3. 如果这张表也没有唯一非空索引，那么 InnoDB 内部会生成一个隐藏的主键作为聚集索引，这个隐藏的主键是一个 6 个字节的列，该列的值会随着数据的插入自增。

基于以上描述大家可以看到，主键索引和聚集索引并不是一回事，切勿混淆！

聚集索引最主要的优势就是查询快。如果要查询完整的数据行，使用非聚集索引往往需要回表才能实现，而使用聚集索引则能一步到位。

不过聚集索引也有一些劣势：

  1. 聚集索引可以减少磁盘 IO 的次数，这在传统的机械硬盘中是很有优势的，不过要是固态硬盘或者内存（有时候为了提高操作效率，数据库服务器会整一个比较大的内存），这个优势就不明显了。
  2. 聚集索引在插入的时候，最好是主键自增，自增主键插入的时候比较快，直接插入即可，不会涉及到叶子节点分裂等问题（不需要挪动其他记录）；而其他非自增主键插入的时候，可能要插入到两个已有的数据中间，就有可能导致叶子节点分裂等问题，插入效率低（要挪动其他记录）。如果聚集索引在插入的时候不是自增主键，插入效率就会比较低。

回表指先通过索引查询出主键，然后根据主键再去查询出所对应数据

当索引中存在查询所需的的所有数据，就不会再去表中二次查询

聚集索引的本质也是使用 B+Tree。聚集索引会将多个关键字放到同一个节点，并且排序按照关键字排序。

排序规则：

  • 先用name排序
  • name排序相同的数据，再根据age排序
  • age排序相同的数据，再按照position排序

由于聚集索引的数据结构特点，在使用聚合索引查询数据时，必须要先找出最左边索引列的数据，才能继续使用后面列的索引，这就是最左前缀原则。

# B-Tree (B+Tree)的特性

特性：

  • 完全匹配：index(name) => where name = 'mysql'
  • 范围索引：index(age) => where age < 18
  • 前缀匹配：index(name) => where name like 'my%'
    • 注意右模糊不会使用索引

限制：

  • index(name, age, sex)
    • 查询条件中不包括最左列则无法使用索引。
      • 如 where age = 5 and sex = 1则无法使用索引。
    • 跳过了索引中的列，则无法完全使用索引。
      • 如 where name = 'mysql' and sex = 1则sex过滤查找不会使用索引。
    • 查询中有某个列的范围查询(或模糊查询)，则其右边所有列都无法使用索引。
      • 如 where name = 'mysql' and age > 10 and sex = 0，只会用到name、age的索引，不会走sex的索引

以上限制即最左匹配原则/最左前缀原则

非聚集索引

非聚集索引我们一般也称为二级索引或者辅助索引，对于非聚集索引，数据库会有单独的存储空间来存放。非聚集索引在查找的时候要经过两个步骤，例如执行 select * from user where username='zhangsan'（假设 username 字段是非聚集索引），那么此时需要先搜索 username 这一列索引的 B+Tree，这个 B+Tree 的叶子结点存储的不是完整的数据行，而是主键值，当我们搜索完成后得到主键的值，然后拿着主键值再去搜索主键索引的 B+Tree，就可以获取到一行完整的数据。

所以如果我们在查询中用到了非聚集索引，那么就会搜索两棵 B+Tree，第一次搜索 B+Tree 拿到主键值后再去搜索聚集索引的 B+Tree，这个过程就是所谓的回表。

一张表只能有一个聚集索引，但可以有多个非聚集索引。使用聚集索引的时候，数据的查询效率高，但如果对数据进行插入，删除，更新等操作，效率会比非聚集索引低。

比较

聚簇索引的叶子节点就是数据节点，而非聚簇索引的叶子节点不存储数据，而是指向对应数据块的指针。

InnoDB 的主键索引使用的是聚簇索引，而 MyISAM 使用了非聚簇索引。

  • 表数据和主键一起存储的，聚簇索引的数据的物理存放顺序与索引顺序是一致的，即：只要索引是相邻的，那么对应的数据一定也是相邻地存放在磁盘上的。而由于无法同时把数据行同时存放在两个不同的地方，所以一张表只有一个聚簇索引。
    • 聚簇索引的二级索引：叶子节点不会保存引用的行的物理位置，而是保存行的主键值。
  • 非聚簇索引：叶子节点存储的是数据块的指针。表数据和索引分开存储。查询时，先找到索引，再根据索引找到对应的数据行

聚簇索引 vs 非聚簇索引

聚簇索引优点：

  • 查找效率理论上比非聚簇索引要高，但是插入、修改、删除操作的性能比非聚簇索引要低
  • 范围查询方便

聚簇索引缺点：

  • 插入速度严重依赖于插入顺序，因此，对于InnoDB表，我们一般都会定义一个自增的ID列为主键
  • 更新主键的代价很高，因为将会导致被更新的行移动。（当然一般不会更新主键）
  • 聚簇索引增删改操作性能比非聚簇索引性能低

对于InnoDB：

  • 主键使用聚簇索引，并且一张表有且只有1个聚簇索引。如果创建的表没有主键，则InnoDB会隐式定义一个主键来作为聚簇索引。
  • 二级索引（非主键索引）叶子节点存储的是行的主键值，因此使用二级索引命中数据需要查询两次，先用二级索引搜索到主键，再用主键查找数据。

对于MyISAM：

  • 主键/非主键索引的叶子节点都是存储指向数据块的指针（索引和数据分开存储）

# 4.索引失效与解决方案

可能导致索引失效的情况

  • 运算：对索引列加减乘除运算会导致索引失效

  • 函数：对索引字段使用内部函数

    • 应该建立基于函数的索引

  • 左模糊：模糊查询下，like以%开头时，索引失效

  • 数据类型：数据类型不正确，隐式转换导致索引失效

  • 空值：空值，不限制索引列为not null，数据库认为索引列可能存在空值，就不会走索引

    • 建议对空值默认为''，且对索引添加 not null 约束

  • 最左：最左前缀原则，在联合索引中，若不是按照索引列最左列开始进行查找，则索引失效

  • 不走索引更快：若数据预计使用全表扫描比索引更快，则不会使用索引

JOIN 算法2：Block Nested-Loop Join（BNLJ）块循环嵌套join

-- 查询出 t1 中符合条件的数据，进行遍历
for each row in t1 matching range  {
	-- 查询 t2 中符合条件的数据，使用 reference key (即join时的on字段) 匹配，
	for each row in t2 reference key  {
		-- 存储到 join_buffer
		store used column form t1, t2 in join buffer
		-- 如果 join_buffer 存储满了
		if buffer is full {
			-- 用 t3 中的数据与 join_buffer 的数据 比较
    		for each row in t3 {
    			for each row in t1, t2 combination in join buffer {
    				-- 满足 join 条件，则记录该条数据，发送到客户端
    				if row satisfies join conditions, send to client
    			}
    		}
    		-- 清空 join buffer
			empty join buffer		
		}
	}
} 

-- 如果 join_buffer 不为空（即之前没有将 join_buffer 存满）
if buffer is not empty {
	-- 用 t3 中的数据与 join_buffer 的数据 比较
	for each row in t3 {
        for each row in t1, t2 combination in join buffer {
            if row satisfies join conditions, send to client
        }
	}
}

JOIN 算法4：HASH JOIN

  • MySQL 8.0.18 引入，用来代替BNLJ

  • join buffer 缓存外部循环的 hash 表，内层循环遍历时到 hash表匹配

HASH JOIN 注意事项

  • MySQL 8.0.18 引入，且该版本有很多限制，如不能作用于外连接，比如 left join / right join 等等。
  • 从 MySQL 8.0.20 开始，限制少了很多，建议使用 8.0.20 或更高版本。
  • join buffer：从 MySQL 8.0.18 开始，hash join 的 join buffer 是递增分配的，这意味着，你可以吧 join_buffer_size 设置的比较大。而在 MySQL 8.0.18 中，若使用了外连接（left join 、right join），外连接没法用 hash join ，此时 join_buffer_size 会按照你设置的值直接分配内存。因此 join_buffer_size 还是得谨慎设置。
  • 从 MySQL 8.0.20 开始，BNLJ 已经被删除，使用 hash join 替代了 BNLJ。
  • hash join 不同版本区别，官方文档：https://dev.mysql.com/doc/refman/8.0/en/hash-joins.html (opens new window)

JOIN 语句调优

驱动表：外层循环的表是驱动表，内层循环的表是是被驱动表

调优建议：

  • 用小表驱动大表（扫描行数少的表，驱动扫描行数多的表）

    • 一般无需人工考虑，关联查询优化器会自动选择最优的执行顺序
    • 如果优化器抽风，可以使用 STRAIGHT_JOIN

  • 如果有 where 条件，应当要能够使用索引，并尽可能减少外层循环的数据量

    • 因为外层循环次数越多，最后扫描的次数越多

  • join 的字段尽量创建索引

    • join 字段的数据类型要保持一致，否则会在 join 时导致索引失效

  • 尽量减少扫描的行数 (explain 的 rows参数）

    • 扫描的行数尽量控制在百万以内（视情况而定）

  • 参与 join 的表不要太多

    • 阿里巴巴编程规约建议不要超过3张表，不要编写复杂 sql
    • 业务中若 join 表过多，可以将join语句拆分，一条一条语句的查询

  • 如果被驱动表的 join 字段用不了索引，且内存充足，可以考虑将 join buffer 设置的大一些

order by 使用索引的场景

  1. 利用索引本身的有序性，让 Mysql 跳过排序：如果索引的顺序与 order by 语句的顺序一致，这样数据库就会按照索引的顺序直接取数据

排序模式

1.rowid排序

第一步：从表中获取满足 where 条件的记录。

第二步：对于每条记录，将记录的主键及排序键（id, order_column）取出放入 sort buffer （由sort_buffer_size 控制）。

第三步：如果 sort buffer 能存放所有满足条件的（id, order_column），则进行排序；否则 sort buffer 满后，排序后写到临时文件。

  • 排序算法使用的是：快速排序算法

第四步：若排序中产生了临时文件，需要利用归并排序算法，从而保证记录有序。

第五步：循环上述过程，直到所有满足条件的记录全部参与排序。

第六步：扫描排好序的 (id, order_column) 对，并利用 id 去取 select 需要的字段。

第七步：返回结果集。

rowid 排序特点：

  • 看 sort buffer 是否能存放结果集里面所有的 (id, order_column) ，如果不满足就会产生临时文件
  • 一次排序需要两次IO：
    • 把 (id, order_column) 放到 sort_buffer 中
    • 通过 id 获取需要的返回的其他字段。由于返回结果是按照 order_column 排序的，所以 id 是乱序的，会存在随机 IO 的问题。Mysql 内部针对这种情况做了优化，在用id取数据之前，会按照 id 排序，并放到一个缓存中，这个缓存大小由 read_rnd_buffer_size 控制，接着再去取记录，从而把随机IO转换成顺序IO。

2.全字段排序（优化排序）

基于 rowid，不同的是：

  • 直接取出 sql 中需要的所有字段，放到 sort buffer
  • 由于 sort buffer 已经包含了查询所需要的所有字段，因此在 sort bufer 中排序完成后可直接返回

优点：性能提升，与 rowid 相比，无需两次 IO

缺点：一行数据占用的空间一般比 rowid 排序多；如果 sort buffer 比较小，容易导致临时文件。

3.排序算法的选择？

MySQL 会根据 max_length_for_sort_date 字段判断：当 order by 语句中需要的字段总长度小于 max_length_for_sort_date 时，使用全字段排序，否则使用 rowid 排序

4.打包字段排序

  • 该排序由 MySQL 5.7 引入
  • 全字段模式的优化，工作原理与全字段排序一样，但是将字段紧密的排列在一起，而不是使用固定长度空间
    • 如：VARCHAR(255) "yes" ：
      • 全字段排序：需要255字节
      • 打包字段排序：只需要 2+3字节就可以了；这样可以节省大量的空间，从而使得 sort_buffer 可以存储更多的内容

相关参数总结

变量 作用
sort_buffer_size 指定 sort buffer 的大小
max_length_for_sort_data 当 order by sql 中出现字段总长度小于该值，使用全字段排序，否则使用 rowid 排序
read_rnd_buffer_size 按照主键排序后存放的缓存区

order by 调优

  • 利用索引，防止 filesort 的发生。
  • 如果发生了 filesort ，并且没办法避免，想办法优化 filesort 。

优化filesort：

  • 调大 sort_buffer_size，减少/避免临时文件、归并操作
    • OPTIMIZER_TRACE 中 number_of_tmp_files 的值比较大时，就会出现很多归并操作，此时就可以增大 sort_buffer_size
    • 关注 sort_merge_passes 的值，该值表示归并次数。使用 show sttaus like '%sort_merge_passes%'，可以查看归并操作的次数
  • 调大 read_rnd_buffer_size ，让一次顺序 IO 返回的结果更多
  • 设置合理的 max_length_for_sort_data 的值。
    • 该值设置的过大，各种排序 sql 都会使用全字段排序，就会导致大量的内存占用，导致写入临时文件。
    • 但是设置的太小，各种排序 sql 都会使用 rowid 排序，从而产生两次 IO，性能变差
    • 一般不建议随意调整
  • 调小 max_sort_length 。max_sort_length 表示排序时最多取多少字节

使用松散索引扫描的条件

  • 查询作用在单张表上
  • group by 指定的所有字段要符合最左前缀原则，且没有其他字段
    • 比如有索引 index(c1, c2, c3)，如果 sql 为 group by c1, c2 则可以使用松散索引扫描；但是 sql 为 group by c2, c3 或 group by c1, c2, c4 则不能使用松散索引扫描
  • 如果存在聚合函数，只支持 min() 和 max() 函数，并且如果同时使用了 min() 和 max()函数，则参数必须为同一个字段。聚合函数作用的字段必须在索引中，并且要紧跟 group by 所指定的字段
    • 如有索引 index(c1, c2, c3)，查询语句：SELECT c1, c2, MIN(c3), MAX(c3) FROM t1 GROUP BY c1, c2 可以使用松散索引扫描
  • 如果查询中存在除了 group by 指定的列以外的其他部分，则必须以常量的形式出现
    • 如有索引 index(c1, c2, c3)。查询语句： SELECT c1,c3 FROM t1 GROUP BY c1,c2 不能使用松散索引扫描。而查询语句： SELECT c1,c3 FROM t1 WHERE c3 = 3 GROUP BY c1,c2 可以使用松散索引扫描
  • 索引必须索引整个字段的值，不能是前缀索引
    • 如有字段 c1，类型为 varchar(32)，但是如果该字段使用的是前缀索引index(c1(10))，而不是index(c1)，无法使用松散索引扫描

SELECT c1, c2 FROM t1 GROUP BY c1, c2;
SELECT DISTINCT c1, c2 FROM t1;
SELECT c1, MIN(c2) FROM t1 GROUP BY c1;
SELECT c1, c2 FROM t1 WHERE c1 < [const] GROUP BY c1, c2;
SELECT c2 FROM t1 WHERE c1 < [const] GROUP BY c1, c2;
SELECT MAX(c3), MIN(c3), c1, c2 FROM t1 WHERE c2 > [const] GROUP BY c1, c2;
SELECT c1, c2 FROM t1 WHERE c3 = [const] GROUP BY c1, c2;

表设计原则

  • 字段应该少而精，建议不超过20个字段（经验之谈）。字段过多可以把一些字段拆分到拓展表
    • 把常用的字段放到一起
    • 把不常用的字段独立出去
    • 大字段（ text | blob | clob 等 ）独立出去
  • 尽量使用占用空间小的字段

  • 避免使用允许为 NULL 的字段

  • 合理平衡范式和冗余
  • 如果表的数据量非常大，考虑分库分表

# 3.4 pt-online-schema-change

  • 官方文档：pt-online-schema-change (opens new window)
  • MySQL 从 5.6 开始，已支持 online DDL 功能，pt-online-schema-change 越来越弱化了。
    • 有关 Online DDL，可详见 Online DDL Operations (opens new window)
    • 有关 online DDL 和 pt-online-schema-change 之间的对比详见《 MySQL ONLINE DDL 和PT-ONLINE-SCHEMA-CHANGE对比 (opens new window) 》

  • 作用

    在线修改表结构，无需锁表地ALTER表结构

  • 原理

    • 创建一张一模一样的表，表名一般是_new后缀
    • 在新表上执行更改字段操作
    • 在原表上加三个触发器，分别对应于DELETE/UPDATE/INSERT操作，并将原表中要执行的语句也在新表中执行
    • 将原表的数据拷贝到新表中
    • 使用原子的RENAME TABLE操作同时重命名原始表和新表，完成此操作后，删除原始表。

  • 语法

    pt-online-schema-change [OPTIONS] DSN
    1

  • 常用OPTIONS

    --dry-run			创建并修改新表的结构，但不会创建触发器、拷贝旧表数据也不会替换旧表

--charset			指定编码

--no-version-check	是否检查版本
--alter-foreign-keys-method		处理带有外键约束的表，以保证它们可以引用到正确的表。取值：auto（自动选择最佳策略）、rebuild_constraints（适用于删除和重新添加引用新表的外键约束）、drop_swap（禁用外键检查，然后在重命名新表之前将其删除）、none（无）

# 七、MySQL 开发36条军规

# 1.核心军规

  1. 尽量不在数据库做运算
  2. 控制单表数据量 纯 INT 不超过10M条，含 Char 不超过5M条
  3. 保持表身段苗条
  4. 平衡范式和冗余
  5. 拒绝大SQL，复杂事务，大批量任务

# 2.字段类军规

  1. 用好数值字段，尽量简化字段位数
  2. 把字符转化为数字
  3. 优先使用 Enum 或 Set
  4. 避免使用 NULL 字段
  5. 少用并拆封 Text/Blob
  6. 不在数据库中存图片

# 3.索引类军规

  1. 谨慎合理添加索引
  2. 字符字段必须建立前缀索引？
  3. 不在索引列做运算
  4. 自增列或全局 ID 做 InnoDB 主键
  5. 尽量不用外键

# 4.SQL类军规

  1. SQL 尽可能简单
  2. 保持事务连接短小
  3. 尽可能避免使用 SP/Trigger/Function
  4. 尽量不用 SELECT *
  5. 改写 OR 为 IN()
  6. 改写 OR 为 UNION
  7. 避免负向查询和%前缀模糊查询
  8. COUNT 不要使用在可 NULL 的字段上面
  9. 减少 COUNT(*)
  10. LIMIT 高效分页。SELECT * FROM message WHERE id > 9527 (or sub select) LIMIT 10
  11. 使用 UNION ALL 而不用 UNION
  12. 分解链接，保证高并发
  13. GROUP BY 去除排序
  14. 同数据类型的列值比较
  15. LOAD DATA 导入数据，比 INSERT 快20倍
  16. 打散大批量更新，尽量凌晨操作

# 5.约定类军规

  1. 隔离线上线下
  2. 禁止未经 DBA 认证的子查询
  3. 永远不在程序段显式加锁
  4. 表字符集统一使用 uft8mb4
