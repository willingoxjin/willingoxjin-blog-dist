# 一、React介绍

# 1.出现的背景

React 是 Facebook 在 2013 年推出的一款前端框架。

由于当时开发一个简单功能上线后：状态栏的待处理消息个数：好友请求、新消息、新动态个数，这个功能总是出现问题，如当出现新消息时，未读消息的个数总是不更新，或者你点开消息未读列表后，消息的个数并不会变为0。

经过 Facebook 内部对问题的分析，最终总结问题的根源：

  1. 传统 UI 操作关注太多细节
  2. 应用程序状态分散在各处，难以追踪和维护

# 2.始终整体刷新页面（UI问题）

当时 Facebook 使用的是JQuery 来编写网页，但是 JQuery API 非常多，必须充分熟悉这些 API 才能去局部的更新网页，这也说明了比较复杂的界面操作需要关注很多的细节。

# 3.受控组件与非受控组件

看个例子：

定义 TabSelector 组件

export default class TabSelector extends PureComponent {
  // prop 的类型
  static propTypes = {
    value: PropTypes.string,// 当前所选中的值
    options: PropTypes.array,// 渲染按钮的数据列表
    onChange: PropTypes.func// 点击按钮发生的回调
  };

  // prop 的默认值
  static defaultProps = {
    value: null,
    options: [],
    onChange: () => {}
  };
  
  render() {
    const { options, value, onChange } = this.props;
    return (
      <div className="tab-selector">
        <ul>
          <!-- map 渲染数据列表 -->
          {options.map(opt => (
            <!-- onClick 点击时触发，会调用 该组件TabSelector中onChange对应的方法 -->     
            <li
              key={opt.value}
              className={`tab-item ${
                opt.value === this.props.value
                  ? "selected"
                  : ""
              }`}
              onClick={() => this.props.onChange(opt.value)}
            >
              {opt.name}
            </li>
          ))}
        </ul>
      </div>
    );
  }
}

# 4.创建组件的原则

单一职责原则

  • 每个组件只做一件事
  • 如果组件变得复杂，那么应该拆分成小组件
    • 组件拆分后，组件的复杂度降低
    • 组件拆分后，性能更好（组件复杂时，任何的状态变化，整个组件都需要重新渲染，拆分为小组件后，只需要重新渲染状态变化所对应的小组件即可）

# 5.数据状态管理

DRY原则

  • 能计算得到的状态，在使用时去计算，不要单独存储
  • 组件尽量无状态，所需数据通过 props 获取

# 四、React 组件生命周期

# 1.生命周期流程

React 组件生命周期图：

图片来自：https://projects.wojtekmaj.pl/react-lifecycle-methods-diagram/ (opens new window)

声明周期的三个阶段：

阶段一：Render 阶段

纯净且不包含副作用。可能会被 React 暂停，中止或重新启动

阶段二：Pre-commit阶段

Pre-commit 并未更新DOM节点，但是在这个阶段，可以去获取到DOM节点的内容

commit 更新DOM节点

阶段三：Commit 阶段

React 实际把状态都更新到了真实的DOM节点上，

在此节点，我们可以对DOM节点进行修改.

生命周期三种类型

1.创建时

  • 首先使用组件的构造函数 constructor 创建出来

  • 然后使用 getDerivedStateFromProps， 用于从外部属性 props，来初始化内部的状态 state

  • 其次 render，该方法是组件必须定义的方法，因为你需要使用该方法去描述的你DOM结构（UI）

  • 更新 DOM 和 refs。

  • 最后进入 componentDidMount 方法，你可以在此方法中进行 ajax 请求来获取后端数据，添加监听时间等处理

2.更新时

  • 若外部属性 props 、内部的state发生变化或者使用了forceUpdate() 强制更新，就会触发组件更新
  • getDerivedStateFromProps 和创建时一样， 用于从外部属性 props，来初始化内部的状态 state
  • shouldComponentUpdate 会告诉组件是否需要Render，该函数可以被用户接入，你可以在此方法中做性能优化，如你的props虽然发生了变化，但是界面的UI可能无须更新，此时就可以在此方法中返回false，就会告诉组件无须更新
  • render 与创建时相同
  • getSnapshotBeforeUpdate 16.3 新引入的声明周期方法
  • 更新 DOM 和 refs。
  • componentDidUpdate会在更新后会被立即调用。可以在此处对 DOM 进行操作。如果你对更新前后的 props 进行了比较，也可以选择在此处进行网络请求
    • 你也可以在 componentDidUpdate() 中直接调用 setState()，但请注意它必须被包裹在一个条件语句里，否则会导致死循环。它还会导致额外的重新渲染，虽然用户不可见，但会影响组件性能。不要将 props “镜像”给 state，请考虑直接使用 props。

3.卸载时

组件在页面上消失时，进行销毁，会进行一些资源释放的处理。

  • componentWillUnmount componentWillUnmount() 会在组件卸载及销毁之前直接调用。在此方法中执行必要的清理操作，例如，清除 timer，取消网络请求或清除在 componentDidMount() 中创建的订阅等

# 2.constructor

constructor(props)
1

# 3.getDerivedStateFromProps

  1. 当 state 需要从 props 初始化时使用
  2. 尽量不要使用：维护两者状态一致性会增加复杂度
  3. 每次 render 都会调用
  4. 典型场景：表单控件获取默认值

# 4.componentDidMount

  1. UI 渲染完成后调用
  2. 只执行一次
  3. 典型场景：获取外部资源—请求后端接口

你可以在 componentDidMount() 里直接调用 setState()。它将触发额外渲染，但此渲染会发生在浏览器更新屏幕之前。如此保证了即使在 render() 两次调用的情况下，用户也不会看到中间状态。请谨慎使用该模式，因为它会导致性能问题。通常，你应该在 constructor() 中初始化 state。如果你的渲染依赖于 DOM 节点的大小或位置，比如实现 modals 和 tooltips 等情况下，你可以使用此方式处理

# 5.getSnapshotBeforeUpdate

  1. 在页面 render 之前调用，state 已更新
  2. 典型场景：获取 render 之前的 DOM 节点

# 6.componentDidUpdate

  1. 每次 UI 更新时被调用
  2. 典型场景：页面需要根据 props 变化重新获取数据
    • 如获取到路由参数，如果路由参数发生变化，重新从后端接口获取数据

# 7.shouldComponentUpdate

  1. 决定 Virtual DOM 是否要重绘
  2. 一般可以由 PureComponent 自动实现
    • PureComponent (opens new window) 是 React 内置的组件，其自动重写了shouldComponentUpdate 方法，其会对 props 和 state 进行浅层比较，并减少了跳过必要更新的可能性。
  3. 典型场景：性能优化

使用 PureComponent 时需要注意，其在 shouldComponentUpdate 方法中进行了浅比较，对于对象类型或者函数，进行浅比较时，得到的结果可能会不是你所期望的

export default class Clock extends React.Component {
  // 构造方法
  constructor(props) {
    super(props);
    console.log("Clock constructed");
    this.state = { date: new Date() };
  }

  // 组件挂载时
  componentDidMount() {
    // 计时器，每秒调用一次
    console.log("Clock did mount");
    this.timerID = setInterval(() => this.tick(), 1000);
  }

  // 组件卸载时
  componentWillUnmount() {
    // 清除计时器
    console.log("Clock will unmount");
    clearInterval(this.timerID);
  }

  // 组件更新时
  componentDidUpdate() {
    console.log("Clock did update");
  }

  tick() {
    // 更新最新的时间
    this.setState({
      date: new Date()
    });
  }

  render() {
    return (
      <div>
        <h2>
          It is {this.state.date.toLocaleTimeString()}.
        </h2>
      </div>
    );
  }
}

  • 使用 getSnapshotBeforeUpdate 获取原容器的高度
  • 在 componentDidUpdate 中根据原容器高度和新容器高度来更新滚动条高度
  • 最后使得页面相对位置的数据不会发生滚动
export default class SnapshotSample extends PureComponent {
  state = {
    messages: [],
  };

  // 添加新消息
  handleNewMessage() {
    this.setState(prev => ({
      messages: [`msg ${prev.messages.length}`, ...prev.messages],
    }));
  }

  componentDidMount() {
    for (let i = 0; i < 20; i++) this.handleNewMessage();
    this.interval = window.setInterval(() => {
      // 超过200条消息清除计时器
      if (this.state.messages.length > 200) {
        window.clearInterval(this.interval);
        return;
      }
      this.handleNewMessage();
    }, 1000);
  }

  componentWillUnmount() {
    window.clearInterval(this.interval);
  }

  getSnapshotBeforeUpdate() {
    // 当前容器节点的滚动条高度
    return this.rootNode.scrollHeight;
  }

  /**
   * componentDidUpdate
   * @param prevProps 更新前的 props
   * @param prevState 更新前 state
   * @param prevScrollHeight 更新前-当容器节点的滚动条高度
   */
  componentDidUpdate(prevProps, prevState, prevScrollHeight) {
    const scrollTop = this.rootNode.scrollTop;
    if (scrollTop < 5) return;

    // 保持滚动条位置不变: 滚动条位置 + (新的容器高度 - 原容器高度)
    this.rootNode.scrollTop =
      scrollTop + (this.rootNode.scrollHeight - prevScrollHeight);
  }

  render() {
    return (
      <div className="snapshot-sample" ref={n => (this.rootNode = n)}>
        {this.state.messages.map(msg => (
          <div>{msg}</div>
        ))}
      </div>
    );
  }
}

# 五、JSX 的运行基础：虚拟DOM

# 1.DOM 过于复杂

在 JS 中实现重新渲染的最简单办法是：当任何内容改变时，都重新构建整个 DOM，然后用新 DOM 取代旧 DOM 。

但是DOM 是复杂的，对它的操作（尤其是查询和创建）是非常慢非常耗费资源的。

解决这个问题，可以将没有改变的 DOM 节点，让它保持原样不动，仅仅创建并替换变更过的 DOM 节点，实现了 DOM 节点复用（Reuse）

因此就需要差异对比（与我们使用的 git 原理相似，使用 Diff 算法比较不同之处），但是比较两个 Tree 的 Diff 标准算法的复杂度为 O(n3)，复杂度很高。

Diff 算法的作用是用来计算出 Virtual DOM 中被改变的部分，然后针对该部分进行原生DOM操作，而不用重新渲染整个页面。

# 2.Virtual DOM和React的Diff算法

React 组件在内部维护了一套虚拟 DOM 状态， 这套虚拟 DOM 状态最终会映射到真实的 DOM 节点上，但是当内部的虚拟 DOM 发生变化时，需要去计算虚拟 DOM 之间的区别，产生一个 Diff。最终在真是的 DOM 节点上并不是整体刷新所有的 DOM ，而是只会把 Diff 的部分，以一种高效的方式更新到UI上。

React 则对 Diff 算法进行优化，将复杂度降低到 O(n)

其过程如下：

  1. 维护一个使用 JS 对象表示的 Virtual DOM，与真实 DOM 一一对应
  2. 对前后两个 Virtual DOM 做 diff ，生成变更（Mutation）
  3. 把变更应用于真实 DOM，生成最新的真实 DOM

React的Diff算法优化：

  • 广度优先分层比较：拿到 DOM 树后，对前后的树一层一层的进行比较

  • 顺序是否发生变化：根据DOM 节点的属性判断顺序

  • 若DOM的类型变化：删除原节点，创建新节点

  • 节点跨层移动

    • 如节点 C 原来在 B 节点下（B——>C），后面移动到了 B 节点下的 Z 节点下（B——>Z——>C）。对于此种情况 React 会直接删除 C 节点。等到达 Z 节点下的的节点比较时，会创建 C 节点。（不关心该节点是否被其他节点用到，此时会性能比较低）
    • 一般不会出现这种情况，UI层面一般来说比较稳定，因此针对这种很少的应用场景，使用此种方式也是比较合理的

Diff 理解可参考文章：

  • https://segmentfault.com/a/1190000013365426 (opens new window)
  • https://zhuanlan.zhihu.com/p/20346379 (opens new window)

# 3.虚拟 DOM 的两个假设

  1. 组件的 DOM 结构是相对稳定的，很少出现跨层移动的情况
    • 开发时尽量使得 DOM 结构保持稳定。例如，可以通过 CSS 隐藏或显示节点，而不是真的移除或添加 DOM 节点。
  2. 类型相同的兄弟节点可以被唯一表示
    • 对两个相邻的同组件，Diff 算法比较时，会根据其属性来判断他们是不是顺序发生了变化，因此在对相同的兄弟组件应该给他们加上KEY，且KEY不能相同，以便区分两个组件的顺序

  // ...并返回另一个组件...
  return class extends React.Component {
    state = { time: new Date() };
    componentDidMount() {
      // 开启计时器
      this.timerID = setInterval(() => this.tick(), 1000);

    componentWillUnmount() {
      // 清除计时器  
      clearInterval(this.timerID);

    tick() {
      // 更新 time 为最新的时间  
      this.setState({
        time: new Date()
      });
    }
    render() {
      // 传递属性 time  
      return <WrappedComponent time={this.state.time} {...this.props} />;
    }
  };
}

# 七、Content API

官方文档：Context (opens new window)

Context 是 React 16.3 提供的新特性

Context 提供了一种在组件之间共享此类值的方式，而不必显式地通过组件树的逐层传递 props。

使用 Content API：

  • React.createContext 创建一个 Context 对象。当 React 渲染一个订阅了这个 Context 对象的组件，这个组件会从组件树中离自身最近的那个匹配的 Provider 中读取到当前的 context 值。
  • Context.Provider Provider 接收一个 value 属性，传递给消费组件。一个 Provider 可以和多个消费组件有对应关系。多个 Provider 也可以嵌套使用，里层的会覆盖外层的数据。
  • Class.contextType 挂载在 class 上的 contextType 属性可以赋值为由 React.createContext() (opens new window) 创建的 Context 对象。此属性可以让你使用 this.context 来获取最近 Context 上的值。你可以在任何生命周期中访问到它，包括 render 函数中。
  • Context.Consumer一个 React 组件可以订阅 context 的变更，此组件可以让你在函数式组件 (opens new window)中可以订阅 context。
  • Context.displayName context 对象接受一个名为 displayName 的 property，类型为字符串。React DevTools 使用该字符串来确定 context 要显示的内容。

示例：切换主题/语言

const enStrings = {
  submit: "Submit",

  render() {
    return (
      <LocaleContext.Provider value={this.state.locale}>
        <button onClick={this.toggleLocale}>
          切换语言

        {this.props.children}

    );

}

# 1.Redux 介绍

Redux 是用于管理 JavaScript 应用程序中的数据状态和 UI 状态的工具。它非常适合随着时间的推移管理状态可能很复杂的单页应用程序 (SPA)。它也与框架无关，因此虽然它是在考虑 React 的情况下编写的，但它甚至可以与 Angular (opens new window)或 jQuery 应用程序一起使用。

Redux 让组件通信更加容易：平级的组件通信必须通过父组件作为连接者进行通信

图片来自：https://css-tricks.com/learning-react-redux/

Redux 的特性：

  1. Single Source of Truth

    • Redux 将所有的状态都放在唯一的一个 store 中，这样所有的

  2. 可预测性

    • state + action = new state
    • 会重新new一个state，而不是在原来的基础上进行修改

  3. 纯函数更新 Store

    • 使用 action 触发，通过函数 reducer 去产生一个新的 store

  return bindActionCreators({ plusOne, minusOne }, dispatch);

# 6.异步 Action

异步 Action

我们在上面看到的 action 都是同步的 action，异步 action 一般用于发送一个异步请求，如下图

图片来源：http://cn.redux.js.org/tutorials/fundamentals/part-6-async-logic

在视图 UI 中，用户点击了 button，触发了 Action ，这个Action 并不是一个对象，而是描述了发送 ajax 请求的过程。

在同步 Action 中，当 Action 发出之后，会进如到 Dispatcher 这一层。但是我们发现异步 Action 会多了一个 Middlewares 这样一个中间件，这个中间会截获某种类型的 Action，然后中间件根据 Action 去访问 API，API 处理完成之后，会根据处理后的最终结果再去 Dispatcher

相比同步 Action，异步 Action 多了一个 Middleware，Middleware 会帮助我们实现异步 Action 的流程。

TIPS

需要注意的是，异步 Action 实际上是一种设计模式，UI 层发出的 Action，仍然是标准的 Action，和同步 Action 没有区别。

不同的是 Middleware，异步 Action 则是将 同步Action 的 Dispatcher 分成了不同阶段，最后从整体上看我们完成了异步 Action

Middlewares

Middlewares 有两个作用

  1. 截获 action
    • 判断 action 是否是一个函数/Promise，仅为函数时才会截获
  2. 发出 action
    • 截获 action 后，会做一些处理，处理完成之后，还是会发出 action

以下是 Middlewares 的一些示例：

  }

export default function onAction(state = defaultState, action) {
    switch (action.type) {
        case Types.POPULAR_REFRESH_SUCCESS://下拉刷新成功
            return {
                ...state,
                [action.storeName]: {
                    ...state[action.storeName], 
                    items: action.items, // 原始数据
                    projectModes: action.projectModes,// 此次要展示的数据
                    isLoading: false,
                    hideLoadingMore: false,
                    pageIndex: action.pageIndex
                }
            };
        case Types.POPULAR_REFRESH://下拉刷新
            return {
                ...state,
                [action.storeName]: {
                    ...state[action.storeName],
                    isLoading: true,
                    hideLoadingMore: true,
                }
            };
        default:
            return state;
    }

}
