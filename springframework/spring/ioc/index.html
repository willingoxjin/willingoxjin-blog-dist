02.IoC 容器介绍

# 一、IoC 概述

# 1.IoC 介绍

控制反转（Inversion of Control，缩写为 IoC）是一种设计原则。可以用来降低代码之间的耦合度。其中最常见的方式叫做依赖注入（Dependency Injection，简称 DI），还有一种方式叫“依赖查找”（Dependency Lookup）。

IoC 类似于好莱坞原则（Hollywood principle）：不要打电话给我，我会打打电话给你。在程序中，依赖和数据由其他方提供，而非你自己读取。

# 2.IoC 主要实现策略

在面向编程中，有以下技术可以实现 IoC：

  1. 使用服务定位模式（Service location patten）：如服务定位模式可以通过 JNDI（Java Naming and Directory Interface）动态地查找和获取远程服务或对象的引用。
  2. 使用依赖注入。如：
    • 构造注入
    • 参数注入
    • Setter 注入
    • Interface 注入
  3. 使用上下文的依赖查询：在 Java Beans 中，可以使用 BeanContext 实现。
  4. 使用模版方法的设计模式：在 Spring 中，JdbcTemplate 就是使用了模版方法，程序不必关心 Callback 从哪里得到，以此实现反转控制。
  5. 使用策略模式。

# 3.IoC 容器的职责

通用职责：

  1. 依赖处理
    • 依赖查找（Dependency Lookup）
    • 依赖注入（Dependency Injection）
  2. 生命周期管理
    • 容器的生命周期
    • 托管的资源（Java Beans 或其他资源）的生命周期
  3. 配置
    • 容器的配置
    • 外部化配置
    • 托管的资源（Java Beans 或其他资源）的配置

# 二、IoC 容器的实现

# 1.IoC 容器的实现方式

Java SE

  1. Java Beans (opens new window)
  2. Java ServiceLoader SPI：JDK 1.6 开始引入，通过此方式加载组件，如 NetBeans。
  3. JNDI（Java Naming and Directory Interface）

Java EE

  1. EJB（Enterprise Java Beans）：EJB 1.0 和 EJB 2.0，主要使用依赖查找；从 EJB 3.0 开始，同时支持依赖注入和依赖查找。
  2. Servlet：通过 Model2 的设计模式，获取数据库源、线程池、消息服务时，通过 JNDI 的方式在 Servlet 、EJB、或 Java EE 容器中进行获取。

开源实现：

  1. Apache Avalon (opens new window)：开源的轻量级依赖查找框架。
  2. PicoContainer (opens new window)：一个轻量级的 Java 容器框架，用于实现依赖注入（DI）和控制反转（IoC）等设计模式
  3. Goole Guice (opens new window)：支持依赖注入和依赖查找
  4. Spring Framework (opens new window)

Spring 作为 IoC 容器的优势？

实际上，Spring 并不单纯的提供了 IoC 容器的功能，其优势如下：

  1. 典型的 IoC 管理，依赖查找和依赖注入；
  2. AOP 抽象
  3. 事务抽象
  4. 事件机制（ApplicationEvent）
  5. SPI 扩展（BeanPostProcessor、BeanFactoryPostProcessor、Spring Factories）
  6. 强大的第三方整合
  7. 易测试
  8. 更好的面向对象实现

# 2 Java Beans

Java Beans 作为 Ioc 容器的特性如下：

  1. 依赖查找
  2. 生命周期管理
  3. 配置元信息
  4. 事件
  5. 自定义
  6. 资源管理
  7. 持久化

相关规范：

  • Java Beans：https://www.oracle.com/technetwork/java/javase/tech/index-jsp-138795.html (opens new window)
  • BeanContext：https://docs.oracle.com/javase/8/docs/technotes/guides/beans/spec/beancontext.html (opens new window)

Java Benas 的部分 API 如下：

  1. BeanInfo：用于提供有关Java Bean的元数据信息
  2. Introspector：该类提供了一种标准的方式获取 JavaBean 支持的属性、事件和方法。Introspector 会通过自省的方式，根据目标类获取到 BeanInfo 信息。
  3. PropertyEditor：用于将一种数据类型转换为另一种数据类型。它通常用于在图形用户界面中编辑 Java Bean 的属性值。PropertyEditor 提供了一种可视化的方式来编辑属性，并且可以与 BeanInfo 一起使用来提供更加灵活和优化的属性编辑功能。
  4. FeatureDescriptor：FeatureDescriptor 类是 PropertyDescriptor、EventSetDescriptor 和 MethodDescriptor 等类的父类。它支持一些可以为任何自省描述符设置和检索的常见信息。此外，它还提供了一种扩展机制，以便可以将任意属性值对与设计特征相关联。
    • PropertyDescriptor：描述了 Java Bean 的属性
    • EventSetDescriptor：描述给定了 Java Bean 的事件
    • MethodDescriptor：描述了 Java Bean 的方法。
    • ……

程序运行时我们需要知道 Java Bean 支持哪些属性、事件和方法，我们称这个过程为内省。 内省是对反射的包装，不直接使用反射的原因是要提供一种标准化的方式去读取 bean 的信息，这种标准化值的就是 JavaBeans 的标准

如下是使用 Java Beans 的一个简单示例：

/**
 * Java Beans的特性：可写方法(Writable) / 可读方法(Readable)
 */
public class Person {

    private String username; // 将属性称为 Property

    private Integer age;

    public String getUsername() {
        return username;
    }

    public void setUsername(String username) {
        this.username = username;
    }

    public Integer getAge() {
        return age;
    }

    public void setAge(Integer age) {
        this.age = age;
    }
}


public class BeanInfoDemo {

    public static void main(String[] args) throws IntrospectionException {
        Person person = new Person();
        person.setUsername("zhangsan");
        person.setAge(24);

        // 使用自省的方式获取 BeanInfo
        // - beanClass: 指定Java Bean的类
        // - stopClass: 指定需要停止递归的父类
        BeanInfo beanInfo = Introspector.getBeanInfo(Person.class, Object.class);
        // 获取并遍历所有的 PropertyDescriptor
        Stream.of(beanInfo.getPropertyDescriptors())
                .forEach((PropertyDescriptor propertyDescriptor) -> {
                    System.out.println(propertyDescriptor);

                    // PropertyDescriptor 允许添加属性编辑器 -> PropertyEditor
                    // GUI -> text(String) -> PropertyType
                    String propertyName = propertyDescriptor.getName();
                    // 为 "age" 属性增加 PropertyEditor
                    if ("age".equals(propertyName)) {
                        // age 属性的输入值 String 类型, 输出要为 Integer, 可以这么做:
                        propertyDescriptor.setPropertyEditorClass(StringToIntegerPropertyEditor.class);
                        System.out.println(propertyDescriptor);
                        // ...

                        Method writeMethod = propertyDescriptor.getWriteMethod(); // Setter 方法
                        Method readMethod = propertyDescriptor.getReadMethod(); // Getter 方法
                        try {
                            writeMethod.invoke(person, 18);
                            System.out.println(readMethod.invoke(person, null)); // print: 18
                        } catch (IllegalAccessException | InvocationTargetException e) {
                            throw new RuntimeException(e);
                        }
                    }
                });
    }

    static class StringToIntegerPropertyEditor extends PropertyEditorSupport {

        @Override
        public void setAsText(String text) throws java.lang.IllegalArgumentException {
            Integer value = Integer.valueOf(text);
            setValue(value);
        }

    }

}

# 3.轻量级 IoC 容器的实现

IoC 容器如何区分轻量级和重量级？

《Expert One-on-One™ J2EE™ Development without EJB™》一书中对于轻量级容器的特征如下：

  1. 可以管理应用程序代码
  2. 可以快速的启动
  3. 容器不需要特殊的配置来进行操作。
  4. 容器的内存占用较少，以及所依赖的 API 较少。
  5. 在部署工作量和性能开销方面，将添加托管对象的标准设置得非常低的容器，从而可以部署和管理细粒度对象以及粗粒度组件

# 4.依赖注入和依赖查找

依赖注入和依赖查找的对比如下：

依赖处理 实现便利性 代码侵入性 API 依赖性 可读性
依赖查找 主动获取 相对繁琐 侵入业务逻辑 依赖容器 API 良好
依赖注入 被动获取 相对便利 低侵入性 不依赖容器 API 一般

依赖查找是主动或手动的依赖查找方式，通常需要依赖容器或者标准 API 实现。而依赖注入则是手动或自动依赖绑定的方式，无需依赖特定的容器和 API。

# 5.构造器注入和 Setter 注入

依赖注入的实现主要由构造器注入和 Setter 注入实现。下面我们来对比这两种方式。

Spring 的团队鼓励使用构造函数注入，因为它允许您将应用程序组件实现为不可变对象，并确保所需的依赖项不为空。此外，构造器注入的组件总是以完全初始化的状态返回给客户端（调用）代码。并且，大量的构造函数参数是一种不友好的代码风格，这意味着类可能有太多的职责，应该重构以更好地解决适当的关注点分离问题。

Setter 注入的方式，主要用于可选的依赖项，这些依赖项可以在类中被分配合理的默认值。否则，必须在代码使用依赖项的任何地方执行非空检查。Setter 注入的一个好处是，Setter 方法使该类的对象可以在以后重新配置或重新注入。

Setter 注入的优点如下：

  1. 在 IDE 中很好地支持 JavaBean 属性。
  2. JavaBean 属性的 Setter 方法不需要过多的文档进行说明。
  3. JavaBean 属性由子类继承而不需要修改代码。
  4. 如果需要，可以使用标准 JavaBeans 属性编辑器机制进行类型转换（即 PropertyEditor API）。
  5. 许多现有的 JavaBeans 可以在面向 JavaBeans 的 loC 容器（指 BeanContext）中使用，而无需修改。
  6. 如果每个 Setter 都有对应的 Getter （使属性可读，也可写），则可以询问组件当前的配置状态。如果我们希望持久化该状态，例如，在 XML 表单或数据库中，这是特别有用的。若使用构造函数注入，无法找到当前状态。
  7. Setter 注入适用于具有默认值的对象，这意味着并非所有属性都需要在运行时提供。

Setter 注入的缺点：

  1. 调用 Setter 的顺序没有在任何契约中表示。因此，有时需要在调用最后一个 Setter 之后调用一个方法来初始化组件。

    • Spring 为此提供了 org.springframework.beans.factory.InitializingBean 接口；它还提供了调用任意init() 方法的能力。但是，本契约必须以文件形式记录，以确保在容器外正确使用。

  2. 并不是所有必要的 Setter 在使用之前都被调用了。因此，该对象可以保留部分配置。

构造器注入的优点：

  1. 构造函数注入中，能够防止构造属性的不可变，如果要更改构造属性，必须重新实例化对象，以此证实例化对象的状态配置一致。(但是，也可以通过依赖项检查在 JavaBeans 中实现相同的结果，因为 Spring 可以选择性地执行。）不需要初始化方法。
  2. 与使用多个 JavaBean 方法的结果相比，代码可能会稍微少一些，尽管代码复杂性没有区别（如使用 Builder 模式构建对象和 Setter 注入的代码复杂性没有多少区别）。

构造器注入的缺点：

  1. 尽管多参数构造函数是 Java 语言特性，但在现有代码中可能不如使用 JavaBean 属性那么常见。
  2. Java 构造函数参数没有名称，以供自省（目前，JavaBeans 可以通过注解的方式解决）。
  3. IDE 对构造函数参数列表的支持不如 JavaBean Setter 方法。
  4. 较长的构造函数参数列表和较大的构造函数体可能会变得笨拙。
  5. 具体继承可能会出现问题。
  6. 与 JavaBeans相比，对可选属性的支持较差
