03.Spring IoC 容器概述

# 一、Spring IoC 依赖查找

依赖查找有如下方式：

  1. 根据 Bean 名称查找

    • 实时查找
    • 延迟查找

  2. 根据 Bean 类型查找

    • 单个 Bean 对象
    • 集合 Bean 对象

  3. 根据 Bean 名称 + 类型查找

  4. 根据 Java 注解查找

    • 单个 Bean 对象
    • 集合 Bean 对象

下面我们使用代码演示，如何进行依赖查找：

我们新建一个 maven 项目，添加 spring-context 依赖：

<dependency>
  <groupId>org.springframework</groupId>
  <artifactId>spring-context</artifactId>
</dependency>

如下代码，将演示 Spring 中如何进行依赖查找：

/**
 * <p>spring 依赖查找示例</p>
 */
public class DependencyLookupDemo {

    public static void main(String[] args) {
        // 配置 XML 配置文件
        // 启动 Spring 应用上下文
        BeanFactory beanFactory = new ClassPathXmlApplicationContext(
                "classpath:/META-INF/dependency-lookup-context.xml");
        lookupByBeanName(beanFactory);
        lazyLookupByBeanName(beanFactory);
        lookupByBeanType(beanFactory);
        lookupCollectionByBeanType(beanFactory);
        lookupByBeanTypeAndName(beanFactory);
        lookupByAnnotationType(beanFactory);
    }

    /**
     * 根据 Bean 名称实时查找
     */
    private static void lookupByBeanName(BeanFactory beanFactory) {
        User user = (User) beanFactory.getBean("user");
        System.out.println("通过名称查找：" + user);
    }

    /**
     * 根据 Bean 名称延迟查找
     */
    @SuppressWarnings("unchecked")
    private static void lazyLookupByBeanName(BeanFactory beanFactory) {
        ObjectFactory<User> objectFactory = (ObjectFactory<User>) beanFactory.getBean("objectFactory");
        System.out.println("通过名称延迟查找：" + objectFactory.getObject());
    }

    /**
     * 根据 Bean 类型查找单个对象
     */
    private static void lookupByBeanType(BeanFactory beanFactory) {
        User user = beanFactory.getBean(User.class);
        System.out.println("通过类型查找：" + user);
    }

    /**
     * 根据 Bean 类型查找集合对象
     */
    private static void lookupCollectionByBeanType(BeanFactory beanFactory) {
        if (beanFactory instanceof ListableBeanFactory listableBeanFactory) {
            Map<String, User> users = listableBeanFactory.getBeansOfType(User.class);
            System.out.println("通过类型，查找到的所有的 User 集合对象：" + users);
        } else {
            System.err.println("当前beanFactory不支持集合类型查找");
        }
    }

    /**
     * 根据 Bean 名称 + 类型查找
     */
    private static void lookupByBeanTypeAndName(BeanFactory beanFactory) {
        User user = beanFactory.getBean("user", User.class);
        System.out.println("通过类型和名称查找：" + user);
    }

    /**
     * 根据 Java 注解查找
     */
    private static void lookupByAnnotationType(BeanFactory beanFactory) {
        if (beanFactory instanceof ListableBeanFactory listableBeanFactory) {
            Map<String, User> users = (Map) listableBeanFactory.getBeansWithAnnotation(ManagerFlag.class);
            System.out.println("查找标注 @Manager 所有的 User 集合对象：" + users);
        }
    }

}

# 二、Spring IoC 依赖注入

Spring IoC 依赖注入支持以下方式：

  1. 根据 Bean 名称注入
  2. 根据 Bean 类型注入
    • 单个 Bean 对象
    • 集合 Bean 对象
  3. 注入容器内建 Bean 对象
  4. 注入非 Bean 对象
  5. 注入类型
    • 实时注入
    • 延迟注入

下面，我们来通过代码示例依赖注入：

@Getter
@Setter
public class UserRepository {

    private Collection<User> users; // 自定义 Bean

    private BeanFactory beanFactory; // 內建非 Bean 对象（依赖）

    private ObjectFactory<ApplicationContext> objectFactory; //

}

这里我们使用 <import> 标签来导入我们在上一节”Spring IoC 依赖查找“中的示例配置，这也是 Spring 的一个特性，能够方便的复用 Bean 定义文件。

上面的 userRepository 的 Bean 定义 中，我们也可以手动的配置注入 users 、beanFactory 等属性。实际上，这样并不方便，因为一旦我们添加了一个新的 User 类型的 Bean 实例，仍需要手动的为 userRepository 的 users 属性添加该 Bean。 不过，Spring 提供了 autowire 属性用来自动注入，autowire 属性的可选值有：no（默认）、byName、byType、constructor。并且，当 autowire="byType" 和我们手动配置注入同时存在时，优先使用手动配置。

下面代码用来测试依赖注入：

public class DependencyInjectionDemo {

    public static void main(String[] args) {
         // 配置 XML 配置文件，启动 Spring 应用上下文
        ApplicationContext applicationContext = new ClassPathXmlApplicationContext("classpath:/META-INF/dependency-injection-context.xml");

        // Collection<User> users
        UserRepository userRepository = applicationContext.getBean("userRepository", UserRepository.class);
        System.out.println(userRepository.getUsers());

        // BeanFactory beanFactory
        System.out.println(userRepository.getBeanFactory());

        // 依赖注入的来源和依赖查找的来源并不是同一个来源
        System.out.println(userRepository.getBeanFactory() == applicationContext); // false
        System.out.println(userRepository.getObjectFactory().getObject() == applicationContext); // true

        // 抛出异常，说明 UserRepository 中注入的 BeanFactory Bean 并不来源于 ioc 容器
        // BeanFactory bean = applicationContext.getBean(BeanFactory.class); // throw NoSuchBeanDefinitionException
    }

}

# 三、Spring IoC 依赖来源

Spring IoC 依赖来源可分为如下三方面：

  1. 自定义 Bean
  2. 容器内建 Bean 对象：容器内容自动初始化的一些 Bean 对象。如 Environment 类型的 Bean 实例。
  3. 容器内建依赖：无法通过依赖查找获取。如 BeanFactory 类型的 Bean 实例。

下面的代码中，注明了三种依赖来源的方式：

public class DependencyInjectionDemo {

    public static void main(String[] args) {
        // 配置 XML 配置文件，启动 Spring 应用上下文
        ApplicationContext applicationContext = new ClassPathXmlApplicationContext("classpath:/META-INF/dependency-injection-context.xml");

        // 依赖来源一：自定义 Bean
        UserRepository userRepository = applicationContext.getBean("userRepository", UserRepository.class);
        System.out.println(userRepository.getUsers());

        // 依赖来源二：內建依赖
        System.out.println(userRepository.getBeanFactory());
        // 抛出异常，说明 UserRepository 中注入的 BeanFactory Bean 并不来源于 ioc 容器
        // BeanFactory bean = applicationContext.getBean(BeanFactory.class); // throw NoSuchBeanDefinitionException

        // 依赖来源三：容器內建 Bean
        Environment environment = applicationContext.getBean(Environment.class);
        System.out.println("Environment Bean：" + environment);
    }

}

# 五、BeanFactory 和 ApplicationContext

# 1.BeanFactory 和 ApplicationContext 的区别

上面的示例代码中，我们即使用了 BeanFactory，也用到了 ApplicationContext。那么，Spring 中的 BeanFactory 和 ApplicationContext 到底哪个是真正的 IoC 容器？

在Spring Framework (opens new window)官方文档中，进行了如下解释：

org.springframework.bean 和 org.springframework.context 包是 Spring Framework 的 IoC 容器的基础。BeanFactory 接口提供了一种高级配置机制，能够管理任何类型的对象。ApplicationContext 是 BeanFactory 的子接口。它增加了如下功能：

  • 简化了与 Spring 的 AOP 特性集成
  • 消息资源处理（用于国际化）
  • 事件发布
  • 应用程序层特定的上下文，例如用于 web 应用程序的 WebApplicationContext

简单来说，BeanFactory 提供了配置框架和基本功能，而 ApplicationContext 则添加了更多特定于企业的功能。ApplicationContext 是 BeanFactory 的一个完整的超集。

上面讲到 BeanFactory 能够管理任何类型的对象，而并非说”管理任何类型的 Bean“。因为依赖注入的来源并不仅限于 Bean。

我们再来进入 ClassPathXmlApplicationContext 的源码看看。

在 ClassPathXmlApplicationContext 的父类 AbstractRefreshableApplicationContext 中，对 getBeanFactory() 方法进行了实现。我们发现，其使用组合模式，当我们调用 getBeanFactory() 方法获取 ConfigurableListableBeanFactory 时，实际上获取的是当前类的内部属性 DefaultListableBeanFactory。

所以，我们在前面的示例中，打印的 UserRepository 的 BeanFactory 属性时，控制台输出的类型是 DefaultListableBeanFactory。

public abstract class AbstractRefreshableApplicationContext extends AbstractApplicationContext {    
    // ...

    @Nullable
	private volatile DefaultListableBeanFactory beanFactory;

	@Override
	public final ConfigurableListableBeanFactory getBeanFactory() {
		DefaultListableBeanFactory beanFactory = this.beanFactory;
		if (beanFactory == null) {
			throw new IllegalStateException("BeanFactory not initialized or already closed - " +
					"call 'refresh' before accessing beans via the ApplicationContext");
		}
		return beanFactory;
	}
    
    @Override
	protected final void refreshBeanFactory() throws BeansException {
		if (hasBeanFactory()) {
			destroyBeans();
			closeBeanFactory();
		}
		try {
			DefaultListableBeanFactory beanFactory = createBeanFactory();
			beanFactory.setSerializationId(getId());
			customizeBeanFactory(beanFactory);
			loadBeanDefinitions(beanFactory);
			this.beanFactory = beanFactory;
		}
		catch (IOException ex) {
			throw new ApplicationContextException("I/O error parsing bean definition source for " + getDisplayName(), ex);
		}
	}

    /**
     * 为此上下文创建内部 Bean 工厂, 每次调用 refresh() 都会尝试调用此方法
     * 默认实现创建 DefaultListableBeanFactory，
     * 并使用此上下文的父 bean 的内部 bean 工厂作为父 bean 工厂，
     * 可以在子类中重写此方法，例如自定义 DefaultListableBeanFactory 的设置。
     */
    protected DefaultListableBeanFactory createBeanFactory() {
		return new DefaultListableBeanFactory(getInternalParentBeanFactory());
	}

    // ...
}

BeanFactory 和 FactoryBean 的区别？

  • BeanFactory 是 IoC 容器。
  • FactoryBean 是创建 Bean 的一种方式，帮助实现复杂的初始化逻辑。

# 2.ApplicationContext 的特性

通过上面的学习，我们知道 BeanFactory 和 ApplicationContext 的区别如下：

  • BeanFactory 是 Spring 底层 IoC 容器

  • ApplicationContext 是具备应用特性的 BeanFactory 超集。

ApplicationContext 除了 IoC 容器的角色，还提供了如下的特性：

  1. 面向切面（AOP）
  2. 配置元信息（Configuration Metadata）
  3. 资源管理（Resources）
  4. 事件（Events）
  5. 国际化（i18n）
  6. 注解（Annotations）
  7. Environment 抽象（Environment Abstraction）

下面我们使用代码来直观地看看这两者的区别：

首先，我们手动初始化 DefaultListableBeanFactory，并使用 XmlBeanDefinitionReader 读取 xml 文件配置：

public class BeanFactoryAsIocContainerDemo {

    public static void main(String[] args) {
        // 创建 BeanFactory 容器
        DefaultListableBeanFactory beanFactory = new DefaultListableBeanFactory();
        // 加载配置
        String locationClassPath = "classpath:/META-INF/dependency-injection-context.xml";
        new XmlBeanDefinitionReader(beanFactory).loadBeanDefinitions(locationClassPath);
        // 依赖查找
        UserRepository userRepository = beanFactory.getBean(UserRepository.class);
        System.out.println(userRepository.getUsers());
        System.out.println(userRepository.getBeanFactory());
        System.out.println(userRepository.getObjectFactory());
    }

}

public class AnnotationContextAsIocContainerDemo {

    public static void main(String[] args) {
        // 创建 BeanFactory 容器
        AnnotationConfigApplicationContext context = new AnnotationConfigApplicationContext();
        // 将当前类 AnnotationContextAsIocContainerDemo 作为配置类（Configuration Class）
        context.register(AnnotationContextAsIocContainerDemo.class);
        // 启动应用上下文
        context.refresh();
        // 依赖查找
        Map<String, User> beans = context.getBeansOfType(User.class);
        System.out.println("查找到的所有 User 类型的集合对象：" + beans);
    }

    @Bean
    public User user() {
        User user = new User();
        user.setId(2L);
        user.setUsername("李四");
        return user;
    }

}

# 六、Spring IoC 容器生命周期

Spring IoC 容器生命周期可以分为如下阶段：

  1. 启动
  2. 运行
  3. 停止

在 AbstractApplicationContext 的 refresh() 方法中，进行了 IoC 容器 的初始化和启动处理。虽然 refresh 方法表示刷新的意思，但实际进行的是启动处理。

部分源码如下：

public abstract class AbstractApplicationContext extends DefaultResourceLoader
		implements ConfigurableApplicationContext {
    
    @Override
	public void refresh() throws BeansException, IllegalStateException {
        // 问：为什么加锁？
        // 答：加对象锁是为了避免多线程同时刷新 Spring 上下文，
        // 也避免在调用 refresh() 方法时。同时调用 close() 方法
		synchronized (this.startupShutdownMonitor) {
			StartupStep contextRefresh = this.applicationStartup.start("spring.context.refresh"); // 计时器

			// 准备此上下文以进行刷新。
			prepareRefresh();

			// 通知子类刷新内部 bean 工厂。
			ConfigurableListableBeanFactory beanFactory = obtainFreshBeanFactory();

			// 准备在此上下文中使用的 bean 工厂。
            // 这里对内建依赖及内建bean等进行初始化
			prepareBeanFactory(beanFactory);

			try {
				// 允许在上下文子类中对bean工厂进行后处理。
                // 注册 BeanPostProcessor，可在此定制 BeanFactory
				postProcessBeanFactory(beanFactory);

				StartupStep beanPostProcess = this.applicationStartup.start("spring.context.beans.post-process"); // 计时器
                // 调用 BeanFactory 的后置处理器：BeanFactoryPostProcessor
                // BeanFactoryPostProcessor 的作用是在 BeanDefinition 创建之后，Bean 实例化对象前，对 BeanDefinition 对象相关属性进行修改
				invokeBeanFactoryPostProcessors(beanFactory);

				// 注册用户自定义的 BeanPostProcessors（这里只是注册，调用是在 BeanFactory 中处理的）
                // invokeBeanFactoryPostProcessors 是对 BeanFactory 进行后置处理
                // registerBeanPostProcessors 是对 Bean 进行后置处理
				registerBeanPostProcessors(beanFactory);
				beanPostProcess.end(); // 计时器

				// 为此上下文初始化国际化处理（i18n）、
				initMessageSource();

				// 为此上下文初始化事件。
				initApplicationEventMulticaster();

				// 初始化特定上下文子类中的其他特殊bean。
				onRefresh();

				// 检查事件监听器bean并注册它们。
				registerListeners();

				// 实例化所有剩余的(非lazy-init)单例。
				finishBeanFactoryInitialization(beanFactory);

				// 最后一步:发布相应的事件。
				finishRefresh();
			}

			catch (BeansException ex) {
				if (logger.isWarnEnabled()) {
					logger.warn("Exception encountered during context initialization - " +
							"cancelling refresh attempt: " + ex);
				}

				// Destroy already created singletons to avoid dangling resources.
				destroyBeans();

				// Reset 'active' flag.
				cancelRefresh(ex);

				// Propagate exception to caller.
				throw ex;
			}

			finally {
				// 重置Spring核心的公共自省缓存，因为我们可能不再需要单例bean的元数据了…
				resetCommonCaches();
				contextRefresh.end();
			}
		}
	}

    protected void prepareRefresh() {
		// Switch to active.
		this.startupDate = System.currentTimeMillis(); // 记录启动的时间戳，为了计算启动总耗时
        // 设置状态为启动激活状态
		this.closed.set(false);
		this.active.set(true);

		if (logger.isDebugEnabled()) {
			if (logger.isTraceEnabled()) {
				logger.trace("Refreshing " + this);
			}
			else {
				logger.debug("Refreshing " + getDisplayName());
			}
		}

		// 初始化 PropertySources。
		initPropertySources();

		// 验证 Environment 中所有的 Properties 必需设置的属性是否已经设置:
		getEnvironment().validateRequiredProperties();

        // 对早期的 ApplicationListeners 进行处理
		// Store pre-refresh ApplicationListeners...
		if (this.earlyApplicationListeners == null) {
			this.earlyApplicationListeners = new LinkedHashSet<>(this.applicationListeners);
		}
		else {
			// Reset local application listeners to pre-refresh state.
			this.applicationListeners.clear();
			this.applicationListeners.addAll(this.earlyApplicationListeners);
		}

		// Allow for the collection of early ApplicationEvents,
		// to be published once the multicaster is available...
		this.earlyApplicationEvents = new LinkedHashSet<>();
	}

	protected ConfigurableListableBeanFactory obtainFreshBeanFactory() {
		refreshBeanFactory();
		return getBeanFactory();
	}
    
}

实际上，Spring Ioc 容器启动前的准备细节远比我们想象的要复杂，其中会有很多预留扩展点的处理，如 IoC 配置元信息读取和解析、注解扫描、事件发布、国际化、Aware 接口的回调等等，这里的源码只是主要的流程。

调用 AbstractApplicationContext 的 close() 方法用于 IoC 容器的关闭 ，停止的相关源码如下：

@Override
public void close() {
    synchronized (this.startupShutdownMonitor) {
        doClose();
        // If we registered a JVM shutdown hook, we don't need it anymore now:
        // We've already explicitly closed the context.
        if (this.shutdownHook != null) {
            try {
                Runtime.getRuntime().removeShutdownHook(this.shutdownHook);
            }
            catch (IllegalStateException ex) {
                // ignore - VM is already shutting down
            }
        }
    }
}

protected void doClose() {
    // Check whether an actual close attempt is necessary...
    if (this.active.get() && this.closed.compareAndSet(false, true)) {
        if (logger.isDebugEnabled()) {
            logger.debug("Closing " + this);
        }

        try {
            // 发布 shutdown 事件.
            publishEvent(new ContextClosedEvent(this));
        }
        catch (Throwable ex) {
            logger.warn("Exception thrown from ApplicationListener handling ContextClosedEvent", ex);
        }

        // 停止所有生命周期 bean，以避免单个销毁过程中的延迟。
        if (this.lifecycleProcessor != null) {
            try {
                this.lifecycleProcessor.onClose();
            }
            catch (Throwable ex) {
                logger.warn("Exception thrown from LifecycleProcessor on context close", ex);
            }
        }

        // 在上下文的 BeanFactory 中销毁所有缓存的单例。
        destroyBeans();

        // 关闭此上下文本身的状态。
        closeBeanFactory();

        // 让子类做一些最后的清理，如果他们愿意的话…
        onClose();

        // 将本地应用程序侦听器重置为预刷新状态。
        if (this.earlyApplicationListeners != null) {
            this.applicationListeners.clear();
            this.applicationListeners.addAll(this.earlyApplicationListeners);
        }

        // 切换应用为非活动状态。
        this.active.set(false);
    }
}
