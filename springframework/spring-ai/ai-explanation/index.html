01.AI 名词

参考：

  • 从 LLM 到 Agent Skill，一期视频带你打通底层逻辑！ (opens new window)

# 一、LLM 大语言模型

LLM 全称：Large Language Model（大语言模型），简称大模型。目前市面上的大模型基本都基于 Transformer 的架构训练出来。Transformer 是 Google 团队 2017年 在论文《Attention is All You Need》中提出。

时间 内容
2017年 Google 团队提出 Transformer
2022年底 openai 发布 GPT-3.5，第一个达到可以级别的大模型
2023年3月 openai 发布 GPT-4，大模型能力大幅度提升
2023年后 Claude、Gemini 等模型逐渐涌出。

LLM 本质是文字接龙游戏，通过预测下一个概率最高的词生成连续文本。

# 二、Token

大模型的运作如下：

  • 编码：我们向大语言模型提问的文字，通过 Tokenizer （分词器），把文字切分成单个的词语，这些单个的词语就是 Token。Tokenizer 将每个 Token 转换成大语言模型认识的 Token ID，然后将转换后的 Token ID 给到大模型中。
  • 运算：大模型经过运算后，返回 Token ID。
  • 解码：最后，Tokenizer 将 Token ID 翻译成 Token。

模型每次只会返回一个 Token，然后通过返回的 Token 预测下一个 Token，这样逐渐形成一段文字。因此，Token 是大模型处理文本的最基本单位。

一般情况下，一个 Token 约等于0.75 和英文单词，或1.5-2个汉字。

# 三、Context 上下文

Context 即上下文，它是大模型每次处理任务时，所接受到的信息总和。我们平时和大模型对话，它可以带出历史的对话内容。在调用大语言模型时，会把历史的内容一起发给大模型。

Context Window 即上下文窗口，指的是 Context 能容纳的最大 Token 数量。

# 四、RAG 检索增强生成

如果公司内部有一份上万字的文档需要交给大模型处理，我们不可能每次都把文档的所有内容都发给大模型，这效率很低，并且成本也很高。

这种场景就可以使用 RAG（Retrieval-Augmented Generation，检索增强生成）技术，RAG 可以从文档中抽出与用户问题最匹配的几个片段内容，只需要将这些片段发给大模型回答问题。

# 五、Prompt 提示词

Prompt 是大模型接收的具体问题或指令。Prompt 的指令越清晰，大模型处理的结果反馈越好。

Prompt 分以下类型：

  • User Prompt：由用户输入的，说明具体任务的 Prompt。比如向大模型提问问题。
  • System Prompt：由开发人员输入，告诉大模型人设和做事规则。比如告诉大模型你是一个 AI 助手，名字叫做铁蛋。
  • Assistant Prompt / Model Prompt：即 上一轮 AI 的回答，放在上下文里，让模型记住对话，用来做续写、多轮对话、记忆。

# 六、Tool

Tool 即工具，其本质是给大模型提供一套可以调用外部工具的能力，让大模型能够感知和影响外部环境。你可以理解为函数，大模型可通过调用工具，返回用户所需要的数据。

例如天气查询工具，大模型无法自己得知天气，就要借助天气查询工具。天气查询工具的参数为城市和日期，返回值为天气信息。需要注意的是，大模型自身无法调用工具，只能知道选择哪种工具处理，它只能借助我们开发者的集成的系统调用工具。

# 七、MCP

MCP（Model Context Protocol，模型上下文协议） 是 Anthropic 于 2024 年底推出的AI 领域统一连接协议，被称为 AI 的 USB-C，核心是让大模型以标准化方式连接外部工具（Tool）、数据源与系统，解决接口碎片化、集成成本高、模型无法安全操作外部世界的问题。

MCP 解决了不同 AI 平台（openai、anthropic、google）接入标准不同的问题，通过 MCP 协议开发者只需要按照协议规范，开发一次工具，就可以被所有的平台使用。

MCP 分三层架构：

  • MCP Host（主机）：用于发起请求、管理会话、协调模型与工具，你可以理解为用户界面。如 Claude Desktop、Cursor IDE、豆包等。
  • MCP Client（客户端）：将模型指令转为标准化 JSON-RPC 请求，与 MCP Server 通信。
  • MCP Server（服务器）：暴露工具 / 数据接口给大模型，可执行 SQL、读写文件、调用 API 等操作。

通过 RAG + MCP 组合，让 AI 既能查知识，又能执行操作。

# 八、Agent

如果让 AI 执行以下任务：查询今天的天气，如果下雨，查询我附近的卖伞的便利店。这时，AI 需要进行如下操作：

  1. AI 思考后，先调用定位工具获取当前用户的经纬度。
  2. 拿到经纬度后，进行思考，调用天气查询工具获取天气情况。
  3. 获取天气情况后，进行思考，判断天气是否是下雨天
  4. 如果是下雨天，进行思考，调用地图工具查询便利店，得到附近的便利店信息
  5. 思考后返回。

以上这种能够自主规划，自主调用工具，最终能够完成用户任务的系统就是 Agent。目前市面上流行的 Claude Code、Codex、Gemini CLI 等就是 Agent。

Agent = LLM（大脑） + 记忆 + 规划能力 + 工具调用能力

流行的 Agent 架构：

  • ReAct（最常用）：Reason + Act 同步进行
  • Plan-Act（先规划后执行）：先生成完整步骤，再一步步执行
  • Self-Reflect（带反思）：执行完自动检查，错了就重来，准确率大幅提升。

# 九、Agent Skill

Agent Skill 是我们提前写好，塞给 Agent 的一份说明文档，Agent Skill 能够让 AI 执行某一类具体任务是，有标准化、可复用、可插拔能力。

Agent Skill 文档：https://agentskills.io/skill-creation/quickstart

示例：

outgoing-check

---
name：outgoing-check
description: 根据用户所在位置的实时天气，提醒用户外出需要携带的物品
---

# 技能目标

你是一个贴心的”出门物品清单助手“，你的任务是根据用户所在位置的实时天气，提醒用户外出需要携带的物品

# 执行步骤：

1.调用”定位工具“，获取用户当前实时位置经纬度。
2.将获取到的经纬度作为参数，调用“天气查询工具”，获取日照情况、降雨概率、空气质量、风力大小、温度数据。
3.根据天气数据结果，及当天是否为工作日，按照下方的“判断规则”整理出需要携带的物品。
4.严格按照下方的“输出格式”向用户输出最终结果。

# 判断规则：
1.手机：必须携带。
2.伞：当“天气工具”返回“有雨时”或“光照强”时，必须携带。
3.口罩：当“天气工具”返回“空气质量差”，或”紫外线强度高“时，必须携带。
4.厚外套：当“天气工具”最高温度返回“温度”小于15摄氏度时，必须携带。
5.耳机：如果当天是工作日，必须携带。
6.水杯：如果当天是非工作日，必须携带。

# 输出格式

你必须输出两段内容：

【结论】
	用一句话总结今天出门天气的关键点，例如“有雨”、“紫外线强”等。
【出门清单】
    - 物品（原因）
    - 物品（原因）

# 示例

## 用户问题
我马上出门了，帮我想下需要准备些什么？

## 工具返回

### 定位工具
{
	"经度": "东经116°23'51.889",
	"维度": "北纬39°54'12.654"
}

### 天气工具
{
	"降雨情况": "小雨",
	"紫外线强度": "较弱",
	"空气质量": "优",
	"风力大小": "微风",
	"最低温度": "9摄氏度",
	"最高温度": "20摄氏度"
}


## 最终输出
【结论】
	有小雨
【出门清单】
    - 手机（必带）
    - 伞（有雨）
    - 水杯（非工作日）
