JVM 理论知识

# 一、JVM 内存结构

# 1.JVM 内存结构

JVM 内存分为5大块：

  • 堆（Heap）
  • 虚拟机栈（VM Stack）
  • 本地方法栈（Native Method Stack）
  • 程序计数器（Program Counter Register）
  • 方法区（Method Area）。

所有的线程都共享堆内存和方法区，且每个线程都有独立的虚拟机栈、本地方法栈、程序计数器。

# 2.堆

堆是 JVM 中最大的一块内存空间，绝大部分对象都存储在堆内存中。

堆内存可以分为新生代、老年代及持久代。新生代又被分为伊甸园（Eden）和存活区（Survivor），存活区由 From Survivor 和 To Survivor 组成。

JDK8之后，持久代被废弃，由元空间代替。但元空间并不是堆内存的一部分，而是一块本地内存。

# 3.虚拟机栈

虚拟机栈占用的是操作系统内存，每个线程对应一个虚拟机栈。

虚拟机栈由栈帧组成，每次方法调用都会创建一个栈帧，压入到栈中（入栈），当方法调用结束，返回执行的结果，就会将该方法弹出（出栈）。

栈帧中存放着一些列的数据：

  • 局部变量表
  • 操作数栈：可以理解为存放临时数据的地方
  • 指向运行时常量池的引用
  • 方法返回地址
  • 动态链接

# 4.本地方法栈

用于支持 native 方法的执行。存储了每个 native 方法的执行状态。本地方法栈和虚拟机栈他们的运行机制一致，唯一的区别是，虚拟机栈执行 Java 方法，本地方法栈执行 native 方法。在很多虚拟机中（如 Sun 的 JDK 默认的 HotSpot 虚拟机），会将虚拟机栈和本地方法栈一起使用。

native 方法都是由 c 语言实现的。

# 5.程序计数器

程序计数器是一个很小的内存区域，不在 RAM 上，而是直接划分在 CPU 上，程序员无法操作它。

它的作用是：JVM 在解释字节码（.class）文件时，存储当前线程执行的字节码行号（地址），只是一种概念模型，各种 JVM 所采用的方式不一样。字节码解释器工作时，就是通过改变程序计数器的值来取下一条要执行的指令，分支、循环、跳转、线程恢复等基础功能都要依赖程序计数器。

# 6.方法区

方法区存放了要加载的类的信息（如类名、修饰符等）、静态变量、构造函数、final定义的常量、类中的字段和方法等信息。

方法区主要包括类信息、运行时常量池、字符串常量池、静态变量。

常量池：

  • 静态常量池：静态常量池也叫 class 文件常量池，主要存放字面量和符号引用
    • 字面量：例如文本字符串、final 修饰的常量
    • 符号引用：例如类和接口的全限定名、字段的名称和描述符、方法的名称和描述符
  • 运行时常量池：
    • 当类加载到内存中后，JVM 就会将静态常量池中的内容存放到运行时常量池中
    • 运行时常量池里面存储的主要是编译期间生成的字面量、符号引用等等
  • 字符串常量池：字符串常量池可以理解成运行时常量池分出来的一部分，类加载到内存的时候，字符串会存到字符串常量池里面

在 Hotspot 虚拟机中，JDK8之前使用持久代实现方法区。

# 2.加载

加载步骤如下：

  • 读取类的二进制流
  • 将二进制流转化为方法区的数据结构，并存放到方法区
  • 在堆中，产生 java.lang.Class 对象

# 3.链接

链接步骤可以细分为：验证、准备、解析三步。

# 3.1 验证

验证1：验证 .class 文件是否符合规范

验证步骤如下：

  • 文件格式的验证
    • .class 文件的内容是否以 0xCAFEBABE 开头（可以使用16进制编辑器查看）
    • 版本号是否合理

.class 文件的内容使用16进制打开如下：

验证2：元数据验证

  • 是否有父类
  • 是否继承的类被 final 修饰（final 修饰的类不能被继承）
  • 非抽象类是否实现了父抽象类或接口的所有抽象方法

验证3：字节码验证

  • 运行检查
  • 栈数据类型和操作码操作参数是否吻合（比如栈空间只有2字节，但其实却需要大于2字节，此时就认为这个字节码是有问题的）
  • 跳转指令是否指向合理的位置

如果.class文件通过了字节码验证，不能说明该文件存在问题；如果没通过字节码验证，那这个文件一定是有问题的。

验证4：符号引用验证

  • 常量池中描述类是否存在
  • 访问的方法或字段是否存在，且是否有足够的权限

可以配置 JVM 启动参数 -Xverify:none 以关闭验证。关闭验证可以加快类的加载速度。

# 3.2 准备

准备的作用：为类的静态变量分配内存，初始化为系统的初始值。

final static 修饰的变量：直接赋值为用户定义的值。如 :

// 准备阶段：直接给value1赋值123
private final static int value1 = 123;

// 准备阶段：value2的值仍为0
private static int value2 = 123;

# 3.3 解析

解析的作用：符号引用转换为直接引用。

符号引用：在编译器，Java 类还不知道引用的对象的实际地址，所以会使用符号引用。

# 4.初始化

# 初始化1

执行<clinit> 方法。clinit 方法由编译器自动收集类里面的所有静态变量的赋值动作及静态语句块合并而成，也叫类构造器方法。

  • 初始化的顺序和源文件中的顺序一致
  • 子类的 <clinit> 方法被调用前，会先调用父类的 <clinit> 方法
  • JVM 会保证 clinit 方法的线程安全性

# 初始化2

初始化时，如果实例化一个新对象，会调用 <init> 方法对实例变量进行初始化，并执行对应的构造方法内的代码。

    static int i = 0;

    public static void main(String[] args) {
        System.out.println("i=" + i); // i=0
    }
}

# 三、编译器优化机制

# 1.字节码是如何运行的？

  • 解释执行：由解释器一行一行翻译执行
    • 优点：没有编译的等待时间
    • 缺点：性能相对比较差
  • 编译执行：把字节码编译成机器码，直接执行机器码
    • 优点：运行效率会高很多，一般认为比解释执行快一个数量级
    • 缺点：带来了额外的开销（内存、CPU开销等）

查询运行模式，可以使用 java -version 查询使用的哪种方式。默认情况使用混合模式（mixed mode）

# 查看java当前的执行模式
java -version

# -Xint 设置JVM执行模式为解释模式(interpreted mode)
java -Xint -version

# -Xcomp JVM优先以编译模式(compiled mode)运行，不能编译的，以解释模式运行
java -Xcomp -version

# -Xmixed JVM以混合模式(mixed mode)运行
java -Xmixed -version

一般情况下执行的流程：

  • 一开始一般由解释器解释执行。

  • 当虚拟机发现某个方法或代码块的运行特别频繁的时候，就会认为这些代码是“热点代码”。

  • 为了提高“热点代码”的执行效率，会用即时编译器（JIT）把这些热点代码编译成与本地平台相关的机器码，并进行各层次的优化（各层次的优化见分层编译）。

# 2.Hotspot 的即时编译器

Hotspot 有两个编译器：C1 编译器和 C2 编译器

C1 编译器：

  • C1 编译器是一个简单快速的编译器
  • 其主要关注局部性的优化
  • 适用于执行时间较短或对启动性能有要求的程序。如 GUI 应用对界面启动速度就有一定要求
  • C1 编译器也称为 Client Compiler

C2 编译器

  • C2 编译器是为长期运行的服务端应用程序做性能调优的编译器
  • 适用于执行时间较长或对峰值性能有要求的程序
  • C2 编译器也称为 Server Compiler

# 3.分层编译

分层编译可分为5种编译级别：

  • level0-解释执行
  • level1-简单 C1 编译：会用C1编译器进行一些简单的优化，不开启 Profiling
  • level2-受限的 C1 编译：仅执行带方法调用次数以及循环回边执行次数 Profiling 的 C1 编译
  • level3-完全 C1 编译：会执行带有所有 Profiling 的 C1 代码
  • level4-C2 编译：使用 C2 编译器进行优化，该级别会启动一些编译耗时较长的优化，一些情况下会根据性能监控信息进行一些非常激进的性格优化

级别越高，应用启动越慢，优化的开销越高，峰值性能也就越高。

分层编译-JVM参数配置示例

  • 只开启 C2 编译器，不使用 C1 编译器：使用参数 -XX:-TieredCompilation 禁用中间编译层（123层）
  • 只开启 C1 编译器，不使用 C2 编译器：使用参数 -XX:-TieredCompilation -XX:-TieredStopAtLevel=1 （TieredStopAtLeve1=1代表仅会使用0，1级别编译；如果设置TieredStopAtLeve1=3，则仅会使用0123级别的编译）

# 4.热点代码

# 4.1 如何找到热点代码

  • 基于采样的热点探测：周期检查各个线程的栈底，如果某些方法总是出现在栈底，说明这些方法是热点方法
  • 基于计数器的热点探测：为每一个方法或代码块设置计数器，统计执行的次数，如果方法或代码块执行的次数超过一定阈值就判定为热点方法。

# 4.2 Hotspot 内置两类计数器

  • 方法调用计数器（Invocation Counter）
    • 方法调用计数器用来统计方法被调用的次数。
    • 在不开启分层编译的情况下，在 C1 编译器下的默认阈值是 1500 次，在 C2 编译器下的默认阈。值是 10000 次。也可以用 JVM 参数 -XX:CompileThreshold=xxx 手动执行阈值大小。
  • 回边计数器（Back Edge Counter）
    • 回边计数器用于统计一个方法中循环体代码执行的次数。
    • 在字节码中遇到控制流向后跳转的指令称为“回边”（Back Edge）。
    • 在不开启分层编译的情况下，C1编译器的默认阈值为 13995 ，C2默认为10700。也可以使用JVM 参数 -XX:OnStackReplacePercentage=xxx 手动执行阈值大小。
    • 建立回边计数器的主要目的是为了触发 OSR 编译（On Stack Replacement）。（OSR编译机制：OSR（On-Stack Replacement）是怎样的机制 (opens new window)）

当开启分层编译器时，JVM 会根据当前待编译的方法数以及编译线程数来动态调账阈值。此时，参数 -XX:CompileThreshold、-XX:OnStackReplacePercentage 都会失效。

方法统计的半衰周期：

  • 如果不做任何设置，方法调用计数器统计的并不是被调用的绝对次数，而是一个相对的执行频率，即一段时间之内方法被调用的次数。
  • 当超过一定的时间限度，如果方法的调用次数仍然不足以让它提交给即时编译器编译，那这个方法的调用计数器就会被减少一半，这个过程称为方法调用计数器热度的衰减，而这段时间就被称为此方法统计的半衰周期。
  • 进行热度衰减的动作是在虚拟机进行垃圾收集时顺便进行的，可以使用虚拟机参数 -XX:-UseCounterDecay 来禁止热度衰减，让方法计数器统计方法调用的绝对次数。这样，只要系统运行时间足够长，绝大部分方法都会被编译成本地代码。
  • 另外，可以使用虚拟机参数 -XX:CounterHalfLifeTime 设置半衰周期的时间（单位：秒）。

# 四、方法内联

# 1.方法内联介绍

方法内联：把目标方法的代码复制到发起调用的方法之中，避免发生真实的方法调用。

方法内联是即时编译器的一种编译优化方式。

  • 当方法进入时，会从栈中压入元素，方法结束返回执行结果时，则会从栈中弹出元素。压栈和出栈操作会有内存、时间的开销，如果入栈出栈比较频繁，则会影响性能。
  • 而 JVM 会自动识别热点方法，并自动的进行方法内联，以此提高性能。

方法内联示例

如下面的代码中，经过方法内联，就会将两个方法中的逻辑合并到一个方法中。

private static int add1(int x1, int x2, int x3, int x4) {
    return add2(x1, x2) + add2(x3, x4);
}

private static int add2(int x1, int x2) {
    return x1 + x2;
}

# 2.方法内联的条件

  • 条件一：方法体足够小
    • 对于热点方法：如果方法体小于 325 字节会尝试内联，可设置 JVM 参数 -XX:FreqInlineSize 修改热点方法尝试内联的字节大小。
    • 对于非热点方法：如果方法体小于 35 字节，会尝试内联，可设置 JVM 参数 -XX:MaxInlineSize 修改非热点方法尝试内联的字节大小。
  • 条件二：被调用的方法运行时的实现可以被唯一确定
    • 对于被 static 、private、final 修饰的方法，JIT即时编译器可以唯一确定具体的实现代码。
    • 对于 public 修饰的实例方法，指向的实现可能是自身、父类、子类的代码（多态），当且仅当 JIT 即时编译器能够唯一确定方法的具体实现时，才有可能完成内联。

# 3.方法内联注意点

  • 尽量让方法的行数少一些。
  • 尽量使用 static 、private、final 关键字修饰的方法，避免因为多态，需要对方法做额外的检查。
  • 在一些场景下，可通过 JVM 参数修改阈值，从而让更多方法内联。

# 4.内联可能带来的问题

方法内联也是存在缺点的，内联优化的本质是使用空间换时间，通过方法联结，减少方法调用导致的入栈出栈的开销，但是经过内联后方法的代码量会变多。而增加的代码量取决于调用次数和方法本身大小，在极端的场景下，方法内联有可能会导致如下问题：

  • CodeCache 的溢出，导致 JVM 退化成解释执行模式
    • CodeCache 是热点代码的缓存区，即时编译器编译过的代码和本机方法代码都会存放在 CodeCache 中。JDK8 中 CodeCache 默认为 240MB。

# 5.方法内联 JVM 参数

参数 默认值 说明
-XX:+PrintInlining - 打印内联详情，该参数需和 -XX:+UnlockDiagnosticVMOptions 配合使用
-XX:+UnlockDiagnosticVMOptions - 打印 JVM 诊断相关的信息
-XX:MaxInlineSize=n 35 如果非热点方法的字节码超过该值，则无法内联。单位：字节
-XX:FreqInlineSize=n 325 如果热点方法的字节码超过该值，则无法内联。单位：字节

-XX:MaxInlineLevel=n 9 内联方法的最大调用帧数（嵌套调用的最大内联深度）

# 五、逃逸分析

逃逸分析是指分析变量能否能逃出它的作用域。

逃逸分析可以细分为以下场景：

  • 全局变量赋值逃逸
  • 方法返回值逃逸
  • 实例引用逃逸
  • 线程逃逸：当赋值给类变量或其他可以在其他线程中访问的实例变量

通过设置 JVM 参数 -XX:+DoEscapeAnalysis 可以开启逃逸分析（JDK8 中默认开启）

逃逸状态标记

  • 全局级别逃逸：一个对象可能从方法或者当前线程中逃逸（即其他方法或其他线程也能访问到该对象）。主要有以下场景：
    • 对象被作为方法的返回值
    • 对象作为静态字段（static field）或成员变量（field）
    • 如果重写了某个类的 finalized() 方法，那么这个类的对象都会被标记为全局逃逸状态并且一定会放在堆内存中
  • 参数级别逃逸：对象被作为参数传递给一个方法，但是这个方法之外无法访问，且对其他线程不可见
  • 无逃逸状态：没有发生逃逸

示例

全局变量赋值逃逸

public class EscapeTest1 {
     
    public static SomeClass someClass;
	
    public void globalVariablePointerEscape() {
        // 将局部变量赋值给静态变量，作用域被方法了，发生了全局变量赋值逃逸
        someClass = new SomeClass();
    }
     
 }

# 六、标量替换

标量是指不能被进一步分解的量，像 Java 中的基础数据类型和对象引用都属于标量。

与标量相对应的是聚合量，可以进一步分解的量。如字符串，因为字符串是通过字符类型的数字实现的，可以被进一步分解。

标量替换指通过逃逸分析确定该对象不会被外部访问，并且对象可以被进一步分解时，JVM 不会创建该对象，而是创建它的成员变量来代替。

可以设置 JVM 参数 -XX:+EliminateAllocations 开启标量替换（JDK8 默认开启）

标量替换示例：

class ScalarTest {
    int id;
    int age;
}

class Test {
    public void someTest() {
        ScalarTest scalarTest = new ScalarTest();
        scalarTest.age = 1;
        scalarTest.id = 1;
    }
}

# 七、栈上分配

Java 中，大多数对象都是存放在堆中，垃圾回收器会自动回收对象。而栈上分配指的是通过逃逸分析，能够确认对象不会被外部访问，就在栈上分配对象。

在栈上分配的对象，会在栈帧出栈时被销毁，回收内存空间。通过栈上分配可以降低垃圾回收的压力。
