05.存储

# 一、Volume

其实 Volume 和 Docker 的数据卷 Data Volume 的概念是一脉相承的。通过将系统的特定目录以 Volume 的形式 mount 到 Kubernetes 的 POD 中。其中又分成五种形式：

  • emptyDir
  • hostPath
  • storage provider
  • PersistentVolume（PV） -> PersistentVolumeClaim (PVC)
  • StorageClass -> PersistentVolumeClaim 动态的PV

emptyDir 最形象的比喻，就是靶场射击模式的 Docker Volumes，完全由系统管理分配资源，将一个系统目录以指定的读写形式 mount 到容器指定目录。数据将保留到 Pod 消失为止。

hostPath 类似狩猎模式的 Docker Bind Mounts，将用户指定的主机系统目录 mount 到容器内。数据将始终在物理节点上保留。缺点是和节点的目录有紧耦合的限制，使用较少。

stoarge provider 是将公有云或分布式存储（Ceph等）上的物理卷映射给容器，具体的配置方式可以参见各云平台文档。因为是脱离于服务器的物理卷，所以数据不会因为 Pod 和服务器节点的故障而销毁。

后续的两种模式是最常见的 Kubernetes Volume 管理模式。

# 配置示例：emptyDir

apiVersion: v1
kind: Pod
 元信息
matadata:
  name: my-nginx-emptydir
spec:
  containers:
  - name: nginx
    image: nginx:1.20.1
  	volumeMounts:
  	- mountPath: /mymount
  		name: mount-volume
  	volumes:
  	- name: mount-volume
  		emptyDir: {}

2.PV

新增配置文件mypv.yaml：

apiVersion: v1
kind: PersistentVolume
metadata:
	name: mypv
spec:
	capacity:
		storage: 1Gi # 空间大小
	accessModes: # 权限
	- ReadWriteOnce	# 一次读写
	persistentVolumeReclaimPolicy: Recycle # 回收策略
	storagClassName: nfs	# 名称
	nfs:
		path: /mnt
		server: 196.168.31.86

3.PVC

新增配置文件mypvc.yaml：

apiVersion: v1
kind: PersistentVolumeClaim
metadata:
	name: mypvc
spec:
	accessModes: # 权限
	- ReadWriteOnce	# 一次读写
  resources:
  	requests:
  		storage: 1Gi
  storageClassName: nfs

# 二、Secret

方式：

  • --from-literal 从命令行直接被输入密码
  • --from-file 从文件读取密码
  • --from-env-file
  • yaml 文件 （推荐）

Secret 是 Kubernetes 容器内密码密钥等敏感信息的传递方式。

下面使用 yaml 文件使用 Secret 。

  1. YAML 方式创建 Secret

    行base64编码命令获得admin（用户名）和123456（密码）的数值

[root@~]# echo -n admin | base64

[root@~]# echo -n 123456 | base64

# 三、Configmap

Configmap 是 Kubernetes 容器内应用配置文件的传递方式。Configmap 的创建和使用方式也有好几种。而每一种都和 Secret 几乎一致。不同处是 Configmap 的 YAML 文件中的 kind 类型为Configmap，同时所有的数值都是明文输入，不需要 base64 编码的。
