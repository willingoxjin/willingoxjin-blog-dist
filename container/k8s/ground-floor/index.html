03.原理与任务

# 一、Pod 调度原理和生命周期

# 1.Pod 调度原理

Pod 应用中启动了两个容器，一个用来从Content Manage 中拉取文件，放到磁盘 Volume 中，另外一个则是web容器，里面运行着web服务，用来提供给其他服务或客户端访问。

两个容器共享同一个 Volume。多个容器共同组成 Pod ，Pod内的数据、环境、网络等可以共享，但是Pod之间会相互隔离。

Pod 中可以添加很多非功能性应用，并且与功能性应用隔离，就像边车一样（三轮单边摩托车），可以在其中添加熔断、健康检查等非功能应用，使用应用组合的方式达到目的。

# 5.健康检查

  • livenessProbe （容器级别）

    • 若发现容器出问题，会尝试重新容器

    • 适合高可用的场景

  • readinessProbe （容器级别）

    • 若发现容器出问题，则停止该容器对外服务

    • 适合在服务上线，升级的场景

  • restartPolicy （Pod级别）

readinessProbe 配置示例：

spec:
  containers:
  - name: nginx # 容器名
    image: nginx:1.20.1 # 容器镜像
    ports:
    - name: http-port
      containerPort: 80
    # 健康检查
    readinessProbe:
      # 向地址index.html发送get请求，检查是否正常运行
      httpGet:
        port: http-port
        path: /index.html
      initialDelaySeconds: 60 # 初始化运行的延迟时间
      periodSeconds: 3  # 检查成功之后，每3秒检查一次

livenessProbe 配置示例：

spec:
  containers:
  - name: nginx # 容器名
    image: nginx:1.20.1 # 容器镜像
  # 创建一个文件，若出错后删除该文件
  command: ["/bin/sh", "-c", "touch /tmp/test"] 
  livenessProbe:
    exec:
      command: ["test", "-e", "/tmp/test"] # /tmp/test文件若不存在，则重新应用
    initialDelaySeconds: 60 # 初始化运行的延迟时间
    periodSeconds: 3  # 检查成功之后，每3秒检查一次

# 二、Controller 控制器原理

# 1.kubectl apply 命令

执行 kubectl apply 命令之后的流程：

  • 首先生生成一个名为 nginx-deployment 的 Deployment

  • Deployment 的下一级会生成 ReplicaSet，ReplicaSet即当前deployment的状态。ReplicaSet的名称会继续沿用 Deployment的名称，然后加一串16进制的数字

  • 然后 ReplicaSet 会根据配置的 replica 数量，去启动对应数量的 pod。pod的名称会继续沿用 ReplicaSet的名称，然后加 一串16进制的数字

# 2.滚动升级

支持多个 ReplicaSet，进行滚动升级，在升级时，老的 ReplicaSet 不会被删除，只有新的 ReplicaSet 发布成功之后，老的 ReplicaSet 才会被删除。

升级：更新镜像 + kubectl apply

查询历史：

kubectl rollout history deployment
1

TIPS

在使用 kubectl apply 命令启动的时候，可以添加参数 --record 来记录到 history 中，否则我们在查询发布 历史时，会出现 <none> 的情况.

建议添加 --record 参数

回滚：

kubectl rollout undo deployment --to-revision=[回滚到的具体版本号]
1

# 3.Daemonset

由于 Daemon 引擎很多运行在系统的 namespace 中，因此使用 kubectl get daemonset 命令获取对应的 Daemon 时，会出现查询不到的情况，因此需要指定一个默认的 namespace，如下：

kubectl get daemonset --namespace=kube-system
1

kube-proxy 是用来进行网络转换的 daemonSet，我们可以尝试运行以下命令，进入修改界面来查看一下配置：

# 进入 daemonset=kube-proxy 的修改页面
kubectl edit daemonset kube-proxy --namespace=kube-system
1
2

DaemonSet 在每一个物理节点都只会跑一个，可以使用以下命令来查看 Pod，发现名为 kube-proxy-xxx 的 Pod 只会在每个物理节点运行一个。

kubectl get pod --namespace=kube-system -o wide | grep kube-proxy
1

      name: hello
    spec:
      containers:
      - name: hello
        image: busybox
        commond: ["echo", "hello kube !"]

# 5.Cron Job

定时器任务

新建配置文件：hello-cron-job.yaml

apiVersion: batch/v1
kind: CronJob
metadata:
  name: hello
spec:
  schedule: "*/1 * * * *" # corn表达式 每分钟执行一次
  jobTemplate:
    spec:
      template:
        spec:
          containers:
           - name: hello
           image: busybox
           commond: ["echo", "hello kube cron-job!"]
          restartPolicy: Never
