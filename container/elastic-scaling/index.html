容器化之弹性扩缩容

# 微服务扩容

# 扩容

扩容的方向：

  • 基础设施（底层扩容）：服务器资源、存储资源等
  • 系统资源（容器层面扩容）
  • 网络引流

我们这里主要以容器层面扩容和网络引流方向讨论

# 无状态服务扩缩容

# 无服务化 Serverless

  • 指的是应用无状态，并非无服务运行
  • 后台的服务的位置、集群数量不需要我们关心
  • 常见的编程方式：函数式编程、响应式编程
  • 弹性：从 0 资源到无限大扩缩容器

Serverless的实现：观察->决策->执行

  • 观察：通过容器的编排工具
  • 决策：在容器的编排层面，定义一些规则，来决策
  • 执行：在容器引擎本身执行。如网络流量导入到新添加的节点

# 以电商场景为例

  • 热点业务

    • 找到热点业务，对热点业务对应的热点应用根据秒杀预期扩容
    • 在秒杀、抢购中的热点业务：支付、下单、添加购物车（某些秒杀商品不支持添加到购物车）、商品详情页（在某个时间点前，不会给用户访问到详情页）、搜索

  • 热点数据

    • 秒杀产品、动态数据、静态数据（redis缓存）

  • 思路

    • 隔离：把秒杀系统独立出来、将数据独立出来存储到redis缓存中
    • 时间与空间的转化
      • 防击穿、防雪崩
      • 全链路压测：将之前热卖的商品数据复制一份、在测试环境压测
      • 反腐层：防止某个服务故障，导致整个系统不可用
      • 限流
      • 排队
      • 降级
      • 熔断
      • 无服务化
      • 弹性扩容
      • ......

# 扩缩容工具的实现方式

  • Cloud Foundry：Metric监控，规则决策，容器&路由伸缩
  • Mesos Marathon：负载均衡，规则触发，弹性负载均衡
  • Kubernates：事件触发，配置声明，自动部署与路由伸缩

# 无状态服务扩缩容

将有状态的应用进行区分：共享磁盘模式和 Share Nothing

  • 共享磁盘模式：可以转成无状态应用
  • Share Nothing 模式：采用合适的集群管理方式和CAP目标

# 共享磁盘模式

如何转成无状态应用？

  • 结构化数据--> 将数据放到共享数据库

  • 非结构化数据 ---> 放到共享缓存(redis)、对象存储(oss)、搜索引擎(Elasticsearch)等等

  • 减少文件系统的依赖（如使用 CDN 直接对接对象存储等）

# Share Nothing 结构

  • CAP：优化可用性和分区性，弱化一致性
  • 集群管理：优化选举、仲裁、阶段提交（事务）、副本、分片管理（分片时考虑特殊的分片算法）
  • 资源预配置：提前把大量的资源加到集群中
