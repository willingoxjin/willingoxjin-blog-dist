public class NacosConsumerTestController {

    private LoadBalancerClient loadBalancerClient;

        System.out.println("服务地址:" + choose.getUri());
        System.out.println("服务名称:" + choose.getServiceId());

# 1.配置中心介绍对比

微服务中配置文件的问题

  • 配置文件的数量会随着服务的增加持续递增
  • 单个配置文件无法区分多个运行环境
  • 配置文件内容若需要修改，必须重启服务，若集群有成百上千个节点，工作量很大

配置中心解决的问题

  • 统一配置文件
  • 提供统一标准接口，服务根据标准接口自动拉取配置
  • 支持动态更新到所有服务

配置中心支持功能

  • 灰度发布：配置的灰度发布是配置中心比较重要的功能，当配置的变更影响比较大的时候，需要先在部分应用实例中验证配置的变更是否符合预期，然后再推送到所有应用实例。

  • 权限管理：配置的变更和代码变更都是对应用运行逻辑的改变，重要的配置变更常常会带来核弹的效果，对于配置变更的权限管控和审计能力同样是配置中心重要的功能。

  • 版本管理&回滚：当配置变更不符合预期的时候，需要根据配置的发布版本进行回滚。

  • 配置格式校验：应用的配置数据存储在配置中心一般都会以一种配置格式存储，比如Properties、Json、Yaml等，如果配置格式错误，会导致客户端解析配置失败引起生产故障，配置中心对配置的格式校验能够有效防止人为错误操作的发生，是配置中心核心功能中的刚需。

  • 监听查询：当排查问题或者进行统计的时候，需要知道一个配置被哪些应用实例使用到，以及一个实例使用到了哪些配置。

  • 多环境：在实际生产中，配置中心常常需要涉及多环境或者多集群，业务在开发的时候可以将开发环境和生产环境分开，或者根据不同的业务线存在多个生产环境。如果各个环境之间的相互影响比较小（开发环境影响到生产环境稳定性），配置中心可以通过逻辑隔离的方式支持多环境。

  • 多集群：当对稳定性要求比较高，不允许各个环境相互影响的时候，需要将多个环境通过多集群的方式进行物理隔离

常见的配置中心

  • Appllo
    • Appllo 是携程框架部门研发的分布式配置中心，能够集中化管理应用不同环境、不同集群的配置，配置修改后能够实时推送到应用端，并且具备规范的权限、流程治理等特性，适用于微服务配置管理场景。
    • https://github.com/apolloconfig/apollo
  • Disconf
    • 百度开源的配置管理中心，同样具备配置的管理能力，不过目前已经不维护
    • https://github.com/knightliao/disconf
  • Spring Cloud Config
    • Spring Cloud 生态组件，可以和Spring Cloud体系无缝整合
    • https://github.com/spring-cloud/spring-cloud-config
  • Nacos
    • Nacos是阿里开源的，支持基于 DNS 和基于 RPC 的服务发现
    • Nacos的注册中心支持CP也支持AP，对他来说只是一个命令的切换，随你玩，还支持各种注册中心迁移到Nacos，反正一句话，只要你想要的他就有。
    • Nacos除了服务的注册发现之外，还支持动态配置服务，一句话概括就是Nacos = Spring Cloud注册中心 + Spring Cloud配置中心。
    • https://nacos.io/zh-cn/docs/what-is-nacos.html

配置中心选型

  • Apollo和Nacos相对于Spring Cloud Config的生态支持更广，在配置管理流程上做的更好。

  • Apollo相对于Nacos在配置管理做的更加全面，不过使用起来也要麻烦一些。

  • Apollo容器化较困难，Nacos有官网的镜像可以直接部署，总体来说，Nacos比Apollo更符合KISS原则。

  • Nacos使用起来相对比较简洁，在对性能要求比较高的大规模场景更适合。

  • 此外，Nacos除了提供配置中心的功能，还提供了动态服务发现、服务共享与管理的功能，降低了服务化改造过程中的难度。

  • 在开发环境的 Data Id 命名为 mall-orders-dev.yml
  • 在测试环境的 Data Id 命名为 mall-orders-test.yml
  • 在预发布环境的 Data Id 命名为 mall-orders-staging.yml
  • 在生产环境的 Data Id 命名为 mall-orders-prod.yml

注意：在启动 spring boot 应用时，应该指定其启动的环境（指定参数 spring.profiles.active=xxx）：

## 如 测试环境下
java -jar -Dspring.profiles.active=test mall-order-0.0.1-SNAPSHOT.jar

# 4.通用配置文件

# 4.1 相同服务在不同环境中的通用配置

不同环境下如果某些配置是一样的，比如redis的超时时间，连接池大小等等，这样在每个配置文件中都写一份，太繁琐了，为了偷懒，我们可以使用通用的配置文件。

比如，对于商城订单模块 mall-orders，你可以在 nacos 中设置配置文件的 Data Id 为 mall-orders.yml

WARNING

如果通用配置文件和环境配置文件都有某个属性的配置，最终应用得到的配置则是 环境配置 中的配置。 如 mall-orders.yml 和 mall-orders-dev.yml 中都有 helloMsg 这个配置，并且启动时配置的参数 spring.profiles.active=dev，最终会使用 mall-orders-dev.yml 中的 helloMsg 配置

多个shard-configs的文件中存在相同的配置，最终会以哪个配置文件中的值为准？

答：根据配置中的顺序，配置列表的顺序越往后，就会后加载，后加载的配置会覆盖之前加载的配置

多个shard-configs的文件中存在相同的配置，更新配置中心的配置，什么时候会发生变化？

  1. 情况一：更新 shard-configs 文件；仅最后加载的配置发生变化，才会更新 Spring Boot 中的配置

如 mysql-common.yml 和 redis-common.yml 都存在 commonConnectTimeout 属性，由于 bootstrap.properties 中的配置，使得 commonConnectTimeout 属性最后使用了 redis-common.yml 中的 commonConnectTimeout 配置。 并且只有 redis-common.yml 中的 commonConnectTimeout 属性发生更新，才会修改 Spring Boot 中的 commonConnectTimeout 属性。 mysql-common.yml 中的 commonConnectTimeout 更新是没有作用的

  1. 情况二：更新除了 shard-configs 之外的配置，根据优先级排列

# 4.3 不同服务的通用配置 extension-configs

骚操作：

如果通用配置文件和环境配置文件都配置了某个属性，更新通用配置文件，最终得到的数据是哪个配置文件中的？

答：会根据配置文件的优先级确定。即使更新了通用配置，若在比通用配置更高优先级的文件中也存在该属性，也不会使用通用配置文件更新的配置

extension-configs和shard-configs存在相同配置，最终会以哪个文件中的值为准？

答：以 extension-configs 为准

# 5.配置文件优先级

若应用名为 mall-orders，越向后越晚加载，优先级越大，相同配置按照后面的为准：

本地配置 --> shard-configs --> extension-configs --> mall-orders --> mall-orders.yml --> mall-orders-${profile}.yml

  • 其中 shard-configs 和 extension-configs 中多个配置的顺序与配置顺序有关，索引越大就会后加载，优先级就越大
