# 什么是锁？

场景描述 锁在JAVA中是一个非常重要的概念，尤其是在当今的互联网时代，高并发的场景下，更是离不开锁。那么锁到底是什么呢？在计算机科学中，锁(lock)或互斥(m 同步机制，用于在有许多执行线程的环境中强制对资源的访问限制。锁旨在强制实施互斥排他、并发控制策略。咱们举一个生活中的例子：大家都去过超市买东 身带了包呢，要放到储物柜里。咱们把这个例子再极端一下，假如柜子只有一个，现在同时来了3个人A，B，C，都要往这个柜子里放东西。这个场景就构造了一个多线程并发的场景，使用多线程自然离不开锁。如下图所示：

A，B，C都要往柜子里放东西，可是柜子只能放一件东西，那怎么办呢？这个时候呢就引出了锁的概念，3个人中谁抢到了柜子的锁，谁就可以使用这个柜子，其他的人等待。比如：C抢到了锁，C可以打用这个柜子。A和B只能等待，等C使用完了，释放锁以后，A和B再争抢锁，谁抢到了，再继续使用柜子。

# 基于锁解决商品超卖

# 超卖现象一：一个库存发生了多个订单

  • 系统中库存只有一个，但是产生了两笔订单
  • 卖家在商品发货时，发现只有一件商品，但是有两笔订单

超卖原因：商品只剩一件，有两个人A、B同时看到商品，下单购买，因为扣减库存的动作是在程序中计算的剩余库存，然后更新数据库中的库存，在并发，同时更新数据库库存，最后将数据库的库存修改为0，但是产生了两笔订单。

超卖的影响：超卖导致商家没有商品发货，或发货时间延长，影响用户和商家体验，可能导致买卖双方发生纠纷。

如何解决：

  • 扣减库存不再程序中执行，而是通过数据库（数据库的库存递减）
  • 在数据库update语句扣减库存，通过update行锁解决并发（update语句在执行时，会自动开启行锁）
update 
	product_item 
set 
	item_count = item_count - 1,
	updated_by = 'xxx',
	updated_time = 'xxxx-xx-xx xx:xx:xx'
where 
	product_item_id = 'xxxx'

# 超卖现象二：负库存

基于超卖现象一的解决之后，并发下仍然会产生多个订单，但库存也会相应的减少，不过库存之后1件，并发生5个订单就会扣减5个库存，最后导致库存数为-4，即负库存

产生原因：并发下校检库存，造成库存充足的假象，更新库存导致库存继续减少

如何解决：

  • 方法一：更新完库存后，再校验一次。若库存数为负数则抛出异常
  • 方法二：校检库存、扣减库存统一加锁，使两者成为原子性操作。
    • 并发下，仅有获得锁的线程才能校检、扣减库存
    • 扣减库存结束后，释放锁
    • 确保库存不会扣成负数
  • 方法三：基于 synchronized 锁解决超卖问题

下面介绍如何实现锁

spring 事务提交的时候synchronized失效的问题

问题：在spring中使用@Transactional注解和synchronized同时使用会出现失效问题：

场景：库存为1，10个线程并行创建订单，会导致订单创建两个，库存为-1.

原因：是因为被synchronized修饰的方法执行完成之后，事务还未完成提交，此时被下一个线程就开始执行，导致查询的库存数仍然为1，则会继续执行订单创建方法。

解决：手动使用事务，不使用@Transactional注解

@Service 
public class OrderServiceImpl implements OrderService {
 	@Autowire
    private PlatformTransactionManager platformTransactionManager;

    @Autowire
    private TransactionDefinition transactionDefinition;

    public synchronized void createOrder() {
        // 获取事务
        TransctionStatus transctionStatus = platformTransactionManager.getTransaction(transactionDefinition);
        try {
            // order created busiesee ...
            // ......

            // 提交事务
            platformTransactionManager.commit(transctionStatus);
        }
        catch(Excetion e) {
            // 事务回滚
            platformTransactionManager.rollback(transctionStatus);
            throw new Exception('exception message ~ ')
        }
    }
    
}

# 类锁与对象锁的区别

  • 对象锁：指定某个变量的对象

    • 类声明后，我们可以 new 出来很多的实例对象。这时候，每个实例在 JVM 中都有自己的引用地址和堆内存空间，这时候，我们就认为这些实例都是独立的个体，很显然，在实例上加的锁和其他的实例就没有关系，互不影响了。

  • 类锁

    • 类锁是加载类上的，而类信息是存在 JVM 方法区的，并且整个 JVM 只有一份，方法区又是所有线程共享的，所以类锁是所有线程共享的。

# ReentrantLock

ReentrantLock是Jdk1.5之后在JUC提供的可重入锁

@Service 
public class OrderServiceImpl implements OrderService {
    
    @Autowire
    private PlatformTransactionManager platformTransactionManager;

    @Autowire
    private TransactionDefinition transactionDefinition;
	
    private Lock lock = new ReentrantLock();
    
    public void createOrder()
        // 获取锁
        lock.lock();
     	// 获取事务
        TransctionStatus transctionStatus = platformTransactionManager.getTransaction(transactionDefinition);
        try {
            // order created busiesee ...
            // ......

            // 提交事务
            platformTransactionManager.commit(transctionStatus);
        }
        catch(Excetion e) {
            // 事务回滚
            platformTransactionManager.rollback(transctionStatus);
            throw new Exception('exception message ~ ')
        }
    	finally {
            // 释放锁
    		lock.unlock();
        }
	}
}

# Java单体应用锁的局限性

上面讲到的锁都是有JDK官方的解决方案，也就是说这些锁只能在一个JVM进程内有效，我们把这种锁叫做单体应用锁。但是，在分布式集群下，单体应用有多个副本，单体应用锁就无法满足使用条件

互联网系统架构的演进：

在互联网系统发展之初，系统比较简单，消耗资源小，用户访问量也比较少，我们只部署一个Tomcat应用就可以满足需求。系统架构图： 网关 ——> Tomcat ——> DB

一个Tomcat可以看作是一个JVM进程，当大量的请求并发到达系统时，所有的请求都落在这唯一的一个Tomcat上，如果某些请求方法是需要加锁的，比如：秒杀 是可以满足需求的，这和我们前面章节所讲的内容是一样的。但是随着访问量的增加，导致一个Tomcat难以支撑，这时我们就要集群部署Tomcat，使用多个Tom 整个系统。系统架构图如下：

# 基于数据库的分布式锁

由于Java的synchronized和ReentrantLock只能在单体中使用，面对分布式集群环境下就会出现失效问题，因此需要使用分布式锁。

# 乐观锁与悲观锁介绍

在数据库当中。当时的应用场景主要是修改数据库的时候，更新数据这个场景也是使用锁的非常主要的场景之一。

更新数据的主要流程如下：

  1. 检索出要更新的数据，供操作人员查看；
  2. 操作人员更改需要修改的数值；
  3. 点击保存，更新数据；

这个流程看似简单，但是我们用多线程的思维去考虑，这也应该算是一种互联网思维吧，就会发现其中隐藏着问题。我们具体看一下，

  1. A检索出数据；
  2. B检索出数据；
  3. B修改了数据；
  4. A修改数据，系统会修改成功吗？

当然啦，A修改成功与否，要看程序怎么写。咱们抛开程序，从常理考虑，A保存数据的时候，系统给出提示，说“您修改的数据已被其他人修改过，请重新查询确认 程序中怎么实现？

  1. 在检索数据，将数据的版本号（version）或者最后更新时间一并检索出来；

  2. 操作员更改数据以后，点击保存，在数据库执行update操作；

  3. 执行update操作时，用步骤1检索出的版本号或者最后更新时间与数据库中的记录作比较；

  4. 如果版本号或最后更新时间一致，则可以更新；

  5. 如果不一致，就要给出上面的提示；

上述的流程就是乐观锁的实现方式。在JAVA中乐观锁并没有确定的方法，或者关键字，它只是一个处理的流程、策略

乐观锁：它是假设一个线程在取数据的时候不会被其他线程更改数据，就像上面的例子那样，但是在更新数据的时候会校验数据有没有被修改过。它是一种比较机制，简称CAS（Compare And Swap）机制。一旦检测到有冲突产生，也就是上面说到的版本号或者最后更新时间不一致，它就会进行重试，直到没有冲突为止

# 悲观锁

在数据查询中通过 select ... for update 访问同一条数据，会锁定数据，其他线程只能等待

在查询库存时，将查询的数据加锁，其他的数据库连接则就无法访问此条数据，直到这条数据的锁释放之后才能被访问

注意：在使用的时候需要将事务开启，在spring中直接使用注解@Transactional(rollbackFor = Exception.class)

select 
	product_item_id,
	item_count
from 
	product_item
where 
	product_item_id = '1'
for update

# 基于Redis实现分布式锁

# 原理

Redis的setnx命令

获取锁的命令：

SET resource_name my_random_value NX PX 30000
1
  • resource_name：key值，资源名称，可根据不同业务区分不同的锁
  • my_random_value：随机值，每个线程的随机值都不同，用与释放锁时的校检
  • NX：key不存在时设置成功，key存在则设置不成功 。NX是原子操作
  • PX：自动失效时间，出现异常情况，锁一段时间后可以过期失效

释放锁：

  • 校检之前设置的随机数，若相同则释放

  • 使用delete命令，删除redis存的值

  • 释放锁采用LUA脚本

    if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else 
    return 0
end

# 实现

# 基于spring-boot-data-redis

@Component
public class RedisOperation {
    

	private final StringRedisTemplate stringRedisTemplate;

    private final RedisTemplate redisTemplate;


    @Autowired
    public RedisUtils(StringRedisTemplate stringRedisTemplate, RedisTemplate redisTemplate) {
        this.stringRedisTemplate = stringRedisTemplate;
        this.redisTemplate = redisTemplate;
    }

	/**
     * 获取锁
     */
    public Boolean getLock(String key, String value) {
        RedisCallback<Boolean> redisCallback = connection -> {
            RedisStringCommands.SetOption setOption = RedisStringCommands.SetOption.ifAbsent();
            Expiration expiration = Expiration.seconds(30);
            byte[] redisByteKey = redisTemplate.getKeySerializer().serialize(key);
            byte[] redisByteValue = redisTemplate.getValueSerializer().serialize(value);
            Boolean result = connection.set(redisByteKey, redisByteValue, expiration, setOption);
            return result;
        };
        Boolean lock = (Boolean) redisTemplate.execute(redisCallback);
        return lock;
    }

    /**
     * 释放锁
     * @param key
     * @param value
     * @return
     */
    public Boolean unlock(String key, String value) {
        final String luaScript = "if redis.call(\"get\", KEYS[1]) == ARGV[1] then\n" +
                "    return redis.call('del', KEYS[1])\n" +
                "else \n" +
                "    return 0\n" +
                "end";
        // RedisScript<Boolean> redisScript = new DefaultRedisScript(luaScript, Boolean.TYPE);
        RedisScript<Boolean> redisScript = DefaultRedisScript.of(luaScript, Boolean.TYPE);
        Boolean result = (Boolean) redisTemplate.execute(redisScript, Collections.singletonList(key), value);
        return result;
    }
    
}

# 基于Zookeeper实现分布式锁

  • Zookeeper是一个分布式一致性协调服务，它是 Apache Hadoop 的一个子项目，它主要是用来解决分布式应用中经常遇到的一些数据管理问题，如统一命名服务、状态同步服务、集群管理、分布式应用配置项的管理等。
  • Zookeeper的数据监听通知机制：客户端注册监听它关心的znode，当znode发生变化（数据改变、删除、子目录增加/删除节点）时，Zookeeper会通知所有的客户端。
  • 我们可以将分布式服务的配置保存在Zookeeper上的某个节点，若需要修改配置文件，其他分布式应用服务都会同步配置信息，达到所有的分布式服务动态更新配置文件的目的。

# Zookeeper实现分布式锁的原理

Zookeeper的观察器：

  • Zookeeper节点数据发生变化后，会将变化发送给客户端
  • 客户端发生变化，会调用观察器的三个方法:
    • getData()
    • getChildren()
    • exists()
  • 观察器只能调用一次，再监控需要重新设置，但是使用Zookeeper的curator客户端可以实现一直监控

实现原理：

  • 利用Zookeeper的瞬时有序节点的特性
    1. 多线程并发创建节点时，会得到有序的序列号
    2. 序号最小的线程获得锁
    3. 其他的线程则监听自己序号的前一个序号
    4. 前一个线程执行完成，删除自己序号的节点
    5. 下一个序号的线程获得通知，继续执行
    6. 以此类推

具体原理见下面读写锁的第二种实现

# 独占锁的实现

独占锁又称排它锁，从字面意思上很容易理解他们的用途。即如果某个操作 O1 对访问资源 R1 的过程加锁，在操作 O1 结束对资源 R1 访问前，其他操作不允许访问资源 R1。以上算是对独占锁的简单定义了

那么这段定义在 Zookeeper 的“类 Unix/Linux 文件系统”的结构中是怎样实现的呢？在锁答案前，我们先看下面的图

独占锁的 Zookeeper 节点结构:

		|
		|——————>lock

对于独占锁，我们可以将资源 R1 看做是 lock 节点，操作 O1 访问资源 R1 看做创建 lock 节点，释放资源 R1 看做删除 lock 节点。这样我们就将独占锁的定义对应于具体的 Zookeeper 节点结构，通过创建 lock 节点获取锁，删除节点释放锁。详细的过程如下：

  1. 多个客户端竞争创建 lock 临时节点
  2. 其中某个客户端成功创建 lock 节点，其他客户端对 lock 节点设置 watcher
  3. 持有锁的客户端删除 lock 节点或该客户端崩溃，由 Zookeeper 删除 lock 节点
  4. 其他客户端获得 lock 节点被删除的通知
  5. 重复上述4个步骤，直至无客户端在等待获取锁了

# 读写锁的实现

读写锁包含一个读锁和写锁，操作 O1 对资源 R1 加读锁，且获得了锁，其他操作可同时对资源 R1 设置读锁，进行共享读操作。如果操作 O1 对资源 R1 加写锁，且获得了锁，其他操作再对资源 R1 设置不同类型的锁都会被阻塞。总结来说，读锁具有共享性，而写锁具有排他性。那么在 Zookeeper 中，我们可以用怎样的节点结构实现上面的操作呢？

|——share_lock

		|————>host1-W-00000000001
		|————>host2-R-00000000002
		|————>host3-W-00000000003

在 Zookeeper 中，由于读写锁和独占锁的节点结构不同，读写锁的客户端不用再去竞争创建 lock 节点。所以在一开始，所有的客户端都会创建自己的锁节点。如果不出意外，所有的锁节点都能被创建成功，此时锁节点结构如图3所示。之后，客户端从 Zookeeper 端获取 /share_lock 下所有的子节点，并判断自己能否获取锁。如果客户端创建的是读锁节点，获取锁的条件（满足其中一个即可）如下：

  1. 自己创建的节点序号排在所有其他子节点前面
  2. 自己创建的节点前面无写锁节点

如果客户端创建的是写锁节点，由于写锁具有排他性。所以获取锁的条件要简单一些，只需确定自己创建的锁节点是否排在其他子节点前面即可。

不同于独占锁，读写锁的实现稍微复杂一下。读写锁有两种实现方式，各有异同，接下来就来说说这两种实现方式。

# 读写锁的第一种实现

第一种实现是对 /share_lock 节点设置 watcher，当 /share_lock 下的子节点被删除时，未获取锁的客户端收到 /share_lock 子节点变动的通知。在收到通知后，客户端重新判断自己创建的子节点是否可以获取锁，如果失败，再次等待通知。详细流程如下：

  1. 所有客户端创建自己的锁节点
  2. 从 Zookeeper 端获取 /share_lock 下所有的子节点，并对 /share_lock 节点设置 watcher
  3. 判断自己创建的锁节点是否可以获取锁，如果可以，持有锁。否则继续等待
  4. 持有锁的客户端删除自己的锁节点，其他客户端收到 /share_lock 子节点变动的通知
  5. 重复步骤2、3、4，直至无客户端在等待获取锁了

上面获取读写锁流程并不复杂，存在性能问题。以图3所示锁节点结构为例，第一个锁节点 host1-W-0000000001 被移除后，Zookeeper 会将 /share_lock 子节点变动的通知分发给所有的客户端。但实际上，该子节点变动通知除了能影响 host2-R-0000000002 节点对应的客户端外，分发给其他客户端则是在做无用功，因为其他客户端即使获取了通知也无法获取锁。所以这里需要做一些优化，优化措施是让客户端只在自己关心的节点被删除时，再去获取锁。

# 读写锁的第二种实现

在了解读写锁第一种实现的弊端后，我们针对这一实现进行优化。这里客户端不再对 /share_lock 节点进行监视，而只对自己关心的节点进行监视。还是以图3的锁节点结构进行举例说明，host2-R-0000000002 对应的客户端 C2 只需监视 host1-W-0000000001 节点是否被删除即可。而 host3-W-0000000003 对应的客户端 C3 只需监视 host2-R-0000000002 节点是否被删除即可，只有 host2-R-0000000002 节点被删除，客户端 C3 才能获取锁。而 host1-W-0000000001 节点被删除时，产生的通知对于客户端 C3 来说是无用的，即使客户端 C3 响应了通知也没法获取锁。这里总结一下，不同客户端关心的锁节点是不同的。如果客户端创建的是读锁节点，那么客户端只需找出比读锁节点序号小的最后一个的写锁节点，并设置 watcher 即可。而如果是写锁节点，则更简单，客户端仅需对该节点的上一个节点设置 watcher 即可。详细的流程如下：

  1. 所有客户端创建自己的锁节点
  2. 从 Zookeeper 端获取 /share_lock 下所有的子节点
  3. 判断自己创建的锁节点是否可以获取锁，如果可以，持有锁。否则对自己关心的锁节点设置 watcher
  4. 持有锁的客户端删除自己的锁节点，某个客户端收到该节点被删除的通知，并获取锁
  5. 重复步骤4，直至无客户端在等待获取锁了

# 编写通用类

@Slf4j
public class ZkLock implements AutoCloseable, Watcher {

    private ZooKeeper zooKeeper;

    private String zNode;

    public ZkLock() throws IOException {
        this.zooKeeper = new ZooKeeper("localhost:2181", 5000, this);
    }

    /**
     * 获取锁
     * @param businessCode 节点name
     * @return
     */
    public boolean getLock(String businessCode) throws InterruptedException, KeeperException {
        StringBuilder stringBuilder = new StringBuilder();
        stringBuilder.append("/").append(businessCode);
        String parentNode = stringBuilder.toString();
        Stat existsNode = zooKeeper.exists(parentNode, false);

        if (existsNode == null) {
            // 创建业务父节点 权限-完全开放 模式-持久节点
            log.info("=====>创建父节点：{}", parentNode);
            zooKeeper.create(parentNode, businessCode.getBytes(), ZooDefs.Ids.OPEN_ACL_UNSAFE,
                    CreateMode.PERSISTENT);
        }

        // 创建瞬时有序节点 (子节点-锁) /my_order/my_order_0000000002
        stringBuilder.append("/").append(businessCode).append("_");
        this.zNode = zooKeeper.create(stringBuilder.toString(), businessCode.getBytes(), ZooDefs.Ids.OPEN_ACL_UNSAFE,
                CreateMode.EPHEMERAL_SEQUENTIAL);
        log.info("=====> 创建瞬时节点：{}", this.zNode);

        // 获取子节点
        List<String> childrenNodes = zooKeeper.getChildren(parentNode, false);
        Collections.sort(childrenNodes);

        // 获得锁成功
        String firstNode = childrenNodes.get(0);
        if (this.zNode.endsWith(firstNode)) {
            log.info("=====>获取锁成功：{}", this.zNode);
            return true;
        }
        // 等待中，监听自己的前一个节点
        String beforeNode = firstNode;
        for (String node : childrenNodes) {
            // 是否是自己的上一个节点
            if (zNode.endsWith(beforeNode)) {
                String watchNode = parentNode + "/" + beforeNode;
                log.info("=====> 等待中, 监听节点：{}", watchNode);
                zooKeeper.exists(watchNode, true);
                break;
            }
            beforeNode = node;
        }
        // 等待节点通知
        synchronized (this) {
            wait();
        }

        return true;
    }

    /**
     * 释放锁
     * @throws Exception
     */
    @Override
    public void close() throws Exception {
        // version=-1 所有版本号
        log.info("=====> 释放锁：{}", this.zNode);
        zooKeeper.delete(this.zNode, -1);
    }

    @Override
    public void process(WatchedEvent watchedEvent) {
        // 等待通知，唤醒等待进程
        if (watchedEvent.getType() == Event.EventType.NodeDeleted) {
            synchronized (this) {
                log.info("=====> 节点被删除，结束等待，开始获得锁");
                notify();
            }
        }
    }

}

            }
        }

# 利用分布式锁解决定时任务集群下任务重复执行问题

若在使用Elastic-Job这样的分布式定时任务时，因为ES-Job框架已经实现了分布式环境下分片执行任务，因此不会重复执行某一任务，即不会出现重复执行问题。

若使用的是Spring Task来实现定时任务，则分布式下还有多个相同的服务，为了避免重复执行，我们需要使用分布式锁。

原理：在定时任务执行之前获取分布式锁，哪个服务获取到锁就执行，没获取到锁的不再执行

这里使用spring-boot-data-redis来实现，有需要可自行替换其他Redis客户端

# 封装锁

@Slf4j
public class RedisLock implements AutoCloseable {

    private RedisTemplate redisTemplate;
    private String key;
    private String value;
    // 单位：秒(s)
    private int expireTime = 30;

    public RedisLock(RedisTemplate redisTemplate,String key,int expireTime){
        this.redisTemplate = redisTemplate;
        this.key = key;
        this.expireTime=expireTime;
        this.value = UUID.randomUUID().toString();
    }
    
      public RedisLock(RedisTemplate redisTemplate,String key){
        this.redisTemplate = redisTemplate;
        this.key = key;
        this.value = UUID.randomUUID().toString();
    }

    /**
     * 获取分布式锁
     * @return
     */
    public boolean getLock(){
        RedisCallback<Boolean> redisCallback = connection -> {
            //设置NX
            RedisStringCommands.SetOption setOption = RedisStringCommands.SetOption.ifAbsent();
            //设置过期时间
            Expiration expiration = Expiration.seconds(expireTime);
            //序列化key
            byte[] redisKey = redisTemplate.getKeySerializer().serialize(key);
            //序列化value
            byte[] redisValue = redisTemplate.getValueSerializer().serialize(value);
            //执行setnx操作
            Boolean result = connection.set(redisKey, redisValue, expiration, setOption);
            return result;
        };

        //获取分布式锁
        Boolean lock = (Boolean)redisTemplate.execute(redisCallback);
        return lock;
    }
    
    /**
     * 释放分布式锁
     * @return
     */
    public boolean unLock() {
        String script = "if redis.call(\"get\",KEYS[1]) == ARGV[1] then\n" +
                "    return redis.call(\"del\",KEYS[1])\n" +
                "else\n" +
                "    return 0\n" +
                "end";
        RedisScript<Boolean> redisScript = RedisScript.of(script,Boolean.class);
        List<String> keys = Arrays.asList(key);

        Boolean result = (Boolean)redisTemplate.execute(redisScript, keys, value);
        log.info("释放锁的结果："+result);
        return result;
    }

    /**
     * 自动关闭-自动释放锁
     * 实现 jdk1.7 的AutoCloseable interface
     * 使用时需要在try() 中声明
     * @throws Exception [description]
     */
    @Override
    public void close() throws Exception {
        unLock();
    }
}

# 分布式锁的对比

实现方式 优点 缺点 说明
数据库 实现简单 对数据库压力大 使用锁的数据库和业务数据库最好分开
Redis 易于理解 自己实现，不支持阻塞
Zookeeper 支持阻塞 需理解Zookeeper、程序复杂
Curator 提供锁的方法 依赖Zookeeper，强一致性
Redission 提供锁的方法，可阻塞
  • 不推荐自己编写的分布式锁
  • 推荐使用Rdisson和Curator实现的分布式锁
