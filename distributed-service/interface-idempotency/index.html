接口幂等性

# 接口幂等性

# 什么是接口幂等性

幂等性：f(f(x))=f(x)，幂等元素运行多次，还等于它原来的运算结果 在后端服务中，一个接口运行多次，与运行一次的效果是一致的。如同一时间下订单提交多次，最后生成的订单数据只有一条

# 什么情况下需要幂等性？

重复提交、接口重试、前端操作抖动（操作一次无响应再操作一次）等

典型场景：

  • 防止订单重复提交：用户多次点击提交订单，后台最后应该只会生成一个订单
  • 支付时，由于网络问题重发，应该只扣一次钱
  • 表单录入页，防止重复提交
  • 微服务接口，客户端重试时

:::WARNING 并不是所有的接口都要求幂等性，要根据业务而定 :::

# 如何保证接口幂等性？

幂等性的核心思想：通过唯一的业务单号保证幂等性

  • 非并发下，查询业务单号有没有操作过，没有则执行操作
  • 高并发下，整个操作过程加锁

数据库操作对幂等性的影响

  • SELECT 不会对数据库的业务数据有影响，因此查询的接口不需要幂等性
  • DELETE 由于对一条数据删除一次和删除多次的结果都是一样的
  • UPDATE 更新操作传入数据版本号，通过乐观锁实现幂等性
  • INSERT 此时没有唯一业务单号，使用Token保证幂等性
  • 混合操作 找到操作的唯一单号，有则可使用分布式锁，没有则通过Token保证幂等性

# 幂等性的实操

# UPDATE操作的幂等性

# 存在唯一的业务号时

# 多次修改结果一直

若业务是根据**唯一业务号（如主键id）**修改数据，并且执行多次结果一致，则不需要做幂等性处理

如下，不管修改名字多少次，最后的结果还是张三

UPDATE user
	SET username = '张三'
WHERE
	id = #{id}
;

# 多次修改结果不一致

若业务虽然使根据唯一业务号修改的数据，但是执行多次对数据还是会有影响的

如，若我们的产品经理说：限制名字的修改次数，每年不超过三次

这时候，我们就需要每年记录用户修改了多少次，到年底的时候，再将名字的修改次数归零

我们把代码改成了下面的语句

UPDATE user
	SET username = '张三', 
	update_name_count = update_count + 1
WHERE
	id = #{id}
;

当然这就属于一个bug了，我们来优化一下上面的代码

UPDATE 
	SET version = version+1,xxx=#(xxx) 
WHERE
	id = #{id}
AND 
	version = #{version}
;

用户查询要修改的数据，系统将数据返回页面，前端则先记录下所有的数据，当然这包括版本号（version）这个字段

当用户修改数据，点击提交，将版本号（version）一同提交给后台

后台使用版本号作为更新条件，这样一来，就算用户疯狂点击，最后成功执行的只会是用户的第一次的请求

TIPS：因为Mysql中UPDATE语句在执行 时会自动加行锁，只有该UPDATE语句执行完成之后，其他的线程才能修改UPDATE所影响的数据

以上使用乐观锁与update行锁，即可保证接口的幂等性

# 不存在唯一的业务号时

若更新操作没有唯一的业务号，可以使用Token机制

用户在进入修改页面时，后端生成一个token返回到前端，并把该token存储到redis

当用户第一次的请求修改数据，我们为该token加锁，然后执行修改，修改结束则释放锁，并删除token

后面的请求过来从redis中验证token的锁都会失败，获取锁失败请求直接返回即可

token的有效期可以设置为几分钟，让锁自动过期即可

不一定非要使用redis来实现锁，也可使用其他方式来实现锁，如Zookeeper

# INSERT操作的幂等性

# 有唯一业务号

如用户注册时点击多次（邮箱、手机号）

有唯一业务号的INSERT操作，例如：秒杀下：商品id+用户id

通过分布式锁，保证接口幂等性

业务执行完成后，不进行锁的释放，让其过期之后自动释放（只能购买一件可以这么做）

# 没有唯一的业务单号

没有唯一的业务单号时

如：添加收货地址点击多次

使用token机制来保证幂等性：

  • 进入注册页面时，后台生成Token（token保存到redis中使用set存储）返回到前端
  • 用户在点击提交，前端发送请求时携带token
  • 后端根据token加分布式锁（key=user:register:[token]），只有获得了分布式锁的请求才能继续执行
  • 获得分布式锁的请求执行成功之后，不释放锁，等待过期自动回收

# 混合操作

多个INSERT和UPDATE可以使用token机制，使用方式同上

# Token机制

进入表单时获取token

// business 业务名/接口名（可以使接口名或接口地址）最好保证唯一
public String getToken(String business, String key) {
    String token = UUID.randomUUID().toString();
    // 不一定要使用string,hash也可以，根据实际业务来使用
    redisUtils.setString("token:" + business + ":" + key, token, 60, TimeUnit.SECONDS)
	return token;
}

public String getAddUserAddressToken(HttpSession session) {
    return this.getToken("addUserAdress", session.getId());

# 总结

  1. 最终的目的就是防止同一用户并发请求同一个接口出现数据不一致问题。

  2. 解决方案就是在这个并发场景下，我们需要找到能唯一确定该次请求的id或序列号。若找不到唯一的序列号就有后端来生成。

  3. 得到这个唯一的序列号，就可以来加锁，达到只能让用户的多个并发请求只有一条成功运行业务逻辑。
