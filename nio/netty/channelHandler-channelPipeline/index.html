07.ChannelHandler 和 ChannelPipeline

本文章是翻译了《Netty in Action》中的内容。

# 一、ChannelHandler 家族

# 1.Channel 的生命周期

Channel 有个简单但强大的状态模型，与 ChannelInboundHandler API 密切相关。下面表格是 Channel 的四个状态：

状态 描述
ChannelUnregistered Channel 创建但未注册到一个 EventLoop.
ChannelRegistered Channel 注册到一个 EventLoop.
ChannelActive Channel 的活动的(连接到了它的 remote peer（远程对等方），现在可以接收和发送数据了
ChannelInactive Channel 没有连接到 remote peer（远程对等方）

通道的正常生命周期如下图所示。当这些状态发生变化时，将生成相应的事件。这些被转发到 ChannelPipeline 中的 ChannelHandlers，然后 ChannelPipeline 可以对它们进行操作。

# 2.ChannelHandler 的生命周期

下表列出了接口 ChannelHandler 定义的生命周期操作，在 ChannelHandler 被添加到 ChannelPipeline 或从 ChannelPipeline 中移除之后调用。每个方法接受一个 ChannelHandlerContext 参数。

类型 描述
handlerAdded 当 ChannelHandler 添加到 ChannelPipeline 调用
handlerRemoved 当 ChannelHandler 从 ChannelPipeline 移除时调用
exceptionCaught 当 ChannelPipeline 执行发生错误时调用

Netty 定义了 ChannelHandler 两个子接口，如下：

  • ChannelInboundHandler——处理所有类型的入站数据和状态更改
  • ChannelOutboundHandler——处理出站数据并允许拦截所有操作

# 3.ChannelInboundHandler 接口

下表列出了接口 ChannelInboundHandler 的生命周期方法。当接收到数据或关联通道的状态发生变化时，将调用这些方法。

方法 描述
channelRegistered 当 Channel 被注册到它的 EventLoop，并且能够处理 I/O 时调用。
channelUnregistered 当 Channel 从 EventLoop 注销并且不能处理任何 I/O 时调用。
channelActive 当 Channel 处于活动状态时调用；通道已连接/绑定并就绪。
channelInactive 当 Channel 离开活动状态并且不再连接到其远程对等端时调用。
channelRead 当从 Channel 读取数据时调用。
channelReadComplete 当 Channel 上的读取操作完成时调用。
userEventTriggered 当用户调用 Channel.fireusereventtrigger(…) 通过 ChannelPipeline 传递 pojo （实体模型）时调用。这可以用来通过 ChannelPipeline 传递用户特定的事件，从而允许处理这些事件。
channelWritabilityChanged 当 Channel 的可写状态改变时调用。用户可以确保写操作不会太快（有发生 OutOfMemoryError 的风险），或者在通道再次可写时恢复写操作。Channel.iswritable() 可用于检测通道的实际可写性。可写性的阈值可以通过Channel.config().setwritehighwatermark() 和 Channel.config().setwritelowwatermark() 设置。

当 ChannelInboundHandler 重写 channelRead() 方法时，它负责显式释放与池化 ByteBuf 实例相关的内存。Netty 为此提供了一个实用程序方法：ReferenceCountUtil.release()。如下所示：

Netty 会使用警告级别的日志消息记录未释放的资源，这使得在代码中查找违规实例变得相当简单。但是以这种方式管理资源非常麻烦。一个更简单的替代方法是使用 SimpleChannelInboundHandler。

SimpleChannelInboundHandler 中的 channelRead() 方法中会自动调用 ReferenceCountUtil.release() 释放资源，如下：

public abstract class SimpleChannelInboundHandler<I> extends ChannelInboundHandlerAdapter {
    
    // ...

    @Override
    public void channelRead(ChannelHandlerContext ctx, Object msg) throws Exception {
        boolean release = true;
        try {
            if (acceptInboundMessage(msg)) {
                @SuppressWarnings("unchecked")
                I imsg = (I) msg;
                channelRead0(ctx, imsg);
            } else {
                release = false;
                ctx.fireChannelRead(msg);
            }
        } finally {
            // 自动释放资源
            if (autoRelease && release) {
                ReferenceCountUtil.release(msg);
            }
        }
    }
 	
    // ...
}

# 4.ChannelOutboundHandler 接口

出站操作和数据由 ChannelOutboundHandler 处理。它的方法由 Channel、ChannelPipeline 和ChannelHandlerContext 调用。

ChannelOutboundHandler 有一个强大功能是按需延迟操作或事件，这允许使用复杂的方法来处理请求。例如，如果挂起对远程对等体的写操作，则可以推迟刷新操作，稍后再恢复。

下表列出了 ChannelOutboundHandler 中定义的所有方法（不包括从 ChannelHandler 继承的方法）。

方法 描述
bind(ChannelHandlerContext, SocketAddress,ChannelPromise) 在请求将 Channel 绑定到本地地址时调用
connect(ChannelHandlerContext,SocketAddress,SocketAddress,ChannelPromise) 在请求将 Channel 连接到远程对等点服务时调用
disconnect(ChannelHandlerContext, ChannelPromise) 在请求将 Channel 与远程对等点的连接断开时调用
close(ChannelHandlerContext,ChannelPromise) 在请求将 Channel 关闭时调用
deregister(ChannelHandlerContext, ChannelPromise) 在请求将 Channel 在其对应的 EventLoop 中注销时调用
read(ChannelHandlerContext) 在请求将 Channel 读取更多数据时调用
flush(ChannelHandlerContext) 在请求将 Channel 排队的数据刷新到远程对等点时调用
write(ChannelHandlerContext,Object, ChannelPromise) 在请求将 Channel 向远程对等端写入数据时调用

上面多数方法都涉及到了 ChannelPromise 参数，关于 ChannelPromise 和 ChannelFuture 的区别如下：

  • ChannelOutboundHandler 中的大多数方法都接受 ChannelPromise 参数，以便在操作完成时得到通知。
  • ChannelPromise 是 ChannelFuture 的子接口，它定义了可写的方法，如 setSuccess() 或 setFailure()，从而使 ChannelFuture 不可变。

# 5.ChannelHandler 适配器

你可以使用 ChannelInboundHandlerAdapter 和 ChannelOutboundHandlerAdapter 类作为我们自定义 ChannelHandlers 的基类。

  • 这些适配器分别提供了 ChannelInboundHandler 和 ChannelOutboundHandler 的基本实现。
  • 它们通过扩展抽象类 ChannelHandlerAdapter 获得它们的公共超接口 ChannelHandler 的方法。

生成的类层次结构如下图所示：

ChannelHandlerAdapter 还提供了 isSharable() 实用方法。如果实现被 @Sharable 修饰，这个方法返回 true，表明它可以被添加到多个 ChannelPipelines。

ChannelInboundHandlerAdapter 和 ChannelOutboundHandlerAdapter 中的方法体（方法实现）中，会调用ChannelHandlerContext 中的等效方法，从而将事件转发到管道中的下一个 ChannelHandler。

如果想要在自己的处理程序中使用这些适配器类，只需扩展它们并覆盖要自定义的方法即可。

# 6.资源管理

# 6.1 资源管理

当我们通过调用 ChannelInboundHandler.channelRead() 或 ChannelOutboundHandler.write() 对数据进行操作时，我们需要确保没有资源泄漏。Netty 使用引用计数器来处理池化的 ByteBufs。因此，在使用完 ByteBuf 后调整引用计数器是很重要的。

为了帮助我们诊断潜在的问题，Netty 提供了 ResourceLeakDetector 类，它将抽样大约 1% 的应用程序缓冲区分配来检查内存泄漏。所涉及的开销非常小。

如果检测到泄漏，将产生类似以下的日志消息：

LEAK: ByteBuf.release() was not called before it's garbage-collected. Enable
advanced leak reporting to find out where the leak occurred. To enable
advanced leak reporting, specify the JVM option
'-Dio.netty.leakDetectionLevel=ADVANCED' or call
ResourceLeakDetector.setLevel().

# 6.2 泄漏检测等级

Netty 目前定义了四个泄漏检测级别，如下：

等级 描述
DISABLED 关闭泄漏检测。只有在经过广泛的测试后才能使用它。
SIMPLE 使用默认的 1% 采样率报告发现的任何泄漏。这是默认级别，适合大多数情况。
ADVANCED 报告发现的泄漏以及访问消息的位置。使用默认采样率。
PARANOID 与 ADVANCED 类似，只是每次访问都是采样的。这对性能有很大的影响，应该只在调试阶段使用。

在启动时，通过以下方式，可以定义泄漏检测的级别：

java -Dio.netty.leakDetectionLevel=ADVANCED
1

这样，我们就能在 ChannelInboundHandler.channelRead(...) 和 ChannelOutboundHandler.write(...) 避免泄漏。

  • 当我们处理 ChannelInboundHandler.channelRead() 操作，并在消费消息时（不是通过 ChannelHandlerContext.fireChannelRead(...) 来传递它到下个 ChannelInboundHandler），要释放它，如下：
@ChannelHandler.Sharable
public class DiscardInboundHandler extends ChannelInboundHandlerAdapter { // 1.继承 ChannelInboundHandlerAdapter

    @Override
    public void channelRead(ChannelHandlerContext ctx, Object msg) {
        ReferenceCountUtil.release(msg); // 2.使用 ReferenceCountUtil.release(...) 来释放资源
    }

}

@ChannelHandler.Sharable

        promise.setSuccess();  // 3.通知 ChannelPromise 数据已经被处理
    }

}

# 6.3 总结

  • 如果消息被消耗或丢弃，并且没有传递给 ChannelPipeline 中的下一个 ChannelOutboundHandler，此时，我们必须调用 ReferenceCountUtil.release()。
  • 如果消息到达实际的传输层，它将在写入或通道关闭时自动释放。

# 二、ChannelPipeline 接口

下图展示了用于入站和出站 ChannelHandler 的 典型 ChannelPipeline 布局：

ChannelPipeline 主要是一系列 ChannelHandler。ChannelPipeline 提供了方法传播事件本身。如果一个入站事件被触发，它将被传递的从 ChannelPipeline 开始到结束。例如上图中出站 I/O 事件将从 ChannelPipeline 右端开始一直处理到左边。

# 修改 ChannelPipeline

ChannelHandler 可以通过添加、删除或替换其他 ChannelHandler 来实时修改 ChannelPipeline 的布局。（它也可以将自己从 ChannelPipeline 中移除。）这是 ChannelHandler 最重要的功能之一，因此我们将仔细研究它是如何实现的。

下表是 ChannelHandler 中，用于修改 ChannelPipeline 的方法：

名称 描述
addFirst addBefore addAfter addLast 添加 ChannelHandler 到 ChannelPipeline.
remove 从 ChannelPipeline 移除 ChannelHandler.
replace 在 ChannelPipeline 替换另外一个 ChannelHandler.

ChannelHandler 的执行和阻塞：

  • 当程序处理网络数据时，每个处理器 ChannelHandler 都会在一个 I/O 线程（EventLoop）中处理传递给它的事件。很重要的一点是不要在这个线程中阻塞，否则会影响整个网络数据的处理效率。
  • 有时候我们可能需要与使用阻塞式编程接口的旧代码进行交互。为了解决这个问题，Netty 提供了一个叫做 DefaultEventExecutorGroup 的实现，可以帮助我们将事件交给其他线程池来处理，从而避免阻塞 I/O 线程。

除了以上操作外，ChannelPipeline 中还有其他一些操作可以通过类型或名称访问 ChannelHandler。如下：

方法 描述
get(…) 按类型或名称返回 ChannelHandler
context(…) 返回绑定到 ChannelHandler 的 ChannelHandlerContext。
names() 返回 ChannelPipeline 中所有 ChannelHandler 的名称。

# 发送事件

ChannelPipeline API 公开了用于调用入站和出站操作的其他方法。下表列出了入站操作，这些操作将ChannelPipeline 中发生的事件通知 ChannelInboundHandler。

方法 描述
fireChannelRegistered 在 ChannelPipeline 中的下一个 ChannelInboundHandler 上调用channelRegistered(ChannelHandlerContext)
fireChannelUnregistered 在 ChannelPipeline 中的下一个 ChannelInboundHandler 上调用channelUnregistered(ChannelHandlerContext)
fireChannelActive 在 ChannelPipeline 中的下一个 ChannelInboundHandler 上调用channelActive(ChannelHandlerContext)
fireChannelInactive 在 ChannelPipeline 中的下一个 ChannelInboundHandler 上调用 channelInactive(ChannelHandlerContext)
fireExceptionCaught 在 ChannelPipeline 中的下一个 ChannelHandler 上调用 exceptionCaught(ChannelHandlerContext, Throwable)
fireUserEventTriggered 在 ChannelPipeline 中的下一个 ChannelInboundHandler上调用 usereventtrigger (ChannelHandlerContext, Object)
fireChannelRead 在 ChannelPipeline 中的下一个 ChannelInboundHandler 上调用 channelRead(ChannelHandlerContext, Object msg)
fireChannelReadComplete 在 ChannelPipeline 的下一个 ChannelStateHandler 上调用 channelReadComplete(ChannelHandlerContext)

在出站端，处理事件将导致在底层套接字上采取某些操作。下表列出了 ChannelPipeline API 的出站操作：

方法 描述
bind 将 Channel 绑定到本地地址。这将在 ChannelPipeline 中的下一个 ChannelOutboundHandler 上调用 bind(ChannelHandlerContext, SocketAddress, ChannelPromise)。
connect 将 Channel 连接到远程地址。这将在 ChannelPipeline 中的下一个 ChannelOutboundHandler 上调用 connect(ChannelHandlerContext, SocketAddress, ChannelPromise)。
disconnect 断开 Channel 连接。这将在 ChannelPipeline 中的下一个 ChannelOutboundHandler 上调用 disconnect(ChannelHandlerContext, ChannelPromise)。
close 关闭 Channel。这将在 ChannelPipeline 中的下一个 ChannelOutboundHandler 上调用 close(ChannelHandlerContext, ChannelPromise)。
deregister 从先前分配的 EventExecutor（EventLoop）中注销 Channel。这将在 ChannelPipeline 中的下一个ChannelOutboundHandler 上调用 deregister(ChannelHandlerContext, ChannelPromise)。
flush 刷新 Channel 中所有挂起的写操作。这将在 ChannelPipeline 中的下一个 ChannelOutboundHandler 上调用 flush(ChannelHandlerContext)。
write 向 Channel 写入消息。这将在 ChannelPipeline 中的下一个 ChannelOutboundHandler 上调用 write(ChannelHandlerContext, Object msg, ChannelPromise)。
注意：这不会将消息写入底层 Socket，而只是将其排队。如果要将其写入 Socket，请调用flush() 或 writeAndFlush()。
writeAndFlush 这是调用 write() 然后调用flush() 的方便方法。
read 请求从通道读取更多数据。这将在 ChannelPipeline 中的下一个 ChannelOutboundHandler 上调用 read(ChannelHandlerContext)。

总结：

  1. ChannelPipeline 保存与通道关联的 ChannelHandler。
  2. 可以通过根据需要添加和删除 ChannelHandler 来动态修改 ChannelPipeline。
  3. ChannelPipeline 有一个丰富的 API，用于调用响应入站和出站事件的操作。

# 三、ChannelHandlerContext 接口

ChannelHandlerContext 表示 ChannelHandler 和 ChannelPipeline 之间的关联，并且在 ChannelHandler 被添加到 ChannelPipeline 时创建。ChannelHandlerContext 的主要功能是管理其关联的 ChannelHandler 与同一 ChannelPipeline 中的其他 ChannelHandler 之间的交互。

ChannelHandlerContext 有许多方法，其中一些方法也存在于 Channel 和 ChannelPipeline 本身，但有一个重要的区别：

  • 如果在 Channel 或 ChannelPipeline 实例上调用这些方法，它们将通过整个管道传播。
  • 在 ChannelHandlerContext 中调用的相同方法将从当前关联的 ChannelHandler 开始，并仅传播到管道中能够处理事件的下一个 ChannelHandler。

下表中总结了 ChannelHandlerContext API：

方法 描述
bind 绑定到给定的 SocketAddress 并返回一个 ChannelFuture。
channel 返回绑定到此实例的 Channel
close 关闭 Channel 并返回 ChannelFuture
connect 连接到给定的 SocketAddress 并返回一个 ChannelFuture
deregister 从先前分配的 EventExecutor 中注销，并返回一个 ChannelFuture
disconnect 断开与远程对等端的连接并返回 ChannelFuture
executor 返回调度事件的 EventExecutor
fireChannelActive 触发对下一个 ChannelInboundHandler 的 channelActive() （已连接）的调用
fireChannelInactive 触发对下一个 ChannelInboundHandler 的 channelInactive() （已关闭）的调用
fireChannelRead 触发对下一个 ChannelInboundHandler 的 channelRead() （收到的消息）的调用
fireChannelReadComplete 触发 channelWritabilityChanged 事件到下一个 ChannelInboundHandler
handler 返回绑定到此实例的 ChannelHandler
isRemoved 如果关联的 ChannelHandler 从 ChannelPipeline 中移除，则返回 true
name 返回此实例的唯一名称
pipeline 返回关联的 ChannelPipeline
read 从 Channel 读取数据到第一个入站缓冲区；如果成功触发 channelRead 事件，并通知处理程序 channelReadComplete
write 通过这个实例通过管道写入一条消息

当使用ChannelHandlerContext API时，请记住以下几点：

  • 与 ChannelHandler 关联的 ChannelHandlerContext 永远不会改变，所以缓存对它的引用是安全的。
  • 正如我们在本节开头所解释的那样，ChannelHandlerContext 方法涉及的事件流比其他类上可用的同名方法要短。应该尽可能地利用这一点，以提供最大的性能。

# 2.ChannelHandler 和 ChannelHandlerContext 的高级用法

我们可以通过调用 ChannelHandlerContext 的 pipeline() 方法来获取对封闭的 ChannelPipeline 的引用。这允许对管道的 ChannelHandlers 进行运行时操作，可以利用它来实现复杂的设计。例如，您可以向管道添加 ChannelHandler 以支持动态协议更改。

其他高级用途可以通过缓存对 ChannelHandlerContext 的引用来支持，以供以后使用，这可能发生在任何 ChannelHandler 方法之外，甚至可能来自不同的线程。

缓存 ChannelHandlerContext 的方式如下：

public class WriteHandler extends ChannelHandlerAdapter {

    private ChannelHandlerContext ctx;

    @Override
    public void handlerAdded(ChannelHandlerContext ctx) {
        this.ctx = ctx; // 1.存储 ChannelHandlerContext 的引用供以后使用
    }

    public void send(String msg) {
        ctx.writeAndFlush(msg); // 2.使用之前存储的 ChannelHandlerContext 来发送消息
    }
}

因为 ChannelHandler 可以属于多个 ChannelPipeline，所以它可以绑定到多个 ChannelHandlerContext 实例。用于此用途的 ChannelHandler 必须用 @Shareable 修饰；否则，Netty 在尝试将其添加到多个 ChannelPipeline 时，将触发异常。显然，为了安全地使用多个并发通道（即连接），这样的 ChannelHandler 必须是线程安全的。

下面的代码中，显示了此模式的正确实现：

@ChannelHandler.Sharable // 1.添加 @Sharable 注解
public class SharableHandler extends ChannelInboundHandlerAdapter {
    @Override
    public void channelRead(ChannelHandlerContext ctx, Object msg) {
        System.out.println("channel read message " + msg);
        ctx.fireChannelRead(msg); // 2.日志方法调用， 并专递到下一个 ChannelHandler
    }
}

# 四、异常处理

# 1.处理入站异常

如果在处理入站事件期间抛出异常，它将从触发它的 ChannelInboundHandler 中的点开始流经 ChannelPipeline。要处理这样的入站异常，您需要在 ChannelInboundHandler 实现中重写以下方法。

public void exceptionCaught(ChannelHandlerContext ctx, Throwable cause) throws Exception;
1

关闭 Channel，并打印异常堆栈跟踪的简单示例如下：

public class InboundExceptionHandler extends ChannelInboundHandlerAdapter {
    @Override
    public void exceptionCaught(ChannelHandlerContext ctx, Throwable cause) {
        cause.printStackTrace();
        ctx.close();
    }
}
1
2
3
4
5
6
7

由于异常将继续在入站方向上流动（就像所有入站事件一样），因此实现上述处理异常逻辑的 ChannelInboundHandler 通常放在 ChannelPipeline 的最后。这确保了所有入站异常总是被处理，无论它们可能发生在 ChannelPipeline 的任何地方。

如果不实现对入站异常的任何处理（或者不使用异常），Netty 将记录未处理异常。

异常处理的流程如下：

  • ChannelHandler.exceptionCaught() 的默认实现将当前异常转发给管道中的下一个处理程序。
  • 如果异常到达管道的末端，则将其记录为未处理。
  • 要定义自定义处理，我们可以覆盖 exceptionCaught()。然后由我们来决定是否在这个点之外传播异常。

# 2.处理出站异常

处理出站操作中的正常完成和异常的选项基于以下通知机制：

  • 每个出站操作都会返回一个 ChannelFuture。在 ChannelFuture 中注册的 ChannelFutureListeners 会在操作完成时收到成功或错误的通知。

  • ChannelOutboundHandler 的方法基本都会通过参数传递一个 ChannelPromise 的实例。作为 ChannelFuture 的子类，ChannelPromise 也可以被分配异步通知的监听器。但是 ChannelPromise 也有提供即时通知的可写方法:

    ChannelPromise setSuccess();
ChannelPromise setFailure(Throwable cause);
    1
    2

出站的异常处理可以使用 ChannelFutureListener 处理，要添加 ChannelFutureListener，可以在 ChannelFuture 实例上调用 addListener(ChannelFutureListener) ，有两种方法可以做到这一点。

最常用的一种方法是在出站操作（例如 write()）返回的 ChannelFuture 上调用addListener() 方法。下面的示例中，将使用这种方法添加一个 ChannelFutureListener，它将打印堆栈跟踪，然后关闭通道。

// 向 ChannelFuture 添加 ChannelFutureListener
ChannelFuture future = channel.write(someMessage);
future.addListener(new ChannelFutureListener() {
    @Override
    public void operationComplete(ChannelFuture f) {
        if (!f.isSuccess()) {
            f.cause().printStackTrace();
            f.channel().close();
        }
    }
});

public class OutboundExceptionHandler extends ChannelOutboundHandlerAdapter {

ChannelPromise 的可写方法

通过在 ChannelPromise 上调用 setSuccess() 和 setFailure()，您可以在 ChannelHandler 方法返回给调用者时立即知道操作的状态。

我们该如何选择异常处理的方式？

  • 对于异常的详细处理，您可能会发现在调用出站操作时添加 ChannelFutureListener 更为合适。
  • 对于处理异常的不太专门化的方法，您可能会发现自定义 ChannelOutboundHandler 实现更简单

如果 ChannelOutboundHandler 本身抛出异常，Netty 将通知注册到 ChannelPromise 的侦听器进行处理。
