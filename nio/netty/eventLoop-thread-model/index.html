08.EventLoop 和线程模型

# 一、线程模型

# 1.线程模型

下面，我们来介绍一下常见的线程模型，以及 Netty 在 3.x 和 4.x 版本中的线程模型。

在 Java 的早期，我们的多线程方法只不过是根据需要创建和启动新的线程来执行并发工作单元，这是一种原始的方法，在繁重的负载下效率很差。Java 5 引入了 Executor API，它的线程池通过线程缓存和重用极大地提高了性能。

基本的线程池模式如下：

  1. 从池的空闲列表中选择一个线程，并将其分配给运行提交的任务（Runnable的实现）。
  2. 任务完成后，线程将返回到空闲列表中，并可供重用。

这种模式如下图所示：

需要注意的是，池化和重用线程不能消除上下文切换的成本，并且随着线程数量的增加，这一成本很快就会显现出来，在高负载下可能会很严重。此外，在项目的生命周期中，由于应用程序的总体复杂性或并发性需求，可能会带来其他与线程相关的问题。简单来说，多线程可能很复杂。

# 二、EventLoop

# 1. EventLoop 接口

在网络连接的生命周期中，运行任务来处理发生的事件，这是网络框架的基本功能，这种编程结构通常被称为事件循环，Netty 在接口 io.netty.channel.EventLoop 中采用了这个术语。

Netty 的 EventLoop 采用了两个基本的 API：并发和网络

  • 首先，包 io.netty.util.concurrent 基于 JDK 的并发包 java.util.concurrent 进行设计，以提供线程执行器。
  • 其次，包 io.net.Channel 中的类扩展了这些类，以便与 Channel 事件进行接口。

生成的类层次结构如图：

在这个模型中，EventLoop 由一个永远不会改变的线程驱动，可以直接提交任务（Runnable 或 Callable）给 EventLoop，以便立即或计划执行。根据配置和可用的 CPU 核心数，可以创建多个 EventLoop 以优化资源使用，并且可以分配单个 EventLoop 来服务多个通道。

需要注意的是，Netty 的 EventLoop，虽然扩展了 ScheduledExecutorService，但其只定义了一个方法 parent()，该方法会返回当前 EventLoop 实例所属 EventLoopGroup 的引用。

EventLoop 代码如下：

public interface EventLoop extends OrderedEventExecutor, EventLoopGroup {
    @Override
    EventLoopGroup parent();
}

事件和任务会按照先进先出的顺序执行。线程模型保证字节内容按正确顺序处理，防止数据（消息）被损坏的可能性。

# 2. Netty 4.x 的 I/O 和事件处理

Netty 的 I/O 操作触发的事件流经一个 ChannelPipeline，该 ChannelPipeline 有一个或多个已安装的ChannelHandler。然后，传播这些事件的方法调用可以被 ChannelHandler 拦截，并根据需要处理事件

在 Netty 4 中，所有的 I/O 操作和事件都由分配给 EventLoop 的 Thread 来处理。

# 3. Netty 3.x 的 I/O 和事件处理

Netty 3.x 版本中使用的线程模型只保证入站（以前称为 upstream，即上游）事件将在所谓的 I/O 线程中执行（对应于 Netty 4 的 EventLoop）。所有出站（以前称为 downstream，即下游）事件都由调用线程处理，调用线程可能是 I/O 线程或任何其他线程。

起初，这种设计看起来是一个不错的想法，但后来发现存在问题：

  • 问题1：因为在 ChannelHandler 中需要对出站事件小心的的进行同步处理，因为它无法保证多个线程不会同时尝试访问同一个出站事件。

    • 这导致在处理出站事件时需要进行复杂的同步和并发控制。因此，无法确保线程安全性，也无法防止多个线程同时访问出站事件可能引发的问题。
    • 例如，如果您通过在不同线程中调用 Channel.write() 方法为同一通道触发同时的出站事件，就有可能发生这种情况。

  • 问题2：当出站事件导致入站事件被触发时，会出现另一个问题。例如当出站 Channel.write() 操作导致异常时，会生成并抛出一个 exceptionCaught 。但 exceptionCaught 是入站事件，在 Netty 3的线程模型中，事件的处理会在调用线程中执行，但 exceptionCaught 事件必须交给 I/O 线程来执行，由 I/O 线程执行实际的异常处理逻辑，这就导致了从调用线程到 I/O 线程的上下文切换，可能会对性能产生一些影响。

exceptionCaught 是 Netty 中的一个方法，用于处理异常事件。它是 ChannelHandler 接口的一个方法，用于捕获和处理在处理 I/O 操作期间发生的异常。

当在 ChannelHandler 的方法中发生异常时，Netty会调用 exceptionCaught 方法来通知开发者。开发者可以通过重写 exceptionCaught 方法来自定义异常处理逻辑，例如记录日志、关闭连接或发送错误响应等。

相比之下，Netty 4 中的线程模型通过在同一个线程中处理给定 EventLoop 中发生的所有事情来解决这些问题。这提供了一个更简单的执行体系结构，并且消除了在多个 ChannelHandler 中同步的需要（除了可能在多个 Channel 之间共享的同步）。

# 三、任务调度

有时候，我们需要一个任务延迟或定期执行。例如，注册一个定时任务，以便在连接客户端成功时，五分钟之后触发。常见的场景就是心跳检测，以检查连接是否仍然存在。如果没有回应，则关闭连接和释放资源。

在 Java 中我们可以使用 ScheduledExecutorService 启动一个定时的任务，如下代码：

public static void main(String[] args) {
    // 1.创建 ScheduledExecutorService 使用10个线程
    ScheduledExecutorService executor = Executors.newScheduledThreadPool(10);
    ScheduledFuture<?> future = executor.schedule(
            // 新建 runnable 调度执行
            new Runnable() {
                @Override
                public void run() {
                    // 3.任务的处理内容
                    System.out.println("Now it is 60 seconds later");
                }
            }, 60, TimeUnit.SECONDS);  // 4.设置调度任务60秒后执行
    // 5.关闭 ScheduledExecutorService，释放任务完成的资源
    executor.shutdown();
}

上面的 schedule() 方法存在于 inteface EventExecutorGroup 中，EventExecutorGroup 继承自 ScheduledExecutorService。

EventExecutorGroup 中与定时任务相关的方法如下：

  • schedule(Runnable command, long delay, TimeUnit unit)：在指定的延迟时间后执行任务。任务将在指定的时间间隔之后执行一次。

  1. schedule(Callable<V> callable, long delay, TimeUnit unit)：在指定的延迟时间后执行任务，并返回一个Future对象，用于获取任务的执行结果。

  2. scheduleAtFixedRate(Runnable command, long initialDelay, long period, TimeUnit unit)：在指定的初始延迟时间后开始执行任务，并以固定的时间间隔重复执行任务。

  3. scheduleWithFixedDelay(Runnable command, long initialDelay, long delay, TimeUnit unit)：在指定的初始延迟时间后开始执行任务，并在每次任务完成后等待指定的延迟时间，然后再次执行任务。

ScheduledFuture 提供一个方法用于取消一个调度了的任务或者检查它的状态。一个简单的取消操作如下：

ScheduledFuture<?> future = ch.eventLoop()
    .scheduleAtFixedRate(..); // 调度任务并获取返回的 ScheduledFuture
// Some other code that runs...
future.cancel(false); // 取消任务，阻止它再次运行

Netty 的定时器实现基于 George Varghese 提出的 “Hashed and hierarchical timing wheels: Data structures to efficiently implement timer facility（散列和分层定时轮：数据结构有效实现定时器）”。这种实现只保证一个近似执行，也就是说任务的执行可能不是100%准确。不过对于大多数情况下影响并不大。

# 四、实现细节

# 1.线程管理

当一个线程需要与 Channel 进行交互时，无需在通道的处理器中同步：

  • 当调用线程是 EventLoop 的线程时，它会直接执行与通道相关的代码块。这意味着，如果当前线程是 EventLoop 的线程，它可以直接处理通道的事件和操作，而无需进行额外的同步操作。
  • 如果调用线程不是 EventLoop 的线程，而是其他线程，那么 EventLoop 会安排任务以供以后执行，并将这些任务放入内部的任务队列中。当 EventLoop 下次处理它自己的事件时，它将按顺序执行队列中的任务。

注意。每个 EventLoop 都有自己的任务队列，独立于任何其他 EventLoop。下图显示了 EventLoop 用于调度任务的执行逻辑。这是 Netty 线程模型的关键组成部分。

我们在前面提到了不阻塞当前 I/O 线程的重要性。我们将以另一种方式再说一遍：“永远不要将长时间运行的任务放在执行队列中，因为它会阻止在同一线程上执行任何其他任务”。如果你必须进行阻塞调用或执行长时间运行的任务，我们建议使用专用的 EventExecutor。

抛开这种极限情况不谈，所使用的线程模型会严重影响排队任务对整体系统性能的影响，所采用的传输的事件处理实现也是如此。

# 2.EventLoop 和 Thread

为 Channel 提供 I/O 和事件服务的 EventLoop 包含在 EventLoopGroup 中。 EventLoops 的创建和分配方式与基于传输的实现不同。

# 2.1 异步传输

异步实现只使用少数几个 EventLoop （和它们相关的线程）共享于 Channel 之间。这允许使用尽可能少的线程来服务多个通道（Channel），不需要为每个通道（Channel）分配一个线程。

下图中，显示了一个固定大小为三个 EventLoops 的 EventLoopGroup （每个 EventLoops 由一个 Thread 驱动）。EventLoop（及其线程）在创建 EventLoopGroup 时直接分配，以确保它们在需要时可用。

EventLoopGroup 负责为每个新创建的 Channel 分配一个 EventLoop。在当前的实现中，使用循环方法实现平衡分布，并且可以将相同的 EventLoop 分配给多个通道（Netty 后续版本可能会改变）。

一旦 Channel 被分配了 EventLoop，它将在其整个生命周期中使用这个 EventLoop （以及相关的线程）。请记住这一点，因为它使您不必担心 ChannelHandler 实现中的线程安全和同步问题。

但是，要注意 EventLoop 分配对 ThreadLocal 使用的影响。因为 EventLoop 通常支持多个 Channel，所以 ThreadLocal 对于所有相关的 Channel 都是相同的。这使得它成为实现状态跟踪等功能的糟糕选择。然而，在无状态上下文中，它仍然可以用于在通道之间共享重的或昂贵的对象，甚至事件。
