09.Bootstrap 引导程序

# 一、Bootstrap 类

类 Bootstrap 的层次结构由一个抽象的父类和两个具体的引导子类组成：

在客户端和服务器中，都使用了 Bootstrap 引导程序，客户端和服务器的应用程序共同的引导步骤由 AbstractBootstrap 处理，而那些特定于客户端或服务器的引导步骤分别由 Bootstrap 或 ServerBootstrap 处理。

Bootstrap 类是可克隆的

  • 有时你需要创建多个具有相似或相同设置的通道。为了支持这种模式，而不需要为每个通道创建和配置新的引导实例，AbstractBootstrap 被标记为 Cloneable。在已经配置的引导程序上调用 clone() 将返回另一个立即可用的引导程序实例。
  • 注意，在克隆时，引导程序中的 EventLoopGroup 只进行了浅拷贝，因此后者将在所有克隆通道之间共享。这是可以接受的，因为克隆通道通常是短暂的，典型的情况是创建用于发出 HTTP 请求的通道。

下面是类 AbstractBootstrap 的声明，其中 B 是超类的类型参数，可以返回对运行时实例的引用来支持方法链接。

public abstract class AbstractBootstrap<B extends AbstractBootstrap<B, C>, C
    extends Channel>

2

AbstractBootstrap 的子类声明如下：

public class Bootstrap extends AbstractBootstrap<Bootstrap, Channel>

public class ServerBootstrap extends AbstractBootstrap<ServerBootstrap, ServerChannel>

# 二、引导客户端和无连接协议

Bootstrap 用于客户端或使用无连接协议的应用程序中。下表列出了该类的方法，其中许多方法继承自AbstractBootstrap。

方法 描述
Bootstrap group(EventLoopGroup) 设置 EventLoopGroup 来处理 Channel 的所有事件。
Bootstrap channel(Class<? extends C>)Bootstrap channelFactory(ChannelFactory<? extends C>) channel() 指定 Channel 实现类。如果指定的类没有提供默认构造函数，则可以调用 channelFactory() 来指定要由 bind() 调用的工厂类。
Bootstrap localAddress(SocketAddress) 指定 Channel 应该绑定到的本地地址。如果没有提供，操作系统将随机创建一个。也可以使用 bind() 或 connect() 指定localAddress。
<T> Bootstrap option(ChannelOption<T> option, T value) 设置 ChannelOption 以应用于新创建通道的 ChannelConfig。这些选项将由 bind() 或 connect() 在 Channel 上设置，以先调用的方式为准。此方法在创建通道后不起作用。所支持的 ChannelOptions 取决于所使用的 Channel 类型。
<T> Bootstrap attr(Attribute<T> key, T value) 指定新创建 Channel 的属性。这些是由 bind() 或 connect() 在 Channel 上设置的，具体取决于先调用哪个。此方法在创建通道后不起作用。
Bootstrap handler(ChannelHandler) 设置添加到 ChannelPipeline 中的 ChannelHandler 以接收事件通知。
Bootstrap clone() 克隆当前引导程序实例，具有与原始引导程序相同的设置。
Bootstrap remoteAddress(SocketAddress) 设置远程地址。也可以使用 connect() 指定它。
ChannelFuture connect() 连接到远程对等端并返回一个 ChannelFuture，一旦连接操作完成，就会通知它。
ChannelFuture bind() 绑定通道并返回一个 ChannelFuture，一旦绑定操作完成就会通知该future，之后必须调用 Channel.connect() 来建立连接。

# 1.引导客户端

Bootstrap 类负责为客户端和使用无连接协议的应用程序创建通道，如下图所示。

下面代码演示了引导客户端，使用的是 NIO TCP 传输：

EventLoopGroup group = new NioEventLoopGroup();
Bootstrap bootstrap = new Bootstrap(); // 1.创建一个新的 Bootstrap 来创建和连接到新的客户端管道
bootstrap.group(group) // 2.指定 EventLoopGroup
    .channel(NioSocketChannel.class) // 3.指定 Channel 实现来使用
    .handler(new SimpleChannelInboundHandler<ByteBuf>() { // 4.设置处理器给 Channel 的事件和数据
        @Override
        protected void channeRead0(
            ChannelHandlerContext channelHandlerContext,
            ByteBuf byteBuf) throws Exception {
                System.out.println("Received data");
                byteBuf.clear();
            }
        });
ChannelFuture future = bootstrap.connect(
    new InetSocketAddress("www.willingoxjin.com", 80)); // 5.连接到远端主机
future.addListener(new ChannelFutureListener() {
    @Override
    public void operationComplete(ChannelFuture channelFuture)
        throws Exception {
            if (channelFuture.isSuccess()) {
                System.out.println("Connection established");
            } else {
                System.err.println("Connection attempt failed");
                channelFuture.cause().printStackTrace();
            }
        }
    });

在启动时，在调用 bind() 或 connect() 之前，必须调用以下方法来设置所需的组件：

  • group()
  • channel() or channnelFactory()
  • handler()

# 三、引导服务器

# 1.ServerBootstrap 类

ServerBootstrap 类的方法如下：

方法 描述
group 设置 EventLoopGroup 由 ServerBootstrap 使用。这个 EventLoopGroup 服务于 ServerChannel 和被接受的 Channel 的 I/O
channel 设置要实例化的 ServerChannel 类。
channelFactory 如果不能通过默认构造函数创建通道，则可以提供 ChannelFactory。
localAddress 指定 ServerChannel 应该绑定到的本地地址。如果没有指定，操作系统将随机使用一个。或者，您可以使用 bind() 或 connect() 指定 localAddress。
option 指定一个 ChannelOption 以应用于新创建的 ServerChannel 的 ChannelConfig。这些选项将通过 bind() 或 connect() 在 Channel 上设置，具体取决于先调用哪个。在调用这些方法之后设置或更改 ChannelOption 是没有效果的。支持哪个 ChannelOptions 取决于所使用的通道类型。请参考您正在使用的 ChannelConfig 的 API 文档。
childOption 指定一个 ChannelOption，当通道被接受时应用于通道的 ChannelConfig。支持哪个 ChannelOptions 取决于所使用的通道类型。请参考您正在使用的 ChannelConfig 的 API 文档。
attr 指定 ServerChannel 上的一个属性。属性将通过 bind() 在通道上设置。在调用 bind() 之后更改它们没有效果。
childAttr 将属性应用于接受的通道。后续调用没有效果。
handler 设置添加到 ServerChannel 的 ChannelPipeline 中的 ChannelHandler。请参阅 childHandler() 了解更常用的方法。
childHandler 设置添加到已接受通道的 ChannelPipeline 的 ChannelHandler。handler() 和 childHandler() 之间的区别在于，前者添加了一个由接受的 ServerChannel 处理的处理程序，而 childHandler() 添加了一个由接受的 Channel 处理的处理程序，后者表示绑定到远程对等体的套接字。
clone 克隆 ServerBootstrap 以连接到具有与原始 ServerBootstrap 相同设置的不同远程对等体。
bind 绑定 ServerChannel 并返回一个 ChannelFuture，一旦连接操作完成（带有成功或错误结果），就会通知该 future。

# 2.引导服务

您可能已经注意到，ServerBootstrap 中存在 Bootstrap 中没有的几个方法：childHandler()、childdattr() 和childOption()。这些调用支持服务器应用程序的典型操作。具体来说，ServerChannel 实现负责创建子通道，这些子通道代表可接受的连接。因此，引导 ServerChannels 的 ServerBootstrap 提供了这些方法来简化将设置应用到已接受 Channel 的 ChannelConfig 成员的任务。

下图显示了一个 ServerBootstrap 在 bind() 上创建一个 ServerChannel，并且 ServerChannel 管理许多子Channel。

下面的代码实现了上图所示的服务器引导：

NioEventLoopGroup group = new NioEventLoopGroup();
ServerBootstrap bootstrap = new ServerBootstrap(); // 1.创建要给新的 ServerBootstrap 来创建新的 SocketChannel 管道并绑定他们
bootstrap.group(group) // 2.指定 EventLoopGroup 用于从注册的 ServerChannel 中获取EventLoop 和接收到的管道
    .channel(NioServerSocketChannel.class) // 3.指定要使用的管道类
    .childHandler(new SimpleChannelInboundHandler<ByteBuf>() { // 4.设置子处理器用于处理接收的管道的 I/O 和数据
        @Override
        protected void channelRead0(ChannelHandlerContext ctx,
            ByteBuf byteBuf) throws Exception {
                System.out.println("Reveived data");
                byteBuf.clear();
            }
        }
    );
ChannelFuture future = bootstrap.bind(new InetSocketAddress(8080)); // 5.通过配置引导来绑定管道
future.addListener(new ChannelFutureListener() {
    @Override
    public void operationComplete(ChannelFuture channelFuture)
        throws Exception {
            if (channelFuture.isSuccess()) {
                System.out.println("Server bound");
            } else {
                System.err.println("Bound attempt failed");
                channelFuture.cause().printStackTrace();
            }
        }
    }
);

# 四、从 Channel 引导客户端

假如 Netty 服务器正在处理一个客户端请求，并且需要将该请求代理到其他服务器（例如 Spring Cloud Gateway 组件）。此时，我们需要从 ServerChannel 引导客户端 Channel。

我们可以创建一个新的 Bootstrap，但这不是最有效的解决方案。因为它需要你创建一个 EventLoop，来给新的客户端 Channel 使用。这将产生额外的线程，需要在接受的 Channel 和客户端 Channel 之间交换数据时进行上下文切换。

一个更好的解决方案：将接收到的 Channel 的 EventLoop 传递给 Bootstrap 的 group() 方法来共享它。因为分配给 EventLoop 的所有 Channel 使用相同的线程，这避免了前面提到的额外的线程创建和相关的上下文切换。这个共享解决方案如下图所示：

  1. 当 bind() 调用时，ServerBootstrap 创建一个新的 ServerChannel 。 当绑定成功后，这个 ServerChannel 就能接收子 Channel 了
  2. ServerChannel 接收新连接并且创建子 Channel 来服务它们
  3. Channel 用于接收到的连接
  4. Channel 自己创建了 Bootstrap，用于当 connect() 调用时创建一个新的管道
  5. 新 Channel 连接到远端
  6. 在 EventLoop 接收通过 connect() 创建后就在管道间共享

实现 EventLoop 共享包括通过调用 group() 方法来设置 EventLoop 的示例代码如下：

ServerBootstrap bootstrap = new ServerBootstrap(); // 1.创建一个新的 ServerBootstrap 来创建新的 SocketChannel 管道并且绑定他们
bootstrap.group(new NioEventLoopGroup(), // 2.指定 EventLoopGroups 从 ServerChannel 和接收到的管道来注册并获取 EventLoops
    new NioEventLoopGroup()).channel(NioServerSocketChannel.class) // 3.指定 Channel 类来使用
        .childHandler( // 4.设置处理器用于处理接收到的 Channel 的 I/O 和数据
            new SimpleChannelInboundHandler<ByteBuf>() {
            ChannelFuture connectFuture;

            @Override
            public void channelActive(ChannelHandlerContext ctx) throws Exception {
                Bootstrap bootstrap = new Bootstrap(); // 5.创建一个新的 Bootstrap 来连接到远程主机
                bootstrap.channel(NioSocketChannel.class) // 6.设置管道类
                        .handler(new SimpleChannelInboundHandler<ByteBuf>() { // 7.设置处理器来处理 I/O
                            @Override
                            protected void channelRead0(ChannelHandlerContext ctx, ByteBuf in) throws Exception {
                                System.out.println("Reveived data");
                            }
                        });
                bootstrap.group(ctx.channel().eventLoop()); // 8.使用相同的 EventLoop 作为分配到接收的管道
                connectFuture = bootstrap.connect(new InetSocketAddress("www.willingoxjn.com", 80)); // 9.连接到远端
            }

            @Override
            protected void channelRead0(ChannelHandlerContext channelHandlerContext, ByteBuf byteBuf) throws Exception {
                if (connectFuture.isDone()) {
                    // do something with the data  // 10.连接完成处理业务逻辑 (比如, proxy)
                }
            }
        });
ChannelFuture future = bootstrap.bind(new InetSocketAddress(8080));  // 11.通过配置了的 Bootstrap 来绑定到管道
future.addListener(new ChannelFutureListener() {
    @Override
    public void operationComplete(ChannelFuture channelFuture) throws Exception {
        if (channelFuture.isSuccess()) {
            System.out.println("Server bound");
        } else {
            System.err.println("Bound attempt failed");
            channelFuture.cause().printStackTrace();
        }
    }
});

# 五、在一个引导中添加多个 ChannelHandler

在所有示例代码中，我们在引导过程中调用 handler() 或 childHandler() 来添加单个 ChannelHandler。这对于简单的应用程序可能已经足够了，但是它不能满足更复杂的应用程序的需要。例如，一个必须支持多种协议的应用程序将有许多 ChannelHandlers。

我们可以通过在 ChannelPipeline 中将它们链接在一起来部署任意数量的 ChannelHandlers。但是，如果在引导过程中只能设置一个 ChannelHandler，您如何做到这一点呢?”

对于这个用例，Netty 提供了 ChannelInboundHandlerAdapter 的一个特殊子类 ChannelInitializer，ChannelInitializer 的声明如下：

public abstract class ChannelInitializer<C extends Channel>
    extends ChannelInboundHandlerAdapter
1
2

其定义了 initChannel 方法来添加多个 ChannelHandlers 到 ChannelPipeline：

protected abstract void initChannel(C ch) throws Exception;
1

我们只需要提供你的 ChannelInitializer 的实现给 Bootstrap，当 Channel 用它的 EventLoop 注册时，initChannel() 就会被调用。方法执行后，ChannelInitializer 实例将自己从 ChannelPipeline 中移除。

下面的代码定义了 ChannelInitializerImpl 类，并使用引导程序的 childHandler() 对其进行注册。这个看似复杂的操作其实非常简单：

ServerBootstrap bootstrap = new ServerBootstrap(); // 1.创建一个新的 ServerBootstrap 来创建和绑定新的 Channel
bootstrap.group(new NioEventLoopGroup(), new NioEventLoopGroup())  // 2.指定 EventLoopGroups 从 ServerChannel 和接收到的管道来注册并获取 EventLoops
    .channel(NioServerSocketChannel.class) // 3.指定 Channel 类来使用
    .childHandler(new ChannelInitializerImpl()); // 4.设置处理器用于处理接收到的管道的 I/O 和数据
ChannelFuture future = bootstrap.bind(new InetSocketAddress(8080));  // 5.通过配置的引导来绑定管道
future.sync();


final class ChannelInitializerImpl extends ChannelInitializer<Channel> { // 6.ChannelInitializer 负责设置 ChannelPipeline
    @Override
    protected void initChannel(Channel ch) throws Exception {
        // 7.实现 initChannel() 来添加需要的处理器到 ChannelPipeline。一旦完成了这方法 ChannelInitializer 将会从 ChannelPipeline 删除自身
        ChannelPipeline pipeline = ch.pipeline();
        pipeline.addLast(new HttpClientCodec());
        pipeline.addLast(new HttpObjectAggregator(Integer.MAX_VALUE));
    }
}

# 六、使用 Netty 的 ChannelOption 和属性

在创建每个 Channel 时，手动配置可能会变得非常繁琐。幸运的是，我们可以使用 option() 将 ChannelOptions 应用于引导程序中。我们提供的配置将自动应用于在引导中创建的所有通道。

可用的 ChannelOptions 包括低级连接细节，如保持活动或超时属性和缓冲区设置。

Netty 应用程序通常与组织的专有软件集成，而 Channel 等组件甚至可以在正常的 Netty 生命周期之外使用。在某些常用属性和数据不可用的情况下，Netty 提供了 AttributeMap 抽象（由通道和引导类提供的集合）和AttributeKey<T>（用于插入和检索属性值的泛型类）。使用这些工具，可以安全地将任何类型的数据项与客户端和服务器 Channel 相关联。

例如，考虑一个跟踪用户和 Channel 之间关系的服务器应用程序。这可以通过将用户 ID 存储为 Channel 的属性来实现。类似的技术可以根据用户的 ID 将消息发送给用户，或者在活动较少时关闭频道。

使用 ChannelOptions 配置 Channel 和属性来存储整数值的码示例如下：

final AttributeKey<Integer> id = new AttributeKey<Integer>("ID");　// 1.新建一个 AttributeKey 用来存储属性值

Bootstrap bootstrap = new Bootstrap(); // 2.新建 Bootstrap 用来创建客户端管道并连接他们
bootstrap.group(new NioEventLoopGroup()) // 3.指定 EventLoopGroups 从和接收到的管道来注册并获取 EventLoop
        .channel(NioSocketChannel.class) // 4.指定 Channel 类
        .handler(new SimpleChannelInboundHandler<ByteBuf>() { // 5.设置处理器来处理管道的 I/O 和数据
            @Override
            public void channelRegistered(ChannelHandlerContext ctx) throws Exception {
               Integer idValue = ctx.channel().attr(id).get();  // 6.检索 AttributeKey 的属性及其值
                // do something with the idValue
            }

            @Override
            protected void channelRead0(ChannelHandlerContext channelHandlerContext, ByteBuf byteBuf) throws Exception {
                System.out.println("Reveived data");
            }
        });
bootstrap.option(ChannelOption.SO_KEEPALIVE, true)
    .option(ChannelOption.CONNECT_TIMEOUT_MILLIS, 5000); // 7.设置 ChannelOption 将会设置在管道在连接或者绑定
bootstrap.attr(id, 123456); // 8.存储 id 属性

ChannelFuture future = bootstrap.connect(new InetSocketAddress("www.willingoxjin.com", 80)); // 9.通过配置的 Bootstrap 来连接到远程主机
future.syncUninterruptibly();

# 七、无连接协议的 Channel

前面的 Bootstrap 代码示例使用了基于 TCP 的 SocketChannel，但是 Bootstrap 也可以用于无连接协议。Netty 为此提供了各种 DatagramChannel 实现。唯一的区别是您不调用 connect()，而只调用 bind()，如下：

Bootstrap bootstrap = new Bootstrap();
bootstrap.group(new OioEventLoopGroup()) // 1.设置为处理 Channel 事件提供 EventLoops 的首选程序
        // 2.创建一个 Bootstrap 来创建和绑定新的数据报通道
        .channel(OioDatagramChannel.class) // 3.特别的 Channel 实现类，用于无连接的数据报传输
        .handler(new SimpleChannelInboundHandler<DatagramPacket>() {
            @Override
            protected void channelRead0(io.netty.channel.ChannelHandlerContext ctx, DatagramPacket msg)
                    throws Exception { // 4.设置用于处理 Channel 的 I/O 以及数据的 ChannelInboundHandler
                // Do something with the packet
            }
        });
ChannelFuture future = bootstrap.bind(new InetSocketAddress(0));  // 5.调用 bind() 方法，因为是无连接的，所以不需要连接到远程地址
future.addListener(new ChannelFutureListener() {
    @Override
    public void operationComplete(ChannelFuture future) throws Exception {
        if (future.isSuccess()) {
            System.out.println("Channel bound");
        } else {
            System.err.println("Bind attempt failed");
            future.cause().printStackTrace();
        }
    }
});

# 八、关闭客户端或服务器

引导让你的应用程序启动并运行，但我们也需要优雅地关闭它。虽然我们可以让 JVM 在退出时处理所有事情，但这将不符合优雅的定义，优雅指的是干净地释放资源。关闭 Netty 应用程序不需要太多魔法，但有几件事要记住。

其中，最重要的是关闭 EventLoopGroup，它将处理任何挂起的事件和任务，并随后释放所有活动线程。这是调用 EventLoopGroup.shutdownGraceful() 的问题。这个调用将返回一个 Future，在关闭完成时通知它。

请注意，shutdownGracefully() 是一个异步操作，因此需要阻塞直到它完成，或者向返回的 Future 注册一个侦听器以获得完成通知。

如下示例代码进行优雅的关闭服务：

EventLoopGroup group = new NioEventLoopGroup() // 1.创建 EventLoopGroup 用于处理 I/O
Bootstrap bootstrap = new Bootstrap(); // 2.创建一个新的 Bootstrap 并且配置他
bootstrap.group(group)
    .channel(NioSocketChannel.class);
...
...
Future<?> future = group.shutdownGracefully(); // 3.最终优雅的关闭 EventLoopGroup 释放资源。这个也会关闭中当前使用的 Channel
// block until the group has shutdown
future.sync();
