04.Netty 组件与设计

# 一、Channel, EventLoop 和 ChannelFuture

# 1.Channel

在基于 Java 的网络编程中，用于基本连接的是 Socket 类，可用于连接、绑定、读取和写入等 IO 操作。

Netty 为了降低了直接使用 Socket 的复杂性，提供了 io.netty.channel.Channel 接口。

Channel 提供了如下功能：

  1. Channel 的当前状态（例如，是否打开？是否已连接），
  2. Channel 的配置参数（例如接收缓冲区大小）
  3. Channel 支持的 I/O 操作（例如读、写、连接和绑定）
  4. 处理与 Channel 相关的所有 I/O 事件和请求的 ChannelPipeline。

Channel 在 Netty 中有许多预定义的、专门的实现，例如：

  1. EmbeddedChannel
  2. LocalServerChannel
  3. NioDatagramChannel
  4. NioSctpChannel
  5. NioSocketChannel

# 2.EventLoop

EventLoop 定义了 Netty 用于处理在连接生命周期中发生的事件的核心抽象。我们将在第7章的Netty线程处理模型中详细讨论EventLoop。目前，图3.1在较高的层次上说明了Channels、EventLoops、Threads和EventLoopGroups之间的关系。

# 3.ChannelFuture

Netty 中的所有 I/O 操作都是异步的。所以，操作可能不会立即返回结果，因此，我们需要一种方法获取其结果。因此，Netty 提供了ChannelFuture，它的 addListener() 方法注册了一个 ChannelFutureListener，以便在操作完成（无论是否成功）时得到通知。

# 二、ChannelHandler 和 ChannelPipeline

# 1.ChannelHandler

对于直接使用 Netty 的开发者来说，ChannelHandler 是 Netty 的主要组件。它充当用于处理入站和出站数据的所有应用程序逻辑的容器。

因为 ChannelHandler 的方法是由网络事件触发的。ChannelHandler 几乎可以用于任何类型的操作，例如将数据从一种格式转换为另一种格式，或者处理任务过程中抛出的异常。

ChannelHandler 的部分实现如下：

可以看到，ChannelHandler 的直接子接口为： ChannelInboundHandler 和 ChannelOutboundHandler 。

# 2.ChannelPipeline

ChannelPipeline 为 ChannelHandler 链提供了一个容器，并定义了一个 API，用于沿链传播入站和出站事件流。当一个 Channel 被创建时，它会被自动分配到对应的 ChannelPipeline。

ChannelHandler 在 ChannelPipeline 中的安装方式如下:

  1. 将 ChannelInitializer 的实现注册到 ServerBootstrap 中。
  2. 当 ChannelInitializer.initChannel() 被调用时，ChannelInitializer 会在管道中安装一组自定义ChannelHandlers。
  3. 初始化完成后，ChannelInitializer 将自己从 ChannelPipeline 中移除。

# 3.ChannelPipeline 和 ChannelHandler 的关系

在应用程序初始化或引导阶段安装的 ChannelHandler 的。这些对象接收事件，执行为其实现的处理逻辑，并将数据传递给链中的下一个 ChannelHandler 处理器。

这些 ChannelHandler 执行的顺序由它们被添加时的顺序决定。我们将这些有序排列的 ChannelHandler 称为 ChannelPipeline。

从客户端的角度来看 ，如果在应用程序中的移动是从客户端到服务器，则事件称为出站事件，反之则称为入站事件。

带有入站和出站通道处理器的 ChannelPipeline 如下图：

入站和出站的处理器都可以安装在同一管道中。

入站：

  • 如果消息或任何其他入站事件被读取时，将从管道的头部开始，传递到第一个 ChannelInboundHandler。这个处理程序的特定函数中，可以进行数据修改操作，

  • 之后数据将被传递到链中的下一个 ChannelInboundHandler。

  • 最后，数据将到达管道的尾部，此时所有处理都将终止。

出站流程如下：

  • 在这种情况下，数据从尾部流过ChannelOutboundHandler 链，直到到达头部。
  • 然后，出站数据到达网络传输（图中显示为 Socket），通常会触发一个写操作。

Netty 提供了基类 ChannelInboundHandlerAdapter 和 ChannelOutboundHandlerAdapter，并且在这两个类的所有方法中，都可以获取到 ChannelHandlerContext 参数，我们可以使用 ChannelHandlerContext 参数将事件传递给当前链中的下一个处理器。

需要注意的是，入站和出站的处理器链，数据仅会在相同方向类型的处理程序之间传递。

# 4.ChannelHandlerContext

当 ChannelHandler 被添加到 ChannelPipeline 时，它被分配一个 ChannelHandlerContext，它表示ChannelHandler和 ChannelPipeline 之间的绑定。ChannelHandlerContext 对象主要用于写入出站数据，并且也可以用于获取底层 Channel。

在 Netty 有两种方式发送消息：

  1. 直接写入消息到 Channel 。
    • 即使用 Channel.write() 方法
    • 该方式消息会被插入到 ChannelPipeline的尾部，并从尾部开始进行处理。这意味着消息会经过出站处理器链，按照出站处理器的顺序进行处理，直到到达链的头部，最终被发送到网络中。
  2. 写入消息到与 ChannelHandler 关联的 ChannelHandlerContext 对象。
    • 即使用 ChannelHandlerContext.write() 方法。
    • 该方式导致消息从 ChannelPipeline 下一个处理器开始，这可能会导致已经执行过的处理器不会处理写入的消息。

# 5.ChannelHandler 的常用实现

Netty 提供了许多 ChannelHandler 的默认处理器的实现。这些实现以适配器类的形式，用于简化应用器处理逻辑的开发。我们可以直接继承这些适配器类（及其子类），并覆盖想要需要定制的方法和事件即可。

常调用的适配器如下：

  1. ChannelHandlerAdapter：其是一个抽象类。它提供了一组默认的方法实现，方便我们自定义的ChannelHandler只需覆盖感兴趣的方法。它既可以作为入站处理器，也可以作为出站处理器。
  2. ChannelInboundHandlerAdapter：专门用于处理入站事件。它提供了一组默认的方法实现，用于处理连接建立、数据读取、异常处理等入站事件
  3. ChannelOutboundHandlerAdapter：专门用于处理出站事件。它提供了一组默认的方法实现，用于处理数据写入、连接关闭、异常处理等出站事件。
  4. ChannelDuplexHandlerAdapter ：同时处理入站和出站事件。它提供了一组默认的方法实现，既可以处理入站事件，也可以处理出站事件。我们可以选择性地覆盖感兴趣的方法。

# 6.编解码处理器

当我们使用 Netty 发送或接收消息时，就会发生数据转换。

  • 如果是入站消息，消息将被解码（反序列化）：从字节转换为 Java 对象；
  • 如果消息是出站的，则相反：它将从当前格式编码为字节（序列化）。

Netty 在 netty-codec 模块中，为编解码提供了基类：ByteToMessageDecoder 或 MessageToByteEncoder。并且提供了 json、marshalling、protobuf、string、xml、Serializable（Java 序列化）、compression（压缩）等编解码方式的默认实现。

  • ByteToMessageDecoder 是基于 ChannelInboundHandler 实现的。
    • 对于从入站通道读取的每条消息，都会调用 channelRead() 方法。然后，它将调用所提供的解码器的 decode() 方法，并将解码的字节转发给管道中的下一个ChannelInboundHandler
  • 同理，MessageToByteEncoder 是基于 ChannelOutboundHandler 实现的。
    • 出站通道的消息，编码器将消息转换为字节，并将其转发给下一个 ChannelOutboundHandler。

# 7.SimpleChannelInboundHandler

多数情况下，Netty 的应用程序将使用一个处理程序来接收解码消息并将业务逻辑应用于数据。

要创建这样一个 ChannelHandler，我们只需要扩展基类 SimpleChannelInboundHandler<T>，其中 T 是我们预期消息的 Java 类型。

在我们基于 SimpleChannelInboundHandler 处理器的实现中，可以重写父类的一个或多个方法，并获得对 ChannelHandlerContext 的引用，该引用作为方法的入参数传递给所有处理器的方法。

# 三、Bootstrapping

Netty 的引导类为应用程序的网络层配置提供了容器：

  1. 引导服务器：这涉及到将一个进程绑定到一个给定的端口（bind()），用于监听传入该端口的连接。
  2. 引导客户端：将一个进程连接到另一个运行在指定端口上的指定主机上的进程（connect()）。

严格来说，术语 “连接”（connection）只适用于面向连接的协议，如 TCP， 保证在连接的端点之间有序地传递消息。

因此，两种类型的引导程序，一种用于客户端（ Bootstrap），另一种用于服务器（ServerBootstrap）。无论应用程序使用哪种协议或执行哪种类型的数据处理，决定它使用哪个引导类的唯一因素是它作为客户机或服务器的功能。

两者区别如下：

  1. ServerBootstrap 绑定到端口，用于监听连接；Bootstrap 想要连接到远程对等体的客户端应用程序使用。
  2. Bootstrap只需要一个 EventLoopGroup；但是 ServerBootstrap 需要两个（可以是同一个 EventLoopGroup 实例）。

为什么ServerBootstrap 需要两个EventLoopGroup 实例？

因为服务器需要两组不同的通道。第一组将包含单个 ServerChannel，用于服务器监听 Socker。第二组用于处理已建立连接的I/O操作和业务逻辑。

下图说明了这个模型，并说明了为什么需要两个不同的 EventLoopGroup

一般我们称作第一组为 Boss EventLoopGroup，第二组为 Worker EventLoopGroup。

  1. Boss EventLoopGroup：这个 EventLoopGroup 负责处理连接的接收。它绑定在服务器的监听端口上，监听客户端的连接请求。当有新的连接请求到达时，Boss EventLoopGroup 会接收并将连接分配给 Worker EventLoopGroup 中的某个 EventLoop 进行处理。
  2. Worker EventLoopGroup：这个 EventLoopGroup 负责处理已经建立的连接，执行实际的 I/O 操作和业务逻辑。它负责读取数据、写入数据以及执行其他业务逻辑。Worker EventLoopGroup 中的每个 EventLoop 都维护一个事件循环，不断地从注册的 Channel 中选择已就绪的事件进行处理。
