2.BPMN 2.0

# 一、BPMN 介绍

BPMN （Business Process Model and Notation 业务流程模型和符号）由 BPMI（Business Process Management Initiative）开发的一套标准的业务流程建模符号。它是一套图形化表示法，用图形来表示业务模型。

BPMN 是流程图的统一标准（OMG BPMN标准 (opens new window)），不论你使用上面三个主流工作流的哪个引擎进行开发，它们都使用 BPMN 的规范。

# 二、BPMN 2.0 结构

一个流程图中主要包含以下内容：

  • 事件
  • 顺序流（连线）
  • 网关
  • 任务
  • 活动

当然还有其他内容，更多流程图结构见官方文档。

可以在官方在线网站中绘制流程图：https://demo.bpmn.io/new (opens new window)

# 三、事件

事件（event）通常用于为流程生命周期中发生的事情建模。事件总是图形化为圆圈。

一个流程图中总是有启动事件和结束事件，如上图中的圆圈即为开始事件和结束事件。另外还有中间事件、边界事件等等。

# 四、顺序流（连线）

顺序流是流程中两个元素间的连接器。在流程执行过程中，一个元素被访问后，会沿着其所有出口顺序流继续执行。

# 五、网关

网关用于控制执行的流向。

网关用其中带有图标的菱形表示：

# 1.排他网关（互斥网关）

排他网关用内部带有“X”图标的标准网关（菱形）表示，“X”图标代表异或的含义。请注意内部没有图标的网关默认为排他网关。

BPMN 2.0 规范不允许在同一个流程中混合使用有及没有“X”的菱形标志

# 2.并行网关

并行网关一般都是成对出现。并行网关不计算条件，如果连接到并行网关的顺序流上定义了条件，会直接忽略该条件。

并行网关，用内部带有“加号”图标的网关（菱形）表示，代表与（AND）的含义。

# 3.包容网关（相容网关）

包容网关会有多个出口，只要满足条件，都会执行。

包容网关，用内部带有“圆圈”图标的网关（菱形）表示。

# 4.事件网关

事件网关通过中间事件驱动，它在等待的事件发生后才会触发决策。

事件网关，用内部带有特殊图标的网关（菱形）表示：

# 六、任务

任务分类非常多，不过大概可以分为两类：

  • 用户任务
    • 表示需要人工介入的任务，如审批的同意，或输入相关的参数。用户任务需要用户向引擎提交一个完成任务的动作，否则，流程会暂停继续执行，一直等待。
  • 服务任务
    • 表示可以自动化处理的事，调用服务任务，服务可以是 Spring 的 JavaBean 对象，也可以是一个 HTTP 服务。

# 1.接收任务

接收任务是等待特定消息到达的简单任务。

接收任务用左上角有一个消息图标的标准 BPMN 2.0 任务（圆角矩形）表示。消息图标是白色的。

# 2.发送任务

# 3.服务任务

服务任务用于调用 Java 类或外部的 Web 服务，

服务任务用左上角有一个小齿轮图标的圆角矩形表示。

# 4.脚本任务

脚本任务是自动执行的活动。当流程执行到达脚本任务时，会执行相应的脚本。

脚本任务用左上角有一个小“脚本”图标的标准BPMN 2.0任务（圆角矩形）表示。

# 5.业务规则任务

业务规则任务用于同步地执行一条或多条规则。

业务规则任务显示为带有表格图标的圆角矩形。

# 6.用户任务

用户任务用于对需要人工执行的任务进行建模。当流程执行到达用户任务时，会为指派至该任务的用户或组的任务列表创建一个新任务。

用户任务用左上角有一个小用户图标的标准任务（圆角矩形）表示。

# 七、活动

活动可以理解为一种特殊的任务，活动可以调用另外一个流程使之作为当前流程的子流程去运行。活动也可以分为用户活动、脚本活动等。

# 八、流程图示例

下面是制造沙发的流程图：

  1. 启动事件

  2. 用户任务：设计师设计产品

  3. 用户任务：设计师设计好产品后，经理进行审批

  4. 排他网关：

    • 审批不通过，则打回重新设计

    • 审批通过，继续后续流程

  5. 服务任务：载入图纸准备生产

  6. 任务：准备生产线（这个可能是服务任务、用户任务等多种类型混合）

  7. 接收任务：等待生产线准备完毕

  8. 并行网关：开始制造

    • 生产时可以并行生产沙发骨架、沙发海绵、沙发外套

  9. 并行网关：制造结束

  10. 服务任务：然后根据前一步生产的东西，进行组装

  11. 服务任务：组装后，添加外包装

  12. 结束事件

# 九、CMMN 与 DMN

除了 BPMN，还有 CMMN、DMN 标准。

# 1.CMMN

CMMN（Case Management Model and Notation 案例管理模型与标记法）

CMMN 是一种图形化表示法，用于建模和管理复杂、动态和不可预见的业务流程。与 BPMN 侧重于定义结构化、可预测的流程不同，CMMN专注于灵活和非结构化的工作流程，通常这些流程需要根据具体情况动态调整。

CMMN 的使用场景如下：

  • 案例管理：处理需要灵活和自适应的业务流程，如客户服务、保险理赔、法律事务等。
  • 事件驱动的流程：管理基于事件触发的任务和活动。
  • 知识密集型流程：处理需要专家判断和知识密集的流程，如医疗诊断、研发项目等。
  • 长期流程：管理持续时间较长且需要多次人为决策和干预的流程。

核心元素如下：

  • 任务（Task）：表示需要完成的工作，可以是手动任务、用户任务、阶段任务等。
    • 用户任务（Human Task）：需要人工执行的任务。
    • 过程任务（Process Task）：嵌入BPMN流程的任务。
    • 案例任务（Case Task）：嵌入另一个CMMN案例的任务。
  • 事件（Event）：表示案例中的某个事件，可以是状态变更事件或外部触发事件。
    • 计划项启动事件（Plan Item Start Event）：当计划项启动时触发。
    • 阶段完成事件（Stage Complete Event）：当阶段完成时触发。
  • 阶段（Stage）：表示一个包含多个任务和事件的分组，可以嵌套其他阶段或任务。
    • 阶段（Stage）：一个包含多个任务、事件和其他元素的分组。
    • 里程碑（Milestone）：表示达到某个重要进展的点。
  • 计划项（Plan Item）：表示任务、阶段或里程碑等可以执行的元素。
  • 案例文件（Case File）：表示与案例相关的文档和数据。
  • 声明（Sentry）：用于控制任务、阶段等元素的执行条件，可以是进入或退出条件。

# 2.DMN

DMN（Decision Model and Notation 决策模型与标记法）用于描述和建模业务决策。DMN 的目标是使业务分析师和技术开发者能够协作定义和管理业务规则和决策逻辑，确保决策过程透明、可理解和可执行。

DMN 的使用场景如下：

  • 决策建模：帮助企业定义和管理复杂的业务规则和决策逻辑。
  • 自动化决策：将决策模型集成到业务流程中，实现自动化决策。
  • 业务规则管理：提供一种标准化的方法来记录和维护业务规则。
  • 分析和优化：通过对决策模型的分析，优化业务决策，提高效率和一致性。

核心元素如下：

  • 决策（Decision）：表示一个业务决策节点，定义了决策逻辑。
    • 决策节点（Decision Node）：用来表示一个具体的决策过程，包含决策逻辑。
    • 输入数据（Input Data）：表示决策所需的输入信息，可以是外部数据、业务参数等。
  • 业务知识模型（Business Knowledge Model）：表示决策过程中使用的业务知识或规则，通常是复用的逻辑单元。
    • 知识源（Knowledge Source）：表示提供决策依据的信息源，可能是政策、法规、指南等。
    • 决策表（Decision Table）：一种常见的决策逻辑表示方法，通过表格定义不同条件下的决策输出。
  • 规则（Rule）：决策表中的每一行表示一个规则，包括条件和对应的行动。
    • 输入条目（Input Entry）：决策表中用于判断的条件。
    • 输出条目（Output Entry）：决策表中对应条件满足时的输出。

BPMN 适合明确顺序和流程的业务。例如，物流配送业务，首先，客户下单（开始事件），然后订单信息被录入系统（任务），接着包裹被分拣（任务）、运输（任务），最后送达客户手中（结束事件）。

CMMN 适合处理灵活多变、并行、需要根据具体情况决策的业务。例如，保险理赔中，报案、定损、审核、赔付流程中，都会有不确定的因素，如材料齐全、不符合理赔条件等不确定因素。

DMN 专注于定义和清晰表达决策的逻辑和规则。例如，银行的信用卡审批决策。银行会根据申请人的信用评分（高于 600 分）、收入水平（月收入大于 5000 元）、负债情况（负债比例小于 50%）等因素来决定是否批准信用卡申请。使用 DMN 可以清晰地定义这些决策规则和条件
