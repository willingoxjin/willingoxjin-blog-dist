1.工作流介绍

# 一、什么是工作流？

工作流（Workflow）是指将一系列业务活动（即工作）按照一定规则与程序连接成一个整体，以完成某个特定的业务过程。

它主要解决的是“使在多个参与者之间按照某种预定义的规则自动进行传递文档、信息或任务的过程，从而实现某个预期的业务目标，或者促使此目标的实现”。

工作流一般在 OA 、ERP、CRM 系统中使用，具体典型的应用有请假审批、合同审核、订单、付款请求、日常报销处理等等。

相关需求如下：

  • 启动请假流程需要提供一些信息，例如雇员名字、请假时长以及说明。

  • 如果领导批准，则需要将申请注册至某个外部系统，并跟着另一个用户任务，将经理的决定通知给申请人。当然也可以改为发送邮件。

  • 如果领导驳回，则给该雇员发送一封邮件。

如果让我们使用程序实现这个需求，就算不使用工作流，也能够实现：

  • 设置一张请假表，如下：
CREATE TABLE holiday_request (
    `id` bigint(11) UNSIGNED NOT NULL AUTO_INCREMENT COMMENT 'Primary key',
    `user_id` bigint(11) NOT NULL COMMENT '用户id',
  	`employee_id` bigint(11) NOT NULL COMMENT '雇员id',
    `start_date` datetime NOT NULL COMMENT '请假开始时间',
    `end_date` datetime NOT NULL COMMENT '请假结束时间',
    `content` text NOT NULL COMMENT '请假说明',
    `check_status` int(2) NOT NULL COMMENT '请假状态 0-待审批 1-已审批 2-已拒绝',
  	`check_user_id` int(2) NOT NULL COMMENT '审批人',
    `check_time` int(2) NOT NULL COMMENT '审批时间',
    PRIMARY KEY (`id`),
    KEY user_id (`user_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT '请假申请表';

  • 代码逻辑如下（仅主逻辑）：
    • 雇员发出请假申请：我们就在 holiday_request 表中添加一条数据，将请假相关信息保存起来。
    • 领导驳回请假申请：我们将请假表中的 check_status 修改为 2-已审批。并且给雇员发送一封通知邮件。
    • 领导同意请假申请：我们将请假表中的 check_status 修改为 1-已审批。并且将请假申请注册到外部系统后，完成请假审批。

这个请假流程非常简单，不过，实际的业务需求却是非常复杂的。例如在报销审批中，需要多人审批（部门领导审批、财务审批、财务部门领导审批、出纳付款等等流程，也可能更加复杂），当报销请求被驳回后，用户还可以修改相关的报销申请资料后，再继续后续的流程，如果仍然使用 check_status 来表示审批的状态，那这个状态用来表示的枚举值会非常多。

并且有时候，流程也存在并行的情况，这种情况更加复杂。例如项目立项申请：项目立项申请可能需要跨多个部门进行审批，如需要同时向市场、人事、财务等部门提交审核，以便更快地得到审批结果。

对于更加复杂的流程，相对于自己实现，使用工作流会更通用、更方便维护。

# 二、工作流引擎

JBPM（Java Business Process Model）是一个基于Java的业务流程管理框架，它是业界比较早期的开源工作流引擎之一，由 JBoss 公司开发。JBPM 的创建者是 Tom Baeyens 离开 JBoss 后，加入到 Alfresco，并推出了 Activiti。Activiti 基于 JBPM4 实现。而 JBPM 后续完全放弃了 JBPM4 的代码。

而 Activiti5 维护没多久，就从 Activiti 中又分裂 Camunda 和 Flowable。

目前主流的工作流引擎对比：

  • Activiti

    • 侧重云原生，目前设计向 Spring Cloud、Docker 靠拢。

  • Flowable

    • 除了最基础的工作流，还提供了很多扩展点，可以基于 Flowable 实现自己想要的功能。

  • Camunda

    • 相对 Activiti 、Flowable 更加轻量级。
    • 提供了一个小巧的编辑器，基于 bpmn.io 实现。
