# 二、flowable 的表

使用 Flowable 引擎的时候，我们调用的是 Flowable 提供的各种 API ，不过这些 API 本质上操作的是数据库，即对数据库的增删改查。

如果你将 Flowable 中的 databaseSchemaUpdate 配置为 true 则 Flowable 在构建引擎的时候，会检查数据库中的表，并在必要时对数据库的表进行更新，

在代码执行后，会创建很多表。这些表的名称命名是有规律的，我们根据前缀、中间、后缀来看一下规律。

# 1.表名前缀

所有的表一共有两种前缀：

  • ACT_
    • 由于 Flowable 是基于 Activiti 开源的流程引擎，因此有些表是 Activiti 中的。因此以 ACT_ 为前缀的表，是 Activiti 中原有的表。
  • ``FLW_`
    • 以 FLW_ 为前缀的表，则是 Flowable 特有的表。

# 2.表名中间

  • ADM（ADMIN）：仅 flowable-ui 应用。与 flowable-ui 中 Admin 应用相关的表。
  • APP：表示都是跟应用程序相关的表。
  • CMMN：表示都是跟内容引擎相关的表。
  • CO（CONTENT）：表示都是跟内容引擎相关的表。
  • DE：仅 flowable-ui 应用。 存储流程定义的数据
  • DMN：表示都是跟 DMN 协议相关的表。
  • EVT（EVENT）：表示都是跟事件相关的表。
  • FO（FORM）：表示都是跟表单相关的表。
  • GE（GENERAL）：表示都是通用数据的表。如系统配置、业务关键对象等。
  • HI（HISTORY）：表示都是包含历史数据的表，即使从运行时的表中删除了数据，历史表仍然包含这些已完成实例的所有信息。如历史流程实例、历史任务、历史流程变量、历史活动等。
  • ID（IDENTITY）：表示都是跟用户身份认证相关的表。
  • PROCDEF（PROCESS DEFINE）：表示都是跟记录流程相关的表。
  • RE（RESOURCE）：表示都是跟流程定义、流程资源（图片，规则等）等包含静态信息相关的表。如流程定义、用户任务、连线、网关、流程变量等。
  • RU（RUNTIME）：表示都是运行时相关的表。如当前正在执行的流程实例、用户任务、多实例任务、消息等。 。
  • CHANNEL：表示都是跟泳道相关的表。
  • EV：EV 这个是与 FLW_ 为前缀的的表中的。这里相关的表都是跟 Liquibase 相关的表。
  • EVENT：表示都是跟事件相关的表。

Liquibase 是一个免费、开源的数据库变更管理系统，可以帮助开发人员在整个应用程序开发周期中自动化和跟踪数据库架构变更。

Liquibase 使用 XML 文件来描述数据库变更集，这些变更集包括创建、修改和删除表、插入和更新数据以及添加和删除索引等操作。当应用程序需要升级到新版本时，Liquibase 可以自动识别要应用的变更，并确保变更是按正确的顺序应用的。

# 3.表名后缀

表名后缀以下是通用的后缀名：

  • DATABASECHANGELOG：表中包含该名称，表示这个表是 Liquibase 执行的记录。
  • DATABASECHANGELOGLOCK：表中包含该名称，表示这个表记录 Liquibase 执行锁的，用来确保一次只运行一个 Liquibase 实例。

# 4.其他

# ACT_APP_*

以 ACT_APP_ 为前缀的表，负责应用引擎存储和应用部署定义。相关的表如下：

  • ACT_APP_APPDEF：用于存储应用程序定义的信息。每个应用程序定义对应一个流程定义，定义了一个应用程序的结构，它包含了应用程序的名称、描述、版本等信息，以及与流程定义相关的信息，如流程定义的 ID、名称、版本等。
  • ACT_APP_DEPLOYMENT：用于存储每个应用的部署信息，包括部署时间、部署用户、应用名称、应用分类、应用资源文件名称等。
  • ACT_APP_DEPLOYMENT_RESOURCE：用于存储每个应用的各种资源文件。ACT_APP_DEPLOYMENT 存储部署信息，ACT_APP_DEPLOYMENT_RESOURCE 存储部署的实际内容。

# ACT_CMMN_*

以 ACT_CMMN_ 为前缀的表，都是与 Flowable CMMN 相关的表

CMMN（Case Management Model and Notation）是一种可视化建模语言和标准，用于描述案例管理过程和方案。它扩展了 BPMN（Business Process Model and Notation）标准，允许在案例管理过程中处理更多不确定性和变化。

CMMN 这类表一般用于存储处理 BPMN 所不能适用的业务场景数据，CMMN 通常与 BPMN 搭配使用，但是只有符合 CMMN 规范的模型数据才会使用这类表。

相关表说明：

  • ACT_CMMN_HI_CASE_INST：记录由 CMMN 引擎启动的每个案例实例

  • ACT_CMMN_HI_MIL_INST：记录了在案例的实例中达到的每个里程碑的数据。

  • ACT_CMMN_HI_PLAN_ITEM_INST：此表记录了作为案例实例执行的一部分创建的每个计划项实例的数据。

  • 下面是以 ACT_CMMN_RU_ 为前缀的表。代表运行时的数据，这些数据包含案例实例、计划项等的运行时数据。Flowable 仅在案例实例执行期间存储运行时数据，并在案例实例结束时删除记录，这使运行时表保持小且查询速度快。

    • ACT_CMMN_RU_CASE_INST：包含每个已启动但尚未完成的案例实例的条目。

    • ACT_CMMN_RU_MIL_INST：包含作为运行案例实例的一部分达到的每个里程碑的条目。

    • ACT_CMMN_RU_PLAN_ITEM_INST：案例实例执行由案例定义中定义的计划项的多个实例组成，此表包含在案例实例执行期间创建的每个实例的条目。

    • ACT_CMMN_RU_SENTRY_PART_INST：计划项目实例可以有守卫状态转换的哨兵，哨兵在状态改变之前可以包含多个部分，这个表就是专门用来存储这种哨兵。

# ACT_DMN_*

以 ACT_DMN_ 为前缀的表，都是与 Flowable DMN 相关的表。

DMN（Decision Model and Notation）是一种用于描述和建模业务决策过程的标准。Flowable DMN 可以帮助企业有效地管理和执行决策过程，提高业务决策的准确性和效率。

相关表说明：

  • ACT_DMN_DECISION：包含已部署决策表的元数据，并与来自其他引擎的定义相对应。

  • ACT_DMN_HI_DECISION_EXECUTION：包含有关 DMN 决策表执行的审计信息。

下面两张表和 ACT_APP_DEPLOYMENT、ACT_APP_DEPLOYMENT_RESOURCE 表的功能一样。只不过这里部署的是 DMN 相关的实例：

  • ACT_DMN_DEPLOYMENT
  • ACT_DMN_DEPLOYMENT_RESOURCE

# ACT_RU_*

以 ACT_RU_ 为前缀的表，主要存储流程引擎运行时的信息。

相关表说明：

  • ACT_RU_ACTINST：流程实例中的每个活动在此表中都有一行来指示活动的当前状态。

  • Flowable 引擎使用作业表来实现异步逻辑、计时器或历史处理。这些表存储每个作业所需的数据：

    • ACT_RU_JOB

    • ACT_RU_TIMER_JOB

    • ACT_RU_SUSPENDED_JOB

    • ACT_RU_EXTERNAL_JOB

    • ACT_RU_HISTORY_JOB

    • ACT_RU_DEADLETTER_JOB

  • ACT_RU_ENTITYLINK：存储有关实例的父子关系的信息。如：若流程实例启动子案例实例，则此关系存储在此表中。这样可以轻松查询关系。

  • ACT_RU_EVENT_SUBSCR：当流程定义使用事件（信号/消息/等或启动/中间/边界）时，引擎将对该表的引用存储在该表中。这简化了查询哪些实例正在等待某种类型的事件。

  • ACT_RU_EXECUTION：存储流程实例和指向流程实例当前状态的指针（称为执行）。

  • ACT_RU_IDENTITYLINK：此表存储有关用户或组的数据及其与（流程/案例/等）实例相关的角色。该表也被其他需要身份链接的引擎使用。

  • ACT_RU_TASK：此表包含正在运行的实例的每个未完成用户任务的条目。然后在查询用户的任务列表时使用此表。CMMN 引擎也使用此表。

  • ACT_RU_VARIABLE：此表存储与实例相关的变量。 CMMN 引擎也使用此表。

# ACT_HI_*

以 ACT_HI_* 为前缀的表，包含正在运行和已完成的实例的历史数据，这些表的名称遵循其运行时对应的名称。

相关表说明：

  • ACT_HI_ACTINST：历史活动信息。这里记录流程流转过的所有节点，与 ACT_HI_TASKINST 不同的是， ACT_HI_TASKINST 只记录 Task 内容。

  • ACT_HI_ATTACHMENT：历史附件表。

  • ACT_HI_COMMENT：流程的历史评论表。

  • ACT_HI_DETAIL：历史详情表。流程中产生的变量详细，包括控制流程流转的变量，业务表单中填写的流程需要用到的变量等。

  • ACT_HI_ENTITYLINK：历史参与的人员表。

  • ACT_HI_IDENTITYLINK：任务参与者数据表，主要存储历史节点参与者的信息，可能是 Group 也可能是 User。

  • ACT_HI_PROCINST：保存每一个历史流程，创建时就生成，一条流程实例对应一个记录。

  • ACT_HI_TASKINST：记录每一个历史节点，一个 Task 对应一个记录。

  • ACT_HI_TSK_LOG：每一次执行可能会带上数据，存在这里。

  • ACT_HI_VARINST：流程历史变量表。

# ACT_GE_*

以 ACT_GE_ 为前缀的表，主要存储一些通用信息表

相关表说明：

  • ACT_GE_BYTEARRAY：存储每个流程的部署记录，bytes_ 字段中保存流程的具体内容。

  • ACT_GE_PROPERTY：存储 Flowable 自身的一些变量，主要是版本号。

# ACT_RE_*

以 ACT_RE_ 为前缀的表，主要存储流程的定义、流程的资源等等包含了静态信息相关的信息。

相关表说明：

  • ACT_RE_DEPLOYMENT：流程部署记录。每次服务重启，流程都会部署一次，而这里则会新增一条记录。
  • ACT_RE_MODEL：创建模型时，额外定义的一些模型相关信息，存在这张表，默认不保存。
  • ACT_RE_PROCDEF：记录流程的变更，流程每变更一次存一条记录，VERSION_ 字段加 1。

# FLW_EVENT_*

相关表说明：

  • FLW_EVENT_DEFINITION：已部署事件定义的元数据。
  • FLW_EVENT_DEPLOYMENT：已部署事件部署元数据。
  • FLW_EVENT_RESOURCE：事件所需资源。

# 其他表

  • 批量迁移流程时使用的表：
    • FLW_RU_BATCH
    • FLW_RU_BATCH_PART
  • ACT_PROCDEF_INFO：流程定义信息，对流程的说明。
  • ACT_CO_CONTENT_ITEM：每项内容在此表中都有个条目。
  • ACT_EVT_LOG：Flowable 引入了事件日志机制，默认会在数据库中创建 ACT_EVT_LOG 表保存事件日志，如果不使用事件日志，则可以删除这个表。
  • FLW_CHANNEL_DEFINITION：泳池管道定义表。

# 四、流程相关概念

在学习流程之前，我们先来看下这几个概念，防止混淆：

  • 流程定义 ProcessDefinition
    • 一个流程的定义就是指一个 XML 文件，它定义了这个流程的结构。它包括了流程中所有的任务节点以及这些节点之间的顺序和条件。
  • 流程实例 ProcessInstance
    • 在创建流程时，会根据流程定义创建一个流程实例。它代表了流程中的一个具体实例。它包含了流程当前的状态、执行到哪个任务节点、哪些节点已经完成、下一步将要执行哪些节点等信息。
  • 流程部署
    • 流程部署是指将流程定义发布到 Flowable 引擎中，以使得引擎能够根据定义的规则来处理流程实例。
    • 部署的过程包括将流程定义的 XML 文件、BPMN 图像等资源文件发布到引擎的数据库中，并在需要时对它们进行解析和编译。一旦部署成功，该流程定义就可以被实例化并执行。
  • 执行实例 Execution
    • 执行实例是指流程实例中的一个具体的执行过程。它代表了流程实例中的一个任务节点的执行过程。
    • 在一个流程实例中，可能会有多个执行实例，这些执行实例都属于同一个流程实例。执行实例包含了节点的运行状态、节点的入参和出参等信息。

# 1.流程部署

# 1.1 自动部署

在 flowable-spring-boot-starter 中，默认会自动部署 /resources/processes/ 目录下的流程文件。流程文件的后缀默认为 .bpmn20.xml 和 .bpmn。

也可以手动指定自动部署的文件目录和文件后缀。如下在 application.properties 中配置：

# 若为true，则在启动时，将检查文件目录下是否有流程定义文件，并将流程定义文件部署到数据库中
flowable.check-process-definitions=false
# 流程文件的位置，默认为classpath*:/processes/
flowable.process-definition-location-prefix=classpath*:/processes/
# 流程文件的后缀名，默认为**.bpmn20.xml,**.bpmn
flowable.process-definition-location-suffixes=**.bpmn20.xml,**.bpmn

# 1.2 手动部署

我们也可以手动的调用代码进行部署流程，如下代码中，testDeployProcess() 方法将部署一个流程定义文件：

@SpringBootTest
class ProcessDefinitionDeployTest {

  @Autowired
  private RepositoryService repositoryService;

  @ParameterizedTest
  @CsvSource({
          "leave, 请假, 请假流程, leave.bpmn20.xml",
  })
  @DisplayName("部署流程")
  void testDeployProcess(String key, String name, String category, String resourceName) {
    Deployment deploy = repositoryService.createDeployment()
            .key(key)
            .name(name)
            .category(category)
            .addClasspathResource("processes/" + resourceName)
            .deploy();
    log.info("流程部署成功, 流程部署ID: {}, name: {}", deploy.getId(), deploy.getName());
  }

}

addClasspathResource() 方法通过类路径添加资源。可以通过该方法将资源添加到流程定义中。

除了 addClasspathResource() 方法，还有如下方法：

/**
 * 通过输入流添加资源。可以通过该方法将资源以输入流的形式添加到流程定义中。例如：
 * <code>
 *     InputStream is = new FileInputStream("processes/myProcess.bpmn20.xml");
 *     deploymentBuilder.addInputStream("myProcess.bpmn20.xml", is);
 * </code>
 */
DeploymentBuilder addInputStream(String resourceName, InputStream inputStream);

/**
 * 通过类路径添加资源。可以通过该方法将资源添加到流程定义中。例如：
 * <code>
 *     deploymentBuilder.addClasspathResource("processes/myProcess.bpmn20.xml");
 * </code>
 */
DeploymentBuilder addClasspathResource(String resource);

/**
 * 通过字符串添加资源。可以通过该方法将字符串转化为流程定义的形式添加到流程定义中。例如：
 * <code>
 *     String xmlString = "<bpmn2:process>...</bpmn2:process>";
 *     deploymentBuilder.addString("myProcess.bpmn20.xml", xmlString);
 * </code>
 */
DeploymentBuilder addString(String resourceName, String text);

/**
 * 通过二进制字节内容添加资源。可以通过该方法将字符串转化为流程定义的形式添加到流程定义中。例如：
 * <code>
 *     byte[] bytes = "<bpmn2:process>...</bpmn2:process>".getBytes();
 *     deploymentBuilder.addBytes("process.bpmn20.xml", bytes);
 * </code>
 */
DeploymentBuilder addBytes(String resourceName, byte[] bytes);

/**
 * 通过zip包添加资源。可以通过该方法将zip包中的资源添加到流程定义中。例如：
 * <code>
 *     InputStream is = new FileInputStream("processes/processes.zip");
 *     ZipInputStream zipInputStream = new ZipInputStream(is);
 *     deploymentBuilder.addZipInputStream(zipInputStream);
 * </code>>
 */
DeploymentBuilder addZipInputStream(ZipInputStream zipInputStream);

/**
 * 通过BPMN模型添加资源。可以通过该方法将BPMN模型转换为流程定义的形式添加到流程定义中。例如：
 * <code>
 *     BpmnModel bpmnModel = new BpmnModel();
 *     ...
 *     deploymentBuilder.addBpmnModel("myProcess.bpmn20.xml", bpmnModel);
 * </code>
 */
DeploymentBuilder addBpmnModel(String resourceName, BpmnModel bpmnModel);

# 4.相关表

流程部署过程，相关的表有三张：

  • ACT_RE_PROCDEF 流程定义表
  • ACT_RE_DEPLOYMENT 流程部署表
  • ACT_GE_BYTEARRAY 通用数据存储表

ACT_RE_PROCDEF 和 ACT_RE_DEPLOYMENT 表我们上面已经看到了。

ACT_RE_PROCDEF 表中存储定义的流程。

  • DEPLOYMENT_ID_ 字段：该字段和 ACT_RE_DEPLOYMENT 的主键 ID_ 字段关联。ACT_RE_PROCDEF 表和 ACT_RE_DEPLOYMENT表的数据时一对一的关系，即流程定义表的数据和流程部署表的数据是一一对应的。
  • CATEGORY_ 字段：该字段用来表示流程的分类。需要注意的是，这个分类和流程部署的分类不一样。流程部署的分类是在部署时配置的，而流程定义的分类对应的是流程定义 XML 文件中 <definitions> 标签的 targetNamespace 属性。
  • VERSION_ 字段：该字段用来描述记录的版本号。
    • 每次修改流程定义，ACT_RE_PROCDEF 和 ACT_RE_DEPLOYMENT 表均会自动增加一条记录。其中，ACT_RE_PROCDEF 表的 VERSION_ 字段则会自动加 1，这样就可以看到流程定义的历史版本了。
    • 系统是根据流程定义 XML 文件中 <process> 标签的 id 属性来判断是否是同一个流程定义。

ACT_RE_DEPLOYMENT 表中存储的都是我们在部署流程时的相关参数：ID_、NAME_、CATEGORY_ 、KEY_、TENANT_ID_、DEPLOY_TIME_等等。

ACT_GE_BYTEARRAY 表：该表用来存储通用的数据。

  • DEPLOYMENT_ID_ 字段：
    • 这个字段和 ACT_RE_DEPLOYMENT 表的 ID_ 字段相关联。
    • ACT_RE_DEPLOYMENT 表中的一条数据（即一条流程部署记录）对应 ACT_GE_BYTEARRAY 表的两条数据，这两条数据分别是流程部署 XML 文件和记录流程图片。
  • BYTES_ 字段：该字段主要用来保存文件。部署的流程 XML 文件就保存在该字段中。并且在部署时，系统根据流程定义 XML 文件生成一张流程图片，也保存该字段中。
