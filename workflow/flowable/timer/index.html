5.Flowable 定时器

# 一、流程定义定时激活与挂起

# 1.流程定义定时部署

如下代码，将部署一个流程定义，并且使用了 activateProcessDefinitionsOn 指定三分钟后进行激活：

@ParameterizedTest
@CsvSource({
        "test_user_task, 测试用户任务, test_user_task.bpmn20.xml, willingoxjin, 3"
})
@DisplayName("定时部署流程定义")
void testDeploy(String key, String name, String bpmn, String tenantId, long minutes) {
    Date activateDate = new Date(System.currentTimeMillis() + 1000L * 60L * minutes);
    Deployment deployment = repositoryService.createDeployment()
            .key(key)
            .name(name)
            .category("定时器")
            .addClasspathResource("processes/" + bpmn)
            .tenantId(tenantId)
            .activateProcessDefinitionsOn(activateDate) /* 激活流程定义的时间 */
            .deploy();
    log.info("部署成功, deploymentId={}", deployment.getId());
}

activateProcessDefinitionsOn() 方法接受一个时间类型的参数，会在这个指定的时间才激活流程定义。

此时，在 ACT_RU_TIMER_JOB 表，就会发现该表中有一条定时任务的执行计划的数据：

ACT_RU_TIMER_JOB 表中有一个 DUEDATE_ 字段，该字段表示定时任务执行的时间。当到达这个时间时，就会执行定时任务，将 ACT_RE_PROCDEF 表中的 SUSPENSION_STATE_ 的值修改为 1 （原来为 2）。

这说明流程定义部署的时候，是先暂时挂起的。等到定时任务执行之后，才会将流程定义激活。

如果我们调用上面代码后，在定时任务激活的时间之前使用 RuntimeService.startProcessInstanceByKey 方法启动流程，则控制台就会打印出“流程定义挂起”的异常错误：

org.flowable.common.engine.api.FlowableObjectNotFoundException: No process definition found for key 'test_timer'

# 2.流程定义定时挂起

如下代码，将在 3 分钟后挂起流程定义，并挂起流程定义对应的所有实例：

@ParameterizedTest
@CsvSource({
        "test_timer, willingoxjin, 3"
})
@DisplayName("定时挂起流程定义")
void testSuspend(String key, String tenantId, Integer minutes) {
    Date suspensionDate = new Date(System.currentTimeMillis() + 1000L * 60L * minutes);
    repositoryService.suspendProcessDefinitionByKey(key, true, suspensionDate, tenantId);
}

RepositoryService#suspendProcessDefinitionByKey 参数：

  • processDefinitionKey：流程定义 key
  • suspendProcessInstances： 是否挂起该流程定义下的实例。
  • suspensionDate：挂起的时间。如果为空，则立即挂起。
  • tenantId：租户 id

上面代码执行之后，就会在 ACT_RU_TIMER_JOB 表中新增一条定时任务执行计划的数据，并在 3 分钟后（指定挂起流程实例的时间），自动挂起流程定义及相关的流程实例。

# 3.流程定义定时激活

如下代码，将在 3 分钟后激活流程定义，并激活流程定义对应的所有实例：

@ParameterizedTest
@CsvSource({
        "test_timer, willingoxjin, 3"

@DisplayName("定时激活流程定义")
void testActivate(String key, String tenantId, Integer minutes) {
    Date activationDate = new Date(System.currentTimeMillis() + 1000L * 60L * minutes);
    repositoryService.activateProcessDefinitionByKey(key, true, activationDate, tenantId);
}

# 二、定时任务执行的过程

# 1.定时任务的表

需要注意的是，虽然 ACT_RU_TIMER_JOB 表存储了定时任务的信息，但是定时任务执行的时候，并非是从 ACT_RU_TIMER_JOB 表中查询数据并进行执行的。而是将 ACT_RU_TIMER_JOB 表中的数据移动到 ACT_RU_JOB 表中，然后定时器会从 ACT_RU_JOB 表中查询出数据，然后执行任务。

# 2.手动执行案例

首先，为了测试，我们先把流程定义和流程实例全部挂起：

        "test_timer, willingoxjin"
})
@DisplayName("挂起流程定义")
void testSuspend(String key, String tenantId) {
    repositoryService.suspendProcessDefinitionByKey(key, true, null, tenantId);

如果想取消某个定时任务，可以通过以下方式将定时任务移入死信队列：

@ParameterizedTest
@CsvSource({
        "e79d45c7-f871-11ed-bcf6-465aefa58ff4"
})
@DisplayName("取消执行定时任务（将定时任务放入死信队列）")
void testMoveJobToDeadLetterJob(String jobId) {
    managementService.moveJobToDeadLetterJob(jobId);
}

已经移入私信队列的定时任务，也可以通过如下方式移出来并直接执行。

@ParameterizedTest
@CsvSource({
        "e79d45c7-f871-11ed-bcf6-465aefa58ff4"
})
@DisplayName("将定时任务从死信队列移动到执行队列，并立即执行（即使时间没到也会马上执行）")
void testMoveDeadLetterJobToExecutableJob(String jobId) {
    managementService.moveDeadLetterJobToExecutableJob(jobId, 2);
}
