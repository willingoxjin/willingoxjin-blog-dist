3.Flowable 任务

# 一、用户任务

用户任务（UserTask），顾名思义，就是需要用户人工干预的任务。当流程执行到达用户任务时，会为指派至该任务的用户或组的任务列表创建一个新任务。

这里我们继续沿用之前的请假审批的流程来作为示例。

# 1.设置任务处理人

给用户任务设置任务处理人，意思就是把任务交给谁来处理。

给“用户任务”设置用户有以下方式：

  • 直接指定

    1. 分配给流程发起人

    2. 分配给单个用户

    3. 候选用户：指定多个用户，用户仅领取了任务才进行，才可进行处理。

    4. 候选组：指定多个组，这些组下的用户领取了任务，才可进行处理。

  • 动态指定

    1. 通过变量指定
    2. 通过监听器指定

# 1.1 分配给流程发起人

任务是由谁发起的，就由谁来处理该任务。

设置后，生成的流程定义 XML 文件中的 UserTask 的 flowable:assignee 属性就自动设置成嘞 $INITIATOR：

# 1.6 通过监听器指定

动态指定任务处理人，也可以使用监听器指定。

通过监听器指定任务处理人，就不需要给任务设置分配用户。而是在任务监督器中配置我们的 Java 代码。

首先，我们来编写任务监听器的 Java 代码：

package org.willingoxjin.flowable.listener;

import org.flowable.engine.delegate.TaskListener;
import org.flowable.task.service.delegate.DelegateTask;

public class UserManagerTaskListener implements TaskListener {

    @Override
    public void notify(DelegateTask delegateTask) {
        delegateTask.setAssignee("zhaoyun");
    }

}

ACT_RU_IDENTITYLINK 表的 TYPE_ 的可选值如下：

@ParameterizedTest

            .list();

# 2.4 修改任务候选人或候选组

TaskService 中有如下方法，可以对候选人或候选组进行删除或新增操作：

  • deleteCandidateUser
  • addCandidateUser
  • deleteCandidateGroup
  • addCandidateGroup

以下方法可以删除或新增：任务处理人、候选人、任务拥有人、启动任务人、会签人、或签人。只需要传递 identityLinkType 参数即可。

  • deleteUserIdentityLink(String taskId, String userId, String identityLinkType)
  • addUserIdentityLink(String taskId, String userId, String identityLinkType)

identityLinkType 参数的可选值在 org.flowable.identitylink.api.IdentityLinkType 中可以查看。

方法使用示例：

@ParameterizedTest
@CsvSource({
        "holiday_request, zhaoyun, zhugeliang"
})
@DisplayName("修改任务候选人")
void testAddCandidateUser(String processDefinitionKey, String userId1, String userId2) {
    List<Task> list = taskService.createTaskQuery()
            .processDefinitionKey(processDefinitionKey)
            .taskAssignee(userId1)
            .list();
    list.forEach(task -> {
        log.info("taskId: {}, taskName: {}", task.getId(), task.getName());
        // 删除候选人
        taskService.deleteCandidateUser(task.getId(), userId1);
        // 添加候选人
        taskService.addCandidateUser(task.getId(), userId2);
    });
}

# 二、脚本任务

ScriptTask 即脚本任务，是自动执行的活动。当流程执行到达脚本任务时，会执行相应的脚本。

Flowable 的脚本任务支持 JavaScript、Groovy、Juel、Python 等语言。

如果想要知道当前项目有哪些脚本的实现，可以查看 javax.script.AbstractScriptEngine 的子类。该类的子类实现都是遵循 JSR-223 规范的。反过来说，只要是引入实现了 JSR-223 规范的依赖的脚本，Flowable 就支持该脚本的执行。

TIPS

如果你想知道使用某个脚本，但默认没有相关的实现，就可以下载 flowable-engine 的源码，然后查看 AbstractScriptEngine 类的子类都有哪些实现类，并查看实现类项目的 maven 坐标 id 即可。

# 四、接收任务

# 1.接收任务介绍

ReceiveTask 也称为接收任务或等待任务，ReceiveTask 可以算 Flowable 中最简单的一种任务，当任务到达 ReceiveTask 节点时，它不做任何处理，而是被动的等待用户 Trigger。

ReceiveTask 任务适用于一些不明确的阻塞，如一个复杂计算需要等待很多个条件，这些条件需要人为判断是否可以执行。如果用户认为条件满足了，可以继续执行，则 Trigger 一下就可以使流程继续往后执行。

# 五、多实例任务

# 1.多实例

多实例是一种流程行为，它允许在流程中多次重复执行一个任务。这个任务可以是用户任务或脚本任务等。多实例任务可以在流程中创建多个实例，并且每个实例都可以有自己的输入和输出数据。在多实例任务中，可以使用集合或表达式来定义实例数量和迭代条件。这种行为可以用于处理需要重复执行的任务，例如审核、审批或通知等。

定义看起来不是那么好理解，下面我们通过会签与或签来一起玩一下多实例任务。

# 2.会签

会签在流程中的意思是，多个用户需要对同一任务审批，仅所有的用户都审批通过时，才算通过。例如有一个用户请假审批，需要直接主管、人事、老板都同意之后才算审批通过。

上面的配置说明：

  • 多实例类型：这里选项有两个：Parallel 和 Sequential。
    • Parallel 表示任务是并发执行。
    • Sequential 表示任务是顺序执行（多个用户之间任务执行有先后顺序）。
  • 集合(多实例)：表示当流程执行到这个节点时，会使用某个变量来表示多实例任务，这个变量是一个集合。
    • 例如，这里配置的是 ${checkResultTasks}，表示 checkResultTasks 变量包含的实例任务。这个变量后面我们会传所有要审批这个任务节点的用户 id。
  • 元素变量(多实例)：和集合(多实例)相对应，这里相当于集合的单个元素变量。可以理解为 Java 中增强 for 循环中的单个 item 的元素名称。
  • 完成条件(多实例)：这里配置的是${nrOfCompletedInstances == nrOfInstances}，表示完成的任务等于总任务数才算这个任务节点完成。 nrOfCompletedInstances 和 nrOfInstances ，是 Flowable 内置的变量多实例相关的内置变量如下：
    • nrOfCompletedInstances 表示完成任务的实例个数。
    • nrOfInstances 表示实例的总个数。
    • nrOfActiveInstances 表示未完成审批的实例个数。
  • 分配用户：任务的执行用户 id，即将这个任务分配给哪个用户。

然后，我们给”发送请假成功通知“服务任务，设置表达式为 ${sendRequestHolidayMsgTask.sendAcceptMsg()}，表示流程走到这个节点时，自动执行 sendRequestHolidayMsgTask.sendAcceptMsg() 方法。

给”发送请假失败通知“服务任务，设置表达式为 ${sendRequestHolidayMsgTask.sendRejectMsg()}，表示流程走到这个节点时，自动执行 sendRequestHolidayMsgTask.sendRejectMsg() 方法。。

# 3.或签

会签在流程中的意思是，多个用户需要对同一任务审批，仅需要某个用户审批通过，就算任务通过。例如有一个用户请假审批，只需要直接主管、人事、老板中的其中一人同意，就算审批通过。

这里不在进行示例，或签其实就是将上面会签中流程判断的条件改为 ${nrOfCompletedInstances >= 1}。

会签和或签不一定非要使用 nrOfCompletedInstances 、nrOfInstances 变量判断，我们也可以通过自定义变量实现，实现起来是非常灵活的。

# 六、评论

# 1.评论介绍

评论是一个用于在流程或任务中添加注释或评论的文本字段。 评论可以包含任何相关信息，例如任务的说明、用户反馈或其他有关流程的补充说明。在任务中添加评论可以帮助其他参与者更好地理解任务的背景和要求，从而更好地完成任务。

有关任务评论的数据存储在 ACT_HI_COMMENT 表中。

ACT_HI_COMMENT 表不仅存储 Comment 评论数据，还会存储历史的 Attachment 附件、UserLink 用户链接：

  • Attachment：用于向任务或流程实例的附件。
  • UserLink：用于将任务或流程实例链接到特定的用户。

该表中的数据，若为评论数据，则 _TYPE=comment；若为附件和用户链接数据，则 _TYPE=event。这些数据根据表中的 ACTION_ 字段区分，ACTION_具体的类型见 org.flowable.engine.task.Event。

# 2.评论相关 API

评论相关的方法都在 TaskService 中：

  • addComment 添加评论
  • saveComment 保存评论（修改评论）
  • getComment 获取评论信息
  • getTaskComments 获取某个任务的评论列表
  • getProcessInstanceComments 获取某个实例的评论列表
  • deleteComments 删除评论
  • ……
