1.Flowable UI

# 一、flowable-ui 安装启动

# 1.war 包方式启动

到 github 中下载编译好的压缩包： https://github.com/flowable/flowable-engine/releases/ (opens new window)。

下载 flowable-x.y.z.zip，解压之后，可以在路径 flowable-6.8.0/wars/flowable-ui.war 下看到 war 包。

然后直接启动即可：

java -jar --server.port=8888 flowable-ui.war
1

# 2.docker 方式启动

更加推荐 docker 的方式启动，方便快捷。

如下命令将启动一个 flowable-ui 的服务：

docker run \
-d \
-p 8888:8080 \
--name flowable-ui \
flowable/flowable-ui

Flowable UI 默认使用的是 h2 数据库，如果想要使用 MySQL 作为数据库，那么需要添加 MySQL 的 JDBC 驱动映射到镜像，并在 Docker 启动参数中添加数据库相关的环境变量。

切换到其他数据库也是一样的逻辑，只需要添加数据库对应的 JDBC，并修改相关数据库配置即可。

为了方便这里使用 docker-compose，新建 flowable-ui.yml 文件

version: '3'
services:
  flowable-ui:
    image: flowable/flowable-ui:6.8.0
    container_name: flowable-ui
    # restart: always
    volumes:
      - "./lib/mysql-connector-j-8.0.33.jar:/app/WEB-INF/lib/mysql-connector-j-8.0.33.jar"
    environment:
      - spring.datasource.driver-class-name=com.mysql.cj.jdbc.Driver
      - spring.datasource.url=jdbc:mysql://host.docker.internal:3306/flowable_test?useUnicode=true&characterEncoding=UTF-8&useSSL=false
      - spring.datasource.username=root
      - spring.datasource.password=123456
      # 当Flowable发现库与数据库表结构不一致时，会自动将数据库表结构升级至新版本
      - flowable.database-schema-update=true
    ports:
      - "8888:8080"

# 二、flowable-ui 的使用

flowable-ui 有以下应用：

  • Workflow （也可以叫做 Task）：任务应用，提供了启动流程实例、编辑任务表单、完成任务、查询任务、查询流程实例等功能。
  • Modeler：建模器应用，提供了创建流程模型、表单、选择表、应用定义等功能。
  • Admin：管理员应用，提供了查询 BPMN、DMN、Form、Content 引擎的查询功能，并提供了许多选项用于修改流程实例、任务、作业等应用。可通过 REST API 连接至引擎，并与 Flowbale Task 应用和 Flowbale REST 应用一同部署。
  • IDM：身份管理应用，对用户、组、应用（Workflow、Modeler、Admin、IDM、REST API）权限授权等功能的管理。

在编辑弹窗的一般菜单中，相关配置说明如下：

  • 标签：文本框展示的信息（相当于 HTML 中的 <label> 标签）
  • 覆盖id？：若不选标签和id的值是一样的；若选中后，就可以自定义id。
  • id：组件的唯一标识，可以用在代码中或表达式判断中。如果我们想要在顺序流中获取到这个组件的值，就可以根据id获取。
  • 必填：该字段内容是否是必填
  • 只读：该字段内容是否是只读
  • 默认值：相当于 HTML 表单标签（如 input、textarea 等）的 placeholder 属性

# 2.5 互斥网关——判断直接主管是否同意请假

为了方便，这里先把所有的流程都画出来后，再解释相关的配置。

添加互斥网关，然后配置网关后面的连线：

  • “同意”连线
    • 条件表达式： ${check_result=="同意"}
  • “拒绝”连线
    • 条件表达式： ${check_result=="同意"}

这里的 check_result 变量就是我们上一步（直接主管审批的用户任务）中的表单配置的，主管审核后，完成任务，就能够获得 check_result 变量。

# 2.6 脚本任务——计算请假时长

如果直接主管同意了请假申请，就会走到这一步。否则就走到结束事件，请假失败。

我们添加一个脚本任务，并设置脚本类型设置为 groovy。

脚本内容如下：

org.joda.time.Days days = org.joda.time.Days.daysBetween(startDate, endDate);
execution.setVariable("amountOfVacationDays", days.getDays());
1
2

脚本逻辑是计算请假天数，并且设置变量。在这里设置了 amountOfVacationDays 变量，在后面的判断中就可以使用该变量了。

这里的脚本支持 JavaScript、Groovy、Juel 等格式。

# 2.7 互斥网关——根据请假时长判断是否需要多级主管审批

在计算请假时长任务后面，添加互斥网关，网关后的连线如下：

  • 小于等于三天：
    • 流条件：${amountOfVacationDays <= 3}
  • 大于三天：
    • 流条件：${amountOfVacationDays > 3}

这里的 amountOfVacationDays 变量就是我们在计算请假时长的脚本任务中设置的。

如果请假天数小于等于三天，则直接流程完成，请假成功。否则需要更高一级的主管审批。

这里将多级主管审批的“分配用户”设置为候选组——丞相。

多级主管审批的表单实际上和直接主管审批是相同的，为了学习更多的功能，这里我们不再自定义表单字段。

先添加表单，名称为：“请假—多级主管审批表单”，key 为“holiday_request_manager2_check”

然后在表单的结果中添加可能的结果。如下图：

  1. 互斥网关——判断上级的上级是否同意请假

多级主管审批任务后面，添加互斥网关，网关后的连线如下：

  • 同意
    • 流条件：${form_holiday_request_manager2_check_outcome=="同意"}
  • 拒绝
    • 流条件：${form_holiday_request_manager2_check=="同意"}

这里的 form_holiday_request_manager2_check_outcome 变量，是多级主管审批任务的输出结果。这个输出结果的命名规则为：form_表单key_outcome。
