02.Hadoop 之 HDFS

# 一、HDFS 介绍

HDFS（Hadoop Distributed File System）是一种允许文件通过网络在多台主机上分享的文件系统，可以让多台机器上的多个用户分享文件和存储空间。

分布式文件系统的典型实现：

  • HDFS：适合大文件存储，不适合小文件存储
  • GFS（谷歌）
  • TFS（淘宝）
  • S3（亚马逊）

HDFS 支持主从结构，主节点被称为 NameNode，从节点称为 DataNode。NameNode 和 DataNode 都支持多个节点。

HDFS 中还包含一个 SencondaryNameNode 进程。

# 2.编写代码

主要使用 FileSystem 这个类的方法来操作文件

import org.apache.hadoop.conf.Configuration;
import org.apache.hadoop.fs.FSDataOutputStream;
import org.apache.hadoop.fs.FileSystem;
import org.apache.hadoop.fs.Path;
import org.apache.hadoop.io.IOUtils;

import java.io.FileInputStream;
import java.net.URI;

public class HdfsOperatorDemo {
    private static void put(FileSystem fileSystem, String hdfsPath, String localPath) throws IOException {
        // 获取HDFS文件系统的输出流
        FSDataOutputStream fos = fileSystem.create(new Path(hdfsPath));
        // 获取本地文件的输入流
        FileInputStream fis = new FileInputStream(localPath);

        // 上传文件：把输入流拷贝到输出流，实现本地文件上传到HDFS
        // 也可以使用
        IOUtils.copyBytes(fis, fos, 1024, true);
    }

    private static void get(FileSystem fileSystem, String hdfsPath, String localPath) throws IOException{
        // 获取HDFS文件系统的输入流
        FSDataInputStream fis = fileSystem.open(new Path(hdfsPath));
        // 获取本地文件的输出流
        FileOutputStream fos = new FileOutputStream(localPath);
        // 下载文件
        IOUtils.copyBytes(fis,fos,1024,true);
    }

    public static void main(String[] args) throws Exception {
        // 设置连接hadoop的用户名
        System.setProperty("HADOOP_USER_NAME", "root");
        // 创建配置对象
        Configuration conf = new Configuration();
        // 指定HDFS的地址
        conf.set("fs.defaultFS", "hdfs://bigdata01:9000");
        // 设置通过域名访问datanode
        // conf.set("dfs.client.use.datanode.hostname", "true");
        // 获取操作HDFS的对象
        FileSystem fileSystem = FileSystem.get(conf);

        // 创建文件夹
        fileSystem.mkdirs(new Path("/test/abc"));
        // 上传-方式一
        put(fileSystem, "/test/java/hello.txt", "/Users/willingoxjin/tmp/hello.txt");
        // 上传-方式二
        fileSystem.copyFromLocalFile(new Path("/Users/willingoxjin/tmp/hello.txt"), new Path("/test/java/a.txt"));

        // 下载-方式一
        get(fileSystem, "/test/java/hello.txt", "/Users/willingoxjin/tmp/test.txt");
        // 下载-方式二
        fileSystem.copyToLocalFile(new Path("/test/java/hello.txt"), new Path("/Users/willingoxjin/tmp/test2.txt"));

        // 删除文件或目录，第二个参数为true则会递归删除
        boolean isDeleted = fileSystem.delete(new Path("/test/java/a.txt"), false);
        System.out.println("删除结果：" + isDeleted);
    }
}

# 四、NameNode

# 1.NameNode 介绍

NameNode 是整个文件系统的管理节点。它主要维护整个文件系统的文件目录树，文件/目录的信息 和每个文件对应的数据块列表，并且还负责接收用户的操作请求。

  • 目录树：表示目录之间的层级关系，即在 HDFS 上执行 ls 命令看到的目录结构信息。
  • 文件/目录的信息：表示文件/目录的的一些基本信息，所有者 属组 修改时间 文件大小等信息
  • 每个文件对应的数据块列表：如果一个文件太大，那么在集群中存储的时候会对文件进行分片，即将文件分成一块一块的，存储到不同机器上面。因此 HDFS 还需要要记录每个文件有多少个块，每个块存储都在什么位置

我们一起在集群的 web 界面（http://bigdata01:9870/）中打开 Browse the file system 页面（或直接访问：http://bigdata01:9870/explorer.html）

点击文件名称，可以看到Block information 但是文件太小，只有一个块 叫 Block 0。因为这个文件比较小，所以只被分了一个块。

我们把 hadoop 的安装包上传上去，再查看分片情况：

这些 Block 的分片信息就是在 NameNode 中进行维护。

# 2.NameNode 的文件

由于文件信息都在 NameNode 上维护和存储，因此我们在命令行使用 hdfs 操作文件时，会先和 NameNode 通信之后，才会去操作数据的。

NameNode 主要包括以下文件：

  • fsimage 文件的元数据信息，记录文件分块存储的信息
  • edits 事务文件，记录文件操作的状态。当文件在上传的时候使用 ls 命令查看是查看不到的
  • seen_txid 保存 edits 文件的最大编号。
  • VERSION 集群版本信息

这些文件所在的路径是由 hadoop-3.x.x/share/hadoop/hdfs/hadoop-hdfs-3.x.x.jar （jar 包需要解压）中的 hdfs-default.xml 的 dfs.namenode.name.dir 属性配置决定，默认配置如下：

<property>
  <name>dfs.namenode.name.dir</name>
  <value>file://${hadoop.tmp.dir}/dfs/name</value>
  <description>Determines where on the local filesystem the DFS name node
      should store the name table(fsimage).  If this is a comma-delimited list
      of directories then the name table is replicated in all of the
      directories, for redundancy. </description>
</property>

# seen_txid 文件

seen_txid 文件用来存储文件编号。

NameNode 启动的时候会把文件中的元数据信息加载到内存中，按照文件名最后的编码从小到大顺序读取文件，直到读取到 seen_txid 中保存的编号为止。

HDFS 不适合存储小文件

由于 NameNode 启动的时候会把文件中的元数据信息加载到内存中，每一个文件的元数据信息会占用 150 字节的内存空间，即文件的元数据信息均为 150 字节，和文件大小没有关系。

不管是大文件还是小文件，一个文件的元数据信息在 NameNode 中都会占用 150 字节，NameNode 节点的内存是有限的，所以它的存储能力也是有限的，如果我们存储了很多的小文件，最后发现NameNode 的内存占满了，确实存储了很多文件，但是文件的总体大小却很小。

# 3.SencondaryNameNode

SencondaryNameNode 服务会定期把 edits 文件中的信息转换后，生成新的内容合并到 fsimages 文件中。这个操作称为 checkpoint。

SencondaryNameNode 只做这一件事。实际生产环境一般都将 SencondaryNameNode 独立部署一台单独的机器上。

在 NameNode 的 HA 架构中（多个 NameNode 服务存在，会有一个主 NameNode），是没有 SencondaryNameNode 进程的，edits 文件的信息转换合并操作将由 standby NameNode （备用的 NameNode）负责实现。

# Block

Block 的概念

  • HDFS 会按照固定的大小，顺序对文件进行划分并编号，划分好的每一个块称一个 Block，HDFS 默认 Block 大小是 128MB

  • Block 块是 HDFS 读写数据的基本单位，不管文件的类型是文本、视频 或音频，针对 HDFS 而言都是字节。

  • HDFS 中，若一个文件小于一个 Block 块的大小，那么并不会占用整个数据块的存储空间，其占用空间的大小就等于文件的真实大小。

Block 块存放在哪些 DataNode 上，只有 DataNode 自己知道，当集群启动的时候，DataNode 会扫描自己节点上面的所有 Block 块信息，然后把节点和这个节点上的所有 Block 块信息告诉给 NameNode。这个关系是每次重启集群都会动态加载。因此集群数据越多，启动就会越慢

# 七、HDFS的高可用和高扩展

# 1.HDFS的高可用（HA）

NameNode 负责接收用户的操作请求，所有的读写请求都会经过它，但是 NameNode 如果出现问题或服务宕机就会造成整个集群服务不可用。

对于此问题，我们可以使用多个 NameNode 服务来解决这个问题。NameNode 是支持多个的服务高可用的，支持一个主节点服务，多个备用节点服务。

HDFS的高可用，简称为 HA（High Available）架构图：

HDFS 的高可用，指的是在一个集群中存在多个 NameNode，分别运行在独立的物理节点上。在任何时间点，只有一个 NameNode 是处于 Active 状态，其它的是处于 Standby 状态。 Active NameNode 负责所有的客户端的操作，而 Standby NameNode 用来同步 Active NameNode 的状态信息，以提供快速的故障恢复能力。

Active NameNode 与 Standby NameNode 节点状态同步

  • 为了保证 Active NameNode 与 Standby NameNode 节点状态同步，即元数据保持一致。需要 DataNode需要向这些 NameNode 发送Bblock 位置信息，并且还构建了一组独立的守护进程 JournalNodes，用来同步 Edits 信息。
  • 当 Active NameNode 执行任何有关命名空间的修改，它需要持久化到一半以上的 JournalNodes 上。
  • Standby NameNode 负责观察 JournalNodes 的变化，读取从 Active NameNode 发送过来的 Edits 信息，并更新自己内部的命名空间。
  • 一旦 Active NameNode 遇到错误，Standby NameNode 需要保证从JNs中读出了全部的 Edits，然后切换成 Active 状态，如果有多个 Standby NameNode，还会涉及到选主的操作，选择一个切换为 Active 状态。

Active NameNode 与 Standby NameNode 节点状态同步的元数据包含两块，一个是静态的，一个是动态的：

  • 静态的是 fsimage 和 edits。
    • 其实 fsimage 是由 edits 文件合并生成的，所以只需要保证edits 文件内容的一致性。
    • 这个就是需要保证多个 NameNode 中 edits 文件内容的事务性同步。这块的工作是由 JournalNodes 集群进行同步的
  • 动态数据是指 Block 和 DataNode 节点的信息
    • 当 DataNode 启动的时候，上报数据信息的时候需要向每个NameNode都上报一份。
    • 这样就可以保证多个 NameNode 的元数据信息都一样了，当一个 NameNode down 掉以后，立刻会从 Standby NameNode 中选择一个进行接管。

注意

使用 HA 的时候，不能启动 SecondaryNameNode，否则会出现问题。之前是 SecondaryNameNode 负责合并 edits 到 fsimage 文件 那么现在这个工作被 Standby NameNode 负责了。

Active NameNode 可以自动切换，也可以手工切换。如果想要实现自动切换，就要使用 zookeeper 集群。

使用 zookeeper 集群自动切换的原理：

当多个 NameNode 启动的时会向 zookeeper 中注册一个临时节点，当 NameNode 挂掉的时候，这个临时节点也就消失了。此时，zookeeper 就会有一个 watcher 监视器监视到，就知道这个节点 down 掉了，然后会选择一个其他的 NameNode 节点转为 Active，把 down 掉的节点转为 Standby。

# 2.HDFS的高扩展（Federation）

之前提到 NameNode 节点的内存是有限的，只能存储有限的文件个数，那使用一个主 NameNode 服务，多个备用 NameNode 服务可以解决这个问题吗？

不能。一个主 NameNode，多个备用的NameNode的方案只能解决NameNode的单点故障问题，无法解决单个 NameNode 内存不够用的问题。

解决这个问题，可以使用官方提供的 Federation 机制，翻译为“联邦”，它可以解决单节点内存不够用的情况，这个就是 HDFS 的高扩展。

HDFS Federation 使用多个 NameNode，每个 NameNode 负责一个命令空间。

使用 Federation 的好处：

  • HDFS 集群扩展性。多个 NameNode 分管一部分目录，使得一个集群可以扩展到更多节点，不再因内存的限制制约文件存储数目。
  • 性能更高效。多个 NameNode 管理不同的数据，且同时对外提供服务，将为用户提供更高的读写吞吐率。
  • 良好的隔离性。用户可根据需要将不同业务数据交由不同 NameNode 管理，这样不同业务之间影响很小。

一般 Federation 都会和 HA 结合使用，如下图：

# 八、HDFS 源码读写数据分析

# 1.读数据

从 HDFS 中读数据的过程：

  1. 客户端向 NameNode 请求下载指定文件，NameNode 获取文件的元数据信息（主要是 Block 的存放位置信息）并且返回给客户端。
  2. 客户端根据返回的 Block 位置信息，请求对应的 DataNode 节点，获取数据。
  3. DataNode 会向客户端传输数据，客户端会以 Packet 为单位接收，先在本地缓存，然后写入对应的文件。

具体到源码中的流程：

  1. 首先调用 FileSystem 对象的 open 方法，其实是一个 DistributedFileSystem 的实例。
  2. DistributedFileSystem 通过 RPC 请求获得文件的第一个 Block 的 Locations，同一个 Block 按照副本数会返回多个 Locations，这些 Locations 按照 Hadoop 集群网络拓扑结构排序，距离客户端近的排在前面。
  3. 前两步会返回一个 FSDataInputStream 对象，该对象会被封装成 DFSInputStream 对象，DFSInputStream 可以方便的管理 DataNode 和 NameNode 数据流。客户端调用 read 方法，DFSInputStream 会找出离客户端最近的 DataNode 并连接。
  4. 数据从 DataNode 源源不断的流向客户端。
  5. 如果第一块（Block）的数据读完了，就会关闭指向第一块的 DataNode 连接，接着读取下一块。这些操作对客户端来说是透明的，客户端的角度看来只是读一个持续不断的流。
  6. 如果第一批 Block 都读完了，DFSInputStream 就会去 NameNode 获取下一批 Blocks 的 Location，然后继续读，如果所有的块都读完，这时就会关闭掉所有的流。

注意：如果在读数据的时候，DFSInputStream 和 DataNode 的通讯发生异常，就会尝试与正在读的 Block 的排第二近的 DataNode 进行通讯，并且会记录哪个 DataNode 发生错误，剩余的 Blocks 读的时候就会直接跳过该 DataNode。DFSInputStream 也会检查 Block 数据校验和，如果发现一个坏的 Block，就会先报告到 NameNode 节点，然后 DFSInputStream 在其他的 DataNode 上读该 Block 的镜像。

什么是 RPC ？

RPC（Remote Produre Call）翻译中文为远程过程调用。RPC 是一种通过网络从远程计算程序上请求服务，而不需要了解底层网络技术的协议。

RPC 采用客户端（client）/服务端（server）模式。RPC 的请求是跨主机跨进程的。

这样设计的原则就是客户端直接连接 DataNode 来检索数据，NameNode 来负责为每一个 Block提供最优的 DataNode，NameNode 仅仅处理 Block Location 的请求，这些信息都加载在 NameNode 的内存中，HDFS 通过 DataNode 可以承受大量客户端的并发访问。

# 2.写数据

向 HDFS 中写数据的过程：

  1. 客户端首先向 NameNode发送上传文件的 RPC 请求，NameNode 会做一些基础检查，例如：目标文件是否已存在/是否允许覆盖、父目录是否存在（不存在创建目录）、名称长度是否合法、NameNode的当前状态、是否处于安全模式、当前请求的用户是否有权限等。

  2. NameNode如果检查通过，则会返回该文件的所有 Block 块及对应存储的 DataNode 列表，否则，抛异常结束。 Block 块及对应存储的 DataNode 列表示例：

    blk-001:bigdata01,bigdata02,bigdata03
blk-002:bigdata01,bigdata02,bigdata03
    1
    2

  3. 客户端收到 Block 块及对应存储的 DataNode 列表之后，就开始向对应的节点中上传数据。

  4. 客户端首先上传第一个 Block 块：blk-001，它对应的 DataNode 列表是 bigdata01,bigdata02,bigdata03，所以客户端就把 blk-001 先上传到 bigdata01，bigdata01 再把 blk-001 上传到 bigdata02，bigdata02 再把 blk-001 上传到 bigdata03。

  5. 客户端接着再上传第二个 Block 块：blk-002，流程同上。

  6. 当所有的 Block 块都传完以后，客户端会给 NameNode 返回一个状态信息，表示数据已经全部写入成功，或者失败。

  7. NameNode 根据客户端返回的状态信息来判断本次写入数据是成功还是失败，如果成功，则需要对应更新元数据信息。

​ 流程如下图：

具体到源码的流程如下：

  1. 客户端通过调用 DistributedFileSystem 类的 create 方法创建新文件
  2. DistributedFileSystem 通过 RPC 请求调用 NameNode 去创建一个没有 Blocks 关联的新文件。创建之前，NameNode 会做各种校验，比如文件是否存在，客户端有无权限去创建等。如果校验通过，NameNode 就会记录下新文件，否则就会抛出 IO 异常。
  3. 前两步结束后会返回 FSDataOutputStream 类型的对象，FSDataOutputStream 被封装成 DFSOutputStream。DFSOutputStream 可以协调 NameNode 和 DataNode。客户端开始通过 DFSOutputStream 写数据，DFSOutputStream 会把数据切成一个个的小 Packet，然后排成队列（Data Quene）。
  4. 向队列中写数据 会调用 DataStreamer 类。DataStreamer 会去处理接收 Data Queue，它会先问询 NameNode 这个新的 Block 最适合存储的在哪几个 DataNode 里，比如副本数是 3，那么就找到 3 个最适合的 DataNode（会用到机架感知策略选择最适合的 DataNode），然后把 DataNode 排成一个管道（pipeline）。DataStreamer把 Packet按队列输出到管道的第一个 DataNode 中，第一个 DataNode 又把 Packet 输出到第二个 DataNode中，以此类推。
  5. DFSOutputStream 还有一个队列叫 Ack Queue，也是由 Packet组成，其会等待收到 DataNode 的响应，当 Pipeline 中的所有 DataNode 都表示已经收到的时候，这时 Ack Queue 才会把对应的 Packet 包移除掉。
  6. 客户端完成写数据后调用 close 方法关闭写入流。
  7. DataStreamer 把剩余的 Packet 都刷写到 Pipeline 里然后等待 Ack 信息，收到最后一个 Ack 后，通知 NameNode 把文件标记为已完成。

注意

客户端执行 write 操作后，写完的 Block 才是可见的，正在写的 Block 对客户端是不可见的，只有调用 sync 方法，客户端才确保该文件被写操作全部完成，当客户端调用 close 方法时会默认调用 sync 方法。

在 DFSClient（客户端）向 HDFS 中写数据的过程中，有3个数据单位：Block、Packet 和 Chunk。

  • Block 是最大的一个单位，它是最终存储于DataNode上的数据粒度，默认是128M。
  • Packet 是中等的一个单位，它是数据由DFSClient流向DataNode的粒度，默认是64K。
  • Chunk 是最小的一个单位，它是DFSClient到DataNode数据传输中进行数据校验的粒度，默认是512B。

租约（Lease）

  • 在 HDFS 中，客户端往某个文件写入数据时，为了保持数据的一致性，是不允许其他的客户端向此文件中同时写数据的。为了保证这一点，HDFS 使用租约来限制数据的写入。
  • 租约就是 HDFS 给予客户端写数据的一个临时许可，没有获得租约就不能允许写数据操作。

HDFS 的机架感知策略

客户端在上传文件的时候会请求 NameNode，NameNode 会返回所有 Block 块及对应存储的 DataNode列表，其中返回的 DataNode 列表中的节点信息的选择依据是根据机架感知策略而得出来的。

流程如下：

  • 第 1 个副本放在和客户端相同机架的节点上（如果客户端不在集群范围内，第 1 个节点是随机选取不太满或者不太忙的节点，离客户端上传最近的节点，主要是为了减少网络传输的时间）。
  • 第 2 个副本放在与第1个机架不同的机架中的任意节点上。
  • 第 3 个副本放在与第2个节点所在机架的不同的节点上。

        // 执行RPC接口中的方法并获取结果
        String result = proxy.hello("RPC");
        System.out.println("[Client] - 收到的Server的返回结果：" + result);
    }
}

# 示例：定时上传数据至HDFS

在实际工作中会有定时上传数据到 HDFS 的需求，如有一个web项目每天都会产生日志文件，日志文件的格式为 access_2022_10_01.log 这种格式的，每天生成一个，我们需要每天凌晨将昨天生成的日志文件上传至 HDFS 上，按天存储到不同的目录下，HDFS 上的目录格式为 20221001。

我们可以写一个 Shell ，使用 crontab 来定时执行。具体流程如下：

# 1.新建 shell

新建文件 uploadLogData.sh，内容如下：

#!/bin/bash

# 获取昨天日期字符串
yesterday=$1
if [ "$yesterday" = "" ]
then
	yesterday=`date +%Y_%m_%d --date="1 days ago"`
fi

# 拼接日志文件路径信息
logPath=/data/log/access_${yesterday}.log

# 将日期字符串中的_去掉
hdfsPath=/log/${yesterday//_/}
# 在hdfs上创建目录
hdfs dfs -mkdir -p ${hdfsPath}
# 将数据上传到hdfs的指定目录中
hdfs dfs -put  ${logPath} ${hdfsPath}
