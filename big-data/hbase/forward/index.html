# 3.压缩

HFile 可以被压缩并存放在 HDFS 上。这有助于节省硬盘 IO，但是读写数据时压缩和解压缩会增加 CPU使用率。

压缩是表定义的一部分，可以在建表或修改表结构时设定。建议打开表的压缩，除非已经确定不会从压缩中受益。只有在数据不能被压缩或者因为某种原因服务器的 CPU 使用率有限制要求的情况下，有可能会关闭压缩特性。

HBase 可以使用多种压缩编码，包括 LZO、SNAPPY 和 GZIP。

# 5.数据块缓存

如果一张表或表里的某个列族只被顺序化扫描时访问或者很少被访问，这个时候就算 Get 或 Scan 花费时间长对业务没有影响，或者是可接受的。在这种情况下，我们可以选择关闭那些列族的缓存。

  • 如果开启缓存，但只是执行很多顺序化扫描，并且可能会滥用缓存，导致应该放进缓存获得性能提升的数据给排挤出去。

  • 如果关闭缓存，不仅可以避免上述情况发生，而且还可以让出更多缓存给其他表和同一个表的其他列族使用。

# 6.布隆过滤器（Bloom filters）

HBase 中存储额外的索引层次会占用额外的空间。布隆过滤器随着它们的索引对象的数据增长而增长，所以行级布隆过滤器比列标识符级布隆过滤器占用空间要少。当空间不是问题的时候，它们可以帮助你榨干系统的性能潜力。

BLOOMFILTER 参数的默认值是 ROW，表示是行级布隆过滤器。

  • 使用行级布隆过滤器需要设置为 ROW
    • 行级布隆过滤器在数据块里检查特定行键是否不存在，
  • 使用列标识符级布隆过滤器需要设置为 ROWCOL。
    • 列标识符级布隆过滤器检查行和列标识符联合体是否不存在。
  • ROWCOL 布隆过滤器的开销要高于 ROW 布隆过滤器。

命令：

create '表名',{NAME => 'cf1', BLOOMFILTER => 'ROW'}

create '表名',{NAME => 'cf1', BLOOMFILTER => 'ROWCOL'}

# 二、Scan 命令

# 1.Scan 命令

HBase 中的 Scan 命令是用于扫描表中行的命令。类似于SQL中的 select * from ...。

命令：

# 查询全表
scan 'table_name'

# 扫描表的一部分行
# start_row_key和stop_row_key是起始行和结束行的行键。
scan 'table_name', {STARTROW => 'start_row_key', STOPROW => 'stop_row_key'}

# 扫描表的一些特定列族和列
# column_family是列族的名称，column1和column2是列的名称。
scan 'table_name', {COLUMNS => ['column_family:column1', 'column_family:column2']}

# 将扫描的结果保存到文件中
# file_path是结果要保存的文件路径
scan 'table_name', {COLUMNS => ['column_family:column1', 'column_family:column2'], LIMIT => 1000, TIMERANGE => [start_time, end_time], FILTER => "filter_string", BATCHSIZE => 100, STARTROW => 'start_row_key', STOPROW => 'stop_row_key'} > file_path

# 2. Filter 过滤器

HBase的 scan 命令允许在扫描表时使用过滤器（Filters）来过滤出满足条件的行。

常用过滤器如下：

  1. SingleColumnValueFilter：基于某一列的值进行过滤。您可以使用比较运算符（如=、>、<、>=、<=、!=）和值来指定过滤条件。例如，以下过滤器将返回column_family:column1列值为value1的所有行：
SingleColumnValueFilter('column_family', 'column1', =, 'binary:value1')
1
  1. RowFilter：基于行键的值进行过滤。您可以使用比较运算符和值来指定过滤条件。例如，以下过滤器将返回行键以row_key_prefix开头的所有行：
RowFilter(=,'binary:row_key_prefix')
1
  1. PrefixFilter：基于行键前缀进行过滤。例如，以下过滤器将返回所有行键以row_key_prefix开头的行：
PrefixFilter('row_key_prefix')
1
  1. ColumnPrefixFilter：基于列名前缀进行过滤。例如，以下过滤器将返回所有列名以column_prefix开头的列：
ColumnPrefixFilter('column_prefix')
1
  1. MultipleColumnPrefixFilter：基于多个列名前缀进行过滤。例如，以下过滤器将返回所有列名以column_prefix1或column_prefix2开头的列：
MultipleColumnPrefixFilter(['column_prefix1', 'column_prefix2'])
1
  1. FamilyFilter：基于列族进行过滤。例如，以下过滤器将返回列族为column_family的所有列：
FamilyFilter(=,'binary:column_family')
1
  1. TimestampsFilter：基于时间戳进行过滤。例如，以下过滤器将返回时间戳在timestamp1和timestamp2之间的所有列：
TimestampsFilter([timestamp1, timestamp2])
1
  1. ValueFilter：基于列值进行过滤。例如，以下过滤器将返回列值为value1的所有列：
ValueFilter(=, 'binary:value1')
1

这些过滤器是常用的几种过滤器，HBase还提供了其他过滤器类型，如InclusiveStopFilter、RegexStringComparator等。您可以根据需要选择适当的过滤器。

示例：

scan 'table_name', {FILTER => "ValueFilter(=, 'binary:value1') AND ColumnPrefixFilter('column1') AND FamilyFilter('column_family')"}
1
2
  • ValueFilter(=, 'binary:value1') 基于列值过滤出列值为 value1 的所有列。
  • ColumnPrefixFilter('column1') 基于列名前缀过滤出所有列名以 column1 开头的列。
  • FamilyFilter('column_family') 基于列族过滤出列族为 column_family 的所有列。

# 四、 批量导入

批量导入两种方式：

  • 方式一：利用 MapReduce 中封装好的方法。在 Map 阶段，把数据封装成 put 操作，直接将数据入库。

package org.willingoxjin.hbase.imports;

import java.io.IOException;
import org.apache.hadoop.conf.Configuration;
import org.apache.hadoop.fs.Path;
import org.apache.hadoop.hbase.client.Put;
import org.apache.hadoop.hbase.mapreduce.TableMapReduceUtil;
import org.apache.hadoop.io.LongWritable;
import org.apache.hadoop.io.NullWritable;
import org.apache.hadoop.io.Text;
import org.apache.hadoop.mapreduce.Mapper;
import org.apache.hadoop.mapreduce.Job;
import org.apache.hadoop.mapreduce.lib.input.FileInputFormat;

/**
 * MapReduce的方式批量导入数据
 */
public class BatchImportMapReduce {

    /**
     * Mapper
     * 利用MapReduce中封装好的方法，在Map阶段，把数据封装成Put操作，将数据写入到HBase中
     */
    public static class BatchImportMapper extends Mapper<LongWritable, Text, NullWritable, Put> {
        @Override
        protected void map(LongWritable key, Text value, Context context) throws IOException, InterruptedException {
            // 读取一行数据
            String[] strArr = value.toString().split("\t");
            if (strArr.length == 4) {
                String rowKey = strArr[0]; // rowKey
                String columnFamily = strArr[1]; // 列族(info)
                String columnName = strArr[2]; // 列名
                String val = strArr[3]; // 列值
                Put put = new Put(rowKey.getBytes());
                // 封装数据
                put.addColumn(columnFamily.getBytes(), columnName.getBytes(), val.getBytes());
                // 将数据写入到HBase中
                context.write(NullWritable.get(), put);
            }
        }
    }

    public static void main(String[] args) throws IOException, InterruptedException, ClassNotFoundException {
        if (args.length != 2) {
            //如果传递的参数不够，程序直接退出
            System.exit(100);
        }

        // 参数1：导入的源文件路径
        String inPath = args[0];
        // 参数2：表名导入数据到那个表
        String outTableName = args[1];

        // 设置属性对应参数
        Configuration conf = new Configuration();
        conf.set("hbase.zookeeper.quorum", "hbase01:2181");
        conf.set("zookeeper.znode.parent", "/hbase-single");
        conf.set("hbase.rootdir", "hdfs://hbase01:9000/hbase");
        conf.set("hbase.table.name", outTableName);
        conf.set("hbase.zookeeper.quorum", "bigdata01:2181,bigdata02:2181,bigdata03:2181");

        // 封装Job
        Job job = Job.getInstance(conf, "Batch Import HBase Table：" + outTableName);
        job.setJarByClass(BatchImportMapReduce.class);

        // 指定输入路径
        FileInputFormat.setInputPaths(job, new Path(inPath));

        // 指定map相关的代码
        job.setMapperClass(BatchImportMapper.class);
        job.setMapOutputKeyClass(NullWritable.class);
        job.setMapOutputValueClass(Put.class);

        TableMapReduceUtil.initTableReducerJob(outTableName, null, job);
        TableMapReduceUtil.addDependencyJars(job);

        // 禁用Reduce
        job.setNumReduceTasks(0);

        job.waitForCompletion(true);
    }

}

package org.willingoxjin.hbase.imports;

import java.io.IOException;
import org.apache.hadoop.conf.Configuration;
import org.apache.hadoop.fs.FileSystem;
import org.apache.hadoop.fs.Path;
import org.apache.hadoop.hbase.TableName;
import org.apache.hadoop.hbase.client.Connection;
import org.apache.hadoop.hbase.client.ConnectionFactory;
import org.apache.hadoop.hbase.client.Put;
import org.apache.hadoop.hbase.io.ImmutableBytesWritable;
import org.apache.hadoop.hbase.mapreduce.HFileOutputFormat2;
import org.apache.hadoop.io.LongWritable;
import org.apache.hadoop.io.Text;
import org.apache.hadoop.mapreduce.Job;
import org.apache.hadoop.mapreduce.Mapper;
import org.apache.hadoop.mapreduce.lib.input.FileInputFormat;
import org.apache.hadoop.mapreduce.lib.output.FileOutputFormat;

public class BatchImportBulkLoad {

    public static class BulkLoadMapper extends Mapper<LongWritable, Text, ImmutableBytesWritable, Put> {

        @Override
        protected void map(LongWritable key, Text value, Context context)
                throws IOException, InterruptedException {
            String[] strArr = value.toString().split("\t");
            if (strArr.length == 4) {
                String rowKey = strArr[0];
                String columnFamily = strArr[1];
                String name = strArr[2];
                String val = strArr[3];
                ImmutableBytesWritable rowkeyWritable = new ImmutableBytesWritable(rowKey.getBytes());
                Put put = new Put(rowKey.getBytes());
                put.addColumn(columnFamily.getBytes(), name.getBytes(), val.getBytes());
                context.write(rowkeyWritable, put);
            }
        }

    }

    public static void main(String[] args) throws Exception {
        if (args.length != 3) {
            // 如果传递的参数不够，程序直接退出
            System.exit(100);
        }
        // 参数1：导入的源文件路径
        String inPath = args[0];
        // 参数2：生成的HFile文件存放路径
        String outPath = args[1];
        // 参数3：表名导入数据到那个表
        String outTableName = args[2];

        // 设置属性对应参数
        Configuration conf = new Configuration();
        conf.set("hbase.zookeeper.quorum", "hbase01:2181");
        conf.set("zookeeper.znode.parent", "/hbase-single");
        conf.set("hbase.rootdir", "hdfs://hadoop01:9000/hbase");
        conf.set("hbase.table.name", outTableName);

        // 封装Job
        Job job = Job.getInstance(conf, "Batch Import HBase Table：" + outTableName);
        job.setJarByClass(BatchImportBulkLoad.class);

        // 指定输入路径
        FileInputFormat.setInputPaths(job, new Path(inPath));

        // 指定输出路径[如果输出路径存在，就将其删除]
        FileSystem fs = FileSystem.get(conf);
        Path output = new Path(outPath);
        if (fs.exists(output)) {
            fs.delete(output, true);
        }
        FileOutputFormat.setOutputPath(job, output);

        // 指定map相关的代码
        job.setMapperClass(BulkLoadMapper.class);
        job.setMapOutputKeyClass(ImmutableBytesWritable.class);
        job.setMapOutputValueClass(Put.class);

        // 禁用Reduce
        job.setNumReduceTasks(0);

        Connection connection = ConnectionFactory.createConnection(conf);
        TableName tableName = TableName.valueOf(outTableName);
        HFileOutputFormat2.configureIncrementalLoad(job, connection.getTable(tableName),
                connection.getRegionLocator(tableName));

        job.waitForCompletion(true);
    }
}

package org.willingoxjin.hbase.exports;

import java.io.IOException;
import org.apache.hadoop.conf.Configuration;
import org.apache.hadoop.fs.Path;
import org.apache.hadoop.hbase.client.Result;
import org.apache.hadoop.hbase.client.Scan;
import org.apache.hadoop.hbase.io.ImmutableBytesWritable;
import org.apache.hadoop.hbase.mapreduce.TableMapReduceUtil;
import org.apache.hadoop.hbase.mapreduce.TableMapper;
import org.apache.hadoop.io.Text;
import org.apache.hadoop.mapreduce.Job;
import org.apache.hadoop.mapreduce.lib.output.FileOutputFormat;

public class BatchExportTableMapReduce {

    public static class BatchExportMapper extends TableMapper<Text, Text> {

        @Override
        protected void map(ImmutableBytesWritable key, Result result, Context context)
                throws IOException, InterruptedException {
            // key在这里就是hbase的Rowkey
            // result是scan返回的每行结果
            byte[] name = null;
            byte[] age = null;
        	try {
                name = result.getValue("info".getBytes(), "name".getBytes());
            } catch (Exception e) {
                e.printStackTrace();
            }
            try {
                age = result.getValue("info".getBytes(), "age".getBytes());
            } catch (Exception e) {
                e.printStackTrace();
            }

            String v2 = ((name == null || name.length == 0) ? "NULL" : new String(name)) + "\t" + (
                    (age == null || age.length == 0) ? "NULL" : new String(age));

            context.write(new Text(key.get()), new Text(v2));
        }

    }

    public static void main(String[] args) throws Exception{
        if(args.length!=2){
            // 如果传递的参数不够，程序直接退出
            System.exit(100);
        }

        // 参数1：导出的表名
        String inTableName = args[0];
        // 参数2：导出的文件路径
        String outPath = args[1];

        // 设置属性对应参数
        Configuration conf = new Configuration();
        conf.set("hbase.zookeeper.quorum", "hbase01:2181");
        conf.set("zookeeper.znode.parent", "/hbase-single");
        conf.set("hbase.rootdir", "hdfs://hadoop01:9000/hbase");

        // 组装Job
        Job job = Job.getInstance(conf);
        job.setJarByClass(BatchExportTableMapReduce.class);

        // 设置map相关的配置
        job.setMapperClass(BatchExportMapper.class);
        job.setMapOutputKeyClass(Text.class);
        job.setMapOutputValueClass(Text.class);

        // 禁用Reduce
        job.setNumReduceTasks(0);

        // 设置输入信息
        TableMapReduceUtil.initTableMapperJob(inTableName, new Scan(), BatchExportMapper.class, Text.class, Text.class, job);

        // 设置输出路径
        FileOutputFormat.setOutputPath(job, new Path(outPath));

        job.waitForCompletion(true);
    }
}

# 2.预分区

HBase 默认新建的表中只有一个 Region，这个 Region 的 Rowkey 是没有边界的。即没有 startRowkey 和 endRowkey，在数据写入时，所有数据都会写入这个默认的 Region 中。

随着数据量的不断增加，此 Region 已经不能承受不断增长的数据量，会进行 Split，分裂成2个 Region。

在这个过程中，会产生两个问题：

  1. 数据往一个 Region 上写，会有写热点问题。
  2. Region Split 会消耗集群 IO 资源。

基于此我们可以控制在建表的时候，创建多个空 Region，并确定每个 Region 的起始和终止 Rowkey，这样只要我们设计的 Rowkey 能均匀的命中各个 Region，就不会存在写热点问题。Region 分裂的几率也会大大降低。但是，随着数据量的不断增长，该分裂还是要进行分裂的。

像这种预先给 HBase 表创建多个 Region 的方式，称之为预分区。

如下，创建表时，设置预分区（SPLITS）：

create 'table_name', 'columnFamily1', SPLITS => ['10', '20', '30', '40']

# 3.RowKey 的设计原则

# 3.1 Rowkey 长度原则

Rowkey 底层存储是一个二进制流，可以是任意字符串，最大长度 64kb ，实际应用中一般是 10-100 字节，以 byte[]形式保存，一般设计为定长。

Rowkey 建议越短越好，不要超过 16 个字节，原因如下：

  • 数据的持久化文件 HFile 中是按照 KeyValue 存储的，如果 Rowkey 过长，比如超过 100 字节，1000w 行数据，Rowkey 就要占用 100*1000w=10 亿字节，也就是将近 1G 数据。这样会极大影响 HFile 的存储效率；
  • MemStore 会缓存部分数据到内存，如果 Rowkey 字段过长，内存的有效利用率就会降低，系统不能缓存更多的数据，这样会降低检索效率；
  • 目前操作系统都是 64 位系统，内存 8 字节对齐，控制在 16 个字节，8 字节的整数倍利用了操作系统的最佳特性。

# 3.2 Rowkey 散列原则

Rowkey 散列原则，主要是为了避免数据热点问题。

虽然我们可以在建表的时候提前设计预分区，但是假设数据的 Rowkey 都是手机号，那么都是1开头，按照上面 Rowkey 的预分区设计，那么所有的数据都会写到 10-20 之间的 Region 中，数据仍然集中存储在部分 Region 中。

保证数据能够尽可能均匀的分布到预先设计好的分区中的解决思路如下（以手机号为例）：

  • 手机号反转，将手机号的最后一位前置，这样第一位就是 0-9 之间的任意一个数字了。
  • 使用一致性哈希算法
    • 按照一定规则对数据进行 hash 取值（即获取数据的 hashCode）， 然后对得到的 hashCode 和与分配的 Region 个数取余，最后将得到的余数拼在手机号前面。
    • 例如：根据手机号 18809881234 后四位 1234 为 hash 值，预分区 5 个 Region，将 1234 对 5 取余得到 4，则设置该数据的 RowKey 为 418809881234。

# 3.3 Rowkey 唯一性原则

必须在设计上保证其唯一性，因为 Rowkey 相同则会覆盖。

Rowkey 是 HBase 里唯一的索引，对于某些查询频繁的限定条件可以把它的内容存放在 Rowkey 里面，提高查询效率。

例如：需要经常使用姓名和年龄这两个字段进行查询，那么可以考虑把姓名和年龄拼接到一块作为Rowkey。如 name=zhangsan，age=18，则我们就可以设置 Rowkey=zhangsan_18

# 4.列族的设计原则

在设计列族的时候，建议把经常读取的字段存储到一个列族中，不经常读取的字段放到另一个列族中。

这样在读取部分数据的时候，就只需要读取一个列族文件即可，可以提高读取效率。

# 7.HBase核心参数优化

  • hbase.hregion.majorcompaction
    • 配置大合并的间隔时间。默认为 604800000 毫秒（7天）
    • 可设置为0，禁止自动的大合并，大合并的执行可能会持续数小时，为减少对业务的影响，建议在业务低峰期进行手动或者通过脚本或者API定期进行大合并。
  • hbase.hregion.max.filesize
    • 默认为 10737418240 Byte(10G)。
    • 当 Region 达到这个阈值时，会自动分裂。
    • Region 分裂会有短暂的 Region 下线时间（通常在5s以内），为减少对业务端的影响，建议调大该值，并在业务低峰期定时手动进行分裂。
  • hbase.regionserver.handler.count
    • 默认 30。
    • 对于大负载的 put（达到了MB 以上的大小），或是大范围的 Scan 操作，handler 数目不易过大，易造成 OOM（内存溢出）。（handler 属于一个处理器，实现底层数据的发送。）
    • 对于小负载的 put、get，delete 等操作，handler 数要适当调大。
  • hbase.hregion.memstore.flush.size
  • 默认值 134217728 Byte (128M)，单位字节。
  • 该参数是 Memstore 中数据持久化到 Storefile 的时机。超过该阈值，则会把 Memstore 中的数据持久化到 Storefile 中。
  • 如果 Regionserver 的 JVM 内存比较充足（如：16G以上），可以适当调大该值。如：调整为256M。这样可以减少 Memstore 中数据溢写文件的次数。
  • hbase.hregion.memstore.block.multiplier
  • 默认值 4。
  • 如果一个 Memstore 的内存大小已经超过 hbase.hregion.memstore.flush.size 与 hbase.hregion.memstore.block.multiplier 的乘积，则会阻塞该 Memstore 的写操作。
  • 为避免阻塞，可以适当调大。例如 6~8。但如果太大，则会有 OOM 的风险。 如果在 Regionserver 日志中出现 Blocking updates for xxx on region : memstore size <xxxx M> is >= than blocking <xxxx M> size 的信息时，说明这个值需要调整。
  • hbase.hstore.compaction.min
    • 默认值为 3。
    • 如果任何一个 Store 里的 Storefile 总数超过该值，会触发默认的合并操作。
    • 可以设置 5~8，在手动的定期大合并中进行 Storefile 文件的合并，减少合并的次数，不过这会延长合并的时间。

以上参数可在 conf/hbase-site.xml 的配置文件中修改。

这些参数的默认配置在 lib/hbase-common-x.y.z.jar 中的 hbase-default.xml 文件中

以上参数对于开发人员了解即可，无需特别深入研究。

# 八、HBase 其他

# 1.Hive 与 HBase 整合

Hive 提供了 与HBase 的集成，可以在 HBase 表上使用 HQL 语句进行查询，插入操作以及进行 join 和 union 等复杂查询。

Hive 整合 HBase 后的使用场景：

  1. 通过 Hive 把数据加载到 HBase 中，数据源可以是文件也可以是 Hive 中的表。
  2. 通过整合，让 HBase 支持 JOIN、GROUP 等 SQL 查询语法。
  3. 通过整合，不仅可完成 HBase 的数据实时查询，也可以使用 Hive 查询 HBase 中的数据完成复杂的数据分析。

TIPS

Hive 查询 HBase 中的数据，性能一般，并不能发挥 HBase 中根据 Rowkey 查询性能较高的特性。

该功能了解即可，实际工作中基本不这样使用。

如果确实既有海量数据读写需求，还有 SQL 查询需求，可以考虑将数据存储两份，HBase 中维护实时读写的数据，然后定时将数据导出到 HDFS 中，在 Hive 中映射表提供 SQL 查询服务。

# 2.Phoenix

Phoenix 是构建在 HBase 上的一个 SQL 层，可以用标准的 JDBC APIs 来创建表，插入数据和对数据进行查询。

Phoenix 完全使用 Java 编写，作为 HBase 内嵌的 JDBC 驱动。Phoenix 查询引擎会将 SQL 查询转换为一个或多个 HBase 扫描，并编排执行以生成标准的 JDBC 结果集。

直接使用 HBase API、协同处理器与自定义过滤器，对于简单查询来说，其性能量级是毫秒，对于百万级别的行数来说，其性能量级是秒。

Phoenix 通过以下方式使我们可以少写代码，并且性能比我们开发编写代码更好：

  1. 将 SQL 编译成原生的 HBase scans 。
  2. 确定 scan 关键字的最佳开始和结束
  3. 并行执行 scan 命令

# 3.协处理器 coprocessor

HBase 作为列族数据库最经常被人诟病的特性包括：

  • 无法轻易建立“二级索引”
  • 难以执行求和、计数、排序等操作。

在旧版本（<0.92）的 HBase 中，统计数据表的总行数，需要使用 Counter 方法，执行一次MapReduce Job 才能得到。虽然 HBase 在数据存储层中集成了 MapReduce，能够有效用于数据表的分布式计算。然而在很多情况下，做一些简单的相加或者聚合计算的时候，如果直接将计算过程放置在 server 端，能够减少通讯开销，从而获得很好的性能提升。

因此，HBase 在 0.92 之后引入了 coprocessors（协处理器），添加了新特性：能够轻易建立二次索引、复杂过滤器以及访问控制等。

协处理器有两种：Endpoint 和 Endpoint

  • Observer
    • 允许集群在正常的客户端操作过程中可以有不同的行为表现
    • 类似于 RDBMS 中的触发器，主要在服务端工作
    • 可以实现权限管理、优先级设置、监控、ddl 控制、二级索引等功能
  • Endpoint
    • 允许扩展集群的能力，对客户端应用开放新的运算命令
    • 类似于 RDBMS 中的存储过程，主要在服务端工作
    • 可以实现 min、max、avg、sum、distinct、group by 等功能

# 4.Elasticsearch + HBase

HBase 中只有 RowKey 作为一级索引， 如果要对表里的非 RowKey 字段进行数据检索和查询， 往往要通过 MapReduce/Spark 等分布式计算框架进行，硬件资源消耗和时间延迟都会比较高。

由于 HBase 不支持多条件查询，不提供二级索引，难以满足用户对检索功能多样性和高效率两方面的需求。

Elasticsearch + HBase 的方案将数据与索引分离，利用 HBase 数据库的存储模式灵活多变，容纳海量数据等特点，结合 Elasticsearch 的快速建立索引和提供多样化的查询接口等优势，实现 Elasticsearch 的 HBase 二级索引方案。

思路：将索引数据存储于 Elasticsearch 中，做查询时，先到 Elasticsearch 中查询，转换为统一的 RowKey 后，再拿 RowKey 到 HBase 中快速定位数据。

# 5.HBase 分页

使用 PageFilter 过滤器 + 循环动态设置 startRow 实现。

由于默认 startRow 中包含这一行数据，因此在循环动态设置 startRow 时，需要在上一次获取到的 rowkey 后面补0，表示新的开始。

正常情况 PageFilter 返回的结果数量可能大于设定的值，因为服务器集群的 PageFilter 是隔离的，只能保证每个 Region 返回的数据量不会超过 PageFilter 中设置的值。想要解决返回数据超过设置数量的问题，可以考虑对获取到的数据进行截取，只保留需要的数据条数，下一次读取时根据上次截取的位置开始往后读取。

当 PageFilter 和其它 Filter 一起使用时，需要将 PageFilter 加入到 FilterList 的末尾，否则会出现结果个数小于你期望的数量。

package org.willingoxjin.hbase;

import java.io.IOException;
import org.apache.hadoop.conf.Configuration;
import org.apache.hadoop.hbase.CompareOperator;
import org.apache.hadoop.hbase.HBaseConfiguration;
import org.apache.hadoop.hbase.TableName;
import org.apache.hadoop.hbase.client.Connection;
import org.apache.hadoop.hbase.client.ConnectionFactory;
import org.apache.hadoop.hbase.client.Result;
import org.apache.hadoop.hbase.client.ResultScanner;
import org.apache.hadoop.hbase.client.Scan;
import org.apache.hadoop.hbase.client.Table;
import org.apache.hadoop.hbase.filter.FilterList;
import org.apache.hadoop.hbase.filter.PageFilter;
import org.apache.hadoop.hbase.filter.SingleColumnValueFilter;
import org.apache.hadoop.hbase.filter.SubstringComparator;
import org.apache.hadoop.hbase.util.Bytes;
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

public class PageFilterDemo {

    private final static Logger log = LoggerFactory.getLogger(PageFilterDemo.class);

    public static Connection initConnect() throws IOException {
        // 获取配置
        Configuration conf = HBaseConfiguration.create();
        // 指定HBase使用的zk的地址，多个用逗号隔开
        conf.set("hbase.zookeeper.quorum", "hbase01:2181");
        // 设置zookeeper的根路径
        conf.set("zookeeper.znode.parent", "/hbase-single");
        // 指定HBase在hdfs上的根目录
        conf.set("hbase.rootdir", "hdfs://hbase01:9000/hbase");
        // 创建HBase连接，负责对HBase中数据的一些增删改查(DML操作)
        return ConnectionFactory.createConnection(conf);
    }

    public static void main(String[] args) throws IOException {
        // 参数：每页的数据量
        int pageSize = 5;
        // 参数：上次的key，为空代表从第一行开始
        String lastRowKey = null;
        // String lastRowKey = "30001";
        
        Connection conn = initConnect();

        // 获取到test表的连接
        Table table = conn.getTable(TableName.valueOf("test_page"));

        Scan scan = new Scan();

        // 组装Filter列表
        FilterList filterList = new FilterList();

        // 注意：如果用到了多个filter，其中包含pageFilter，那么pageFilter需要放在filterList的最后一个
        // SingleColumnValueFilter singleColumnValueFilter = new SingleColumnValueFilter(
        //         Bytes.toBytes("info"),
        //         Bytes.toBytes("name"),
        //         CompareOperator.EQUAL, new SubstringComparator("张"));
        // filterList.addFilter(singleColumnValueFilter);

        // 注意：pageFilter返回的数据总数会超过这里设置的数量，需要自己在程序内部处理
        PageFilter pageFilter = new PageFilter(pageSize);
        filterList.addFilter(pageFilter);

        scan.setFilter(filterList);

        // 记录上一次返回的分页数据中的最大的Rowkey，最开始为null
        byte[] lastRowKeyByte = lastRowKey != null ? Bytes.toBytes(lastRowKey) : null;
        if (lastRowKeyByte == null) {
            // 第一次查询，查询第一页数据
            // 设置一个不存在的RowKey
            scan.withStartRow(Bytes.toBytes("-1"));
        } else {
            // 查询第二页或者其他页数的数据
            // 注意：在这里需要在lastRowKey后面补0，否则会把当前这条数据也返回过来，这样就重复了，补0之后可以保证返回的都是新数据
            scan.withStartRow(Bytes.add(lastRowKeyByte, Bytes.toBytes("0")));
        }
        // 记录每次迭代的数据条数
        int rowCount = 0;
        // 获取结果
        ResultScanner scanner = table.getScanner(scan);
        for (Result res: scanner) {
            byte[] rowKeyBytes = res.getRow();
            byte[] nameBytes = res.getValue(Bytes.toBytes("info"), Bytes.toBytes("name"));
            String rowKey = new String(rowKeyBytes);
            String name = new String(nameBytes);
            log.info("rowKey:{}，name:{}", rowKey, name);
            rowCount++;
            // scan返回的数据是基于rowKey有序的，直接判断数据条数即可。
            if (rowCount == pageSize) {
                log.info("当前页面数据获取完毕，退出循环, lastRowKey:{}", rowKey);
                break;
            }
        }

        conn.close();
    }

}

# 6.HBaseUtils 工具类封装

package org.willingoxjin.hbase.utils;

import org.apache.hadoop.conf.Configuration;
import org.apache.hadoop.hbase.TableName;
import org.apache.hadoop.hbase.client.*;
import org.apache.hadoop.hbase.util.Bytes;
import java.io.IOException;
import java.util.ArrayList;
import java.util.List;
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;
import org.willingoxjin.hbase.HBaseClientDemo;

/**
 * HBase工具类
 */
public class HBaseUtil {

    private final static Logger log = LoggerFactory.getLogger(HBaseUtil.class);

    private static final Connection conn = initConnection();

    private HBaseUtil() {
    }

    private static Connection initConnection() {
        // 获取hbase的链接
        Configuration conf = new Configuration();
        // 指定HBase使用的zk的地址，多个用逗号隔开
        conf.set("hbase.zookeeper.quorum", "hbase01:2181");
        // 设置zookeeper的根路径
        conf.set("zookeeper.znode.parent", "/hbase-single");
        // 指定HBase在hdfs上的根目录
        conf.set("hbase.rootdir", "hdfs://hbase01:9000/hbase");
        Connection conn = null;
        try {
            // 创建HBase连接，负责对HBase中数据的一些增删改查(DML操作)
            conn = ConnectionFactory.createConnection(conf);
        } catch (IOException e) {
            log.error("获取链接失败：" + e.getMessage());
        }
        return conn;
    }

    /**
     * 对外提供的方法
     */
    public static Connection getInstance() {
        return conn;
    }

    /**
     * 创建表
     *
     * @param tableName 表名
     * @param cfs       列族
     */
    public static void createTable(String tableName, String... cfs) throws IOException {
        Admin admin = conn.getAdmin();
        ArrayList<ColumnFamilyDescriptor> cfArr = new ArrayList<>();
        for (String cf : cfs) {
            ColumnFamilyDescriptor cfDesc = ColumnFamilyDescriptorBuilder
                    .newBuilder(Bytes.toBytes(cf))
                    .build();
            cfArr.add(cfDesc);
        }
        TableDescriptor tableDesc = TableDescriptorBuilder
                .newBuilder(TableName.valueOf(tableName))
                .setColumnFamilies(cfArr)
                .build();
        admin.createTable(tableDesc);
        admin.close();
    }

    /**
     * 添加一个单元格(列)的数据
     *
     * @param tableName    表名
     * @param rowKey       行键
     * @param columnFamily 列族
     * @param column       列
     * @param value        值
     */
    public static void put2HBaseCell(String tableName, String rowKey, String columnFamily, String column, String value)
            throws IOException {
        Table table = conn.getTable(TableName.valueOf(tableName));
        Put put = new Put(Bytes.toBytes(rowKey));
        put.addColumn(Bytes.toBytes(columnFamily), Bytes.toBytes(column), Bytes.toBytes(value));
        table.put(put);
        table.close();
    }

    /**
     * 向hbase中添加一批数据
     *
     * @param tableName 表名
     * @param list      数据集合
     */
    public static void put2HBaseList(String tableName, List<Put> list) throws IOException {
        Table table = conn.getTable(TableName.valueOf(tableName));
        table.put(list);
        table.close();
    }

}

# HBase 常见问题

1.HBase Put 功能初始化数据过慢时，可以考虑使用批量导入。

2.统一各个系统的字符集，非 utf8 的要做转换。

如果使用 hbase shell 时打印中文出现十六进制，可以使用 FORMATTER 参数指定输出格式

scan 'user', {'FORMATTER' => 'toString'}
1

3.在 HBase 客户端节点中需要配置 HBase 集群所有节点的主机名和 IP 的映射关系。

4.每日全量数据入库，数据实际发生变化的条数不多，浪费资源。建议只入库发生变化的数据，在源数据中设置更新时间，根据更新时间判断数据是否发生变化。

6.Scan 大表超时

  • 最好限制一个范围
  • 或尝试调整 RPC 请求的超时时间。适当调大参数 hbase.rpc.timeout 的值

7.HBase 第一次查询数据很慢，建议提前初始化客户端与 HBase 的连接（connection）。
