2.Flume 高级组件使用

https://flume.apache.org/releases/content/1.11.0/FlumeUserGuide.html#configuration (opens new window)

# 一、Flume 高级组件

  • Source Interceptors：Source 可以指定一个或者多个拦截器按先后顺序依次对采集到的数据进行处理。
  • Channel Selectors：Source发往多个 Channel 的策略设置，如果 Source 后面接了多个 Channel，到底是给所有的 Channel 都发，还是根据规则发送到不同 Channel，这些是由 Channel Selectors 来控制的
  • Sink Processors：Sink 发送数据的策略设置，一个channel后面可以接多个sink，channel中的数据是被哪个sink获取，这个是由Sink Processors控制的

# 1.Event

因为高级组件会用到 Event，所以学习高级组件之前，先来看看 Event。

Event 是 Flume 传输数据的基本单位，也是事务的基本单位，在文本文件中，通常一行记录就是一个Event。

Event 中包含 header 和 body：

  • body 是采集到的那一行记录的原始内容
  • header 的类型为 Map<String, String>。header 可以存储一些属性信息，方便后面使用 我们可以在 Source 中给每一条数据的 header 中增加 key-value，然后就可以在 Channel 和 Sink 中使用 header 中的值。

# 2.Source Interceptors

# Source Interceptors 介绍

系统中已经内置提供了很多 Source Interceptors。常见的 Source Interceptors 类型有 Timestamp Interceptor、Host Interceptor、Search and Replace Interceptor 、Static Interceptor、Regex Extractor Interceptor 等。

下面看几个常用的：

  • Timestamp Interceptor：向 Event 中的 header 里面添加 timestamp 时间戳信息
  • Host Interceptor：向 Event 中的 header里面添加 host 属性，host 的值为当前机器的主机名或者IP
  • Search and Replace Interceptor：根据指定的规则查询 Event 中 body 里面的数据，然后进行替换
  • Static Interceptor：向 Event 中的 header 添加固定的 key 和 value
  • Regex Extractor Interceptor：根据指定的规则从 Event 中的 body 里面抽取数据，生成 key 和value，再把 key 和 value 添加到 header 中

Search and Replace Interceptor和Regex Extractor Interceptor 比较常用。

Agent 配置

在 flume 的目录下的 conf 目录下， 新建配置文件 file-to-hdfs-type.properties，内容如下：

# agent的名称是a1
# 指定source组件、channel组件和Sink组件的名称
a1.sources = r1
a1.channels = c1
a1.sinks = k1

# 配置source组件
a1.sources.r1.type = exec
a1.sources.r1.command = tail -F /opt/data/flume-test-logs/recordLog.log

# 配置sources的拦截器 [多个拦截器按照顺序依次执行]
a1.sources.r1.interceptors = i1 i2 i3 i4
# 替换video_info为videoInfo
a1.sources.r1.interceptors.i1.type = search_replace
a1.sources.r1.interceptors.i1.searchPattern = "type":"video_info"
a1.sources.r1.interceptors.i1.replaceString = "type":"videoInfo"

# 替换user_info为userInfo
a1.sources.r1.interceptors.i2.type = search_replace
a1.sources.r1.interceptors.i2.searchPattern = "type":"user_info"
a1.sources.r1.interceptors.i2.replaceString = "type":"userInfo"

# 替换gift_record为giftRecord
a1.sources.r1.interceptors.i3.type = search_replace
a1.sources.r1.interceptors.i3.searchPattern = "type":"gift_record"
a1.sources.r1.interceptors.i3.replaceString = "type":"giftRecord"

a1.sources.r1.interceptors.i4.type = regex_extractor
a1.sources.r1.interceptors.i4.regex = "type":"(\\w+)"
a1.sources.r1.interceptors.i4.serializers = s1
a1.sources.r1.interceptors.i4.serializers.s1.name = logType


# 配置channel组件
a1.channels.c1.type = file
a1.channels.c1.checkpointDir = /opt/soft/apache-flume-1.11.0-bin/data/recordLog/checkpoint
a1.channels.c1.dataDirs = /opt/soft/apache-flume-1.11.0-bin/data/recordLog/data


# 配置sink组件
a1.sinks.k1.type = hdfs
a1.sinks.k1.hdfs.path = hdfs://172.16.23.129:9000/recordLog/%Y%m%d/%{logType}
a1.sinks.k1.hdfs.fileType = DataStream
a1.sinks.k1.hdfs.writeFormat = Text
a1.sinks.k1.hdfs.rollInterval = 3600
a1.sinks.k1.hdfs.rollSize = 134217728
a1.sinks.k1.hdfs.rollCount = 0
a1.sinks.k1.hdfs.useLocalTimeStamp = true

# 增加文件前缀和后缀
a1.sinks.k1.hdfs.filePrefix = data
a1.sinks.k1.hdfs.fileSuffix = .log

# 绑定组件
a1.sources.r1.channels = c1
a1.sinks.k1.channel = c1

# 3.Channel Selectors

# Channel Selectors 介绍

Channel Selectors 的类型：

  • Replicating Channel Selector 发往所有的 Channel
  • Multiplexing Channel Selector 按照规则发送到 Channel
  • Load Balancing Channel Selector 负载均衡发送
  • Custom Channel Selector 自定义

Replicating Channel Selector 是默认的 channel 选择器，它会将 Source 采集过来的 Event 发往所有的 Channel。

Replicating Channel Selector 的使用示例

a1.sources = r1
a1.channels = c1 c2 c3
a1.sources.r1.selector.type = replicating
a1.sources.r1.channels = c1 c2 c3
a1.sources.r1.selector.optional = c3
1
2
3
4
5

  • 在上面的配置中，c3 是一个可选的 channels 。写入 c3 失败会被忽略。即可以保证c1、c2一定能接收到所有数据，但是无法保证c3可以接受到所有数据。

  • selector.optional 是可选参数，可传多个值，使用空格隔开。

Multiplexing Channel Selector 会根据 Event 中 header 里面的值将 Event 发往不同的 Channel

使用示例：

a1.sources = r1
a1.channels = c1 c2 c3 c4

a1.sources.r1.selector.type = multiplexing
a1.sources.r1.selector.header = state
a1.sources.r1.selector.mapping.CZ = c1
a1.sources.r1.selector.mapping.US = c2 c3
a1.sources.r1.selector.default = c4
1

  • 如果 state 属性的值是 CZ，则发送给 c1
  • 如果 state 属性的值是 US，则发送给 c2 c3
  • 如果 state 属性的值是其它值，则发送给 c4

# 示例：Replicating Channel Selector

需求

  • 使用 Replicating 选择器，将 Source 采集到的数据重复发送给两个 Channl
  • 每个 Channl 后面接一个 Sink，负责把数据存储到不同存储介质中，方便后期使用

场景

数据采集过来后，分别存储到不同的存储介质中，不同存储介质的特点和应用场景是不一样的，典型的就是 HDFS Sink 和 Kafka Sink，

  • 通过 HDFS Sink 实现离线数据落盘存储，方便后面进行离线数据计算
  • 通过 Kafka Sink 实现实时数据存储，方便后面进行实时计算

这里为了简化示例，Kafka Sink 暂时用 Logger Sink 来代替

Agent 配置

在 Flume 的安装目录的 conf 目录下，新建 Agent 配置文件 tcp-to-replicatingchannel.properties，内容如下：

# agent的名称是a1
# 指定source组件、channel组件和Sink组件的名称

a1.channels = c1 c2

a1.sinks.k2.hdfs.fileType = DataStream

a1.sinks.k2.hdfs.rollInterval = 3600
a1.sinks.k2.hdfs.rollSize = 134217728
a1.sinks.k2.hdfs.rollCount = 0
a1.sinks.k2.hdfs.useLocalTimeStamp = true
a1.sinks.k2.hdfs.filePrefix = data
a1.sinks.k2.hdfs.fileSuffix = .log

# 绑定组件
a1.sources.r1.channels = c1 c2
a1.sinks.k1.channel = c1

# 示例：Multiplexing Channel Selector

需求

  • 使用 Multiplexing 选择器，将 Source 采集到的数据按照一定规则发送给两个 Channle
  • 最终再把不同 Channle 中的数据存储到不同介质中

这里使用正则抽取拦截器在 Event 的 header 中生成 key-value，然后将生成的 key-value 作为 Multiplexing 选择器的规则。

数据内容：

{"name":"张三","age":18,"city":"北京","cityCode":"beijing"}
{"name":"李四","age":20,"city":"长沙","cityCode":"hunan_changsha"}
1
2

配置 Agent

在 Flume 的安装目录的 conf 目录下，新建 Agent 配置文件 tcp-to-multiplexingchannel.properties，内容如下：

# agent的名称是a1
# 指定source组件、channel组件和Sink组件的名称
a1.sources = r1
a1.channels = c1 c2
a1.sinks = k1 k2

# 配置source组件
a1.sources.r1.type = netcat
a1.sources.r1.bind = 0.0.0.0
a1.sources.r1.port = 12000

# 配置source拦截器
a1.sources.r1.interceptors = i1
a1.sources.r1.interceptors.i1.type = regex_extractor
a1.sources.r1.interceptors.i1.regex = "cityCode":"(\\w+)"
a1.sources.r1.interceptors.i1.serializers = s1
a1.sources.r1.interceptors.i1.serializers.s1.name = cityCode

# 配置channle选择器
a1.sources.r1.selector.type = multiplexing
a1.sources.r1.selector.header = cityCode
a1.sources.r1.selector.mapping.beijing = c1
a1.sources.r1.selector.default = c2

# 配置channel组件
a1.channels.c1.type = memory
a1.channels.c1.capacity = 1000
a1.channels.c1.transactionCapacity = 100

a1.channels.c2.type = memory
a1.channels.c2.capacity = 1000
a1.channels.c2.transactionCapacity = 100


# 配置sink组件
a1.sinks.k1.type = logger

a1.sinks.k2.type = hdfs
a1.sinks.k2.hdfs.path = hdfs://172.16.23.129:9000/multiplexing
a1.sinks.k2.hdfs.fileType = DataStream
a1.sinks.k2.hdfs.writeFormat = Text
a1.sinks.k2.hdfs.rollInterval = 3600
a1.sinks.k2.hdfs.rollSize = 134217728
a1.sinks.k2.hdfs.rollCount = 0
a1.sinks.k2.hdfs.useLocalTimeStamp = true
a1.sinks.k2.hdfs.filePrefix = data
a1.sinks.k2.hdfs.fileSuffix = .log


# 绑定组件
a1.sources.r1.channels = c1 c2
a1.sinks.k1.channel = c1
a1.sinks.k2.channel = c2

# 4.Sink Processors

# Sink Processors 介绍

Sink Processors 类型包括三种：

  • Default Sink Processor
    • 该类型是默认的，无需配置
  • Load balancing Sink Processor
    • 负载均衡处理器。一个 Channle 后面可以接多个 Sink，这多个 Sink 属于一个 Sink group，根据指定的算法进行轮询或者随机发送，以减轻单个 Sink 的压力
  • Failover Sink Processor
    • 故障转移处理器，一个 Channle 后面可以接多个 Sink，这多个 Sink 属于一个Sink group。在 group 中，按照Sink 的优先级，默认先让优先级高的 Sink 来处理数据，如果这个 Sink 出现了故障，则会使用优先级低一点的 Sink 处理数据，以保证数据不丢失

Load balancing Sink Processor

使用示例：

# sinkgroups的名称
a1.sinkgroups = g1
# 在该group中的所有sink, 使用空格分开
a1.sinkgroups.g1.sinks = k1 k2
# load_balance 代表使用 Load balancing Sink Processor
a1.sinkgroups.g1.processor.type = load_balance
# 默认为false，设置为true后，故障的节点会列入黑名单，过一定时间才会再次发送数据，如果还失败，则等待时间是指数级增长；一直到达到最大的时间
a1.sinkgroups.g1.processor.backoff = true
# 负载均衡策略
a1.sinkgroups.g1.processor.selector = random

  • sinks：在该 group 中的所有 sink, 使用空格分开
  • processor.type：必须指定类型为 load_balance（必须这样配置）
  • processor.selector：属性必须是 round_robin（轮询）、random（随机）或继承自 AbstractSinkSelector 的自定义类的FQCN。默认值为 round_robin
  • processor.backoff：默认为 false，设置为 true 后，故障的节点会列入黑名单，过一定时间才会再次发送数据，如果还失败，则等待时间是指数级增长；一直到达到最大的时间。 如果不开启，故障的节点每次还会被重试发送，如果真有故障节点的话就会影响效率。

该组件可以解决单节点输出能力有限的问题，通过多个 Sink 后面连接多个 Agent 实现负载均衡，如果后面的 Agent 挂掉1个，也不会影响整体流程

# 三、Flume 优化

# 1.修改 JVM 内存参数

调整 Flume 启动的 JVM 内存大小，建议设置1G~2G，太小的话会导致频繁GC

Flume 是基于 Java 开发的，一般建议启动的单个 Flume 进程（或单个 Agent ）内存设置为 1G~2G，内存太小的话会频繁 GC，影响 Agent 的执行效率。

Flume 启动的 JVM 内存大小具体设置多少，需要根据 Agent 读取的数据量的大小和速度有关系，所以需要具体情况具体分析，当 Flume 的 Agent 启动之后，对应就会启动一个进程，我们可以通过 jstat -gcutil [PID] 1000 来看看这个进程 GC 的信息，每一秒钟刷新一次，如果GC次数增长过快，说明内存不够用。

  • YGC：表示新生代堆内存 GC 的次数，如果每隔几十秒产生一次，也还可以接受。如果每秒都会发生一次 YGC，就说明需要增加内存了

  • FGC：FULL GC 发生的次数，注意，如果发生 FUCC GC，则 Flume 进程会进入暂停状态。当 FUCC GC 执行完以后 ，Flume才会继续工作，所 FUCC GC 是非常影响效率的，这个指标的值越低越好，没有更好。

建议 Xms 和 Xmx 设置为一样大，避免进行内存交换，内存交换也比较消耗性能。

# 2.多个 Agent 日志分区

在一台服务器启动多个 Agent 的时候，建议修改配置区分日志文件。

在 Flume 安装包中，conf 目录的 log4j.properties 文件中，可以修改日志文件的名称及配置。

我们可以拷贝多个 conf 目录，然后修改对应 conf 目录 log4j.properties 日志的文件名称，使多个 Agent 的日志分别存储。

这样在启动 Agent 的时候分别通过 –conf参数（或 -c）指定不同 conf 目录，每一个 Agent 都有一个单独的日志文件，后期查看分析日志就方便很多。

可以把日志级别调整为 WARN 级别，以减少日志的产生。

# 四、Flume 进程监控

Flume 的 Agent 服务是一个独立的进程，如果我们需要知道 Agent 服务是否是存活状态，用来监控进程，如果发现进程挂掉了，就尝试重启服务。

# 1.启动脚本

新建脚本 startFlume-failover.sh ，内容如下：

#!/bin/bash

flume_path=/opt/soft/apache-flume-1.11.0-bin

nohup ${flume_path}/bin/flume-ng agent \
--name a1 \
--conf ${flume_path}/conf/ \
--conf-file ${flume_path}/conf/failover-source.properties  \
&

  • 文件中的第一列指定一个 Agent 的唯一标识，我们需要根据这个标识过滤对应的 Flume 进程，所以一定要保证至少在一台机器上是唯一的。

    • 这里的名字需要与配置文件名一致。

# 3.监控脚本

新建脚本 monlist.sh ，用来检查进程是否存在，内容如下：

#!/bin/bash

# 进入shell所造目录
CURRENT_DIR=$(cd $(dirname $0); pwd)
cd $CURRENT_DIR

monlist=`cat /opt/monlist/monlist.conf`
echo "start check"

for item in ${monlist}
do
	# 设置字段分隔符
	OLD_IFS=$IFS
	IFS="="
	# 把一行内容转成多列[数组]
	arr=($item)
	# 获取等号左边的内容
	name=${arr[0]}
	# 获取等号右边的内容
	script=${arr[1]}
	
	echo "time is:"`date +"%Y-%m-%d %H:%M:%S"`" check "$name
	if [ `jps -m|grep $name | wc -l` -eq 0 ]
	then
		# 发短信或者邮件告警
		echo `date +"%Y-%m-%d %H:%M:%S"`$name "is none"
		sh -x ./${script}
	fi
done
