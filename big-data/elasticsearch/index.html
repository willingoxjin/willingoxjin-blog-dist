# 一、搜索引擎介绍

# 什么是分布式搜索引擎

  • 搜索引擎
    • 数据检索过滤，精准的返回用户需要的数据
    • 优化用户搜索体验
    • 商业化，用户可通过搜索来查询需要的信息
    • 运营可根据用户搜索内容推荐相关商品/信息
  • 分布式存储与搜索
    • 多个节点都可以同时提供服务，提高计算能力
    • 增加支持的并发量

# 为什么要使用搜索引擎？

  • 使用sql查询不支持搜索词之间的空格
  • sql不支持拆词查询
  • 搜索内容不支持高亮
  • 当数据库中的数据过多，并发量大，直接在数据库使用like查询性能低

# 相关的搜索引擎

# Lucene

Lucene 是一个开源、免费、高性能、纯 Java 编写的全文检索引擎，可以算作是开源领域最好的全文检索工具包。

在实际开发中，Lucene 几乎适用于任何需要全文检索的场景，所以 Lucene 先后发展出好多语言版本，例如 C++、C#、Python 等。

早在 2005 年，Lucene 就升级为 Apache 顶级开源项目。它的作者是 Doug Cutting，有的人可能没听过这这个人，不过你肯定听过他的另一个大名鼎鼎的作品 Hadoop。

需要注意的是，Lucene 只是一个工具包，并非一个完整的搜索引擎，开发者可以基于 Lucene 来开发完整的搜索引擎。比较著名的有 Solr、ElasticSearch，不过在分布式和大数据环境下，ElasticSearch 更胜一筹。

Lucene 主要有如下特点：

  • 简单
  • 跨语言
  • 强大的搜索引擎
  • 索引速度快
  • 索引文件兼容不同平台
  • 倒排序索引
  • 是一个类库，相当于一个jar包，可以被直接引用，基于Java语言

# Solr

Solr是基于Lucene进行开发，是apache的一个开源项目，可以部署在tomcat或Jetty中，支持分布式集群 、索引、负载均衡的查询、自动故障转移。

可靠性、容错性较高

# ElasticSearch

ElasticSearch （本文简称为es/ES）是一个分布式、可扩展、近实时性的高性能搜索与数据分析引擎。ElasticSearch 基于 Java 编写，通过进一步封装 Lucene，将搜索的复杂性屏蔽起来，开发者只需要一套简单的 RESTful API 就可以操作全文检索。支持PB(1024TB)级别搜索。

ElasticSearch 在分布式环境下表现优异，这也是它比较受欢迎的原因之一。它支持 PB 级别的结构化或非结构化海量数据处理

整体上来说，ElasticSearch 有三大功能：

  • 数据搜集
  • 数据分析
  • 数据存储

ElasticSearch 的主要特点：

  1. 分布式文件存储。
  2. 实时分析的分布式搜索引擎。
  3. 高可拓展性。
  4. 可插拔的插件支持。

# 搜索引擎的技术选型

# 集群（Cluster）

一个或者多个安装了 es 节点的服务器组织在一起，就是集群，这些节点共同持有数据，共同提供搜索服务。

一个集群有一个名字，这个名字是集群的唯一标识，该名字成为 cluster name，默认的集群名称是 elasticsearch，具有相同名称的节点才会组成一个集群。

可以在config/elasticsearch.yml文件中配置集群名称：

cluster.name: my-es
1

在集群中，节点的状态有三种：绿色、黄色、红色：

  • 绿色：节点运行状态为健康状态。所有的主分片、副本分片都可以正常工作。
  • 黄色：表示节点的运行状态为警告状态，所有的主分片目前都可以直接运行，但是至少有一个副本分片是不能正常工作的。
  • 红色：表示集群无法正常工作。

# 节点（Node）

集群中的一个服务器就是一个节点，节点中会存储数据，同时参与集群的索引以及搜索功能。一个节点想要加入一个集群，只需要配置一下集群名称即可。默认情况下，如果我们启动了多个节点，多个节点还能够互相发现彼此，那么它们会自动组成一个集群，这是 es 默认提供的，但是这种方式并不可靠，有可能会发生脑裂现象。所以在实际使用中，建议一定手动配置一下集群信息。

# 索引（Index）

索引可以从两方面来理解：

  • 名词 具有相似特征文档的集合。

  • 动词 索引数据以及对数据进行索引操作。

# 类型（Type）

类型是索引上的逻辑分类或者分区。在 es6 之前，一个索引中可以有多个类型，从 es7 开始，一个索引中，只能有一个类型。在 es6.x 中，依然保持了兼容，依然支持单 index 多个 type 结构，但是已经不建议这么使用。

# 文档（Document）

一个可以被索引的数据单元。例如一个用户的文档、一个产品的文档等等。文档都是 JSON 格式的。

# 分片（Shards）

把索引库拆分为多份，分别放在不同的节点上，比如有3个节点，3个节点的所有数据内容加在一起是一个完整的索引库。分别保存到三个节点上，目的为了水平扩展，提高吞吐量。 一个shard可以理解为一个节点，多个shard组成集群

索引都是存储在节点上的，但是受限于节点的空间大小以及数据处理能力，单个节点的处理效果可能不理想，此时我们可以对索引进行分片（primary shard）。当我们创建一个索引的时候，就需要指定分片的数量。每个分片本身也是一个功能完善并且独立的索引。

默认情况下，一个索引会自动创建 1 个分片，并且为每一个分片创建一个副本。

# 副本（Replicas）

副本（replica shard）也就是备份，是对主分片的一个备份。(每个shard的备份)

# Settings

集群中对索引的定义信息，例如索引的分片数、副本数等等。

# Mapping

Mapping 保存了定义索引字段的存储类型、分词方式、是否存储等信息。

# Analyzer

字段分词方式的定义。

# ElasticSearch集群架构原理

用户请求时，会平均的分摊到三个shard上，若单个shard的吞吐量只有1000MB/s，三个shard组成的集群就可以提供3000MB/s的吞吐量，三个shard可以并行处理请求。 比如，现在有3T的数据量，当进行数据处理的时候，每个shard会平均分配1T

而每个shard都会有一个备份replica，当shard宕机之后，replica就会顶上去

# 正排索引与倒排索引

倒排索引源于实际应用中需要根据属性的值来查找记录。这种索引表中的每一项都包括一个属性值和具有该属性值的各记录的地址。由于不是由记录来确定属性值，而是由属性值来确定记录的位置，因而称为倒排索引(inverted index)。带有倒排索引的文件我们称为倒排索引文件，简称倒排文件

使用倒排索引能够根据单词进行搜索，搜索性能比正排索引更好 倒排索引不仅能够方便搜索，还可以统计单词出现的次数,位置

# 健康值

我们再新建一个索引index_2，分片5，副本每个分片一个副本，发现健康值变为 yellow (10 of 15)。

这是因为我们没有未副本分配节点，副本就不可用，关于健康值请参考下方描述。

关于健康值

green

所有的主分片和副本分片都已分配。你的集群是 100% 可用的。

yellow

所有的主分片已经分片了，但至少还有一个副本是缺失的。不会有数据丢失，所以搜索结果依然是完整的。不过，你的高可用性在某种程度上被弱化。如果 更多的 分片消失，你就会丢数据了。把 yellow 想象成一个需要及时调查的警告。

red

至少一个主分片（以及它的全部副本）都在缺失中。这意味着你在缺少数据：搜索只能返回部分数据，而分配到这个分片上的写入请求会返回一个异常

# 文档的乐观锁控制

es的乐观锁：在文档中的_version，是es自己生成的乐观锁，_version 从1开始，每次操作加1 ,

在es的乐观锁中，进行数据操作时，es会判断乐观锁版本是否一致，仅在一致时才能进行修改

乐观锁的使用

老版本

修改语句?version=<版本号>&
1

新版本

修改语句?if_seq_no=<_if_seq_no>&if_primary_term=<_if_seq_no>
1
  • _seq_no：文档版本号，作用同_version（相当于学号，每个班级的班主任为学生分配编号，效率要比学校教务处分配来的更加高效，管理起来更方便）
  • _primary_term：文档所在位置（相当于班级） 官文地址：https://www.elastic.co/guide/en/elasticsearch/reference/current/optimistic-concurrency-control.html

# analyze分词

分词，就是将一句话中的词分成多个中文/英文的词语单元

Elasticsearch默认不支持中文的分词，只支持英文的分词，solri也是不支持中文的分词的。

es会把每个中文字都会被拆分为单个字。若需要中文分词，只需要增加中文分词器

# 内置分词器

Elasticsearch内置了五种分词器

  • standard：默认分词，单词会被拆分，大小会转换为小写
  • simple：按照非字母分词。大写转为小写
  • whitespace：按照空格分词。大写不会被转化为小写
  • stop：去除无意义单词，比如the/a/an/is…
  • keyword：不做分词。把整个文本作为一个单独的关键词

# minimum_should_match

  • minimum_should_match: 最低匹配精度，至少有 [分词后的词语个数] x 百分百，得出一个数据值取整。举个例子：当前属性设置为70，若一个用户查询检索内容分词后有10个词语，那么匹配度按照 10x70%=7，则desc中至少需要有7个词语匹配，就展示；若分词后有8个，则 8x70%=5.6，则desc中至少需要有5个词语匹配，就展示。

# boost-设置搜索字段的比例

权重，为某个字段设置权重，权重越高，文档相关性得分就越高。通畅来说搜索商品名称要比商品简介的权重更高。

		}
    },

# head可视化插件组合查询

如下事宜head插件进行搜索查询

先只使用must查询：在desc中匹配张刘，得到三条结果

然后增加must_not查询：排除birthday=1999-02-14的结果，得到两条结果

最后增加should查询：在nickname中匹配土哥，最终得到两条结果，但是排序发生改变（排序改变原因：should查询的到的结果会将nickname=土哥 的那条数据的分数提高，分数越高排序越往前，最终导致排序发生变化）

# must、must_not、should组合查询使用的说明

  • 多个must 交集
  • must和should 在must查出来之后，should会在must查询结果中进行查询，若should查询有结果，该结果会被加分（分数高排在前面）
  • must和must_not 差集（must—must_not）
  • 多个should 并集
  • should和must_not shouuld会被认为是must，差集（should—must_not）
  • must, should 和 must_not 首先must和must_not取差集，在must和must_not的结果中should进行查询，若should查询出结果则结果加分（分数高排在前面）

对搜索出来的结果进行数据过滤。不会到es库里去搜，不会去计算文档的相关度分数，所以过滤的性能会比较高，过滤器可以和全文搜索结合在一起使用。 post_filter元素是一个顶层元素，只会对搜索结果进行过滤。不会计算数据的匹配度相关性分数，不会根据分数去排序，query则相反，会计算分数，也会按照分数去排序。

使用规则：

  • query：根据搜索条件检索匹配记录
  • post_filter：用于查询后，对结果数据的筛选
    • range 过滤范围
      • gte：大于等于
      • lte：小于等于
      • gt：大于
      • lt：小于

除此以外还能做其他的match等操作也行

例

在desc字段下匹配搜索张三，然后过滤出 20 ≤ age <30的结果

request

POST 	/user_demo/_doc/_search

{
    "query": {
        "match": {
            "desc": "张三"
        }
    },
    "post_filter": {
        "range": {
            "age": {
                "gte": 20,
                "lt": 30
            }
        }
    },
    "_source": [
        "id",
        "age",
        "desc"
    ]
}

# DSL—模糊搜索fuzzy

模糊搜索，并不是指的sql的模糊搜索，而是用户在进行搜索的时候的打字错误现象，搜索引擎会自动纠正，然后尝试匹配索引库中的数据。

fuzziness的参数

  • AUTO
    • 以下为最大可模糊替换的字符串长度
    • 0 1或2个字符的字符串（1-2个长度的字符串最多匹配0个模糊查询的结果）
    • 1 3、4或5个字符的字符串（如ipzd，可以模糊匹配到为ipad，但是ipzz或ipzzzd不能匹配到ipad）
    • 2 多于5个字符的字符串
  • 1
  • 2 （Elasticsearch 支持最大的编辑距离为2）

当然, 你可能发现编辑距离为2 仍然是太过了, 返回的结果也许并没有什么关联. 把 fuzziness 设置为 1 ，可能会获得更好的结果和性能.

# scroll 滚动搜索

滚动搜索也可以称作游标搜索，每次查询都会返回一个游标，用户根据游标参数查询下一页的数据。

一次性查询1万+数据，往往会造成性能影响，因为数据量太多了。这个时候可以使用滚动搜索，也就是 scroll。 滚动搜索可以先查询出一些数据，然后再紧接着依次往下查询。在第一次查询的时候会有一个滚动id，相当于一个锚标记，随后再次滚动搜索会需要上一次搜索的锚标记，根据这个进行下一次的搜索请求。 每次搜索都是基于一个历史的数据快照，查询数据的期间，如果有数据变更，不会影响搜索结果，因为 搜索的内容还是快照中的数据。

  • scroll=1m，相当于是一个session会话时间，搜索保持的上下文时间为1分钟。

# 基本语法

bulk操作和以往的普通请求格式有区别。不要格式化json，不然就不在同一行了，这个需要注意。

{ action: { metadata }}\n
{ request body        }\n
{ action: { metadata }}\n
{ request body        }\n
...
1
2
3
4
5
  • { action: { metadata }}代表批量操作的类型，可以是新增、删除或修改
  • \n是每行结尾必须填写的一个规范，每一行包括最后一行都要写，用于es的解析
  • { request body }是请求body，增加和修改操作需要，删除操作则不需要

注意：

  • bulk 请求不是原子的，不能使用事务

  • 谨记最后一个换行符不要落下。

  • 删除操作不需要请求body

# 批量操作的类型

action 必须是以下选项之一:

  • create：如果文档不存在，那么就创建它。存在会报错。发生异常报错不会影响其他操作。
  • index：创建一个新文档或者替换一个现有的文档。
  • update：部分更新一个文档。
  • delete：删除一个文档。

metadata 中需要指定要操作的文档的_index、_type 和_id，_index 、_type也可以在url中指定

# 六、ElasticSearch集群

虽然单机ES可以直接使用，但是一般都会搭建ES的集群，2台以上的ES即可成为集群，集群不仅可以实现高可用，也能实现海量数据存储的横向扩展

分片机制

每个索引可以被分片，就相当于吃披萨的时候被切了好几块，然后分给不同的人吃

  • 索引index_doc只有一个主分片；
  • 索引user_demo有3个主分片；
  • 索引user_demo2有5个主分片。
  • 每个主分片都包含索引的数据，由于目前是单机，所以副本分片是没有的，这个时候集群健康值显示为黄色。
  • 副本分片是主分片的备份，主挂了，备份还是可以访问，这就需要用到集群了。
  • 同一个分片的主与副本是不会放在同一个服务器里的，因为一旦宕机，这个分片就没了
  • 分配的规则是有ES系统自己处理分配

假如现在有三台服务器，我们分配5个主分片（0、1、2、3、4），每个主分片分配一个副本，分配规则如下

  • 服务器1：0、3、1（副）、4（副）
  • 服务器2：1、4、2（副）
  • 服务器3：2、0（副）、3（副）

# ES集群脑裂

# 什么是集群脑裂？

当主节点宕机后，剩下的服务器节点会重新选举出新的主节点。 然而原来的主节点由宕机恢复到正常状态的时候，自己又成为主节点，最后出现两个主节点，这就是集群脑裂。

# 出现原因

选举主节点时，设置的最小投票人数为1，也就是自己给自己投票，自己成为主节点。

集群脑裂会出现在es的老版本中，7.x以上版本则由es自行管理，并且在7.x版本已经移除了minimum_master_node放入配置

# 解决方法

设置discovery.zen.mininum_master_nodes=(N+2)+1

# N为服务器节点总数

# ES文档读写原理

# 文档写原理

  1. 首先，客户端的请求到节点时，该节点则被视为协调节点。 协调节点根据算法计算出该数据写入的服务器节点及分片地址，生成路由将请求转发到对应的目标地址
  2. 当请求转发到对应的目标地址后，该节点的分片进行数据修改/增加操作。
  3. 并将该数据的修改/增加同步到其他节点的分片上
  4. 当所有的节点分片的文档数据同步完成之后，协调节点收到通知，将处理结果返回到客户端

# 文档读原理

  1. 客户端读取文档请求到达协调节点，协调节点进行hash算法，将请求转发到对应的路由
  2. 转发的路由地址可能是主分片或副本分片
    1. 读请求会轮询转发主分片和所有的副本分片
  3. 对应的节点分片读取结果之后，将结果返回到协调节点
  4. 协调节点将最终结果返回到客户端

 * @author Jin.Nie

    public static void main(String[] args) throws IOException {
        RestClientBuilder restClientBuilder = RestClient.builder(
                new HttpHost("localhost", 9200, "http")
        );

        // 执行同步请求
        restHighLevelClient.indices().create(createIndexRequest, RequestOptions.DEFAULT);

        // 关闭请求
        restHighLevelClient.close();
    }

# Logstash介绍

Logstash 是免费且开放的服务器端数据处理管道，能够从多个来源采集数据，转换数据，然后将数据同步到“存储库”中。Logstash是elastic推出的，用来解决数据库与elaticsearch的数据同步问题。

Logstash的功能

  • 数据采集：将数据库数据采集到Es中
  • 以id或update_time作为同步边界
    • id：当数据库的id字段有新的id时，同步新增的数据
    • update_time：当数据的数据存在大于update_time字段时，同步变动的数据，由于使用id只能新增数据，一般使用update_time，即可以新增又可以更新数据
  • logstash-input-jdbc 插件
    • 老版本插件需要自己集成
    • 使用此插件时，需要与Es版本一致
  • 使用时需要预先手动创建索引
