1.Spark 基础

# 一、Spark 介绍

# 1.什么是Spark

Spark是一个基于内存的分布式计算框架，旨在解决 Hadoop MapReduce 计算速度慢、不支持实时计算等问题。

Spark 不仅仅可以做类似于 MapReduce 的离线数据计算，还可以做实时数据计算，并且它还可以实现类似于 Hive 的 SQL 计算。

Spark 中一个最重要的特性就是基于内存进行计算，从而让它的计算速度可以达到 MapReduce 的几十倍甚至上百倍。

# 2.Spark的特点

  • Speed：速度快
    • Spark 是基于内存进行计算的，所以它的计算性能理论上可以比 MapReduce 快 100 倍。
    • Spark 使用最先进的 DAG 调度器、查询优化器和物理执行引擎，实现了高性能的批处理和流处理。
    • 注意：批处理其实就是离线计算，流处理就是实时计算，只是说法不一样，意思是一样的。
  • Easy of Use：易用性
    • 可以使用多种编程语言快速编写应用程序，例如 Java、Scala、Python、R 和 SQL。
    • Spark 提供了80多个高阶函数，可以轻松构建 Spark 任务。
  • Generality：通用性
    • Spark 提供了 Core、SQL、Streaming、MLlib、GraphX 等技术组件，可以一站式地完成大数据领域的离线批处理、SQL 交互式查询、流式实时计算，机器学习、图计算等常见的任务 从这可以看出来 Spark 也是一个具备完整生态圈的技术框架。
  • Runs Everywhere：到处运行
    • 我们可以在 Hadoop YARN、Mesos 或 Kubernetes 上使用 Spark 集群，并且可以访问 HDFS、Alluxio、Apache Cassandra、Apache HBase、Apache Hive 和数百个其它数据源中的数据。

# 3.Spark 对比 Hadoop

综合能力：

  • Spark 是一个综合性质的计算引擎

  • Hadoop 既包含 MapReduce（计算引擎），还包含 HDFS（分布式存储）和 YARN（资源管理）

计算模型：

  • Spark 任务可以包含多个计算操作，轻松实现复杂迭代计算
  • Hadoop 中的 MapReduce 任务只包含 Map 和 Reduce 阶段，不够灵活

处理速度：

  • Spark 任务的数据是基于内存的，计算速度很快
  • Hadoop 中 MapReduce 任务是基于磁盘的，速度较慢

Spark 会替代 Hadoop 吗？

  • Spark 是一个通用的计算引擎，而 Hadoop 是一个包含 HDFS、MapRedcue 和 YARN 的框架。所以说 Spark 就算替代也只是替代 Hadoop 中的 MapReduce，也不会整个替代 Hadoop，因为 Spark 还需要依赖于 Hadoop 中的 HDFS 和 YARN。

所以在实际工作中 Hadoop 会作为一个提供分布式存储和分布式资源管理的角色存在 Spark 会在它之上去执行。

# 4.Spark+Hadoop

Spark 和 Hadoop 的组合：

  • 底层基于 Hadoop 的 HDFS 和 YARN
  • Spark Core 指的是 Spark 的离线批处理
  • Spark Streaming 指的是 Spark 的实时流计算
  • Spark SQL 指的是 Spark 中的 SQL 计算
  • Spark MLib 指的是 Spark 中的机器学习库，这里面集成了很多机器学习算法
  • Spark GraphX 是指图计算

# 5.Spark 的应用场景

  1. 低延时的海量数据计算需求，即针对 Spark Core 的应用
  2. 低延时 SQL 交互查询需求，即针对 Spark SQL 的应用
  3. 准实时（秒级）海量数据计算需求，即针对 Spark Streaming 的应用

# 二、Spark 安装部署

# 1.集群安装部署

Spark集群有多种部署方式，常见的有 Standalone 模式和 ON YARN 模式。

  • Standalone 模式：部署一套独立的 Spark 集群，后期开发的 Spark 任务在这个独立的 Spark 集群中执行
  • ON YARN 模式：使用现有的 Hadoop 集群，后期开发的 Spark 任务会在这个 Hadoop 集群中执行。此时这个 Hadoop 集群就是一个公共的了，不仅可以运行 MapReduce 任务，还可以运行 Spark 任务，这样集群的资源就可以共享了，并且也不需要再维护一套集群了，减少了运维成本和运维压力，一举两得。

实际工作中一般使用 ON YARN 模式。

Spark 下载地址：

  • Spark Archived Releases：https://archive.apache.org/dist/spark/ (opens new window)

提交任务的相关参数：

  • --name mySparkJobName：指定任务名称

  • --class：运行的主类名。

  • --master：指定集群的master URL，如“local”、“yarn”、“mesos”等。

  • --deploy-mode：指定在集群上以何种模式运行应用程序，可以是“client”或“cluster”。

  • --num-executors：指定在集群上运行的计算节点数量（分配多少个 Executor 进程）。

  • --executor-cores：指定每个计算节点的 CPU 核心数（一个 Executor 进程分配多少个 CPU Core）。

  • --executor-memory：指定每个计算节点可用的内存量，实际工作中设置2~4G即可。

  • --driver-cores 1 ：Driver 进程分配多少 CPU Core，默认为1即可。

  • --driver-memory：指定 Driver 程序可用的内存量（如果需要使用类似于 collect 之类的 Action 算子向 Driver 端拉取数据，则可以设置大一些）。

  • --jars fastjson.jar,abc.jar：设置 Job 依赖的第三方 jar 包，多个 jar 包之间通过逗号隔开即可。

    • 不建议把第三方依赖 jar 包整体打包进 Spark 的 Job 中，那样会导致任务 jar 包过大。
    • 所有人都统一使用 HDFS 中的共享的这些第三方 jar 包，这样版本就统一了。
    • 这里的 jar 包路径可以指定本地磁盘路径，或者是 HDFS 路径，建议使用 HDFS 路径，因为 Spark 在提交任务的时候会把本地磁盘的依赖 jar 包也上传到 HDFS 的一个临时目录中。如果在这里本来指定的就是 HDFS 的路径，那么 Spark 在提交任务的时候就不会重复提交依赖的 jar 包了，这样可以提高任务提交的效率。
    • 我们可以在 HDFS 中创建一个类似于 common-lib 的目录，统一存放第三方依赖的 jar 包。

  • --conf：设置 Spark 配置选项，使用方式：--conf "spark.app.name=MyApp"。

    1. spark.app.name：指定应用程序的名称。

    2. spark.master：指定 Spark 集群的主节点 URL。

    3. spark.driver.memory：设置应用程序驱动程序的内存分配大小。

    4. spark.executor.memory：设置每个执行程序的内存分配大小。

    5. spark.num.executors：设置执行程序的数量。

    6. spark.executor.cores：为每个执行程序设置 CPU 核心数。

    7. spark.submit.deployMode：确定应用程序的运行模式，可以是 client 或 cluster。

    8. spark.dynamicAllocation.enabled：启用或禁用 Spark 动态分配资源功能。

    9. spark.shuffle.service.enabled：启用或禁用独立的 Shuffle 服务。

    10. spark.eventLog.enabled：启用或禁用事件日志记录。

    11. ……

详细说明见官方文档：https://spark.apache.org/docs/3.3.2/submitting-applications.html#launching-applications-with-spark-submit (opens new window)

# 三、Spark 架构原理

# 1. Spark 任务的工作原理

Spark 架构的工作运行图如下：

上图中的结构：

  • 左边是 Spark 的客户端节点，这个节点主要负责向 Spark 集群提交任务。

  • 中间是一个 Spark 集群，可以理解为 Spark 的 Standalone 集群，集群中有6个节点

  • 右边就是 Spark 的数据源 HDFS。

Spark 任务执行流程：

  1. 任务提交

当我们通过客户端节点向 Spark 集群提交了一个任务，Spark 任务会把 HDFS 中的数据读取出来，然后把 HDFS 中的数据转化为 RDD。RDD 是一个弹性分布式数据集，它是一个逻辑概念，可以先把 RDD 理解成一个数据集合。

RDD 是有分区这个特性的，即一整份数据被分成多份。假如从 HDFS 中读取的这份数据被转化为 RDD 之后，在 RDD 中分成了3份，那这3份数据可能会分布在3个不同的节点上面，对应图中的：节点1、节点2、节点3。

  1. 分片计算

这个 RDD 的3个分区的数据对应的是 partiton-1、partition-2、partition-3。这样的好处是可以并行处理，后期每个节点就对当前节点上分区的数据进行计算。这里的计算逻辑类似于 MapReduce 的计算思想：本地计算。但是与 MapReduce 不同的就是 RDD 的数据是在内存中的。

假设这个 RDD 中每个分区中有 10w 条数据。我们想对这个 RDD 中的数据进行计算，就可以使用 Scala 的高阶函数进行计算，例如：flatMap、map 等

如果使用 flatMap 对数据进行处理，把每一行数据转成多行数据，此时就会在节点1、节点2和节点3上并行执行 flatMap 计算。计算之后的结果还是一个带有分区的 RDD，我们假设把计算结果的 RDD 存在节点4、节点5和节点6上。此时每个节点上面会有一个分区的数据，我们给这些分区数据起名叫 partition-4、partition-5、partition-6。

  1. 数据传输

正常情况下，前面节点1上的数据处理之后会发送到节点4上，其他节点也是一样的。经过计算之后，前面 RDD 的数据传输到后面节点上面这个过程是不需要经过 shuffle 的，可以直接在内存中通过网络传输过去，因为现在这两个 RDD 的分区数量是一一对应的。

  1. 结果存储

后面可能还会通过 map 或其它的高阶函数对数据进行处理，当处理到最后一步的时候是需要把数据存储起来的，在这我们选择把数据存储到 HDFS 上面。在实际工作中，针对这种离线计算，大部分的结果数据都是存储在 HDFS 上，当然也可以存储到其它的存储介质中。

Spark 任务处理的整体流程：先通过 Spark 客户端提交任务到 Spark 集群，然后 Spark 任务在执行的时候会读取数据源 HDFS 中的数据，将数据加载到内存中，转化为 RDD，然后针对 RDD 调用一些高阶函数对数据进行处理，中间可以调用多个高阶函数，最终把计算出来的结果数据写到 HDFS 中。

# 2.RDD

RDD 通常通过 HDFS 文件进行创建，也可以通过程序中的集合来创建。

RDD 是 Spark 提供的核心抽象，全称为 Resillient Distributed Dataset。即弹性分布式数据集

RDD 的特点：

  • 弹性：RDD 数据默认情况下存放在内存中，但是在内存资源不足时，Spark 会自动将 RDD 数据写入磁盘。
  • 分布式：RDD 在抽象上来说是一种元素数据的集合，它是被分区的，每个分区分布在集群中的不同节点上，从而让 RDD 中的数据可以被并行操作。
  • 容错性：RDD 最重要的特性就是提供了容错性，可以自动从节点失败中恢复过来。如果某个节点上的 RDD partition，因为节点故障，导致数据丢了，那么 RDD 会自动通过自己的数据来源重新计算该 partition 的数据。

# 3.Spark 相关进程

以 Spark 的 Standalone 集群为例：

  • Driver
    • 我们编写的 Spark 程序就在 Driver 进程上，由 Driver 进程负责执行。
    • Driver 进程所在的节点可以是 Spark 集群的某一个节点，或者就是我们提交 Spark 程序的客户端节点。具体 Driver 进程在哪个节点上启动是由我们提交任务时指定的参数决定的。
  • Master
    • Master 进程是集群的主节点中启动的进程。
    • Master 进程主要负责集群资源的管理和分配，还有集群的监控等。
  • Worker
    • Worker 进程集群的从节点中启动的进程。
    • Worker 进程主要负责启动其它进程来执行具体数据的处理和计算任务。
  • Executor
    • Executor 进程由 Worker 负责启动，主要为了执行数据处理和计算。
  • Task
    • Task 是一个线程
    • 由 Executor 进程负责启动，它真正进行计算，处理数据的进程。

# 4.Spark 架构

Spark 的架构原理：

  1. 首先我们在 Spark 的客户端机器上通过 Driver 进程执行我们提前编写好的 Spark 代码。当我们通过 spark-submit 脚本提交 Spark 任务的时候 Driver 进程就启动了。
  2. Driver 进程启动之后，会做一些初始化的操作，会找到集群 Master 进程，对 Spark 应用程序进行注册
  3. 当 Master 收到 Spark 程序的注册申请之后，会发送请求给 Worker，进行资源的调度和分配
  4. Worker 收到 Master的请求之后，会为 Spark 应用启动 Executor 进程 会启动一个或者多个 Executor。具体启动 Executor 的个数，会根据配置来启动
  5. Executor 启动之后，会向 Driver 进行反注册，这样 Driver 就知道哪些 Executor 在为它服务了
  6. Driver 会根据我们对 RDD 定义的操作，提交一堆的 Task 去 Executor 上执行 Task 执行的其实就是具体的 map、flatMap 等操作。

# 五、RDD 创建、操作与持久化

# 1.创建 RDD

RDD 是 Spark 编程的核心。在进行 Spark 编程时，首要任务是创建一个初始的 RDD，这样就相当于设置了 Spark 应用程序的输入源数据。

在创建了初始的 RDD 后，才可以通过 Spark 提供的一些高阶函数，对这个 RDD 进行操作，来获取其它的 RDD

Spark 提供三种创建 RDD 方式：集合、本地文件、HDFS 文件

  • 使用程序中的集合创建 RDD，主要用于进行测试，可以在实际部署到集群运行之前，自己使用集合构造一些测试数据，来测试后面的 Spark 应用程序的流程。
  • 使用本地文件创建 RDD，主要用于临时性地处理一些存储了大量数据的文件
  • 使用 HDFS 文件创建 RDD，是最常用的生产环境的处理方式，主要可以针对 HDFS 上存储的数据，进行离线批处理操作。

# 1.1 使用集合创建 RDD

如果通过集合来创建 RDD，需要针对程序中的集合，调用S parkContext 的 parallelize() 方法。Spark 会把集合中的数据拷贝到集群上，形成一个分布式的数据集合，也就是一个 RDD。

相当于，集合中的部分数据会到一个节点上，而另一部分数据会到其它节点上。然后就可以用并行操作数据集合了。

调用 parallelize() 时，可以指定将集合切分成多少个 partition。Spark 会为每一个 partition 运行一个 Task 来进行处理数据。

默认情况下，Spark 会根据集群的配置来设置 partition 的数量。我们也可以在调用 parallelize() 方法时，传入第二个参数，来设置 RDD 的 partition 数量，例如：parallelize(arr, 5)

示例 Scala 代码：

package org.willingoxjin.scala.spark.rdd

import org.apache.spark.{SparkConf, SparkContext}

object CreateRddByArray extends App {

  // 创建SparkContext
  val conf = new SparkConf()
  conf.setAppName("CreateRddByArray-Scala ") // 设置任务名称
    .setMaster("local") // local表示在本地执行
  val sc = new SparkContext(conf)

  // 创建集合
  val arr = Array(1, 2, 3, 4, 5, 6, 7, 8, 9)
  // 基于集合创建RDD
  val rdd = sc.parallelize(arr)
  val sum = rdd.reduce(_ + _)
  println(sum) // 45

  // 停止SparkContext
  sc.stop()

}

# 1.2 使用本地文件和 HDFS 文件创建 RDD

通过 SparkContext 的 textFile() 方法，可以针对本地文件或 HDFS 文件创建 RDD，RDD 中的每个元素就是文件中的一行文本内容。

textFile() 方法

  • 支持针对目录、压缩文件以及通配符创建 RDD
  • minPartitions 参数：Spark 默认会为 HDFS 文件的每一个 Block 创建一个 partition，也可以通过 textFile() 的第二个参数（minPartitions）手动设置分区数量。不过分区数量必须比 Block 数量多，不能比 Block 数量少。比 Block 数量少的话设置不生效。

示例 Scala 代码：

package org.willingoxjin.scala.spark.rdd

import org.apache.spark.{SparkConf, SparkContext}

object CreateRddByFile extends App {

  // 创建SparkContext
  val conf = new SparkConf()
  conf.setAppName("CreateRddByFileScala") //设置任务名称
    .setMaster("local") //local表示在本地执行
  val sc = new SparkContext(conf)

  var path = "hdfs://hadoop01:9000/spark_test/word1.data"
  // 读取文件数据，可以在textFile中指定生成的RDD的分区数量
  val rdd = sc.textFile(path, 2)
  // 获取每一行数据的长度，计算文件内数据的总长度

# 2.操作 RDD

Spark 对 RDD 的操作可以整体分为两类：Transformation 和 Action。

  • Transformation 转换
    • Transformation 表示 针对 RDD 中数据的转换操作，主要会针对已有的RDD创建一个新的RDD。
    • 常见的 Transformation 有 map、flatMap、filter 等。
  • Action 执行
    • Action 表示触发任务执行的操作，主要对 RDD 进行最后的操作，并且还可以把结果返回给 Driver 程序。
    • 常见的 Action 有遍历、reduce、保存到文件等。

不管是 Transformation 的操作还是 Action 的操作，一般都将它们都称之为算子。例如：map 算子、reduce 算子。

特性：

  • Transformation 的特性：lazy。

    • lazy 特性在这里指的是，如果一个 Spark 任务中只定义了 Transformation 算子，那么即使执行这个任务，任务中的算子也不会执行。即 Transformation 是不会触发 Spark 任务执行的，它们只是记录了对 RDD 的操作，不会执行。
    • 只有当 Transformation 之后，接着执行了一个 Action 操作，那么所有的 Transformation 才会执行。
    • Spark 通过 lazy 这种特性，优化底层的 Spark 任务执行，避免产生过多中间结果。

  • Action 的特性

    • 执行 Action 操作才会触发一个 Spark 任务的运行，从而触发这个 Action 之前所有的Transformation 的执行（实际上就是与 Transformation 的特性结合）。

下面是我们之前对单词计数的 Scala 代码。只有当执行到 wordCountRDD.foreach(wordCount => println(s"${wordCount._1}=${wordCount._2}")) 时，任务才会真正开始执行计算，如果任务中没有那一行代码，前面的所有算子是不会执行的。如下：

# 3.RDD 持久化

# 3.1 RDD持久化原理

Spark 中有一个非常重要的功能就是可以对 RDD 进行持久化（RDD Persistence）。

当对 RDD 执行持久化操作时，每个节点都会将自己操作的 RDD 的 partition 数据持久化到内存中，并且在之后对该 RDD 的反复使用中，直接使用内存中缓存的 partition 数据。

这样的话，针对一个 RDD 反复执行多个操作的场景，就只需要对 RDD 计算一次即可，后面直接使用该 RDD，而不需要反复计算多次该 RDD。

正常情况下 RDD 的数据使用过后内存中是不会一直保存的。

如下操作，针对 mapRDD 需要多次使用的情况，我们就需要持久化 mapRDD：

val dataRDD = sc.parallelize(Array(1,2,3,4,5))
val mapRDD = dataRDD.map(...)
mapRDD.foreach(...)
mapRDD.saveAsTextFile(...)
mapRDD.collect()

巧妙使用 RDD 持久化，在某些场景下，能够很大的提升 Spark 应用程序的性能。特别是对于迭代式算法和快速交互式应用来说，RDD 持久化，是非常重要的。

要持久化一个 RDD，只需要调用它的 cache() 或者 persist() 方法就可以了。

当在代码中设置了持久化，在 RDD 第一次被计算出来时，就会直接缓存在每个节点中。而且 Spark 的持久化机制还是自动容错的，如果持久化的 RDD 的任何 partition 数据丢失了，那么 Spark 会自动通过其源 RDD，使用Transformation 算子重新计算该 partition 的数据。

cache() 和 persist() 的区别：cache() 是 persist() 的一种简化方式。cache() 的底层就是调用的 persist() 的无参方法，即调用 persist(MEMORY_ONLY)，将数据持久化到内存中。

如果需要从内存中清除缓存，那么可以使用 unpersist() 方法。

# 3.2 RDD 持久化策略

目前 Spark 支持的持久化策略见官方文档：https://spark.apache.org/docs/3.3.2/rdd-programming-guide.html#rdd-persistence (opens new window)

存储策略（存储等级） 描述
MEMORY_ONLY 以非序列化的方式持久化在JVM内存中
MEMORY_AND_DISK 同上，但是当某些partition无法存储在内存中时，会持久化到磁盘中
MEMORY_ONLY_SER (Java and Scala) 同MEMORY_ONLY，但是会序列化
MEMORY_AND_DISK_SER (Java and Scala) 同MEMORY_AND_DSK，但是会序列化
DISK_ONLY 以非序列化的方式完全存储到磁盘上
MEMORY_ONLY_2, MEMORY_AND_DISK_2, etc. 尾部加了2的持久化级别。表示会将持久化数据复制一份，保存到其他节点
OFF_HEAP (实验性) 同MEMORY_ONLY_SER，但是将数据存储在堆外内存中。这需要启用堆外内存。

如何选择 RDD 的持久化策略？

Spark 提供了多种持久化级别，主要是为了在 CPU 和内存消耗之间进行取舍。下面是一些通用的持久化级别的选择建议：

  1. 优先使用 MEMORY_ONLY，纯内存速度最快，而且没有序列化不需要消耗 CPU 进行反序列化操作，缺点就是比较耗内存
  2. MEMORY_ONLY_SER，将数据进行序列化存储，纯内存操作还是非常快，只是在使用的时候需要消耗 CPU 进行反序列化

WARNING

如果需要进行数据的快速失败恢复，那么就选择带后缀为 _2 的策略，进行数据的备份，这样在失败时，就不需要重新计算了。

能不使用 DISK 相关的策略，就不要使用，除非计算数据集的函数很昂贵，或者它们过滤了大量的数据。因为有的时候，从磁盘读取数据效率，还不如重新计算一次。

# 4.共享变量

默认情况下，如果在一个算子函数中使用到了某个外部的变量，那么这个变量的值会被拷贝到每个Task 中。此时每个 Task 只能操作自己的那份变量数据。如果多个 Task 想要共享某个变量，就要使用 Spark 的共享变量（Shared Variables）。

Spark 提供了两种共享变量：

  • Broadcast Variable（广播变量）
  • Accumulator（累加变量）

# 4.1 Broadcast Variable

Broadcast Variable 会将使用到的变量，仅仅为每个节点拷贝一份，而不会为每个 Task 都拷贝一份副本。因此其最大的作用，就是减少变量到各个节点的网络传输消耗，以及在各个节点上的内存消耗。

通过调用 SparkContext 的 broadcast() 方法，针对某个变量创建广播变量。然后在算子函数内，使用到广播变量时，每个节点只会拷贝一份副本。可以使用广播变量的 value() 方法获取值。

TIPS

广播变量是只读的。

Broadcast Variable 的原理如下：

先看图中的左边：

  • 左边是我们经常使用的 map 算子的代码。map 算子中执行对每一个元素乘以一个固定变量的操作，此时这个固定的变量属于外部变量。
  • 默认情况下，算子函数内，使用到的外部变量，会被拷贝到执行这个算子的每一个 Task 中

再看图中间的 MapTask：

  • 这些都是 map 算子产生的 Task，也就是说外部变量 varable 会被拷贝到每一个 Task 中。如果这个外部变量是一个集合，集合中有上亿条数据，这个网络传输就会很耗时，而且在每个 Task 上占用的内存空间，也会很大。
  • 如果算子函数中使用的外部变量是广播变量，那么每个变量只会拷贝一份到每个 Worker 节点上。节点上所有的 Task 都会共享这一份变量，这样可以减少网络传输消耗的时间，以及减少内存占用。

大家可以想象一个极端情况，如果 map 算子有10 个 Task，恰好这10个 Task 还都在一个 Worker 节点上，那么这时 map 算子使用的外部变量就会在这个 Worker 节点上保存10份，这样是非常占用内存的。

Broadcast Variable 使用示例：

Scala 代码：

# 4.2 Accumulator

Spark 提供的 Accumulator，主要用于多个节点对一个变量进行共享性的操作。

正常情况下的 Spark 任务中，由于一个算子可能会产生多个 Task 并行执行，所以在这个算子内部执行的聚合计算都是局部的，想要实现多个 Task 进行全局聚合计算，此时需要使用到共享的累加变量 Accumulator。

TIPS

Accumulator 只提供了累加的功能。在 Task 只能对 Accumulator 进行累加操作，不能读取它的值。只有在 Driver 进程中才可以读取 Accumulator 的值。

  // 并且foreach算子可能会在多个task中执行，这样foreach内部实现的累加也不是最终全局累加的结果

Scala 代码实现：

import com.alibaba.fastjson2.JSON
import org.apache.spark.{SparkConf, SparkContext}

object TopNState extends App {

  val conf = new SparkConf()
  conf.setAppName("TopNState-Scala")
    .setMaster("local")
  val sc = new SparkContext(conf)
  // 1.使用fastjson包解析数据，获取的需要的字段
  val liveRecordInfoRDD = sc.textFile("/tmp/live_record_json.log")
  val giftRecordRDD = sc.textFile("/tmp/gift_record_json.log")

  // (live_show_id, (uid, area))
  val liveRecordInfoFieldRDD = liveRecordInfoRDD.map(line => {
    val jsonObj = JSON.parseObject(line)
    val live_show_id = jsonObj.getString("live_show_id")
    val uid = jsonObj.getString("uid")
    val area = jsonObj.getString("area")

    (live_show_id, (uid, area))
  })
  // (live_show_id, gold)
  val giftRecordFieldRDD = giftRecordRDD.map(line => {
    val jsonObj = JSON.parseObject(line)
    val liveShowId = jsonObj.getString("live_show_id")
    val gold = Integer.parseInt(jsonObj.getString("gold"))

    (liveShowId, gold)
  })

  // 2.对用户送礼记录数据进行聚合，对相同live_show_id的数据求和
  // (live_show_id, gold_sum)
  val giftRecordFieldAggRDD = giftRecordFieldRDD.reduceByKey(_ + _)

  // 3.把这两份数据join到一块，live_show_id作为join的key
  // (live_show_id, ((uid, area), gold_sum))
  val joinRDD = liveRecordInfoFieldRDD.join(giftRecordFieldAggRDD)

  // 4.使用map迭代数据，最后获取到uid、area、gold_sum字段
  // ((uid, area), gold_sum)
  val joinMapRDD = joinRDD.map(tup => {
    // joinRDD: (live_show_id, ((uid, area), gold_sum))
    // 获取uid
    val uid = tup._2._1._1
    // 获取area
    val area = tup._2._1._2
    // 获取gold_sum
    val goldSum = tup._2._2
    ((uid, area), goldSum)
  })

  // 5.使用reduceByKey算子对数据进行聚合
  // ((uid, area), gold_sum_all)
  val reduceRDD = joinMapRDD.reduceByKey(_ + _)

  // 6.接下来需要使用groupByKey对数据进行分组，所以先使用map进行转换
  // map：(area, (uid, old_sum_all))
  // groupByKey: (area, <(uid, gold_sum_all), (uid, gold_sum_all), (uid, gold_sum_all)>)
  val groupRDD = reduceRDD.map(tup => (tup._1._2, (tup._1._1, tup._2))).groupByKey()
  
  // 7.使用map迭代每个分组内的数据，按照金币数量倒序排序，取前N个，最终输出area,topN
  // (area, topN)
  val top3RDD = groupRDD.map(tup => {
    val area = tup._1
    // toList：把iterable转成list
    // sortBy：排序，默认是正序
    // reverse：反转，实现倒序效果
    // take(3)：取前3个元素
    // mkString：使用指定字符把集合转成字符串
    // uid:gold_sum_all,uid:gold_sum_all,uid:gold_sum_all
    val top3 = tup._2
      .toList // List<(area, <(uid, gold_sum_all), (uid, gold_sum_all), (uid, gold_sum_all)>)>
      .sortBy(_._2)
      .reverse
      .take(3)
      .map(tup => s"${tup._1}:${BigDecimal(tup._2.toString) * BigDecimal("0.01")}") // 金额单位使用的是分，转为元
      .mkString(",")
    (area, top3)
  })

  // 8.使用foreach将结果打印到控制台，多个字段使用制表符分割
  top3RDD.foreach(tup => println(s"${tup._1}\t${tup._2}"))

  sc.stop()

}

import com.alibaba.fastjson2.JSON;
import com.alibaba.fastjson2.JSONObject;
import com.google.common.collect.Lists;
import java.math.BigDecimal;
import java.util.ArrayList;
import java.util.Collections;
import org.apache.spark.SparkConf;
import org.apache.spark.api.java.JavaPairRDD;
import org.apache.spark.api.java.JavaRDD;
import org.apache.spark.api.java.JavaSparkContext;
import org.apache.spark.api.java.function.Function2;
import scala.Tuple2;

public class TopNState {

    public static void main(String[] args) {
        // 创建JavaSparkContext
        SparkConf conf = new SparkConf();
        conf.setAppName("TopNState-Java")
                .setMaster("local");
        JavaSparkContext sc = new JavaSparkContext(conf);

        // 1.使用fastjson包解析数据，获取的需要的字段
        JavaRDD<String> liveRecordInfoRDD = sc.textFile("/Users/willingoxjin/Codes/big-data/bigdata-demo/files/live_record_json.log");
        JavaRDD<String> giftRecordRDD = sc.textFile("/Users/willingoxjin/Codes/big-data/bigdata-demo/files/gift_record_json.log");
        // (live_show_id, (uid, area))
        JavaPairRDD<String, Tuple2<String, String>> liveRecordInfoFieldRDD = liveRecordInfoRDD.mapToPair(line -> {
                JSONObject jsonObj = JSON.parseObject(line);
                String liveShowId = jsonObj.getString("live_show_id");
                String uid = jsonObj.getString("uid");
                String area = jsonObj.getString("area");
                return new Tuple2<>(liveShowId, new Tuple2<>(uid, area));
            });

        // (live_show_id, gold)
        JavaPairRDD<String, Integer> giftRecordFieldRDD = giftRecordRDD.mapToPair(line -> {
                JSONObject jsonObj = JSON.parseObject(line);
                String liveShowId = jsonObj.getString("live_show_id");
                Integer gold = Integer.parseInt(jsonObj.getString("gold"));
                return new Tuple2<>(liveShowId, gold);
            });

        // 2.对用户送礼记录数据进行聚合，对相同live_show_id的数据求和
        // (live_show_id, gold_sum)
        JavaPairRDD<String, Integer> giftRecordFieldAggRDD = giftRecordFieldRDD.reduceByKey((i1, i2) -> i1 + i2);

        // 3.把这两份数据join到一块，live_show_id作为join的key
        // (live_show_id, ((uid, area), gold_sum))
        // 4.使用map迭代join之后的数据，最后获取到uid、area、gold_sum字段
        // ((uid, area), gold_sum)
        JavaPairRDD<Tuple2<String, String>, Integer> joinMapRDD = liveRecordInfoFieldRDD.join(giftRecordFieldAggRDD)
                .mapToPair(tup -> {
                    // joinRDD:(live_show_id, ((uid, area), gold_sum))
                    // 获取uid
                    String uid = tup._2._1._1;
                    // 获取area
                    String area = tup._2._1._2;
                    // 获取gold_sum
                    Integer goldSum = tup._2._2;
                    return new Tuple2<>(new Tuple2<>(uid, area), goldSum);
                });
        // 5.使用reduceByKey算子对数据进行聚合
        // ((uid, area), gold_sum_all)
        JavaPairRDD<Tuple2<String, String>, Integer> reduceRDD = joinMapRDD.reduceByKey(
                (Function2<Integer, Integer, Integer>) (i1, i2) -> i1 + i2);
        // 6.接下来对需要使用groupByKey对数据进行分组，所以先使用map进行转换
        // map：(area, (uid, gold_sum_all))
        // groupByKey: area,<(uid, gold_sum_all),(uid, gold_sum_all),(uid, gold_sum_all)>
        JavaPairRDD<String, Iterable<Tuple2<String, Integer>>> groupRDD = reduceRDD
                .mapToPair(tup -> new Tuple2<>(tup._1._2, new Tuple2<>(tup._1._1, tup._2)))
                .groupByKey();

        // 7.使用map迭代每个分组内的数据，按照金币数量倒序排序，取前N个，最终输出area,topN
        // (area, topN)
        JavaRDD<Tuple2<String, String>> top3RDD = groupRDD.map(tup -> {
                String area = tup._1;
                ArrayList<Tuple2<String, Integer>> tupleList = Lists.newArrayList(tup._2);
                // 对集合中的元素排序
                Collections.sort(tupleList, (t1, t2) -> t2._2 - t1._2);
                StringBuffer sb = new StringBuffer();
                for (int i = 0; i < tupleList.size(); i++) {
                    if (i < 3) {//top 3
                        Tuple2<String, Integer> t = tupleList.get(i);
                        if (i != 0) {
                            sb.append(",");
                        }
                        BigDecimal sumGold = new BigDecimal(t._2).multiply(new BigDecimal("0.01"));
                        sb.append(t._1 + ":" + sumGold.doubleValue());
                    }
                }
                return new Tuple2<>(area, sb.toString());
        });

        // 8.使用foreach将结果打印到控制台，多个字段使用制表符分割
        // area topN
        top3RDD.foreach(tup -> System.out.println(tup._1+"\t"+tup._2));

        sc.stop();
    }

}
