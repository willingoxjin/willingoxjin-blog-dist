1.Hive 基础

# 一、Hive 介绍

# 1.Hive

Hive 是在 Hadoop 分布式文件系统 (HDFS) 之上开发的 ETL 和数据仓库工具，有 Facebook 实现并开源。

Hive 定义了简单的类 SQL 查询语言，称为 HQL，它允许熟悉 SQL 的用户直接查询 Hadoop 中的数据，同时，这个语言也允许熟悉 MapReduce 的开发者开发自定义的 mapreduce 任务来处理内建的 SQL 函数无法完成的复杂的分析任务。

Hive 中包含 SQL 解析引擎，它会将 SQL 语句转译成 M/R Job，然后在 Hadoop 中执行。

Hive 的数据存储：

  • Hive 其数据存储基于 Hadoop 的 HDFS
  • Hive 没有专门的数据存储格式
  • Hive 默认可以直接加载文本文件（TextFile），还支持 SequenceFile、RCFile 等文件格式
  • 针对普通文本数据，我们在创建表时，只需要指定数据的列分隔符与行分隔符，Hive 就能解析里面的数据

# 2.Hive 的系统架构

Hive 主要由 Clients、Serverices、Storage and compute 组成。

  • Clients（客户端）：Hive 为与不同类型的应用程序的通信提供了不同的驱动程序。Hive clinets 可以支持 Thrift、JDBC、ODBC 协议与 Hive services 进行通信。
  • Serverices（服务）：Clients 与 Hive 的交互通过 Hive Services 执行。CLI 是命令行界面，充当 DDL（数据定义语言）操作的 Hive 服务。Serverices 核心的模块是 Driver（驱动程序）。
    • Hive Services、CLI、Hive web Interface 都通过 Driver（驱动程序）对元数据、文件系统进行计算和处理。
  • Storage and compute（存储和计算）：Hive 的元数据存储在元数据数据库中，元数据数据库支持 Mysql 等多种类型。Hive 的查询结果和数据存储在 Hadoop 中。

整个流程：用户通过接口传递 Hive SQL，然后 Driver 对SQL进行分析、编译，生成查询计划，查询计划会存储在HDFS 中，然后再通过 MapReduce 进行计算出结果。

Metastore

  • Metastore 是 Hive元数据的集中存放地。
  • Hive 中的元数据包括：表的名字、表的列和分区及其属性、表的属性（是否为外部表等）、表的数据所在的hdfs目录等。
  • Metastore 默认使用内嵌的 derby 数据库
    • Derby数据库的缺点：在同一个目录下一次只能打开一个会话
    • 使用derby存储方式时，Hive会在当前目录生成一个 derby.log 文件和一个 metastore_db 目录，metastore_db 里面会存储具体的元数据信息。如果下次切换到一个另一个新目录访问 Hive，则会重新生成 derby.log 文件 和metastore_db 目录，这样就没有办法使用之前的元数据信息了。
    • 推荐使用MySQL作为外置存储引擎，可以支持多用户同时访问以及元数据共享。

# 3.Hive VS Mysql

Hive MySQL
数据存储位置 HDFS 本地磁盘
数据格式 用户定义 系统决定
数据更新 不支持(不支持修改和删除) 支持(增删改查)
索引 有，但较弱，一般很少用 有，经常使用
执行 MapReduce Executor
执行延迟 高 低
可扩展性 高 低
数据规模 大 小

# 4.数据库 VS 数据仓库

数据库：

  • 传统的关系型数据库主要应用在基本的事务处理，例如银行交易之类的场景
  • 数据库支持增删改查这些常见的操作。

数据仓库：

  • 主要做一些复杂的分析操作，侧重决策支持，相对数据库而言，数据仓库分析的数据规模要大得多。但是数据仓库只支持查询操作，不支持修改和删除

数据库与数据仓库的本质区别就是 OLTP 与 OLAP 的区别：

  • OLTP(On-Line Transaction Processing)
    • 操作型处理，称为联机事务处理，也可以称为面向交易的处理系统，它是针对具体业务在数据库联机的日常操作，通常对少数记录进行查询、修改。用户较为关心操作的响应时间、数据的安全性、完整性等问题
    • OLTP侧重于事务
  • OLAP(On-Line Analytical Processing)
    • 分析型处理，称为联机分析处理，一般针对某些主题历史数据进行分析，支持管理决策
    • OLAP侧重于分析

Hive 就是一个数据仓库，更像一个传统意义上的SQL数据库，主要用作数据仓库数据分析，更加强调离线的数据分析，为公司各个业务部门提供数据支撑。

import java.sql.Connection;
import java.sql.DriverManager;
import java.sql.ResultSet;
import java.sql.Statement;

public class HiveJdbcDemo {

    public static void main(String[] args) throws Exception {
        // 加载Hive JDBC驱动程序
        Class.forName("org.apache.hive.jdbc.HiveDriver");
        // 指定hiveserver2的连接
        String jdbcUrl = "jdbc:hive2://hive01:10000";
        // 获取jdbc连接，这里的user=root(linux用户名)，password随便指定
        Connection conn = DriverManager.getConnection(jdbcUrl, "root", "");
        // 获取Statement
        Statement stmt = conn.createStatement();
        // 执行查询
        String sql = "select * from test_user";
        ResultSet res = stmt.executeQuery(sql);
        // 处理结果集
        while (res.next()){
            // 处理每一行数据
            System.out.println(res.getInt("id")+"\t"+res.getString("name"));
        }
        // 关闭连接
        // conn关闭后，stmt和res也会自动关闭
        conn.close();
    }

}

# 3.Set 命令

在 hive 命令行中可以使用 set 命令临时设置一些参数的值，其实就是临时修改 hive-site.xml 中参数的值。

通过 set 命令设置的参数只在当前会话有效，退出重新打开就无效了，如果想要对当前机器上的当前用户有效的话可以把命令配置在 ~/.hiverc 环境变量文件中。在 hive-site.xml 中配置则是永久有效。

在 hive-site.xml 中的参数：

  • hive.cli.print.current.db 设置为 true ，可以显示当前所在的数据库名称，默认值为 false。
  • hive.cli.print.header 可以控制获取结果的时候显示字段名称

hive> set hive.cli.print.current.db = true;
hive (default)> set hive.cli.print.header = true;
hive (default)> select * from test_user;

# 1.数据库操作

  • 查看数据库列表

default 是默认数据库，默认就在这个库里面 咱们前面说过 hive 的数据都是存储在 hdfs 上，那这里的 default 数据库在 HDFS 上是如何体现的？

在 hive-site.xml 中有一个参数 hive.metastore.warehouse.dir ，表示 hive 的 default 默认数据库对应的 hdfs 存储目录。默认值是 /user/hive/warehouse。

在 HDFS 上看一下，发现确实有这个目录。这个目录下还有一个目录：test_user，也就是我们之前创建的 test_user 表。

可以看出，hive 中的数据库和 hive 中的表，在 hdfs 上的体现都是目录。

hive-site.xml 中有一个参数 hive.metastore.warehouse.dir 的配置：

<property>
    <name>hive.metastore.warehouse.dir</name>
    <value>/user/hive/warehouse</value>
    <description>location of default database for the warehouse</description>
</property>

Map 和 Struct 的区别

Map：

  • Map 在建表语句中需要指定 key-value 的类型
  • Map 在查询时通过 [key] 取值

  • Struct 的源数据中只需要有 value 即可

# 六、Hive 表

Hive 中的表类型可以分为四种：内部表、外部表、分区表、桶表

# 1.内部表

内部表也可以称为受控表。它是 hive 中的默认表类型，表数据默认存储在 warehouse 目录下

在加载数据（使用 load data 命令）的过程中，实际数据会被移动到 warehouse 目录下。当我们删除表时，表中的数据和元数据将会被同时删除。

实际上，我们前面创建的表都属于受控表，创建一张表，就会在 metastore 中存储该表的元数据信息。当从 hive 中删除某一张表之后，该表中的数据会被删除，在 metastore 中该表存储的元数据信息也会被删除。

# 2.外部表

外部表在加载数据的时候，实际数据并不会移动到 warehouse 目录下，只是与外部数据建立一个链接

（映射关系）。

表的定义和数据的生命周期互相不约束，数据只是表对 hdfs 上的某一个目录的引用，当删除表定义的时候，数据仍然存在的，删除的只是表和数据之间引用关系。所以这种表是比较安全的，就算是误删表，数据也不会丢失。

扩展表在建表语句中增加了 EXTERNAL， 以及在最后通过 locatin 指定表数据的存储位置（存储位置是是 hdfs 中的路径）。如下：

# 3.分区表

分区表：通过分区把不同类型的数据放到不同目录中。

应用场景：

  • web 服务每天都会产生的日志数据。通过 Flume 把日志数据采集到 hdfs 中，根据日志产生的日期（按天分区，如：2023-03-01），将日志存储到对应的 hdfs 目录下。
  • 这样做的好处是优化查询。
    • 一般我们查询日志都是按照天来查询，如果文件都放在一个目录下，hive 在查询数据的时候，会把该目录下的数据都查询一遍，就会导致服务器资源占用变大，效率也很低。
    • 按照分区表来做，就可以根据查询的日期，定位到对应的日期目录，能够提升查询的效率。

分区的标准就是指定分区字段，分区字段可以有一个或多个。

分区表的意义在于优化查询，查询时尽量利用分区字段，如果不使用分区字段，就会全表扫描，最典型的一个场景就是把天作为分区字段，查询的时候指定天

# 4.桶表

桶表是对数据进行哈希取值，根据取值存储到对应的文件中。

在物理卷上，每个桶就是表（或分区）里的一个文件。

桶表的使用场景：

  • 人口统计。如果按照省份来分区，就会导致某些分区数据过多（河南、山东、安徽、江苏等），而某些分区数据很少（如西藏、新疆、海南等），也就是造成数据倾斜。这样对数据存储以及查询不太友好，计算效率也不高，应该相对均匀的存放数据，这时就可以使用桶表。

示例：

step.1 创建桶表

按照 id 进行分桶，分成四个桶：

create table bucket_tbl (
    id int,
    name string
) clustered by (id) into 4 buckets;

桶表的作用：

  • 数据抽样。

    • 如果使用一个大规模的数据集，只想去抽取部分数据进行查看，使用 bucket 表可以变得更加的高效。

    • 如下语句：

      # 语法
select * from bucket_tb
	tablesample (bucket [x] out of [y] on [column_field);

# 示例：根据id对桶表中的数据重新分桶,将桶分为4桶，取出第一桶中的数据
select * from bucket_tb
	tablesample (bucket 1 out of 4 on id);
      1
      2
      3
      4
      5
      6
      7
      • tablesample(xxx) 是抽样语句
      • column_field 分桶因子，即分桶的字段
      • y 表示是把桶表中的数据随机分为多少个桶
      • x 表示取出第几桶的数据
      • y 尽可能是桶表的 bucket 数的倍数或者因子，且 y 大于等于 x

    • select […] tablesample 语句可以在任何表中执行，但在桶表中的执行效率更高。可以使用 explain 命令查看执行计划

  • 提高某些查询效率。

    • 如：join 查询，可以避免产生笛卡尔积的操作

      select a.id,a.name,b.addr from a join b on a.id = b.id;
      1
      • 如果表 a 和表 b 已经是分桶表，且分桶的字段是id字段，那么做这个操作的时候就不需要再进行全表笛卡尔积了。因为分桶之后相同规则的 id 已经在相同的文件里面了，这样表 a 的每个桶就可以和表 b 的每个桶直接 join，以此避免 join 时的全表扫描 。
