3.Hive扩展

# 一、Hive 数据倾斜

# 1.数据倾斜出现场景

可能会触发 Hive 数据倾斜的几种情况如下：

关键字 场景 影响
join 大表与小表，小表中的key集比较集中 分发到某一个或几个Reduce上的数据远高于平均值
join 大表与大表，但是join中指定的字段0值或空值过多 这些空值都由一个reduce处理，非常慢
group by group by维度过小，处理的数量过多 处理某值的reduce非常耗时

出现数据倾斜的原因：

  • key 分布不均匀
  • 业务数据本身的特性
  • 建表时考虑不周
  • SQL 语句本身就有数据倾斜

# 2.数据倾斜问题的表现

  • 任务进度长时间维持在 99%（或100%），查看任务监控页面，发现只有少量（1个或几个）reduce 子任务未完成。因为其处理的数据量和其他 reduce 差异过大。
  • 单个 reduce 的记录数与平均记录数差异过大，通常可能达到 3 倍甚至更多。 最长时长远大于平均时长。

# 3.数据倾斜的解决方案

1.参数调节

  1. 参数 hive.map.aggr Map 端部分聚合，相当于Combiner
hive.map.aggr=true
1
  1. 参数hive.groupby.skewindata 有数据倾斜的时候进行负载均衡。
hive.groupby.skewindata=true
1
  • 当选项设定为 true，生成的查询计划会有两个 MR Job。
    • 第一个 MR Job 中，Map 的输出结果集合会随机分布到 Reduce 中，每个 Reduce 做部分聚合操作，并输出结果，这样处理的结果是相同的 Group By Key 有可能被分发到不同的 Reduce 中，从而达到负载均衡的目的；
    • 第二个 MR Job 再根据预处理的数据结果按照 Group By Key 分布到 Reduce 中（这个过程可以保证相同的 Group By Key 被分布到同一个 Reduce 中），最后完成最终的聚合操作。

2.SQL语句调节

  • 大小表 join：
    • 使用 map join 让小的维度表（1000条以下的记录条数） 先进内存，在 map 端完成 join 操作。
    • 正常情况下 join 是需要在 reduce 端执行的，通过 map join 可以实现在 map 端执行 join 操作，这样可以避免在 shuffle 的时候造成数据倾斜。
  • 大表 join 大表：
    • 把空值的 key 变成一个字符串加上随机数，把倾斜的数据分到不同的 reduce 上，由于 null 值关联不上，处理后并不影响最终结果。

示例

现在有一个表 test_table ，其中有字段 key，key=KEY001 的数据占了所有数据的 90%（key 对应的数据存在数据倾斜）。

我们需要查询出每个 key 对应的数据条数，设置 reduce 任务总数为 50 个。

如果直接使用 SELECT key, COUNT(*) FROM test_table GROUP BY key，就会导致 key=KEY001 的数据所在的 reduce 任务需要执行的数据量占 90% 以上，且其他 reduce 任务仅仅执行了很少的数据计算，那我们设置 50 个 reduce 任务就没啥效果，影响计算效率最终查询会非常的慢。

解决方案：我们将数据尽可能平均的分到每个 reduce 中

  • 由于 key=KEY001 的数据占 90%，所以可以把 key=KEY001 的数据打散，分到每个 reduce 中（50份），进行局部聚合。
  • 最后再通过外层的 select 进行全局的聚合，将所有 reduce 任务执行的结果再分组计算，这样就可以显著提高计算效率

下面的 SQL 用来解决我们遇到的数据倾斜：

SELECT 
	a.key,
    SUM(a.total) AS total
FROM (
    SELECT 
    	`key`
        COUNT(*) AS total
    FROM test_table
    GROUP BY 
   		key,
      	CASE
       		WHEN key = 'KEY001' THEN Hash(random()) % 50
       		ELSE 0
       	END
) a
GROUP BY a.key;

# 二、Hive 数据存储格式

Hive 主要包括下面这些数据存储格式：

  • TextFile

    • Hive 中默认使用 TextFile 格式的数据，这种格式的数据存储占用的空间较大，占用存储空间，也影响计算效率。

  • SequenceFile

  • RCFile

    • RCFile 存储格式是从 Hive 0.6 版本开始支持的

  • Avro

    • Avro 存储格式是从 Hive 0.9 版本开始支持的

  • ORC

    • ORC 存储格式是从 Hive 0.11 版本开始支持的

  • PARQUET

    • PARQUET 存储格式是从 Hive 0.13 版本开始支持的

参考：https://cwiki.apache.org/confluence/display/Hive/FileFormats (opens new window)

实际工作中使用最多的存储格式是 TextFile、ORC 和 Parquet。Avro 基本上用不到。

默认情况下使用 TextFile 即可，想要提高数据存储和计算效率，可以考虑使用 ORC 或者 Parquet。

# 1.TextFile

TextFile 是 Hive的默认数据存储格式，基于行存储。它的主要特点是磁盘存储开销大，数据解析开销大。

  • 磁盘存储开销大：因为存储的是原始文件内容，没有使用压缩，所以存储开销会比较大。
  • 数据解析开销大：因为在反序列化读取数据的过程中，必须逐个字符判断是不是字段分隔符和行结束符，所以数据解析开销大。

一起来看一下，Hive 中如何集成数据压缩格式，以及对数据切分的支持程度。

# 2.SequenceFile

SequenceFile 是一种二进制文件，内部数据是 <Key,Value> 的形式，也属于行存储。

它的特点是使用方便，MapReduce 原生支持这种数据格式，并且它还支持切分，也支持压缩。

SequenceFile 支持 NONE、RECORD、BLOCK 级别的压缩。

  • NONE 表示不压缩
  • RECORD 表示行级别的压缩
  • Block 表示块级别的压缩

由于 Record 是针对每一条数据分别进行压缩，压缩率比较低，所以一般都会选择 Block 压缩。

SequenceFile 文件自身就支持切分，我们进行的压缩是针对文件内部的数据进行压缩，不会改变 SequenceFile 的特性。

在 SequenceFile 中使用压缩的时候就不需要考虑压缩格式自身是否支持切分的特性了，需要考虑的是压缩格式的压缩比、以及压缩解压速度这些指标。

下图是 SequenceFile 存储格式的结构：

  • 图中最上面是 SequenceFile中的数据，每一个 Record 代表一条数据
  • 当没有压缩的时候，Record 内部存储的是 Record 的长度、key 的长度、key 和 value
  • 当使用 Record 级别压缩的时候，Record 内部存储的是 Record 的长度、key的长度、key 和压缩之后的 value。
  • 当使用 Block 级别压缩的时候，SequenceFile 中的数据是以 Block 为单位存储的，每个 Block 中存储多个 Record，并且对 Block 内部的多个 Record 统一压缩存储。所以这里的压缩是针对SequenceFile 内部数据的压缩，并没有改变 SequenceFile 自身的特性，所以文件依然支持切分。

            // 打印文件名称，内容

        }
        reader.close();
    }
}

# 3.RCFile

RCFile 是专门为 Hive 设计的数据存储格式，主要特点是压缩速度快，可分割，支持快速列存取。

RCFile 格式的数据会首先按照行分组，每个组内部按照列存储。其整合了行存储和列存储的优点，可以称为是行列式存储，大层面还是属于列式存储的。

在读取所有列的情况下，RCFile 的性能没有 SequenceFile 高，不过在实际工作中大部分的统计分析需求不会读取所有列。

RCFile 的数据存储格式图：

  • 关系型数据库数据存储格式

    • 左边的 Relation 表示关系型数据库中的数据存储形式，可以把它认为是一张表，表中有A\B\C\D这几个列。

    • 第一行数据中 A的值为101，B的值为111，C的值为121，D的值为131。 后面数据以此类推。

  • RCFile 储格式存

    • 看中间的图，RCFile 是存储在 HDFS 里面的，所以说会有多个 HDFS 的 Block 块。 每个 Block 块内部包含多个 Row Group。RowGroup 可以翻译为：行组。
    • RowGroup 包含的是具体的数据。存储格式看右边图：
      • 第一行数据是所有A字段的值，因为 RowGroup 是行列式存储，所以会把原始表中的每个列的数据存储到一起，便于查询这个列的数据。
      • 第二行数据是B字段所有的值。后面的行以此类推

所以从图里面可以看出来，RCFile 会把数据首先按照行分成 Row Group，在 Row Group 内部按照列存储，每个列的数据存储在一起。

# 4.ORC Files

ORC 是在 RCfile 的基础之上做了一些升级，在性能层面有大幅度提升。

ORC 的存储结构如下：

  • ORC 中的数据首先会被划分为多个Stripe，每个 Stripe 250M。Stripe 表示 ORC 文件存储数据的地方。

  • 在 Stripe 内部包含了 Index Data，Row Data 和 Stripe Footer。

    • Index Data：存储索引数据。

    • Row Data：存储具体的数据，这里面有多个行组，每10000行构成一个行组。

    • Stripe Footer 中存储的是数据所在的文件目录。

  • File Footer：File Footer 里包含了 ORC 文件中 Stripe 的列表、每个 Stripe 的行数，以及每个列的数据类型。他还包含了每个列的最小值、最大值等信息。

  • Postscript：Postscript 包含了压缩参数和压缩大小相关的信息。

ORC 存储格式和 RCFile 存储格式在存储形式上没有特别大的区别，核心思想都是行列式存储。

在创建 ORC 数据存储格式表的时候，所有关于 ORC 的参数都在建表语句中的 tblproperties 指定，不需要在 hive 的命令行中使用 set 命令指定。

如下是在创建表时，指定表的 ORC 示例：

create table t1 (
  id int,
  name string
) stored as orc tblproperties("orc.compress"="NONE")
1
2
3
4

tblproperties 中可以指定多个参数，多个参数之间使用逗号分割即可

ORC 支持的参数如下：

Key Default Notes
orc.bloom.filter.columns "" 表示针对哪些列创建布隆过滤器，默认不创建。
orc.bloom.filter.fpp 0.05 通过布隆过滤器可以通过较少的文件空间快速判断数据是否存在表中，但是也会存在将不属于这个表的数据判定为属于这个表的情况，这其实就是布隆过滤器的特性，这个特性可以称之为假正概率，我们可以调整这个概率。设置的概率越低，出现的误差就越小，但是布隆过滤器所需要的空间就越大。这个概率的取值范围在0~1之间，大于0，小于1。
orc.compress ZLIB 表示 ORC 文件的压缩类型，支持NONE、ZLIB、SNAPPY 和 LZO。这里的 ZLIB 其实就和 Deflate 压缩格式是类似的，因为 Deflate 压缩格式底层就是用的ZLIB。（由于版本迭代，支持类型会有增减，具体支持类型见 hive 源码中的 org.apache.hadoop.hive.ql.io.orc.CompressionKind 枚举）
orc.compress.size 262,144 Byte 表示每个压缩块的大小，默认256KB
orc.stripe.size 67,108,864 stripe 是 ORC 文件中存储数据的单位。这个配置表示在生成 stripe 的时候可以使用的内存缓冲池大小，默认值是64M。这里面显示的是字节。
orc.row.index.stride 10,000 行组级别索引的数据量大小，默认是10000，必须要设置为大于等于10000的数值。表示每10,000行数据，建一次索引。
orc.create.index true 是否创建行组级别索引，默认是true。

见 https://cwiki.apache.org/confluence/display/Hive/LanguageManual+ORC (opens new window)

# 5.Parquet

Parquet 是一种新型的与语言无关的，并且不和任何一种数据处理框架绑定的列式存储结构，适配多种语言和组件。Parquet 数据存储格式可以在 Hive、Impala、Spark 等计算引擎中使用。

Parquet 文档：

  • https://cwiki.apache.org/confluence/display/Hive/Parquet (opens new window)
  • https://github.com/apache/parquet-mr (opens new window)
  • http://parquet.incubator.apache.org/docs/ (opens new window)

Parquet 的存储格式如下：

  • Parquet 格式的文件中包含了多个 Row Group。
  • 每个Row Group 内部包含了多个 Column。
  • Column 内部包含了多个 Page。
  • Page 内部存储了具体的数据内容。

Parquet 存储格式比 ORC 更复杂一些，多封装出来一层 Page。ORC 存储格式是把数据内容直接存储到了 Column 这一层。

Parquet 支持的压缩格式如下

  • uncompressed。默认 uncompressed。
  • snappy
  • gzip
  • lzo
  • brotli
  • ``lz4`
  • zstd

apache/parquet-mr (opens new window) 中的子项目 parquet-hadoop (opens new window) 的文档的 parquet.compression 参数中查看。

可以看到，ORC 格式在数据存储层面性能最优。

在实际工作中，如果希望减少 Hive 中的数据存储量，建议使用 ORC 数据存储格式。

# 6.2 压缩速度

在使用 ORC 数据存储格式的时候，最好再配合上合适的压缩格式，这样可以进一步挖掘 ORC 的存储性能。

但是 ORC 自身存储能力已经非常好了，并且也支持切分。所以在选择压缩格式时只需要重点考虑压缩和解压速度就行了。

常见的压缩格式中，Lzo、Snappy 的压缩和解压速度是最快的。而 Snappy 的压缩和解压速度相对更好一些。如下图：

压缩速度 解压速度
Lzo 135 MB/s 410 MB/s
Snappy 172 MB/s 409 MB/s

注意：在测试解压缩速度时，最好不要使用虚拟机测试。测试速度可能会因为服务器主机性能不同。

所以，如果再考虑减少数据存储和提高压缩速度，建议在工作中选择 ORC + Snappy。

# 6.3 兼容性

但是，有时候在选择数据存储格式的时候还需要考虑它的兼容度，是否支持多种计算引擎。

假设这一份数据，既想要可以在 Hive 中使用，还希望在 Impala 中使用，这个时候就需要重点考虑这个数据格式是否能满足多平台同时使用。

Impala 很早就开始支持 Parquet 类型的数据格式，但是从 Impala 3.x 版本才开始支持 ORC 。

总结：

  • 如果工作中使用的是 Impala 3.x 版本，建议使用 ORC + Snappy
  • 如果工作中使用的是 Impala 2.x 版本，还想要支持 Hive 和 Impala 查询同一份数据，这个时候就需要选择 Parquet 了。

相对来说，Parquet 存储格式在大数据生态圈中的兼容度是最高的。

# 2.Hive客户端日志突然变多

若突然在使用 Hive 客户端时，出现嘞非常多的日志，每次输入命令后执行都会打印非常多的日志。

该问题的原因是日志依赖的冲突导致的，可能是 Flink 客户端或其他的软件导致的。

如，在 Flink 客户端使用时，需要在环境变量中配置 HADOOP_CLASSPATH 环境变量，若再使用 Hive 的时候就会遇到这个问题了。

解决这个问题，有以下方式：

  • 方式一：使用 Hive 中的 beeline 客户端`。（beeline 不受日志影响）
  • 方式二：注释或者删掉 HADOOP_CLASSPATH 环境变量，重启服务器后，即可正常使用 Hive 客户端，但是这样会导致 Flink 客户端无法正常使用。
  • 方式三：将 Flink 客户端和 Hive 客户端分开部署，只要不同时在一个机器中即可。
