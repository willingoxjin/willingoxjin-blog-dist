1.Impala 基础

# 一、Impala 介绍

# 1.Impala 概述

Impala 是一种高性能、分布式 SQL 查询引擎，最初由 Cloudera 开发。它是一种基于 Apache Hadoop 的分布式计算系统，可以处理大规模数据集。

Impala 支持标准 SQL 语言和 Hadoop 生态系统中的各种数据格式，包括 HDFS、HBase、Kudu 等。

Impala 可以直接使用 Hive 的 Metastore。Hive 中创建的表，Impala 可以直接使用。并且兼容 HiveSQL（大部分的 Hive SQL 语法都是兼容的）。

# 2.Impala 的优点

  1. 基于内存运算，中间结果不落盘，节省大量的 I/O 开销；
    • 在 MapReduce 任务中，Map 阶段产生的中间结果需要先写磁盘，再通过 Shuffle 拷贝到 Reduce 阶段，所以会产生大量的 I/O，因此计算效率不如 Impala。
  2. 底层计算引擎由 C++ 编写，通过 LLVM 统一编译运行，效率更高。
    • LLVM 是一个编译器。Impala 底层的计算引擎没有使用大数据生态圈中的计算框架，它是自己单独实现的。
  3. 兼容 HiveSQL 语法，学习成本低，容易上手。
    • ImpalaSQL 支持 SQL92 标准，并具有自己的解析器和优化器。
    • SQL92 是数据库的一个 ANSI 标准，HiveSQL 也支持 SQL92 标准。
  4. 兼容 Hive 的 Metastore，可以直接使用 Hive 中的表。
  5. 支持数据本地化（Data Locality）特性，提高计算效率。
    • 即 Impala 支持数据本地计算，前提是 Impala 的服务需要和 HDFS 的 datanode节 点部署在一起。
  6. 支持列式存储数据格式，可以和 HBase 整合。也就是说 Impala 可以通过 SQL 计算 HBase 中的数据。
  7. 支持多种文件格式。
    • Hive 支持的数据格式，Impala 基本都是支持的。例如：TextFile、SequenceFile、RCFile、ORC和Parquet。
  8. 支持 ODBC/JDBC 远程访问，可以无缝对接到 Web 项目，在页面操作起来更方便。

# 3.Impala 的缺点

  1. 基于内存进行计算，对内存依赖性较大。
    • 基于内存计算，既是优点，也是缺点。
  2. 使用 C++ 编写，意味着维护难度增加。相对而言，Java 程序维护起来比较简单。
  3. 分区数量超过一万时，性能严重下降，容易出现问题。
    • 主要是因为它是基于内存计算，单个 SQL 任务计算的数据量过大，对内存依赖就很大了，所以容易出现问题。
  4. 基于 Hive，与 Hive 共存亡。
    • 因为 Impala 需要使用 Hive 的 Metastore 来存储元数据信息。
  5. 稳定性不如 Hive。
    • 主要也是因为 Impala 是基于内存的，所以稳定性相对来说一般。
  6. 只能读取文本文件，而不能直接读取自定义二进制文件。
  7. 每当新的记录/文件被添加到HDFS中的数据目录时，该表需要被刷新。

# 4.Impala 与 Hive

Impala 和 Hive 之间的关系如下图：

最底层存储使用 HDFS 和 MapReduce，Hive 对 MapReduce 做了封装，提供了 SQL 支持。

Hive 的元数据信息存储在 Metastore 中，Metastore 一般使用 MySQL。

Impala 自己使用 C++ 实现了一套分布式计算引擎，也可以直接操作 HDFS 中的数据，并且和 Hive 共享一个 Metastore。

# 5.Impala 三大核心组件

Impala 架构中主要包含三大核心组件：

  • Impala daemon：简称为 impalad。
    • 它是 Impala 集群中的核心守护进程，需要在多个节点启动，一般会和 DataNode 节点部署在一起；
    • 它主要负责向 Statestore 保持通信，汇报节点健康状态。同时负责接收客户端的请求，执行查询，并把结果返回给客户端。
  • Statestore daemon：简称为 statestored。
    • 主要负责 metadata 的广播，收集和检查集群中各个 Impalad 进程的资源信息、各节点健康状况、同步节点信息，以及负责 Query 的协调调度。
  • Catalog daemon：简称为 catalogd。
    • 主要负责分发表的元数据信息到各个 Impalad 中，以及接收来自 Statestore 的所有请求。

# 6.Impala 架构

Impala 的整体架构如下：

这个图里面包含了客户端、Hive Metastore、HDFS NameNode、HDFS DataNode、HBase 以及 Impala 中的 Statestore、Catalog、Impalad 三个核心组件。

其中 Hive Metastore 负责维护 Impala 元数据信息的，HDFS 和 HBase 都是可以作为 Impala 底层数据存储的组件。

客户端向 Impala 发送请求时的执行流程如下：

  1. 客户端向某一个 Impalad 发送一个 Query（SQL）。
  2. Impalad 将 Query 解析为具体的 Planner（执行计划），然后交给当前机器的 Coordinator（中心协调节点）。
  3. Coordinator 根据 Planner（执行计划），通过本机的 Executor 执行，并转发给其他有数据的 Impalad。
    • 多个 Impalad 的 Executor 之间会进行通信，可能需要一些数据的处理。
  4. 各个 Impalad 的 Executor 执行完成后，会将结果返回给 Coordinator。
  5. 最后由 Coordinator 将汇聚的查询结果返回给客户端。

# 7.Impalad 服务的三种角色

  • Query Planner
    • 由 Java 编写，用来解析 SQL 生成 QueryPlanTree 执行计划树
  • Query Coordinator
    • 用户在 Impala 集群上的某个节点提交数据处理请求（例如 impala-shell 提交SQL），则该Impalad 节点称为 Coordinator Node（协调节点），负责定位数据位置，拆分请求（Fragment），将任务分解为多个可并行执行的小请求，发送这些请求到多个Query Executor，接收Query Executor处理后返回的数据并构建最终结果返回给用户。
  • Query Executor
    • 执行数据计算，比如 Scan，Aggregation，Merge 等，返回数据。

# 二、Impala 安装

由于单独安装 Impala 需要自己编译，并且编译的时候需要依赖很多外部依赖，以及一些依赖冲突的问题，所以不建议单独安装。

一般都通过 CDH 平台安装 Impala，CDH 集成了 Impala，管理、安装都很方便。
