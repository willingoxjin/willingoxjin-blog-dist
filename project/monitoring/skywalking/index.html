Skywalking 使用

本文基于 Skywalking 9.2.0，理论支持 Skywalking 所有版本。

# 一、认识 SkyWalking

官方文档地址：https://skywalking.apache.org/docs/ (opens new window)

我这里使用的使用 9.2.0 的版本：https://skywalking.apache.org/docs/main/v9.2.0/readme/ (opens new window)

中文文档 6.2.0 ：https://www.itmuch.com/books/skywalking (opens new window)

# 1.介绍

SkyWalking 是一个开源的监控应用的平台，用于收集、分析、聚合和可视化来自服务和云原生基础设施的数据。SkyWalking 提供了简单的方法来保持分布式系统的清晰视图。它是一个现代化的 APM，专为云原生、基于容器的分布式系统而设计。

SkyWalking 提供的功能：

  • 服务、服务实例、端点指标分析
  • 根本原因分析
  • 服务拓扑图分析
  • 服务、服务实例、端点依赖性分析
  • 监测慢速服务与端点
  • 性能优化
  • 分布式跟踪与上下文传播
  • 数据库访问指标
  • 告警
  • ……

# 2.架构

SkyWalking 在逻辑上分为四个部分：Probes, Platform backend, Storage 和 UI。

其架构如下：

官网架构图：

  • Probes 探针

    • 收集监测数据，包括各种格式的度量、跟踪、日志和事件（SkyWalking、Zipkin、OpenTelemetry、Prometheus、Zabbix等）。
    • 收集数据后，通过 gRPC 或 HTTP 的方式发送给 Platform backend。

  • Platform backend （SkyWalking 的后端，也叫 OAP）

    • 支持数据聚合、分析和流处理，包括跟踪、度量、日志和事件。作为聚合者角色、接收者角色或两者工作。
    • 其相关文件在 oap-liboap-libs（相关依赖）、conf（相关配置） 等目录中
    • 其相关数据使用 Storage 进行存储

  • Storage 存储

    • 通过开放/可插拔接口存储 SkyWalking 的数据。可以选择一个 SkyWalking 现有的实现，例如 ElasticSearch、H2、MySQL、TiDB、BanyanDB，或者自定义的实现数据的存储
    • 默认使用 H2 的内存模式存储数据

  • UI web 界面

    • 是一个高度可定制的基于 web 的界面，允许 SkyWalking 的终端用户可视化和管理 SkyWalking 的数据
    • web 管理界面通过 HTTP 请求，以 GraphQL 格式向 Platform backend 查询相关数据，然后展示在页面中
    • 相关文件在 webapps 目录下

GraphQL 是一种针对 Graph（图状数据）进行查询特别有优势的 Query Language（查询语言），所以叫做 GraphQL。它跟 SQL 的关系是共用 QL 后缀，就好像「汉语」和「英语」共用后缀一样，但他们本质上是不同的语言。GraphQL 跟用作存储的 NoSQL 没有必然联系，虽然 GraphQL 背后的实际存储可以选择 NoSQL 类型的数据库，但也可以用 SQL 类型的数据库，或者任意其它存储方式（例如文本文件、存内存里等）。

# 3.核心概念

  • Service 服务

  • Service Instance 服务实例

  • Endpoint 端点 访问的路径

  • Trace 追踪 对每个请求都会有一个对应的 Trace ID ，用来表明是同一个请求或调用

# 2.SkyWalking 存储

SkyWalking 可以用 H2 存储，也可以用 ElasticSearch 存储。

H2 是使用 Java 开发的内嵌式数据库，其支持内存、文件等多种模式存储。

H2 支持以下的运行模式：

  • Embedded Mode 内嵌模式

  • Server Model 服务器模式

在 Skywalking 中自带 H2 数据库，其依赖在 Skywalking目录/oap-libs/h2-2.1.212.jar 中

# 三、SkyWalking 的使用

Skywalking 有多种使用方式，目前最流行(也是最强大)的使用方式是基于Java Agent 的。

Java Agent 支持的框架、中间件等，可在链接：https://github.com/apache/skywalking/blob/v6.6.0/docs/en/setup/service-agent/java-agent/Supported-list.md (opens new window) 中查看。

基本上流行的框架都是支持的，就算不支持也没关系，Skywalking 也支持自定义。

除 Java Agent 方式外，Skywalking 还支持其他语言的 Agent，详见：https://skywalking.apache.org/docs/main/v9.2.0/en/setup/service-agent/server-agents/#installing-language-agents-in-services

此外，Skywalking 还支持：

  • 基于 Service Mesh（如 Istio ）
  • Proxy（如 Envoy Proxy）
  • ……

# 四、数据持久化

SkyWalking 默认使用 H2 的 mem 模式（内存存储），在生产环境如果重启了 SkyWalking 的后端服务，会造成数据丢失。所以我们可以将其配置为以数据库存储。

SkyWalking 数据存储可以选择 elasticsearch、h2、mysql、tidb、postgresql、banyandb 等。

使用 elasticsearch 作为存储是常见的选择，下面进行配置示例：

-e "discovery.type=single-node" \
-e xpack.security.enabled=false \
-p 9200:9200 -p 9300:9300 \

如果你使用 elasticsearch 存储数据，你需要根据以下指标，结合你的实际服务器情况修改。

由于存储也会占用服务器资源，例如下面的索引分片数就会占用大量的存储空间，如果你的项目流量不大，或者处于初期阶段，可以适当的减小分片数以及存储数据天数。

storage:
  elasticsearch:
	# 控制索引滚动创建的时间跨度,设置为 1 表示按天创建新索引。例如，今天会创建 sw_segment-20230910，明天则会创建 sw_segment-20230911。这是实现数据过期删除（TTL）的基础。如果设置为 2，则会每两天创建一个新索引。
	dayStep: ${SW_STORAGE_DAY_STEP:1}
	# 设置普通索引的主分片数量,注意改分片根据es的部署情况修改
	indexShardsNumber: ${SW_STORAGE_ES_INDEX_SHARDS_NUMBER:1}
	# 设置普通索引的副本分片数量
	indexReplicasNumber: ${SW_STORAGE_ES_INDEX_REPLICAS_NUMBER:1}
	# 针对“超级数据集”的索引滚动时间跨度，小于0使用dayStep的值
	superDatasetDayStep: ${SW_STORAGE_ES_SUPER_DATASET_DAY_STEP:-1}
	# 如果启用了超级数据集，其索引分片数将是普通索引的多少倍
	superDatasetIndexShardsFactor: ${SW_STORAGE_ES_SUPER_DATASET_INDEX_SHARDS_FACTOR:5}
	# 如果启用了超级数据集，其索引副本数将是普通索引的多少倍
	superDatasetIndexReplicasNumber: ${SW_STORAGE_ES_SUPER_DATASET_INDEX_REPLICAS_NUMBER:0}
   
core:
  default:
    # 记录型数据的保留时间，3代表只保存3天（链路追踪数据）
    recordDataTTL: ${SW_CORE_RECORD_DATA_TTL:3}
	  # 指标型的保留时间，3代表只保存3天（监控指标数据）
	  metricsDataTTL: ${SW_CORE_METRICS_DATA_TTL:7}

# 五、告警

Skywalking 的告警由规则来定义。规则存放在 skywalking安装包目录/config/alarm-settings.yml 中。

alarm-settings.yml 文件结构大致如下：

rules:
  endpoint_percent_rule:
    metrics-name: endpoint_percent
    include-names:
      - dubbox-provider
    exclude-names:
      - dubbox-consumer
    threshold: 75
    op: <
    period: 10
    count: 3
    silence-period: 10
    message: Successful rate of endpoint {name} is lower than 75%
webhooks:
  - http://127.0.0.1/skywalking/alarm/notify/

# 1.规则

规则配置的 key 的含义：

  • endpoint_percent_rule：规则名称，将会在告警消息体中展示，必须唯一，且以 _rule 结尾；
  • metrics-name：度量名称，取值可在 skywalking根目录/config/official_analysis.oal 中找到，填写其中的key即可，对OAL感兴趣的，可前往 https://github.com/apache/skywalking/blob/v6.6.0/docs/en/concepts-and-designs/oal.md 阅读其定义；
  • include-names：将此规则作用于匹配的实体名称上，实体名称可以是服务名称或端点名称等
  • exclude-names：将此规则作用于不匹配的实体名称上，实体名称可以是服务名称或端点名称等
  • threshold：阈值
  • op：操作符，目前支持 >、<、=
  • period：多久检测一次告警规则，即检测规则是否满足的时间窗口，与后端开发环境匹配
  • count：在一个 period 窗口中，如果实际值超过该数值将触发告警
  • silence-period：触发告警后，在 silence-period 这个时间窗口中不告警，该值默认和 period 相同。例如，在时间 T 这个瞬间触发了某告警，那么在 (T+10) 这个时间段，不会再次触发相同告警
  • message：告警消息体，{name} 会解析成规则名称

默认规则的告警规则：

  • 过去3分钟内服务平均响应时间超过1秒
  • 服务成功率在过去2分钟内低于80%
  • 服务90%响应时间在过去3分钟内高于1000毫秒
  • 服务实例在过去2分钟内的平均响应时间超过1秒
  • 端点平均响应时间过去2分钟超过1秒

# 六、动态配置

官方文档：https://skywalking.apache.org/docs/main/v9.2.0/en/setup/backend/dynamic-config/#dynamic-configuration (opens new window)

从 Skywalking 6.5.0开始，Skywalking 的部分配置项支持动态配置——这样修改完配置后，就无需重启Skywalking。

# 1.支持的动态配置项

支持动态配置的配置项如下：

配置 Key 描述 值的格式

core.default.endpoint-name-grouping 端点名称分组设置。该值将会覆盖endpoint-name-grouping.yml。 格式同endpoint-name-grouping.yml (opens new window)

agent-analyzer.default.traceSamplingPolicy 默认和服务维度的采样策略。该值将会覆盖trace-sampling-policy-settings.yml 格式同trace-sampling-policy-settings.yml (opens new window)
configuration-discovery.default.agentConfigurations ConfigurationDiscovery设置 见configuration-discovery.md (opens new window)

要想实现动态配置，需要一个额外的配置服务器。

Skywalking支持使用如下配置服务器：

  • Dynamic Configuration Service (opens new window)
  • Apollo (opens new window)
  • Nacos (opens new window)
  • Zookeeper (opens new window)
  • Etcd (opens new window)
  • Consul (opens new window)

除 Dynamic Configuration Service (opens new window) 尚不完备以外，其他的都可以直接用在生产。
