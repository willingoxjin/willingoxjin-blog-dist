03.ReentrantLock 锁优化

# 一、ReentrantLock 介绍

在 java.util.concurrent 包下提供了很多锁，一般都以 Lock 结尾。如：ReentrantLock、ReentrantReadWriteLock、StampedLock等。

其中最核心的就是 ReentrantLock。

ReentrantLock 的特性：

  • 互斥性：只能有一个线程可以获取到锁
  • 可重入性：同一个线程下，可以调用 lock() 多次（多次获取锁）。但是获取多少次锁就要
  • 公平性：可设置为公平锁

  • 公平锁：如果另一个线程持有锁或者其他线程在等待队列中等待这个锁，那么新发出的请求的线程将被放入到队列中。

  • 非公平锁：当锁被某个线程持有时，新发出请求的线程才会被放入队列。如果在发出请求的时候，锁变成可用状态，这个线程会跳过队列中的等待线程获得锁。

  • 非公平锁的性能大于公平锁。

  • ReentrantLock 默认为非公平锁。

# 二、ReentrantLock 的方法

ReentrantLock 实现了接口 Lock ，接口 Lock 定义了锁相关的方法。

具体的

方法 说明
lock() 获得锁
tryLock() 尝试获得锁。成功返回 true，失败返回 false
tryLock(long timeout, TimeUnit unit) 带超时的 tryLock()
lockInterruptibly() 如果当前线程 interrupt 了就抛出异常，如果没有就去获取锁
unlock() 释放锁（解锁）
newCondition() 创建一个Condition对象并返回。
Condition对象主要用来处理线程让出与唤醒的相关操作

# 三、ReentrantLock 的使用

可重入性示例：

public class ReentrantLockDemo1 implements Runnable {
    private static int i = 0;
    private ReentrantLock lock = new ReentrantLock();

    private void increase() {
        try {
            lock.lock();
            // 可重入性
            lock.lock();
            lock.lock();
            i++;
        } finally {
            // 获取多少次锁，释放多少次锁
            lock.unlock();
            lock.unlock();
            lock.unlock();
        }
    }

    @Override
    public void run() {
        for (int j = 0; j < 10000; j++) {
            increase();
        }
    }

    public static void main(String[] args) throws InterruptedException {
        ReentrantLockDemo1 t = new ReentrantLockDemo1();
        Thread t1 = new Thread(t);
        Thread t2 = new Thread(t);
        t1.start();
        t2.start();
        t1.join();
        t2.join();
        System.out.println(i);
    }
}

# 四、Condition

Condition 用来替代传统线程通信中， Object 的 wait()，notify() 方法。

Condition 的核心方法如下：

  • await() 对应 Object.wait() 方法
  • signal() 对应 Object.notify() 方法

注意：调用 Condition 的方法时，必须获取了锁。

使用示例如下：

@Slf4j
public class ReentrantLockDemo3 {
    private final ReentrantLock lock = new ReentrantLock();
    public Condition condition= lock.newCondition();

    public void awaitCondition() {
        String threadName = Thread.currentThread().getName();
        try {
            lock.lock();
            log.info("{} 调用了awaitCondition1",  threadName);
            long timeBefore = System.currentTimeMillis();
            // 阻塞注册了condition的线程
            condition.await();
            long timeAfter = System.currentTimeMillis();
            log.info("{} 被唤醒",  threadName);
            log.info("{} 等待了: {}ms", threadName, timeAfter - timeBefore);
        } catch (InterruptedException e) {
            e.printStackTrace();
        } finally {
            lock.unlock();
        }
    }


    public void signalCondition() {
        try {
            lock.lock();
            log.info("{} 启动唤醒程序", Thread.currentThread().getName());
            // 唤醒所有注册condition的线程
            condition.signalAll();
        } finally {
            lock.unlock();
        }
    }


    public static void main(String[] args) throws InterruptedException {
        ReentrantLockDemo3 test = new ReentrantLockDemo3();
        new Thread(test::awaitCondition)
                .start();

        Thread.sleep(2000L);
        test.signalCondition();
    }
}

# 五、ReentrantLock 原理

是基于 AbstractQueuedSynchronizer 和 CAS 实现的。

所以我们先来看看 AbstractQueuedSynchronizer。

# 1.AbstractQueuedSynchronizer

AbstractQueuedSynchronizer（AQS）是一个用于构建锁和同步容器的框架。JDK 中很多的并发容器都是基于 AQS 实现的，如 ReentrantLock、CountDownLatch 等。

AQS 是一个抽象类，主要是通过继承的方式来使用，它本身没有实现任何的同步接口，仅仅是定义了同步状态的获取以及释放的方法来提供自定义的同步组件。

AQS 中有个核心的的变量 state，在 ReentrantLock 中是用来记录线程重入的次数。0代表锁是释放状态，当有一个线程拿到 state 锁，其值通过 CAS 将从 0 设置为 1，如果该线程又重入一次锁（再一次获取锁），则 state 就会变成 2。而每次释放，state 就会减去 1，直到为 0。

AQS 提供了一个 FIFO 队列： AbstractQueuedSynchronizer.Node ，当有新的获取锁的线程或当前已经获取锁的线程释放锁，队列中的变化如下：

添加节点：

涉及的变化：

  • 新的线程封装成 Node 节点追加到同步队列中，设置 prev 节点以及修改当前节点的前置节点的 next 节点指向自己
  • 通过 CAS 将 tail 指向新的尾部节点

释放锁移除节点：

涉及的变化：

  • 修改 head 节点指向下一个获得锁的节点
  • 新的获得锁的节点，将 prev 的指针指向 null

# 2.tryLock()

tryLock() 源码如下：

private final Sync sync;

public boolean tryLock() {
    return sync.nonfairTryAcquire(1);
}

abstract static class Sync extends AbstractQueuedSynchronizer {
  	
    @ReservedStackAccess
    final boolean nonfairTryAcquire(int acquires) {
        final Thread current = Thread.currentThread();
        // 获取当前线程的锁状态
        // state 是锁的计数器
        int c = getState();
        // state=0:表示锁没有被占用
        if (c == 0) {
            // 设置state为acquires（即1）
            if (compareAndSetState(0, acquires)) {
                // 若设置成功，代表获取锁成功，就把当前线程设置为独占线程
                setExclusiveOwnerThread(current);
                return true; // 成功拿到锁
            }
        }
        else if (current == getExclusiveOwnerThread()) {
            // 设置重入
            // 如果当前线程就是独占线程的话，就在原来state的基础+acquires(即+1)
            // 这也就是为什么调用几次lock()方法，就需要在最后调用多少次unlock()方法
            int nextc = c + acquires;
            if (nextc < 0) // overflow
                throw new Error("Maximum lock count exceeded");
            setState(nextc);
            return true; // 成功拿到锁
        }
        return false; // 拿到锁失败
    }
    
}

# 3.unlock()

unlock() 源码如下：

public void unlock() {
    sync.release(1);
}

public final boolean release(int arg) {
    // 调用 tryRelease 释放锁
    if (tryRelease(arg)) {
        // 释放成功，则移除队列中的当前head节点，head指向队列中的下一个节点
        Node h = head;
        if (h != null && h.waitStatus != 0)
            unparkSuccessor(h);
        return true;
    }
    return false;
}


abstract static class Sync extends AbstractQueuedSynchronizer {

    @ReservedStackAccess
    protected final boolean tryRelease(int releases) {
        // c=state-releases（即state-1）
        int c = getState() - releases;
        // 当前线程如果@是独占线程，抛出异常
        if (Thread.currentThread() != getExclusiveOwnerThread())
            throw new IllegalMonitorStateException();
        boolean free = false; // 锁是否被完全释放
        // 如果state=0,代表当前线程释放了所有的锁
        if (c == 0) {
            free = true;
            // 设置独占线程为空
            setExclusiveOwnerThread(null);
        }
        // 设置state
        setState(c);
        // 
        return free;
    }
}

# 4.lock()

lock() 源码如下：

// ReentrantLock
public void lock() {
    sync.acquire(1);
}

// AbstractQueuedSynchronizer
public final void acquire(int arg) {
    // 先尝试获取锁tryAcquire(arg)
    // 若获取锁失败，则调用acquireQueued(addWaiter(Node.EXCLUSIVE), arg)
    
    // addWaiter()将线程包装成Node对象，放到队列尾部，tail指向该节点
    // Node.EXCLUSIVE 独占模式; Node.SHARED 共享模式
    if (!tryAcquire(arg) &&
        acquireQueued(addWaiter(Node.EXCLUSIVE), arg))
        selfInterrupt();
}

// AbstractQueuedSynchronizer
final boolean acquireQueued(final Node node, int arg) {
    boolean interrupted = false;
    try {
        // 直到当前线程封装成的Node对象成为排队链表的head节点时
        // 且获取锁成功才会结束该方法
        for (;;) {
            final Node p = node.predecessor();
            if (p == head && tryAcquire(arg)) {
                setHead(node);
                p.next = null; // help GC
                return interrupted;
            }
            if (shouldParkAfterFailedAcquire(p, node))
                interrupted |= parkAndCheckInterrupt();
        }
    } catch (Throwable t) {
        cancelAcquire(node);
        if (interrupted)
            selfInterrupt();
        throw t;
    }
}

sync 变量的类型为 Sync

Sync 的子类如下：

具体使用非公平锁还是公平锁，会根据构造参数来确定。

# FairSync

FairSync 公平锁的 tryAcquire() 方法的源码如下：

static final class FairSync extends Sync {
    
    @ReservedStackAccess
    protected final boolean tryAcquire(int acquires) {
        final Thread current = Thread.currentThread();
        int c = getState();
        // state=0:表示锁没有被占用
        if (c == 0) {
            // 若没有排队进程，则获取锁
            if (!hasQueuedPredecessors() &&
                compareAndSetState(0, acquires)) {
                setExclusiveOwnerThread(current);
                return true;
            }
        }
        else if (current == getExclusiveOwnerThread()) {
            int nextc = c + acquires;
            if (nextc < 0)
                throw new Error("Maximum lock count exceeded");
            setState(nextc);
            return true;
        }
        return false;
    }   

}

# NonfairSync

NonfairSync 的 tryAcquire() 方法实际调用了 nonfairTryAcquire() 方法，逻辑与 tryLock() 相同。

static final class NonfairSync extends Sync {
    private static final long serialVersionUID = 7316153563782823691L;
    
    protected final boolean tryAcquire(int acquires) {
        return nonfairTryAcquire(acquires);
    }
}

# 六、synchronized vs ReentrantLock

比较指标 synchronized ReentrantLock
互斥性 互斥 互斥
可重入性 可重入 可重入
锁实现机制 JVM实现，监视器模式 JDK实现，基于AQS
性能 jdk1.6之前：synchronized性能比ReentrantLock差
jdk1.6之后：synchronized与ReentrantLock性能差不多
使用 方便简洁，由JVM保证加锁解锁 手动加锁解锁
功能性 仅支持非公平锁 - 支持公平锁 & 非公平锁
- 提供了 Condution

官方建议优先使用 synchronized
