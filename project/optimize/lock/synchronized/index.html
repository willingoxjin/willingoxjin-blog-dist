02.Synchronized 锁优化

# 一、synchronized 锁介绍

  • 修饰实例方法：该当前实例加锁，进入同步代码前要获得当前实例的锁

  • 修饰静态方法：给当前类加锁，进入同步代码前要获得指定类对象的锁

  • 修饰代码块：给指定对象或类加锁，进入同步代码前要获得指定对象或类的锁

类锁与对象锁的区别

对象锁：

  • 指定某个变量的对象

  • 类声明后，我们可以 new 出来很多的实例对象。这时候，每个实例在 JVM 中都有自己的引用地址和堆内存空间，这时候，我们就认为这些实例都是独立的个体，很显然，在实例上加的锁和其他的实例就没有关系，互不影响了。

类锁：

  • 类锁是加载类上的，而类信息是存在 JVM 方法区的，并且整个 JVM 只有一份，方法区又是所有线程共享的，所以类锁是所有线程共享的。

以下是未使用锁，使用两个线程，每个线程对变量 i 进行自增 10000 次的操作：

public class UnSynchronizedDemo1 implements Runnable {
    private static int i = 0;

    // 可重入锁
    private void increase() {
        i++;
    }

    @Override
    public void run() {
        for (int j = 0; j < 10000; j++) {
            increase();
        }
    }

    public static void main(String[] args) throws InterruptedException {
        UnSynchronizedDemo1 t = new UnSynchronizedDemo1();
        Thread t1 = new Thread(t);
        Thread t2 = new Thread(t);
        t1.start();
        t2.start();
        // 等待两个线程停止
        t1.join();
        t2.join();
        System.out.println(i);
    }
}

# 二、synchronized 锁的使用

示例如下：

上面的代码存在并发问题，解决该问题，只需要在 increase() 方法上，添加 synchronized 关键字即可。代码如下：

public class SynchronizedDemo1 implements Runnable {
    private static int i = 0;

    private synchronized void increase() {
        i = i + 1;
    }

    @Override
    public void run() {
        for (int j = 0; j < 10000; j++) {
            increase();
        }
    }

    public static void main(String[] args) throws InterruptedException {
        SynchronizedDemo1 t = new SynchronizedDemo1();
        Thread t1 = new Thread(t);
        Thread t2 = new Thread(t);
        t1.start();
        t2.start();
        // 等待两个线程停止
        t1.join();
        t2.join();
        System.out.println(i);
    }
}

上面代码中，synchronized 修饰 increase() 方法，代表所有线程在执行 increase() 方法之前，必须先获取锁，如果获取不到锁就会一直等待。

如果 synchronized 修饰的方法中，所调用的方法也带了 synchronized 锁，那么这个被调用的方法的锁会自动的获取到，如下代码：

public class SynchronizedDemo2 implements Runnable {
    private static int i = 0;

    // 可重入锁
    private synchronized void increase() {
        i = i + 1;
        // 调用someMethod
        // 由于someMethod()方法使用了synchronized修饰，获取到increase()方法的锁，就会自动获取到someMethod()方法的锁
        this.someMethod();
    }

    private synchronized void someMethod() {
        // do some thing...
    }

    @Override
    public void run() {
        for (int j = 0; j < 10000; j++) {
            increase();
        }
    }

    public static void main(String[] args) throws InterruptedException {
        SynchronizedDemo2 t = new SynchronizedDemo2();
        Thread t1 = new Thread(t);
        Thread t2 = new Thread(t);
        t1.start();
        t2.start();
        // 等待两个线程停止
        t1.join();
        t2.join();
        System.out.println(i);
    }
}

# 三、synchronized 原理

# 1.早期 JDK 的 synchronized 实现

  • synchronized 基于对象内部的监视器实现
  • 监视器依赖于操作系统的 Mutex Lock（互斥锁）
  • Mutex Lock 在阻塞或唤醒线程时，都需要操作系统帮忙。就像从用户状态切换到内核态，但切换的开销比较大，因此性能比较差。

# 2.对象在堆内存的存储

JDK 1.6 开始对 synchronized 进行了改造，通过锁升级，来优化锁。为了更好的理解，先来看看对象在堆内存中存储的相关知识。

对象在堆内存的布局分为三部分：

  • 对象头。用来描述对象
  • 实例数据。对象的具体数据
  • 对其填充。占位符，用来补齐不足8位的数据。

对象头包含以下内容：

  • Mark Word。用来存储对象运行时的数据。如 HashCode、GC的分代年龄、锁状态标志、线程组持有的锁等等；
  • 类型指针。JVM 虚拟机通过类型指针来确定对象是哪个类的示例
  • 数组长度。仅当对象为数组的时候，才会包含数组长度。

# 3.锁分级

对象头中和锁相关的是 Mark Word，64位的 Mark Word 数据内容大致如下：

锁状态 56bit 1bit 4bit 1bit（偏向锁标识） 2bit（锁标识）
无锁 25位未使用 31位HashCode 未使用 对象年龄 0 01
偏向锁 54位线程ID 2位epoch 未使用 对象年龄 1 00
轻量级锁 62位指针，指向轻量级锁记录 01
重量级锁 62位指针，指向轻量级锁记录 10
GC标记 空 11

锁分级是对 synchronized 进行的优化处理。

# 4.偏向锁

偏向锁指锁会偏向已经拿到锁的线程，主要用来优化同一个线程用来多次申请同一个锁的情况。若大部分时间只有一个线程使用锁资源，使用偏向锁的性能更好。

如下代码，来测试使用偏向锁和不使用偏向锁的代码耗时：

public class SynchronizedDemo4 {
    private static List<Integer> list = new Vector<>();

    public static void main(String[] args) {
        long start = System.currentTimeMillis();
        for (int i = 0; i < 100000000; i++) {
            list.add(i);
        }
        long end = System.currentTimeMillis();
        System.out.println(end - start);
    }
}

# 6.锁的适用场景

锁 优点 缺点 适用场景
偏向锁 加锁和解锁不需要额外的消耗，和执行非同步方法比较，仅存在纳秒级的差距 如果线程之间存在锁竞争，会带来额外的锁撤销的消耗 适用于只有一个线程访问同步代码块的场景
轻量级锁 竞争的线程不会阻塞，提高了程序的响应速度 如果始终得不到锁竞争的线程使用自旋会消耗 CPU 追求响应时间，锁占用时间很短（多个线程交替进入同步代码块场景）
重量级锁 线程竞争不使用自旋，不会消耗 CPU 线程阻塞，响应时间缓慢 追求吞吐量，锁占用时间较长（多个线程同时竞争同步代码块的场景）

# 2.锁消除

JDK 通过逃逸分析，分析被 synchronized 修饰的代码，监测不存在共享数据，也不存在多个线程同时操作共享数据。如果监测到这样的代码，就会自动的消除 synchronized 锁。

逃逸分析是指分析变量能否能逃出它的作用域。

下面代码中的 object 变量属于局部变量，不可逃逸：

public class SynchronizedClaer {
    public static void main(String[] args) {
        someMethod();
    }

    private static void someMethod() {
        Object object = new Object();
        synchronized(object) {
            System.out.println(object);
        }
    }
}

public class SynchronizedClaer2 {

        Object object = new Object();
        synchronized (object) {
            return object;
        }
    }
}

# 3.锁粗化

锁粗化是指：将多个连续的加锁、解锁操作连在一起，扩展成一个范围更大的锁。

锁粗化的优化伪代码如下：

public void doSomethingMethod1() {
    synchronized (lock) {
        // Do something 1...
    }
    // 能够很快执行，无需同步处理的代码...
    
    synchronized (lock) {
        // Do something 2...
    }
}

// 粗化之后的代码：
public void doSomethingMethod2() {
    // 将代码整合到一次锁请求、同步、释放
    synchronized (lock) {
        // Do something 1...
        // 能够很快执行，无需同步处理的代码...
        // Do something 2...
    }
}
