04.ReentrantReadWriteLock 锁优化

# 一、ReentrantReadWriteLock 介绍

ReentrantReadWriteLock 使用的场景：有两个线程（或更多）对某个数据操作，包含读操作和写操作，且读多写少。当没有写操作的时候，并发读取数据是不会有问题的；但是其中某个线程修改了数据，就不能允许其他线程对数据进行操作（不能读和写），否则就会出现线程安全问题。

针对上面多线程对同一份数据读多写少的情况，JDK 提供了 ReentrantReadWriteLock 来保证数据读写的线程安全。

ReentrantReadWriteLock 包含两个锁：

  • ReadLock 读锁，也称共享锁
  • WriteLock 写锁，也称排他锁

ReentrantReadWriteLock 特性：

  • 公平性
  • 可重入性
  • 锁降级（写锁可降级为读锁）

# 二、获取锁的条件

线程允许获取读锁的条件：

  • 不能有其他线程获取了写锁
  • 没有写请求。或有写请求，但是调用线程和持有锁的线程是同一个线程（即写锁和读锁是同一个线程）

线程允许获取写锁的条件：

  • 不能有线程（包括当前线程）获取了读锁
  • 不能有线程（包括当前线程）获取了写锁

# 三、使用示例

以下代码示例在写入缓存时，获取读写锁：

public class ReentrantReadWriteLockDemo1 {
    private Object data;
    // 缓存是否有效
    private volatile boolean cacheValid;
    private final ReentrantReadWriteLock lock = new ReentrantReadWriteLock();

    public void processCachedData() {
        // 获取读锁
        lock.readLock().lock();
        // 如果缓存无效，更新cache;否则直接使用data
        if (!cacheValid) {
            // 获取写锁前须释放读锁；否则会出现死锁
            lock.readLock().unlock();
            lock.writeLock().lock();
            // Recheck state because another thread might have acquired
            //   write lock and changed state before we did.
            if (!cacheValid) {
                // 更新数据
                data = new Object();
                cacheValid = true;
            }
            // 锁降级：在释放写锁前获取读锁
            lock.readLock().lock();
            // 释放写锁，依然持有读锁
            lock.writeLock().unlock();
        }
        // 使用缓存
        // do something ...

        // 释放读锁
        lock.readLock().unlock();
    }

    public static void main(String[] args) {
        ReentrantReadWriteLockDemo1 demo = new ReentrantReadWriteLockDemo1();
        demo.processCachedData();
    }
}

这里使用了锁降级：将写锁降级为读锁。在持有写锁时，获取读锁，然后释放写锁，就完成了将写锁降级为读锁的过程。

# 四、ReentrantLock vs ReentrantReadWriteLock

ReentrantLock ReentrantReadWriteLock
互斥性 完全互斥 读锁共享，写锁互斥
适用场景 适用于写多读少的场景下 读多写少的场景下，吞吐量和性能高
