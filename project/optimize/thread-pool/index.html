01.线程池

# 一、线程池介绍

线程池，本质上是一种对象池，用于管理线程资源。

在任务执行前，需要从线程池中拿出线程来执行。在任务执行完成之后，需要把线程放回线程池。通过线程的这种反复利用机制，可以有效地避免直接创建线程所带来的性能问题。

好处：

  • 降低资源的消耗。线程本身是一种资源，创建和销毁线程会有CPU开销；创建的线程也会占用一定的内存。
  • 提高任务执行的响应速度。任务执行时，可以不必等到线程创建完之后再执行。
  • 提高线程的可管理性。线程不能无限制地创建，需要进行统一的分配、调优和监控。

坏处：

  • 频繁的线程创建和销毁会占用更多的CPU和内存
  • 频繁的线程创建和销毁会对GC产生比较大的压力
  • 线程太多，线程切换带来的开销将不可忽视
  • 线程太少，多核CPU得不到充分利用，是一种浪费

线程池是特殊的对象池，可以使用 common-pool2 实现一个线程池，可以参考文章进行学习：使用 common-pool2 实现一个线程池 (opens new window)

不过 JDK 已经提供了线程池的功能，并且功能强大，非常好用。

# 二、JDK 的线程池

# 1.ThreadPoolExecutor

ThreadPoolExecutor 是 JDK 提供的核心线程类。

ThreadPoolExecutor 的构造方法的参数说明如下：

  • corePoolSize：核心线程数
  • maximumPoolSize：最大线程数
  • keepAliveTime：允许的空闲时间。
    • keepAliveTime 默认情况下指的是非核心线程的空闲时间
    • 如果设置了 ThreadPoolExecutor 中的 allowCoreThreadTimeOut=true，则 keepAliveTime 就表示核心线程/非核心线程允许的空闲时间
  • unit：keepAliveTime 的单位（毫秒、秒、分钟、小时、天等）
  • workQueue：用来存储等待执行的任务。需传入一个 BlockingQueue 类型的对象
  • threadFactory”：用来创建线程的工厂方法。可选的工厂方法如下：
方法 说明
Executors.defaultThreadFactory() 该方法是创建线程的默认方法，创建线程池常使用此种方式。
其创建的线程拥有相同优先级，并且都不是守护线程（即非守护线程），且线程有线程名称。
ExecutorsprivilegedThreadFactory() 其在 defaultThreadFactory() 的基础上，可以让运行在这个线程中的任务拥有和这个线程相同的访问控制权限和ClassLoader
  • rejectedExecutionHandler：线程的拒绝策略。
    • 当 workQueue 中等待的任务数已经满了，并且当前线程池没有空余的线程，此时仍然有任务提交到线程池，就需要使用到此参数进行处理。
    • 拒绝策略如下
拒绝策略 拒绝行为
AbortPolicy 直接抛出RejectedExecutionException，该异常会阻止系统运行
DiscardPolicy 直接丢弃任务
DiscardOldestPolicy 丢弃执行队列中最靠前的任务（丢弃等待最久的任务）
CallerRunsPolicy 用调用者所在的线程执行任务

核心线程数与非核心线程数的理解

核心线程可以理解为公司中的正式员工，非核心线程可以理解为公司的临时工。当工作量较多时，就会叫临时工过来帮忙。但是最大的员工总数不会超过 maximumPoolSize。

threadFactory 就相当与人才市场，提供员工（创建线程）。

如果工作量过多就会积压到 workQueue 中。像工作量已经超过积压的情况，就会使用拒绝策略 rejectedExecutionHandler 进行拒绝，

如果工作量不是很多，员工就会空闲，就会先解雇临时工。如果设置了 allowCoreThreadTimeOut=true ，解雇员工时就不会区分临时工和正式工。

执行关闭线程池相关方法

方法 说明
execute() 提交任务，交给线程池执行
submit() 提交任务，交给线程池执行。此方法能够返回结果
shutdown() 等待线程的任务执行完后，关闭线程池
shutdownNow() 立刻关闭线程池，不会等待任务执行完

监控相关方法

方法 说明
getTaskCount() 返回线程池已经执行和未执行的任务总数
getCompletedTaskCount() 返回已经完成的任务数
getPoolSize() 返回当前的线程数
getActiveCount() 返回线程池中正在执行任务的线程数

# 3.线程池的状态

  • RUNNING

线程池处于 RUNNING 状态时，能够接收新任务以及对已添加的任务进行处理

线程池的初始状态为 RUNNING。即线程池一旦被创建，就处于 RUNNING 状态，且线程池中的任务数为0。

  • SHUTDOWN

线程池处于 SHUTDOWN 状态时，不接收新任务，但能处理已添加的任务

调用线程池的 shutdown() 方法，状态由 RUNNING->SHUTDOWN

  • STOP

线程池处于STOP状态时，不接收新任务，不处理已添加的任务，并且会中断正在处理的任务

调用线程池的 shutdownNow() 方法，状态由 RUNNING或SHUTDOWN->STOP

  • TIDYING

当所有的任务已终止，任务数为0，线程池的状态会变为 TIDYING 状态；当线程池的状态变为TIDYING状态时，会调用函数 terminated()，该方法在 ThreadPoolExecutor 中是空的，若用户想在线程池变为TIDYING 时进行相应的处理，就需要重载 terminated() 函数实现

当线程池状态为 SHUTDOWN 时，阻塞队列为空并且线程池中执行的任务也为空时，状态就会由SHUTDOWN->TIDYING； 当线程池为 STOP 时，线程池中执行的任务为空时，状态就会由 STOP->TIDYING；

  • TERMINATED

线程池彻底终止，就会变成TERMINATED状态

线程池处于TIDYING状态时，调用terminated()方法，状态就会由 TIDYING->TERMINATED

# 三、线程池的快速使用

public class ThreadPoolDemo {

    public static void main(String[] args) throws ExecutionException, InterruptedException {
        // 创建线程池
        ThreadPoolExecutor executor =
                new ThreadPoolExecutor(
                        5,
                        10,
                        // 默认情况下指的是非核心线程的空闲时间
                        // 如果allowCoreThreadTimeOut=true：核心线程/非核心线程允许的空闲时间
                        10L,
                        TimeUnit.SECONDS,
                        new LinkedBlockingQueue<>(100),
                        Executors.defaultThreadFactory(),
                        new ThreadPoolExecutor.AbortPolicy()
                );
        // 允许核心线程超过空闲时间keepAliveTime后，被回收
        executor.allowCoreThreadTimeOut(true);

        // 提交任务
        executor.execute(() -> System.out.println("任务1"));
        // 提交任务
        executor.execute(() -> System.out.println("任务2"));
        // 提交带返回值的任务
        Future<String> future = executor.submit(() -> "带返回值的任务3");
        String s = future.get();
        System.out.println(s);
    }

}

# 四、BlockingQueue

# 1.BlockingQueue介绍

BlockingQueue 是一个阻塞队列，属于 JDK 提供法人并发容器之一。

BlockingQueue 中，当队列为空时，获取对象会阻塞，当队列满时，放入对象会阻塞。

BlockingQueue 实现了队列的功能，在多线程下会自动管理线程的等待与唤醒。

# 2.核心方法

抛出异常(操作不成功会抛异常) 特殊值(方法的返回值) 阻塞(在调用成功之前会一直阻塞当前线程) 超时(在超时时间内阻塞当前线程，超过超时时间则释放)
插入 add(e) offer(e) put(e) offer(e, time, unit)
移除 remove() poll() take() poll(time, unit)
检查 element() peek() - -

使用 LinkedBlockingQueue ，对上面方法进行简单使用：

public class LinkedBlockQueueDemo {

    public static void main(String[] args) throws InterruptedException {
        // 容量为2
        LinkedBlockingQueue<Object> queue = new LinkedBlockingQueue<>(2);
        queue.add("hello1"); // 插入成功
        queue.add("hello2"); // 插入成功
        boolean def = queue.offer(123); // 插入失败-返回false（超过容器容量大小）
        System.out.println(def);
        boolean def2 = queue.offer(123, 3, TimeUnit.SECONDS); // 阻塞三秒，若三秒后仍插入失败-返回false（超过容器容量大小）
        System.out.println(def2);
        queue.add("666"); // 插入失败-抛出异常Queue full（超过容器容量大小）
    }

}

# 3.BlockingQueue 的实现类

类名 有界 特性
ArrayBlockingQueue 有界 - 内部基于数组实现
- 初始化时必须指定容量大小（没有无参构造方法）
- 一旦指定容量大小，就不能修改容量
LinkedTransferQueue 无界 - 底层基于链表
- 用于数据交换
- 比其他队列多了 transfer() 及 tryTransfer() 方法
DelayQueue 无界 - 其中的元素必须实现 Delayed 接口
- 其中的元素需要排序，一般情况下按照过期的时间的优先级排序
- 使用场景：定时关闭连接、缓存对象、超时处理场景
LinkedBlockingQueue 有界/无界 - 容量可选，默认无界（Integer.MAX_VALUE）
- 内部基于链表实现
PriorityBlockingQueue 无界 - 带优先级的阻塞队列
- 允许插入NULL值
- 元素必须实现Comparable接口，队列的排序规则需要用到
SynchronousQueue 有界 - 不存储元素，内部容量为0
- 一个线程发起插入操作后，就会被阻塞，直到另一个线程发起相应的删除操作才会恢复，因此又被称为同步队列
- 主要实现了take()和put()操作，且需要配对使用

有界，是指队列的容量有上限；

无界是指队列的容量没有上限，可以一直向队列添加元素，直到内存溢出。

# 五、线程池使用注意（调优）

# 1.避免使用无界队列

Executors 中默认提供了创建线程池的方法，详见 八、Executors

不要使用 Executors.newXxxxThreadPool() 快捷方法创建线程池，因为这种方式会使用无界的任务队列，为避免OOM（内存溢出），我们应该使用 ThreadPoolExecutor 的构造方法手动指定队列的最大长度：

ExecutorService executorService = new ThreadPoolExecutor(
        4,
        6,
        0, TimeUnit.SECONDS,
        new ArrayBlockingQueue<>(256), // 使用有界队列，指定排队最大长度
        new ThreadPoolExecutor.DiscardPolicy());

# 2.明确拒绝任务时的行为

线程池默认的拒绝行为是 AbortPolicy，也就是在抛出 RejectedExecutionHandler 异常，该异常是非受检异常，很容易忘记捕获。如果不关心任务被拒绝的事件，可以将拒绝策略设置成DiscardPolicy，这样多余的任务会悄悄的被忽略。

# 3.线程数的配置

ThreadPoolExecutor 的构造方法中 int maximumPoolSize,（ 线程数的上）限参数配置。

根据实际的情况，可以把线程池的任务类型分成：CPU密集型、IO密集型及混合型

  • CPU密集型

计算密集型任务的特点是要进行大量的计算，消耗CPU资源，比如计算圆周率、对视频进行高清解码等等。

CUP密集型任务配置尽可能的少的线程数量，一般的公式：

poolSize=CUP核心数+1
1

如10核CUP使用11个最高线程

为防止某个线程出现问题，因此会增加一个替补线程用来替换其他线程。

可以使用以下代码来获取 CPU 的核心数：

Runtime.getRuntime().availableProcessors()
1
  • IO密集型

涉及到网络、磁盘IO的任务都是 IO 密集型任务，这类任务的特点是 CPU 消耗很少，任务的大部分时间都在等待 IO 操作完成（因为 IO 的速度远远低于CPU和内存的运算或读写速度）。

IO 密集型时，大部分线程都阻塞，故需要多配置线程数，参考公式：

poolSize=2*CUP核心数
1

如10核CPU，使用20个线程数

  • 混合型

混合型是指CPU密集型和IO密集型的合体。并且也是经常遇到的类型

参考公式：

poolSize=N*U*(1+WT/ST)
1
  • N：CPU核心数
  • U：目标CPU利用率，即希望 CPU 的利用率能够达到多少
  • WT：线程等待时间
  • ST：线程运行时间

线程等待时间和线程运行时间可以使用工具 VisualVM 进行分析：

如果本机安装了 JDK ，在命令行执行 jvisualvm ，可以打开 VisualVM—选择项目—在左侧选择 Profiler—点击CPU，此时就会进行 CPU 的分析。

分析完后，就可以查看到线程的运行时间和总时间

  • ST=自用时间。线程的运行时间
  • WT=总时间-自用时间。线程等待时间

若使用 jvisualvm 命令启动出错，可以手动下载 VisualVM： https://visualvm.github.io/download.html (opens new window)

注意

需要注意的是，以上均为经验公式，并非最优标准。

实际使用可以结合业务、压力测试逐步进行调整线程数和队列容量。

# 4.减少线程切换的开销

部分情况下，可以增大队列容量，减少线程的个数，减少线程过多时线程切换的开销。如设置线程数较多，排队队列的容量设置的较小，当任务数非常多的时候，就会由调用多个线程执行，导致线程切换开销变大，最终可能会降低系统吞吐量。

如果任务经常阻塞，队列的容量经常满，可以适当增加线程的数量；

所以线程数和队列容量大小的设置需要根据具体的实际情况进行设置。

# 5.设置线程池核心数大小的工具类

工具来源：Threading stories: about robust thread pools (opens new window)

工具类：

package org.willingoxjin.threadpool;

import java.math.BigDecimal;
import java.math.RoundingMode;
import java.util.Timer;
import java.util.TimerTask;
import java.util.concurrent.BlockingQueue;

/**
 * A class that calculates the optimal thread pool boundaries. It takes the desired target utilization and the desired
 * work queue memory consumption as input and retuns thread count and work queue capacity.
 *
 * @see <a href="https://www.javacodegeeks.com/2012/03/threading-stories-about-robust-thread.html">Threading stories: about robust thread pools</a>
 * @author Niklas Schlimm
 *
 */
public abstract class PoolSizeCalculator {

    /**
     * The sample queue size to calculate the size of a single {@link Runnable} element.
     */
    private final int SAMPLE_QUEUE_SIZE = 1000;

    /**
     * Accuracy of test run. It must finish within 20ms of the testTime otherwise we retry the test. This could be
     * configurable.
     */
    private final int EPSYLON = 20;

    /**
     * Control variable for the CPU time investigation.
     */
    private volatile boolean expired;

    /**
     * Time (millis) of the test run in the CPU time calculation.
     */
    private final long testtime = 3000;

    /**
     * Calculates the boundaries of a thread pool for a given {@link Runnable}.
     *
     * @param targetUtilization
     *            the desired utilization of the CPUs (0 <= targetUtilization <= 1)
     * @param targetQueueSizeBytes
     *            the desired maximum work queue size of the thread pool (bytes)
     */
    protected void calculateBoundaries(BigDecimal targetUtilization, BigDecimal targetQueueSizeBytes) {
        calculateOptimalCapacity(targetQueueSizeBytes);
        Runnable task = creatTask();
        start(task);
        start(task); // warm up phase
        long cputime = getCurrentThreadCPUTime();
        start(task); // test intervall
        cputime = getCurrentThreadCPUTime() - cputime;
        long waittime = (testtime * 1000000) - cputime;
        calculateOptimalThreadCount(cputime, waittime, targetUtilization);
    }

    private void calculateOptimalCapacity(BigDecimal targetQueueSizeBytes) {
        long mem = calculateMemoryUsage();
        BigDecimal queueCapacity = targetQueueSizeBytes.divide(new BigDecimal(mem), RoundingMode.HALF_UP);
        System.out.println("Target queue memory usage (bytes) [目标队列占用总内存]: " + targetQueueSizeBytes);
        System.out.println("createTask() produced " + creatTask().getClass().getName() + " which took " + mem
                + " bytes in a queue [单个任务在队列中占用的内存]");
        System.out.println("Formula: " + targetQueueSizeBytes + " / " + mem);
        System.err.println("* Recommended queue capacity (bytes) [推荐队列容量]: " + queueCapacity);
    }

    /**
     * Brian Goetz' optimal thread count formula, see 'Java Concurrency in Practice' (chapter 8.2)
     *
     * @param cpu
     *            cpu time consumed by considered task
     * @param wait
     *            wait time of considered task
     * @param targetUtilization
     *            target utilization of the system
     */
    private void calculateOptimalThreadCount(long cpu, long wait, BigDecimal targetUtilization) {
        BigDecimal waitTime = new BigDecimal(wait);
        BigDecimal computeTime = new BigDecimal(cpu);
        BigDecimal numberOfCPU = new BigDecimal(Runtime.getRuntime().availableProcessors());
        BigDecimal optimalthreadcount = numberOfCPU.multiply(targetUtilization).multiply(
                new BigDecimal(1).add(waitTime.divide(computeTime, RoundingMode.HALF_UP)));
        System.out.println("Number of CPU [CPU核心数]: " + numberOfCPU);
        System.out.println("Target utilization [CPU目标利用率]: " + targetUtilization);
        System.out.println("Elapsed time (nanos) [总时间]: " + (testtime * 1000000));
        System.out.println("Compute time (nanos) [线程运行时间]: " + cpu);
        System.out.println("Wait time (nanos) [线程等待时间]: " + wait);
        System.out.println("Formula: " + numberOfCPU + " * " + targetUtilization + " * (1 + " + waitTime + " / "
                + computeTime + ")");
        System.err.println("* Optimal thread count [推荐设置的线程数]: " + optimalthreadcount);
    }

    /**
     * Runs the {@link Runnable} over a period defined in {@link #testtime}. Based on Heinz Kabbutz' ideas
     * (http://www.javaspecialists.eu/archive/Issue124.html).
     *
     * @param task
     *            the runnable under investigation
     */
    public void start(Runnable task) {
        long start = 0;
        int runs = 0;
        do {
            if (++runs > 5) {
                throw new IllegalStateException("Test not accurate");
            }
            expired = false;
            start = System.currentTimeMillis();
            Timer timer = new Timer();
            timer.schedule(new TimerTask() {
                public void run() {
                    expired = true;
                }
            }, testtime);
            while (!expired) {
                task.run();
            }
            start = System.currentTimeMillis() - start;
            timer.cancel();
        } while (Math.abs(start - testtime) > EPSYLON);
        collectGarbage(3);
    }

    private void collectGarbage(int times) {
        for (int i = 0; i < times; i++) {
            System.gc();
            try {
                Thread.sleep(10);
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
                break;
            }
        }
    }

    /**
     * Calculates the memory usage of a single element in a work queue. Based on Heinz Kabbutz' ideas
     * (http://www.javaspecialists.eu/archive/Issue029.html).
     *
     * @return memory usage of a single {@link Runnable} element in the thread pools work queue
     */
    public long calculateMemoryUsage() {
        BlockingQueue<Runnable> queue = createWorkQueue();
        for (int i = 0; i < SAMPLE_QUEUE_SIZE; i++) {
            queue.add(creatTask());
        }
        long mem0 = Runtime.getRuntime().totalMemory() - Runtime.getRuntime().freeMemory();
        long mem1 = Runtime.getRuntime().totalMemory() - Runtime.getRuntime().freeMemory();
        queue = null;
        collectGarbage(15);
        mem0 = Runtime.getRuntime().totalMemory() - Runtime.getRuntime().freeMemory();
        queue = createWorkQueue();
        for (int i = 0; i < SAMPLE_QUEUE_SIZE; i++) {
            queue.add(creatTask());
        }
        collectGarbage(15);
        mem1 = Runtime.getRuntime().totalMemory() - Runtime.getRuntime().freeMemory();
        return (mem1 - mem0) / SAMPLE_QUEUE_SIZE;
    }

    /**
     * Create your runnable task here.
     *
     * @return an instance of your runnable task under investigation
     */
    protected abstract Runnable creatTask();

    /**
     * Return an instance of the queue used in the thread pool.
     *
     * @return queue instance
     */
    protected abstract BlockingQueue<Runnable> createWorkQueue();

    /**
     * Calculate current cpu time. Various frameworks may be used here, depending on the operating system in use. (e.g.
     * http://www.hyperic.com/products/sigar). The more accurate the CPU time measurement, the more accurate the results
     * for thread count boundaries.
     *
     * @return current cpu time of current thread
     */
    protected abstract long getCurrentThreadCPUTime();

}

# 六、ScheduledThreadPoolExecutor

# 1.介绍

ScheduledThreadPoolExecutor 是 ThreadPoolExecutor 的 子类，其在 ThreadPoolExecutor 的基础上，增加了延时执行，周期性执行任务的功能。

# 2.核心方法

方法 说明
schedule(Runnable command, long delay, TimeUnit unit) 延迟执行任务，不返回任务结果
ScheduledFuture<V> schedule(Runnable command, long delay, TimeUnit unit) 延迟执行任务，可以返回任务结果
scheduleAtFixedRate - 延迟执行任务，并每隔一段时间都会执行一次（不会等待任务执行完再延时执行）；
- 不返回任务结果
scheduleWithFixedDelay - 延迟执行任务，并每隔一段时间都会执行一次（会等待任务执行完再延时执行）；
- 不返回任务结果

方法的使用：

public class ScheduledThreadPoolExecutorDemo {

    public static void main(String[] args) throws ExecutionException, InterruptedException {
        ScheduledThreadPoolExecutor executor = new ScheduledThreadPoolExecutor(
                10,
                Executors.defaultThreadFactory(),
                new AbortPolicy()
        );
        // 延迟5秒执行，无返回值
        executor.schedule(() -> {
            System.out.println("schedule void");
        }, 5, TimeUnit.SECONDS);

        // 延迟5秒执行，有返回值
        ScheduledFuture<String> future = executor.schedule(() -> {
            return "schedule return";
        }, 8, TimeUnit.SECONDS);
        // 这里会阻塞，等待线程池执行完
        System.out.println(future.get());

        // 第一次延时0秒执行，后续隔每隔3秒执行一次（不会等待任务执行完再延时执行）
        executor.scheduleAtFixedRate(() -> {
            System.out.println("scheduleAtFixedRate, date=" + LocalTime.now());
            try {
                Thread.sleep(3000);
            } catch (InterruptedException e) {
                throw new RuntimeException(e);
            }
        }, 0, 3, TimeUnit.SECONDS);

        // 第一次延时0秒执行，每次执行完任务之后，延时3秒再次执行（会等待任务执行完再延时执行）
        executor.scheduleWithFixedDelay(() -> {
            System.out.println("scheduleWithFixedDelay, date=" + LocalTime.now());
            try {
                Thread.sleep(3000);
            } catch (InterruptedException e) {
                throw new RuntimeException(e);
            }
        }, 0, 3, TimeUnit.SECONDS);
    }

}

# 3.ScheduledThreadPoolExecutor 和 Timer 的区别

Timer：

  • 单线程的
  • 如果一个任务阻塞的比较久，就会影响下一个任务的执行
  • Timer 中如果某个任务抛出异常，则会导致未执行的其他任务不能执行

一般最好使用 ScheduledThreadPoolExecutor，避免使用 Timer。

# 七、ForkJoinPool

# 1.介绍

ForkJoinPool 是 JDK1.7 开始提供的线程池。其将一个大任务拆分成若干个小任务，最终将每个小任务的结果进行汇总。拆分任务的过程称为 Fork，汇总任务结果的过程称为 Join。

ForkJoinPool 适合分而治之，递归计算的 CPU 密集场景。

典型应用

  • 二分搜索
  • 大整数乘法
  • Strassen 矩阵乘法
  • 棋盘覆盖
  • 合并排序
  • 快速排序
  • 线性时间选择
  • 汉诺塔

使用工作窃取算法，具体逻辑如下：

  • 将一个大任务，拆分成多个互不依赖的子任务：ForkJoinTask-[1-8]
  • 在 ForkJoinPool 中，有两个工作线程：ForkJoinWorkerThread-1、ForkJoinWorkerThread-2。每个线程负责一个队列。线程处理自己所负责的任务队列，会以后进先出的形式获取队列中的任务并执行。
  • 当线程处理完自己所在队列的任务后，就会检查其他的队列中是否存在尚未执行的任务。如果检查到存在尚未执行的任务，就会以先进先出的形式获取队列中的任务并执行。
    • 如图中，ForkJoinWorkerThread-1 检查另外一个队列中的任务，并发现存在未执行的任务，然后通过先进先出的方式获取任务，并进行执行。
  • 最后，所有的任务执行完成后，返回结果。

ForkJoinPool 能够并行计算数据，并且使用工作窃取算法充分利用了空闲的线程。

# 2.ForkJoinPool 的使用

ForkJoinPool 的方法使用和 ThreadPoolExecutor 的差不多，不过参数需要时 ForkJoinTask 类型的对象。

ForkJoinPool 的方法：

  • execute 执行没有返回值的任务。该方法返回一个 ForkJoinTask 类型的对象
  • submit 执行有返回值的任务。该方法返回一个 ForkJoinTask 类型的对象
  • invoke 执行有返回值的任务。该方法会直接返回执行的结果
  • invokeAll 接收一个任务列表。

ForkJoinPool 提交任务的方法需要接收一个 ForkJoinTask 类型的任务。关于 ForkJoinTask 的说明如下：

  • ForkJoinTask

    • 提交任务需要继承 ForkJoinTask ，并实现 getRawResult()、setRawResult()、exec() 方法。
      • getRawResult() 用于获取计算结果（即 join 后的结果）
      • setRawResult() 用于设置计算结果
      • exec() 执行计算的具体逻辑

  • RecursiveTask 用于有返回结果的递归任务

    • RecursiveTask 是 ForkJoinTask 的子类，用于递归计算。RecursiveTask 的实现十分简单，直接在 exec() 方法中调用了 compute() 方法
    • 你想要使用 RecursiveTask，只需要继承该类，并实现 compute() 方法即可

  • RecursiveAction 用于没有返回结果的递归任务

    • 同 RecursiveTask ，只不过没有返回值

  • ……

示例：计算1到1000000的数字总和

public class ForkJoinPoolDemo {

    public static void main(String[] args) throws ExecutionException, InterruptedException {
        // 连续阶乘的求和：计算1到1000的数字总和
        ForkJoinPool pool = new ForkJoinPool();
        ForkJoinTask<Integer> task = pool.submit(new ContinuousFactorialTask(1, 1000000));
        Integer sum = task.get();
        System.out.println(sum);
        
        // invoke() 方法会直接获得返回值
        System.out.println(pool.invoke(new ContinuousFactorialTask(1, 1000000)));
    }

}

class ContinuousFactorialTask extends RecursiveTask<Integer> {
    public ContinuousFactorialTask(int start, int end) {
        if (start >= end) {
            throw new IllegalArgumentException("start必须小于end");
        }
        this.start = start;
        this.end = end;
    }

    // 起始位置
    private final int start;
    // 结束位置
    private final int end;
    // 阈值：end-start必须大于此阈值，才会进行递归计算
    private static final int minThreshold = 100;

    @Override
    protected Integer compute() {
        int sum = 0;
        boolean needFork = (end - start) > minThreshold;
        if (needFork) {
            // 拆分任务
            int middle = (start + end) / 2;
            ContinuousFactorialTask leftTask = new ContinuousFactorialTask(start, middle);
            ContinuousFactorialTask rightTask = new ContinuousFactorialTask(middle + 1, end);

            // 执行子任务
            leftTask.fork();
            rightTask.fork();

            // 子任务执行完成之后的结果
            Integer leftResult = leftTask.join();
            Integer rightResult = rightTask.join();

            // 合并任务
            sum = leftResult + rightResult;
        } else {
            for (int i = start; i <= end; i++) {
                sum += i;
            }
        }
        return sum;
    }
}

# 3.ForkJoinPool 总结

问题：

  • 需要自己实现 Fork，Join 的过程，编码比较复杂
  • compute() 方法中的异常，需要自行处理，不可抛出

ThreadPoolExecutor 与 ForkJoinPool 的区别：

  • 使用 ForkJoinPool 能够使用数量有限的线程来完成非常多的具有父子关系的任务
  • 但是 ThreadPoolExecutor 中的 Thread 无法选择优先执行子任务

JDK1.8 中引入的并行流计算，底层就是使用了 ForkJoinPool，如 ：

public class SumArray {

        List<Integer> numberList = Arrays.asList(1, 2, 3, 4, 5, 6, 7, 8);
        numberList.parallelStream().forEach(System.out::println);
    }
}

# 八、Executors

Executors 中默认提供了创建线程池的方法，如下：

方法名 功能 返回的线程池类型
newFixedThreadPool(int nThreads) - 创建固定大小的线程池，任意时间最多只有固定数目的活动线程存在
- 适用于线程数比较稳定的并发线程场景		 ThreadPoolExecutor
newSingleThreadExecutor() - 创建只有一个线程的线程池，保证任务按照指定的顺序执行
- 适用于需要严格控制执行顺序的场景		 ThreadPoolExecutor
newCachedThreadPool() - 缓存型线程池，会先查看池中是否有以前建立的线程，有就复用，没有就创建
- 适用于生存周期很短的异步任务		 ThreadPoolExecutor
newScheduledThreadPool() - 创建定时调度的线程池
- 适用于定时任务、延时任务		 ScheduledThreadPoolExecutor
newWorkStealingPool() - 创建流式（fork-join）线程池
- 适用于分而治之，递归计算的CPU密集场景		 ForkJoinPool

Executors 创建线程池，使用示例如下：

public class ExecutorsDemo {
    public static void main(String[] args) {
        ExecutorService pool = Executors.newCachedThreadPool();
        pool.execute(() -> System.out.println("hello"));
    }
}
