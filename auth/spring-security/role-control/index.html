10.Spring Security 的权限控制

# 一、Spring Security 的权限控制

Spring Security 提供了很多灵活的权限控制方式，不过按照权限控制的范围划分，可以分为两种：

  • HTTP 请求的权限控制
  • 方法的权限控制

下面我们就来看一下，如何使用权限控制吧。

# 二、基于配置对 HTTP 请求的权限控制

Spring Security 中HTTP 请求权限控制的官方文档：Authorize HttpServletRequests (opens new window)

  • Ant 匹配
  • 正则表达式匹配
  • Http 方法匹配
  • 调度程序类型匹配
  • 自定义匹配器

# 1.Ant 匹配

如下示例，将使用 Ant 匹配来定义 HTTP 请求的认证授权规则：

@Bean
public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
    http
        .authorizeHttpRequests((authorize) -> authorize
            // 无需认证,就可以访问该资源
            .requestMatchers("/resource/**").permitAll()
            // 需要有permission:read权限,才允许访问该资源
            .requestMatchers("/resource1/**").hasAuthority("permission:read")
            // 需要是USER角色,才允许访问该资源  
            .requestMatchers("/resource2/**").hasRole("USER")
            // 需要有ROLE_USER或ROLE_ADMIN角色的其中一个,才允许访问该资源  
            .requestMatchers("/resource3/**").hasAnyRole("USER", "ADMIN")
            // 只有当请求中的`name`参数的值与当前用户的认证名字相同时,才允许访问该资源
            // 这里可以使用内置的AuthorizationManager或自定义的AuthorizationManager
            .requestMatchers("/resource4/{name}").access(new WebExpressionAuthorizationManager("#name == authentication.name"))
           	.requestMatchers("/db/**").access(allOf(hasAuthority('db'), hasRole('ADMIN'))) 
          	// 对其它的请求,必须认证后才允许访问资源
            .anyRequest().authenticated()
        );

  	// ...

    return http.build();
}

  • requestMatchers(String) 方法用于配置 ant 风格的 URL 路径表达式
  • 授权规则：
    • permitAll ：允许所有用户访问该 URL 路径或执行该 HTTP 方法，无需进行任何授权验证
      • 请注意，在这种情况下，永远不会从会话中检索 Authentication
    • denyAll ：表示在任何情况下都不允许请求
      • 请注意，在这种情况下，永远不会从会话中检索 Authentication
    • hasAuthority ：要求用户具有特定的权限才能访问该 URL 路径或执行该 HTTP 方法。权限字符串可以是任何自定义的权限标识符。
    • hasAnyAuthority ：要求用户具有给定权限列表中的任意一个权限才能访问该URL路径或执行该HTTP方法。
    • hasRole ：要求用户具有特定的角色才能访问该 URL 路径或执行该 HTTP 方法。
      • 角色名通常以 ROLE_ 前缀。
    • hasAnyRole ：要求用户具有给定角色列表中的任意一个角色才能访问该 URL 路径或执行该 HTTP 方法。
      • 角色名通常以 ROLE_ 前缀。
    • access ： 请求使用将使用自定义或指定的 AuthorizationManager 来确定访问权限。

内置的授权规则详细见 AuthorizeHttpRequestsConfigurer.AuthorizedUrl 类。

需要注意的是，上面使用 access 方法配置 WebExpressionAuthorizationManager 时，使用的 SpEL 表达式是无法调用 Spring 中的 bean 对象的方法校检权限，因为 WebExpressionAuthorizationManager 中默认的 SecurityExpressionHandler 在创建后，未设置对应的 BeanResolver。

解决这个问题，只需要进行如下配置：

@Configuration
public class SecurityConfig {

		@Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        DefaultHttpSecurityExpressionHandler expressionHandler = new DefaultHttpSecurityExpressionHandler();
        ApplicationContext context = http.getSharedObject(ApplicationContext.class);
        Assert.notNull(context, "Could not find ApplicationContext");
        expressionHandler.setApplicationContext(context);
        WebExpressionAuthorizationManager webExpressionAuthorizationManager = new WebExpressionAuthorizationManager(
                "@defaultAuthorizationCheck.check(authentication, request)");
        webExpressionAuthorizationManager.setExpressionHandler(expressionHandler);

        http
            .authorizeHttpRequests(auth -> auth
           					.requestMatchers("/hello").access(webExpressionAuthorizationManager)
                    .anyRequest().authenticated())
            .formLogin(formLogin -> formLogin.permitAll())
          	.csrf(AbstractHttpConfigurer::disable);
    }

}

# 4.调度程序类型匹配

首先，我们了解一下调度类型匹配器：

调度类型匹配器（DispatcherTypeMatcher）是用于确定请求的调度类型的对象。在 Java Servlet 规范中，请求可以通过不同的调度类型进行处理。

调度类型匹配器的类型如下：

  • FORWARD 表示请求是通过 RequestDispatcher 的 forward() 方法转发给 Servlet 的。在转发过程中，Servlet 容器会将请求传递给另一个 Servlet 进行处理。
  • INCLUDE 表示请求是通过RequestDispatcher的include()方法包含在另一个 Servlet 的响应中的。在包含过程中，Servlet 容器会将请求传递给另一个 Servlet 处理，并将其响应内容包含在当前 Servlet 的响应中。
  • REQUEST 表示请求是通过客户端直接发送给Servlet的。这是最常见的调度类型，用于处理普通的HTTP请求。
  • ASYNC 表示请求是通过异步处理机制处理的。在 Servlet 3.0 规范中引入了异步处理请求的功能，允许 Servlet 在接收到请求后，将处理过程交给其他线程处理，而不是立即返回响应。
  • ERROR 表示请求是通过错误处理机制处理的。当 Servlet 容器在处理请求时发生错误时，会将请求转发给一个特殊的错误处理 Servlet 来处理。这个错误处理 Servlet 可以通过<error-page>元素在 web.xml 配置文件中进行配置。当错误处理 Servlet 被调用时，它会接收到一个包含错误信息的 ServletRequest 对象，并可以根据错误类型和状态码来执行相应的错误处理逻辑。

在 Spring Security 中，可以配置调度类型匹配器和权限，如下代码中配置 FORWARD 和 ERROR 类型的调度器无需权限校检：

http.authorizeHttpRequests((authorize) -> authorize
        .dispatcherTypeMatchers(DispatcherType.FORWARD, DispatcherType.ERROR).permitAll()
        .anyRequest().authenticated()
    )

# 5.自定义请求匹配器

我们可以自定义 RequestMatcher 来处理请求的权限控制。

RequestMatcher 接口用于匹配 HttpServletRequest，如果要自定义该接口，只需要实现其 matches 方法即可，我们可以使用 Lambad 的写法：

// 请求中含有print参数,则校检打印权限 
RequestMatcher printview = (request) -> request.getParameter("print") != null;
1
2

然后，配置 RequestMatcher 的所需的权限或角色，如下：

http.authorizeHttpRequests((authorize) -> authorize
        // 仅有print权限的用户才可访问RequestMatcher匹配到的请求
        .requestMatchers(printview).hasAuthority("print")
        .anyRequest().authenticated()
    )

# 三、使用注解对方法的权限控制

Spring Security 中方法权限控制的官方文档：Method Security (opens new window)

如果想要对方法进行权限控制，Spring Security 也提供了注解：

  • @PreAuthorize：用于在方法执行前进行是否有权限验证；
  • @PostAuthorize：用于在方法执行后对方法返回值进行权限验证；
  • @PreFilter：用于在方法执行之前对传入的集合、Map 或流进行过滤；
  • @PostFilter：用于在方法执行之后对方法返回的集合进行过滤；
  • @Secured：类似于 @PreAuthorize。
  • JSR-250 (opens new window) 的注解：JSE-250 是 Java EE规范的一部分，且只能在支持 Java EE规范的应用程序中使用。

上面的 @PreAuthorize 、@PostAuthorize 、@PreFilter 、@PostFilter 注解都支持与 SpEL 表达式结合使用，关于 SpEL 表达式的使用文档见：Expressing Authorization with SpEL (opens new window)。SpEL 表达式的类为 SecurityExpressionRoot。

默认情况下，Spring Boot Security 不会启用方法级授权，要想使上面这些注解生效，需要先配置 @EnableMethodSecurity 注解来开启。

@EnableMethodSecurity
@Configuration
public class MethodSecurityConfig {
}

# 四、注解中使用 SpEL 表达式

上面我们说到，@PreAuthorize 、@PostAuthorize 、@PreFilter 、@PostFilter 注解都支持与 SpEL 表达式，下面一起看看这些注解中如何使用 SpEL 表达式吧：

# 1.内置的方法与域对象

Spring Security 内置的权限校检的方法如下：

  • permitAll
  • denyAll
  • hasAuthority
  • hasRole
  • hasAnyRole
  • hasPermission ：进入 PermissionEvaluator 实例的钩子，用于进行对象级授权

内置的方法都在 SecurityExpressionOperations 的实现中。

以下是最常见的领域：

  • authentication-与此方法调用相关的Authentication实例
  • principal-与此方法调用相关的Authentication#getPrincipal

# 2.使用示例

@Component
public class MyService {

    /**
  	 * 任何用户不得调用该方法
  	 * 使用SecurityExpressionRoot#denyAll方法
  	 */
    @PreAuthorize("denyAll") 
    MyResource myDeprecatedMethod(...);

    /**
  	 * 仅有ROLE_ADMIN角色的用户才可访问该方法
  	 * 使用SecurityExpressionRoot#hasRole方法
  	 */
    @PreAuthorize("hasRole('ADMIN')") 
    MyResource writeResource(...)

    /**
  	 * 必须同时有db权限和ROLE_ADMIN角色的用户才可访问该方法
  	 */
    @PreAuthorize("hasAuthority('db') and hasRole('ADMIN')") 
    MyResource deleteResource(...)

    /**
  	 * 校检Authentication中的principal属性内的数据值
  	 */
    @PreAuthorize("principal.claims['aud'] == 'my-audience'") 
    MyResource readResource(...);

    /**
  	 * 调用Spring容器中name为authorizationLogic的bean的check方法校检
  	 *
  	 * authorizationLogic 的定义如下：
  	 * @Component("authorizationLogic")
     * public class AuthorizationLogic {
     * 		public boolean decide(MethodSecurityExpressionOperations operations) {
     *    		// ... authorization logic
     *    }
     * }
  	 */
		@PreAuthorize("@authorizationLogic.check(authentication, #root)")
    MyResource shareResource(...);

    /**
     * 校检参数contact
  	 */
    @PreAuthorize("hasPermission(#contact, 'permission:write')")
    public void updateContact(Contact contact);

    /**
     * 使用org.springframework.security.core.parameters.P注解定义参数的别名
  	 */
    @PreAuthorize("hasPermission(#c, 'permission:write')")
    public void updateContact2(@P("c") Contact contact);

  	/**
  	 * 使用org.springframework.data.repository.query.Param注解定义参数的别名
  	 * 该注解是spring-data中的注解
  	 */
  	@PreAuthorize("#n == authentication.name")
    Contact findContactByName(@Param("n") String name);

}

# 1.自定义切点校检权限

配置切面及对应的拦截器：

@Bean
@Role(BeanDefinition.ROLE_INFRASTRUCTURE)
static Advisor protectServicePointcut() {
    JdkRegexpMethodPointcut pattern = new JdkRegexpMethodPointcut();
  	// 匹配SecurityAuthorizeTestService中的所有方法
    pattern.setPattern("org.willingoxjin.service.SecurityAuthorizeTestService.*");
  	// 创建拦截器,配置仅有ROLE_ADMIN角色的才允许调用方法
   	AuthorizationManagerBeforeMethodInterceptor interceptor = new AuthorizationManagerBeforeMethodInterceptor(pattern,
          AuthorityAuthorizationManager.hasRole("ADMIN"));
  	// 设置拦截器的执行顺序
    interceptor.setOrder(AuthorizationInterceptorsOrder.PRE_AUTHORIZE.getOrder());
    return interceptor;
}

# 2.与 AspectJ 集成

若要与 Aspect 集成，需要先引入 spring-security-aspects 的依赖：

<dependency>
  <groupId>org.springframework.security</groupId>
  <artifactId>spring-security-aspects</artifactId>
</dependency>

4

然后，配置 @EnableMethodSecurit 注解中的 mode：

@EnableMethodSecurity(
        proxyTargetClass = true,
        mode = AdviceMode.ASPECTJ
)
@Configuration
public class AuthorityConfig {
}

3
4

6
7
  • proxyTargetClass ：
    • 配置为 true 表示创建基于子类 （CGLIB） 的代理。并且仅当配置了 mode = AdviceMode.ASPECTJ 时才有效。
    • 配置为 false 表示创建基于标准 Java 接口的代理，默认为 false。

需要注意的是，配置了 mode = AdviceMode.ASPECTJ ，且配置了 proxyTargetClass=true 时，会影响 Spring 容器中所有需要代理的 bean。如 @Cacheable、@Transactional 所修饰的 bean。因为在执行的时候，如果抛出异常，会影响他们的执行。

比如，有一个同时被 @Transactional 和 @PostAuthorize 注解修饰的方法，如果希望当 @PostAuthorize 运行时，事务仍然处于打开状态，以便 AccessDeniedException 将事务回滚。此时，就可以配置事务执行的顺序放在 @PostAuthorize 之前执行：

// @EnableTransactionManagement执行的默认顺序为Integer.MAX_VALUE
// @PostAuthorize 执行的默认顺序为500
@EnableTransactionManagement(order = 499)

2
3

不过，要注意的是，不要影响到其他拦截器或代理类方法的执行。

Spring Security 中方法权限控制相关注解的执行顺序见：AuthorizationInterceptorsOrder。
