04.Spring Security 实现前后端分离的认证

# 一、Spring Security 与前后端分离

# 1.前后端分离的开发模式

现在，前后端分离已成为互联网项目开发的业界标准了，基本上新项目都会采用前后端分离的模式开发。前端只对用户看到的页面开发，后端也只需要提供接口给前端，前端后端交互只需要通过 RESTFUL API 接口，且都使用 JSON 格式的数据传递信息。

该方式可以有效的在前后端项目之间进行解耦，并且前后端分离会为以后的大型分布式架构、弹性计算架构、微服务架构、多客户端化服务（例如浏览器、移动App、桌面应用、小程序、车载终端等）打下坚实的基础。

当然，现在开发大家都是前后端分离的模式了（虽然很多公司前后端分离，但是干活的人不分离），在权限校检中，登录成功、登录失败、提示登录等场景，也都会返回 JSON 的信息。所以，在 Spring Security 中，我们登录会使用接口直接请求，如果登录成功则返回授权相关的信息，登录失败则提示用户名密码错误或其他错误信息，未登录也会返回 JSON 数据提示用户。

# 2.Spring Security 的配置

继承 WebSecurityConfigurerAdapter 类，并重写 configure(HttpSecurity http) 方法，在 configure 方法中的 http 参数中与登录相关的方法如下：

  • authenticationEntryPoint 未认证时的处理
  • successHandler 登录成功的处理
  • failureHandler 登录失败的处理
  • logoutSuccessHandler 成功退出登录的处理
  • ......

我们可以在以上方法中配置对应 XxxxHandle 或 AuthenticationEntryPoint，并在 XxxxHandle 或 AuthenticationEntryPoint 类中的相关方法中返回相关的 JSON 数据。

下面就一起来具体配置使用一下。

# 二、未认证的处理

# 1.authenticationEntryPoint() 与 AuthenticationEntryPoint

authenticationEntryPoint(AuthenticationEntryPoint authenticationEntryPoint) 方法用来配置 AuthenticationEntryPoint 。而 AuthenticationEntryPoint 专门用来处理未认证的请求。

  • 如果我们没有显式地配置 authenticationEntryPoint()，则默认使用 LoginUrlAuthenticationEntryPoint 作为认证入口点。
    • LoginUrlAuthenticationEntryPoint 默认会将页面中定向到登录页面（/login）。
    • 可以通过构造函数或 setLoginFormUrl() 方法来设置自定义的登录 URL。
  • 除了 LoginUrlAuthenticationEntryPoint，Spring Security还提供了其他的认证入口点实现类，如Http403ForbiddenEntryPoint、BasicAuthenticationEntryPoint等。如果需要使用其他的认证入口点，可以通过authenticationEntryPoint() 方法来进行配置。

我们想要实现前后端分离的配置，所以不需要让页面重定向，只需要直接返回未认证的 JSON 数据即可。

# 2.配置示例

我们下面先来自定义一个 AuthenticationEntryPoint：如下创建 SecurityAuthenticationEntryPoint 类，使其实现 AuthenticationEntryPoint，并重写 commence() 方法。

public class SecurityAuthenticationEntryPoint implements AuthenticationEntryPoint {

    @Override
    public void commence(HttpServletRequest request, HttpServletResponse response,
            AuthenticationException authException) throws IOException, ServletException {
        // TODO: 这里可以自己定义返回的对象
       	Response<Void> result = ResponseFactory.FACTORY.build(HttpStatus.UNAUTHORIZED.value(), "尚未登录，请先登录");

        response.setContentType(MediaType.APPLICATION_JSON_UTF8_VALUE);
        PrintWriter out = response.getWriter();
        out.write(JsonUtils.toJsonStr(result));
        out.flush();
        out.close();
    }

}

# 二、认证成功的处理

# 1.successHandler() 和 AuthenticationSuccessHandler

successHandler() 方法用来配置身份验证成功后的处理器（即 AuthenticationSuccessHandler）

AuthenticationSuccessHandler 接口中只有一个方法 onAuthenticationSuccess()，该方法在身份验证成功时被调用，可以在该方法中进行一些自定义的逻辑处理，如重定向到其他页面、记录日志等。

对于我们前后端分离的模式来说，在这里我们可以返回用户的 token 或用户的相关信息。

# 2.配置示例

我们来编写一个处理器类，使该类实现 SavedRequestAwareAuthenticationSuccessHandler 接口。

代码如下：

public class SecurityAuthenticationSuccessHandler extends SavedRequestAwareAuthenticationSuccessHandler {

    /**
     * 登录成功时的处理
     * @param request 请求对象
     * @param response 响应对象
     * @param authentication 携带登录的用户名及角色等信息
     */
    @Override
    public void onAuthenticationSuccess(HttpServletRequest request, HttpServletResponse response,
            Authentication authentication) throws ServletException, IOException {
        // 直接输出json格式的响应信息 TODO: 这里可以自己定义返回的对象

由于是前后端分离的开发模式，我们就不能使用 Session 管理用户会话状态了，所以在认证成功后，需要生成 token，并缓存（使用 Redis）起来，最后在 JSON 数据中返回给前端。前端将 token 缓存起来，后面的请求都携带该 token。服务端则通过 token 校检是否是已经认证的用户。或者也可以使用 JWT 的方案。

由于是示例，这里服务端并没有生成 token 并缓存起来，因此，大家在实际开发中需要手动加入。

如果要生成 token 返回给前端，则必须添加 token 校检的过滤器，来验证 token 是否有效。具体示例请看 JWT 的认证。

然后我们在 SecurityConfig 配置类中，调用 successHandler() 方法，设置 SecurityAuthenticationSuccessHandler ：

@EnableWebSecurity
public class SecurityConfig {

  	@Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        http.authorizeHttpRequests(auth -> auth.anyRequest().authenticated())
                .exceptionHandling(exceptionHandling ->
                        exceptionHandling.authenticationEntryPoint(new SecurityAuthenticationEntryPoint())
                )
                .formLogin(formLogin ->
                    formLogin.permitAll()
                            // 认证成功时的处理器
                            .successHandler(new SecurityAuthenticationSuccessHandler())
                )
                .csrf(AbstractHttpConfigurer::disable);

        return http.build();
    }

}

# 三、认证失败的处理

# 1.failureHandler() 和 AuthenticationFailureHandler

failureHandler() 方法与 successHandler() 一样，都是用户设置一个处理器。只是 failureHandler() 方法是用来设置认证失败的处理器，并且，其参数为 AuthenticationFailureHandler 类型的对象。

# 2.配置示例

我们编写一个类 SecurityAuthenticationFailureHandler，实现 ExceptionMappingAuthenticationFailureHandler 接口，来专门处理认证失败时的返回结果。

代码如下：

public class SecurityAuthenticationFailureHandler extends ExceptionMappingAuthenticationFailureHandler {

    /**
     * 认证失败时的处理
     * @param request 请求对象,利用该参数我们可以实现服务端的跳转
     * @param response 响应对象,利用该参数我们可以做客户端的跳转，也可以返回 JSON 数据;
     * @param e 异常信息(登录失败的原因)
     */
    @Override
    public void onAuthenticationFailure(HttpServletRequest request, HttpServletResponse response,
            AuthenticationException e) throws IOException, ServletException {
        // TODO: 这里可以自己定义返回的对象
        Response<Void> result = ResponseFactory.FACTORY.build(HttpStatus.BAD_REQUEST.value(), e.getMessage());

        response.setContentType(MediaType.APPLICATION_JSON_UTF8_VALUE);
        PrintWriter out = response.getWriter();
        out.write(JsonUtils.toJsonStr(result));
        out.flush();
        out.close();
    }

}

在 SecurityConfig 配置类中，调用 failureHandler() 方法来关联上面定义的 SecurityAuthenticationFailureHandler 类对象：

@Configuration
public class SecurityConfig {

    @Autowired
    private AuthenticationManager authenticationManager;

    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        http.authorizeHttpRequests(auth -> {
                    // /doLogin无需认证
                    // auth.antMatchers("/doLogin").permitAll()
                    auth.anyRequest().authenticated();
                })
                .exceptionHandling(exceptionHandling ->
                    exceptionHandling.authenticationEntryPoint(new SecurityAuthenticationEntryPoint())
                )
                .formLogin(formLogin ->
                    formLogin.permitAll()
                        // 认证成功时的处理器
                        .successHandler(new SecurityAuthenticationSuccessHandler())
                        // 认证失败的处理器
                        .failureHandler(new SecurityAuthenticationFailureHandler())
                )
                .csrf(AbstractHttpConfigurer::disable);

        http.addFilterBefore(jsonLoginFilter(), UsernamePasswordAuthenticationFilter.class);

        return http.build();
    }

    private JsonLoginFilter jsonLoginFilter() {
        JsonLoginFilter filter = new JsonLoginFilter();
        filter.setAuthenticationManager(authenticationManager);
        filter.setFilterProcessesUrl("/login");
        filter.setSecurityContextRepository(new HttpSessionSecurityContextRepository());

        return filter;
    }

}

                        .deleteCookies("JSESSIONID")
                )
                .csrf(AbstractHttpConfigurer::disable);

        return http.build();
    }

    private void initExceptionConfigure(HttpSecurity http) throws Exception {
        http.exceptionHandling().authenticationEntryPoint(new SecurityAuthenticationEntryPoint());
    }



}

# 五、使用 JSON 登录

上面使用的是 Spring Security 中默认的登录接口。如果我们想要在登录时，使用 JSON 格式的数据作为参数传递，我们可以自定义过滤器，并手动定义接口调用认证逻辑。

示例：

# 1.自定义过滤器的方式

首先我们新建过滤器 JsonLoginFilter，使其继承 UsernamePasswordAuthenticationFilter，并重写 attemptAuthentication() 方法，代码如下：

public class JsonLoginFilter extends UsernamePasswordAuthenticationFilter {

    @Override
    public Authentication attemptAuthentication(HttpServletRequest request, HttpServletResponse response)
            throws AuthenticationException {
        // 获取请求头，据此判断请求参数类型
        String contentType = request.getContentType();
        if (MediaType.APPLICATION_JSON_VALUE.equalsIgnoreCase(contentType) || MediaType.APPLICATION_JSON_UTF8_VALUE.equalsIgnoreCase(contentType)) {
            // 仅支持POST请求
            if (!request.getMethod().equals("POST")) {
                throw new AuthenticationServiceException("Authentication method not supported: " + request.getMethod());
            }
            String username = null;
            String password = null;
            try {
                // 解析请求体中的 JSON 参数
                Map<String, Object> map = JsonUtils.toMap(request.getInputStream());
                username = (String) map.get("username");
              	username = (username != null) ? username.trim() : "";
                password = (String) map.get("password");
              	password = (password != null) ? password : "";
            } catch (IOException e) {
                throw new RuntimeException(e);
            }
            // 构建登录令牌
            UsernamePasswordAuthenticationToken authRequest =
                    UsernamePasswordAuthenticationToken.unauthenticated(username, password);
            // Allow subclasses to set the "details" property
            setDetails(request, authRequest);
            // 执行真正的登录操作
            return this.getAuthenticationManager().authenticate(authRequest);
        }

        return super.attemptAuthentication(request, response);
    }

}

在 SecurityConfig 中配置过滤器 JsonLoginFilter，由于要依赖 AuthenticationManager ，因此我们需要配置 PasswordEncoder、InMemoryUserDetailsManager、AuthenticationManager。

相关代码如下：

@Configuration
public class PasswordEncoderConfig {

    @Bean
    public PasswordEncoder passwordEncoder() {
        return NoOpPasswordEncoder.getInstance();
    }

}

@Configuration
public class AuthenticationManagerConfig {

    @Autowired
    private PasswordEncoder passwordEncoder;

    @Bean
    public InMemoryUserDetailsManager inMemoryUserDetailsManager(SecurityProperties properties) {
        SecurityProperties.User user = properties.getUser();
        return new InMemoryUserDetailsManager(User.withUsername(user.getName())
                .password(passwordEncoder.encode(user.getPassword()))
                .roles(StringUtils.toStringArray(user.getRoles()))
                .build());
    }

    @Bean
    public AuthenticationManager authenticationManager(InMemoryUserDetailsManager userDetailsService) {
        DaoAuthenticationProvider provider = new DaoAuthenticationProvider();
        provider.setUserDetailsService(userDetailsService);
        provider.setPasswordEncoder(passwordEncoder);
        return new ProviderManager(provider);
    }

}

@Configuration
public class SecurityConfig {

    @Autowired
    private AuthenticationManager authenticationManager;

    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        http.authorizeHttpRequests(auth -> auth.anyRequest().authenticated())
                .exceptionHandling(exceptionHandling ->
                        exceptionHandling.authenticationEntryPoint(new SecurityAuthenticationEntryPoint())
                )
                .formLogin(formLogin ->
                    formLogin.permitAll()
                        // 认证成功时的处理器
                        .successHandler(new SecurityAuthenticationSuccessHandler())
                        // 认证失败的处理器
                        .failureHandler(new SecurityAuthenticationFailureHandler())
                )
                .logout(logout -> 
                    logout.logoutSuccessHandler(new SecurityLogoutSuccessHandler())
                        // 使session失效
                        .invalidateHttpSession(true)
                        // 清除认证信息
                        .clearAuthentication(true)
                        // 注销时,删除自定义的cookie
                        //.deleteCookies("myCookie")
                )
                .csrf(AbstractHttpConfigurer::disable);
				
      	// 添加自定义过滤器
      	http.addFilterBefore(jsonLoginFilter(), UsernamePasswordAuthenticationFilter.class);

      	return http.build();
    }

    private JsonLoginFilter jsonLoginFilter() {
        JsonLoginFilter filter = new JsonLoginFilter();
        filter.setAuthenticationManager(authenticationManager);
        filter.setFilterProcessesUrl("/login");
    		// 注意：这里如果使用JWT或无状态的身份验证,就不要设置securityContextRepository
        filter.setAuthenticationSuccessHandler(new SecurityAuthenticationSuccessHandler());
        filter.setAuthenticationFailureHandler(new SecurityAuthenticationFailureHandler());

        return filter;
    }

}

# 2.自定义接口的方式

首先，我们自定义登录的接口，通过 AuthenticationManager 来进行认证相关的逻辑：

@RestController
public class LoginController {

    @Autowired
    private AuthenticationManager authenticationManager;

    @PostMapping("/doLogin")
    public String doLogin(@RequestBody User user, HttpSession session) {
        UsernamePasswordAuthenticationToken unauthenticated = 
        			UsernamePasswordAuthenticationToken.unauthenticated(user.getUsername(), user.getPassword());
        try {
            Authentication authenticate = authenticationManager.authenticate(unauthenticated);
            SecurityContextHolder.getContext().setAuthentication(authenticate);
          	session.setAttribute(HttpSessionSecurityContextRepository.SPRING_SECURITY_CONTEXT_KEY, SecurityContextHolder.getContext());
            return ResponseFactory.FACTORY.buildSuccess();
        } catch (AuthenticationException e) {
            rreturn ResponseFactory.FACTORY.buildFail(e);
        }
    }
}

然后，我们在 SecurityConfig 中配置对应的登录处理的 url，即下面的 auth.antMatchers("/doLogin") 代码：

@Configuration
public class SecurityConfig {

    @Autowired
    private UserDetailsService userDetailsService;

		@Bean
    public AuthenticationManager authenticationManager() {
        DaoAuthenticationProvider provider = new DaoAuthenticationProvider();
        provider.setUserDetailsService(userDetailsService);
        return new ProviderManager(provider);
    }

    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        http.authorizeRequests(auth ->
                        // /doLogin无需认证
                        auth.requestMatchers("/doLogin").permitAll()
                        				.anyRequest().authenticated()
                )
                .exceptionHandling(exceptionHandling ->
                        exceptionHandling.authenticationEntryPoint(new SecurityAuthenticationEntryPoint())
                )
                .formLogin(formLogin ->
                    formLogin.permitAll()
                        // 认证成功时的处理器
                        .successHandler(new SecurityAuthenticationSuccessHandler())
                        // 认证失败的处理器
                        .failureHandler(new SecurityAuthenticationFailureHandler())
                )
                .logout(logout -> 
                    logout.logoutSuccessHandler(new SecurityLogoutSuccessHandler())
                        // 使session失效
                        .invalidateHttpSession(true)
                        // 清除认证信息
                        .clearAuthentication(true)
                        // 注销时,删除自定义的cookie
                        //.deleteCookies("myCookie")
                )
                .csrf(AbstractHttpConfigurer::disable);

        return http.build();
    }

}

# 六、其他方式

如果我们不想定义这些处理器，也有另外一种方式，就是配置未认证、认证成功、认证失败后跳转的接口地址，具体可参考 FormLoginConfigurer 类的配置：

方法 说明
loginProcessingUrl(String loginProcessingUrl) 实际表单向后台提交用户信息的 Action ，再由过滤器 UsernamePasswordAuthenticationFilter 拦截处理，该 Action 其实 不会处理任何逻辑。默认为 /login
usernameParameter(String usernameParameter) 用来自定义用户参数名，默认 username 。
passwordParameter(String passwordParameter) 用来自定义用户密码名，默认 password。
failureUrl(String authenticationFailureUrl) 登录失败后会重定向到此路径， 一般前后分离不会使用它。
failureForwardUrl(String forwardUrl) 登录失败会转发到此，一般前后分离用到它。可在 Controller 中定义一个接口来处理返回值，但是要注意 RequestMethod 。
logoutUrl(String logoutUrl) 退出登录成功的接口地址，默认为 /logout
logoutSuccessUrl(String logoutSuccessUrl) 退出登录成功会转发到此路径，可在 Controller 中定义一个接口来处理返回值，但是要注意 RequestMethod。默认为 /login?logout
defaultSuccessUrl(String defaultSuccessUrl, boolean alwaysUse) 默认登陆成功后跳转到此 。如果 alwaysUse 为 true， 只要进行认证流程而且成功，会一直跳转到此。一般推荐默认值 false
successForwardUrl(String forwardUrl) 效果等同于上面 defaultSuccessUrl 的 alwaysUse 为 true 但是要注意 RequestMethod。
failureHandler(AuthenticationFailureHandler authenticationFailureHandler) 自定义失败处理器，可替代上面所有的 failureXxx 方式
successHandler(AuthenticationSuccessHandler successHandler) 自定义认证成功处理器，可替代上面所有的 successXxx 方式
logoutSuccessHandler(LogoutSuccessHandler logoutSuccessHandler) 自定义认证成功处理器，可替代上面所有的 logoutSuccessXxx 方式
permitAll(boolean permitAll) form 表单登录是否放开
