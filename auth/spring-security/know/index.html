# 一、权限校检框架

权限校检，就是用户在访问接口或者web资源服务时，我们要确定用户是合法的，当前用户访问的数据是在权限范围内的，避免有人恶意访问数据。

Java 领域权限管理框架目前著名的就是 Shiro 和 Spring Security。

# 1.Shiro

Shiro 有着众多的优点，例如轻量、简单、易于集成等。

当然 Shiro 也有不足，例如对 OAuth2 支持不够，在 Spring Boot 面前无法充分展示自己的优势等等，特别是随着现在 Spring Boot 和 Spring Cloud 的流行。

# 2.Spring Security

自从 Spring Boot、Spring Cloud 火起来之后，Spring Security 在 JavaEE 企业级开发的安全管理架构选择越来越优先。

Spring Security 最早不叫 Spring Security ，叫 Acegi Security，叫 Acegi Security 并不是说它和 Spring 就没有关系了，它依然是为 Spring 框架提供安全支持的。事实上，Java 领域的框架，很少有框架能够脱离 Spring 框架独立存在。

Acegi Security 基于 Spring，可以帮助我们为项目建立丰富的角色与权限管理，但是最广为人诟病的则是它臃肿繁琐的配置，这一问题最终也遗传给了 Spring Security。

在 Acegi Security 时代，网上流传一句话：“每当有人要使用 Acegi Security，就会有一个精灵死去。”足见 Acegi Security 的配置是多么可怕。

当 Acegi Security 投入 Spring 怀抱之后，先把这个名字改了，这就是大家所见到的 Spring Security 了，然后配置也得到了极大的简化。

但是和 Shiro 相比，人们对 Spring Security 的评价依然中重量级、配置繁琐。

直到 Spring Boot 火起来，Spring Security 也被选择为企业开发的重要选型方案。Spring Boot 优化了 Spring Security 的配置，使得开发者们快速的对 web 应用进行安全管理

# 二、Spring Security 核心功能

对于一个权限管理框架而言，无论是 Shiro 还是 Spring Security，最核心的功能，无非就是两方面：

  • 认证 authentication
  • 授权 authorization

认证就是我们常说的登录，即我是谁。授权就是权限鉴别，即我能做什么，看看请求是否具备相应的权限。如去游乐园，需要携带身份证和购买门票，身份证则是证明你是谁，门票则是证明你能享受什么项目，138的套餐只能玩游乐园的项目，288元的套餐可以免费吃小吃，428元的项目可以住宿等等。

Spring Security 支持多种不同的认证方式，这些认证方式有的是 Spring Security 自己提供的认证功能，有的是第三方标准组织制订的，主要有如下一些：

一些比较常见的认证方式：

  • HTTP BASIC authentication headers：基于 IETF RFC 标准。
  • HTTP Digest authentication headers：基于 IETF RFC 标准。
  • HTTP X.509 client certificate exchange：基于 IETF RFC 标准。
  • LDAP：跨平台身份验证。
  • Form-based authentication：基于表单的身份验证。
  • Run-as authentication：用户用户临时以某一个身份登录。
  • OpenID authentication：去中心化认证。

除了这些常见的认证方式之外，一些比较冷门的认证方式，Spring Security 也提供了支持。

  • Jasig Central Authentication Service：单点登录。
  • Automatic “remember-me” authentication：记住我登录（允许一些非敏感操作）。
  • Anonymous authentication：匿名登录。
  • ……

作为一个开放的平台，Spring Security 提供的认证机制不仅仅是上面这些。如果上面这些认证机制依然无法满足你的需求，我们也可以自己定制认证逻辑。当我们需要和一些“老破旧”的系统进行集成时，自定义认证逻辑就显得非常重要了。

除了认证，剩下的就是授权了。

Spring Security 支持基于 URL 的请求授权、支持方法访问授权以及对象访问授权。

# 三、前置知识

# 1.什么是 ant 风格路径匹配

# 1.1 什么是 ant 风格路径匹配

ant 风格就是一种路径匹配表达式。主要用来对 uri 的匹配。其实跟正则表达式作用是一样 的，只不过正则表达式适用面更加宽泛，ant 仅仅用于路径匹配。

ant 风格路径匹配在 Spring MVC 和 Spring Security 中是经常用到的。

# 1.2 ant 通配符

ant 中的通配符有三种：

  • ? ：匹配任何单字符

  • *：匹配 0 或者任意数量的字符

  • **：匹配 0 或者多级的目录

ant 中的 * 是在一个目录中进行匹配的，** 是可以匹配多级目录的。

匹配示例：

  • ? ：

    • /ant/p?tten：会匹配到路径 /ant/patten 、/ant/pbtten 等路径，但是不会匹配到 /ant/ptten。

  • *：

    • /ant/*.html：会匹配到 /ant 路径下，所有的 html 文件。
    • /ant/*/path：会匹配到 /ant/path 、/ant/a/path、/ant/abc/path 等路径，但是不会匹配到 /ant/a/b/path。

  • **：

    • ``/ant/**/path：会匹配到/ant/path、/ant/a/path、/ant/abc/path、/ant/a/b/path`等路径。

# 1.3 最长匹配原则

在我们上面的示例中，可以发现 * 和 ** 会有冲突的情况存在。为了解决这种冲突的情况，ant 风格匹配的时候，如果一个 url 匹配到了两个符合条件的 ant 匹配规则，则会匹配字符最多的规则。因为字符越长，匹配信息就越具体。

如 /ant/a/path 同时匹配到了 /ant/*/path 和 **/path，那么就会走 /ant/*/path 。

# 2.RBAC 模型

# 2.1 RBAC 介绍

RBAC（Role-Based Access Control）全称为基于角色的访问控制。在 RBAC 中，权限与角色相关联， 用户通过成为适当角色的成员而得到这些角色的权限。这就极大地简化了权限的管理。这样管理都是层级相互依赖的，权限赋予给角色，而把角色又赋予用户，这样的权限设计很清楚，管理起来很方便。

当你拥有某个角色以后，自然会继承了该角色的所有功能（权限）。对你的一些操作限制不需要直接与你进行沟通，只需要操作你拥有的角色。比如你是一个人事，你在公司系统只有人事的角色和人事相关的权限（招聘、工资、入职、转岗等等功能的权限），突然老板说让你兼顾做行政，于是你在公司系统也有了行政的角色，行政相关的权限（办公用品管理、公司团建、会议室管理、资产管理等权限）你也自然拥有了。

# 2.2 RBAC 模型的分类

RBAC 模型可以分为：RBAC0、RBAC1、RBAC2、RBAC3 四种。其中 RBAC0 是基础，其它三种都是在 RBAC0 基础上的变种。大部分情况下，使用 RBAC0 模型就可以满足常规的权限管理系统设计了。不过 一定不要拘泥于模型，要以业务需要为先导。

RBAC0

RBAC0 是基础，定义了能构成 RBAC 权限控制系统的最小的集合，RBAC0 由四部分构成:

  • 用户（User）：权限的使用主体。
  • 角色（Role）：包含许可的集合。
  • 会话（Session）：绑定用户和角色关系映射的中间通道。而且用户必须通过会话才能给用户设置角色。
  • 许可（Pemission）：对特定资源的特定的访问许可。

RBAC1

RBAC1 在 RBAC0 的基础之上引入了角色继承的概念，有了继承那么角色就有了上下级或者等级关系。 父角色拥有其子角色所有的许可。例如，你有的角色，你的领导一定有这个角色，反过来就不一定了。

RBAC2

“体育比赛中，你不可能既是运动员又是裁判员!”。这是很有名的一句话。反应了我们经常出现的一种职务冲突。有些角色产生的历史原因就是为了制约另一个角色。如公司的会计和出纳不能由一个人承担。

小知识

在财务准则中，出纳和会计是不允许由同一个人担任这两个职位的。除非是老板不怕这个人私自挪用公司的钱，且公司是无需审计的单位。

还有就是我们每个人在不同的场景都会充当不同的角色，在公司你就是特定岗位的员工，在家庭中你就是一名家庭成员。随着场景的切换，我们的角色也在随之变化。

所以 RBAC2 在 RBAC0 的基础上引入了静态职责分离（Static Separation of Duty，简称SSD）和动态职责分离（Dynamic Separation of Duty，简称DSD）两个约束概念。他们两个作用的生命周期是不同的：

  • SSD 作用于约束用户和角色绑定时。
    1. 互斥角色：就像上面的例子你不能既是 A 又是 B，互斥的角色只能二选一；
    2. 数量约束：用户的角色数量是有限的不能多于某个基数；
    3. 条件约束：只能达到某个条件才能拥有某个角色。经常用于用户等级体系，如充钱成为 VIP 才能观看会员视频。
  • DSD 作用于会话和角色交互时。当用户持有多个角色，在用户通过会话激活角色时加以条件约束， 根据不同的条件执行不同的策略。

RBAC3

RBAC3 = RBAC1 + RBAC2 。

# 2.3 RBAC 中概念的理解

  • 用户（User）：
    • 对用户的理解不应该被局限于单个用户，也可以是用户组（类似 Linux 的 UserGroup）也可以成是部门或者公司。
    • 也可以是虚拟的账户、客户、甚至第三方应用。
    • 所以对用户的理解要宽泛一些，只要是有访问资源需求的主体都可以纳入用户范畴。
  • 角色（Role）：
    • 角色是特定许可的集合以及载体。一个角色可以对应多个用户，一个用户同样的也可以属于多个角色。同样的，一个角色可以对应多个用户组，一个用户组也可以具有多个角色。即角色和用户是多对多的关系。
    • 角色是可以细分的，也就是可以继承、可以分组的。
  • 许可（Permission）：
    • 许可一般称作权限。通常我们将访问的目标统称为资源，不管是数据还是静态资源都是资源。
    • 一般访问资源基本上都会通过 HTTP 接口来访问。所以一般权限都体现在对接口的控制上。再细分的话可以将其划分为菜单控制、具体数据增删改查功能控制（前台体现为按钮）。
    • 许可具有原子性，不可再分。我们将许可授予角色时就是粒度最小的单元。

# 3.安全策略

  • CSRF 跨站请求伪造：CSRF 是一种攻击，它通过欺骗受害者在不知情的情况下执行网站上的操作来实现。通常涉及攻击者制作恶意链接或表单，受害者无意中点击，从而代表受害者执行操作。
  • XSS 跨站脚本攻击：XSS 是另一种类型的攻击，它涉及将恶意代码注入到网站中。攻击者可以利用这些代码窃取敏感信息或执行其他恶意操作。
  • OAuth 2.0：OAuth 2.0 是一种用于授权和认证的协议。它允许用户授权另一个网站访问其数据，而无需提供密码。这通常由社交媒体网站用于允许用户使用其社交媒体凭据登录其他网站。如微信一键登录、手机号码一键登录等。
