06.Spring Security 的用户存储

# 一、Spring Security 的用户存储

我们在系统中登录及查询用户权限的前提是用户已经在系统注册，那么用户注册后其信息（用户名、密码、姓名、角色等信息）在 Spring Security 中存储到哪里了？又是怎么存储的呢？

Spring Security 中，默认基于内存的存储、其自身也内置了基于数据库的存储，不过其内置的基于数据库存储对应的表是非常简单的，不能满足开发者的需求。因此，其提供了可以自定义数据库存储的接口 UserDetailsService，我们只需要实现 UserDetailsService 接口，只需要在接口的实现中调用我们自定义操作数据库的代码即可。

当然我们也可以使用 UserDetailsManager 接口，UserDetailsManager 接口继承于 UserDetailsService 接口。其比 UserDetailsService 接口多了 创建用户、修改用户、删除用户、修改密码、判断用户是否存在的方法。

UserDetailsService 接口仅仅有一个 loadUserByUsername 方法，该方法用来在认证时通过用户名查询用户信息（包括用户名、密码、权限、用户禁用状态、是否锁定等信息）。

这样一来，我们就可以使用自己系统中的 ORM 框架（如 Mybatis，Spring Data JPA，Spring JDBC 等等），也可以自己自定义用户、权限、角色等相关的表，使用起来也非常灵活。

# 二、基于内存的存储

# 1.InMemoryUserDetailsManager

InMemoryUserDetailsManager 是基于内存存储用户的类，其类结构图如下：

通过类图可以发现，InMemoryUserDetailsManager 的实现非常简单，其实现了 UserDetailsManager 、UserDetailsPasswordService 、UserDetailsService 的方法。

InMemoryUserDetailsManager 内部维护了一个 Map<String, MutableUserDetails> users 属性，该属性用来存储相关的用户。当我们添加、修改、删除用户，最终都是修改的 users 属性中的值。

当然，users 是存储在内存中的，这就是系统一旦启动，其用户信息就会被重置的原因。

# 2.基于内存的存储的默认配置

实际上，我们前面已经接触到了基于内存的授权，之前配置 InMemoryUserDetailsManager 的 Bean，就是配置关于基于内存的 UserDetailsManager。

Spring Security 中，默认使用基于内存的存储，在 Spring Boot 程序启动时，Spring Security 默认情况下会自动初始化一个

基于内存的存储配置只是用来测试或者供开发人员学习的，实际的开发中，绝大多数不会使用基于内存的存储，因为一旦系统重启，用户的信息就就会丢失，并且这在分布式集群中也不能共享用户信息。

基于内存的存储的默认配置在以下类中：

  • InMemoryUserDetailsManagerConfigurer：其仅有一个构造方法，构造方法中会直接创建 InMemoryUserDetailsManager 实例。
  • DefaultPasswordEncoderAuthenticationManagerBuilder：AuthenticationManagerBuilder 的默认情况的配置。

# 3.手动配置

当然，我们也可以手动配置基于内存的存储：

@Configuration
public class AuthenticationManagerConfig {

    @Autowired
    private PasswordEncoder passwordEncoder;

   	@Bean
    public UserDetailsManager inMemoryUserDetailsManager(SecurityProperties properties) {
        List<UserDetails> users = new ArrayList<>();
        // 获取application.properties配置文件中的用户信息
        SecurityProperties.User user = properties.getUser();
        if (user != null) {
            UserDetails userDetails = User.withUsername(user.getName())
                    .password(passwordEncoder.encode(user.getPassword()))
                    .roles(StringUtils.toStringArray(user.getRoles()))
                    .build();
            users.add(userDetails);
        }
      
      	// 自定义用户配置
        UserDetails userDetails2 = User.withUsername("zhangsan")
                .password(passwordEncoder.encode("123456"))
                .roles("USER")
                .build();
        users.add(userDetails2);
        return new InMemoryUserDetailsManager(users);
    }

    @Bean
    public AuthenticationManager authenticationManager(UserDetailsManager userDetailsService) {
        DaoAuthenticationProvider provider = new DaoAuthenticationProvider();
        provider.setUserDetailsService(userDetailsService);
        provider.setPasswordEncoder(passwordEncoder);
        return new ProviderManager(provider);
    }

}

# 三、基于默认数据库模型的存储

# 1.JdbcUserDetailsManager

Spring Security 除了提供基于内存的 InMemoryUserDetailsManager，还提供了基于数据库的 JdbcUserDetailsManager 。

JdbcUserDetailsManager 的结构类图如下：

JdbcUserDetailsManager 继承自 JdbcDaoImpl，JdbcDaoImpl 的父类 JdbcDaoSupport 属于 spring-jdbc 提供的类。

使用 JdbcUserDetailsManager 可以将用户和授权信息存储和维护到数据库中，通过 JDBC 对用户信息增删改查。

不过，JdbcUserDetailsManager 中用户和授权的数据库模型（表结构）必须使用 Spring Security 官方的提供的默认的数据库模型，这个默认的数据模型比较简单，一般无法满足实际的业务。

# 2.3 配置 JdbcUserDetailsManager

我们新建一个配置类 SecurityConfig，并配置 JdbcUserDetailsManager、PasswordEncoder、SecurityFilterChain 对象到 Spring 容器。

@Configuration
public class SecurityConfig {

		@Bean
    public JdbcUserDetailsManager jdbcUserDetailsManager(DataSource dataSource) {
        return new JdbcUserDetailsManager(dataSource);
    }

  	@Bean
    public PasswordEncoder passwordEncoder() {
        return NoOpPasswordEncoder.getInstance();
    }
    
  	@Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        http.authorizeHttpRequests(auth -> auth
                        .requestMatchers("/hello").permitAll()
                        .requestMatchers("/user/**").hasAnyRole("USER", "ADMIN")
                        .requestMatchers("/admin/**").hasRole("ADMIN")
                        .anyRequest().authenticated()
                )
                .formLogin(formLogin -> formLogin.permitAll());

          return http.build();
    }

}

  `user_id` bigint unsigned NOT NULL DEFAULT '0' COMMENT '用户ID',
  `username` varchar(30) COLLATE utf8mb4_general_ci NOT NULL COMMENT '用户账号',
  `real_name` varchar(50) CHARACTER SET utf8mb4 COLLATE utf8mb4_general_ci DEFAULT NULL COMMENT '真实姓名',

  `password` varchar(100) COLLATE utf8mb4_general_ci DEFAULT '' COMMENT '密码',
  `enable` char(1) COLLATE utf8mb4_general_ci DEFAULT '1' COMMENT '帐号状态 0-停用 1-正常',
  `del_flag` char(1) CHARACTER SET utf8mb4 COLLATE utf8mb4_general_ci DEFAULT '0' COMMENT '删除标志 0-存在 1-删除',

  PRIMARY KEY (`user_id`),

) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_general_ci COMMENT='用户角色表';

# 2.5 自定义 MybatisUserDetailsService

自定义 MybatisUserDetailsService 类，实现 UserDetailsService 接口，实现 loadUserByUsername 方法。

在 loadUserByUsername 方法中，通过调用 SysUserMapper 和 SysUserRoleMapper 接口，来查询数据库，将查询出的用户及权限信息组装到 UserDetails 对象中，并 return。

@Service
public class MybatisUserDetailsService implements UserDetailsService {

    private static final String IS_DISABLED = "0";

    @Autowired
    private SysUserMapper sysUserMapper;

    @Autowired
    private SysUserRoleMapper sysUserRoleMapper;

    @Override
    public UserDetails loadUserByUsername(String username) throws UsernameNotFoundException {
        // 用户名
        SysUser user = sysUserMapper.selectOne(Wrappers.<SysUser>lambdaQuery().eq(SysUser::getUsername, username));
        // 手机号
        if (user == null) {
            user = sysUserMapper.selectOne(Wrappers.<SysUser>lambdaQuery().eq(SysUser::getMobile, username));
        }
        // 邮箱
        if (user == null) {
            user = sysUserMapper.selectOne(Wrappers.<SysUser>lambdaQuery().eq(SysUser::getEmail, username));
        }
        if (user == null) {
            throw new UsernameNotFoundException(username);
        }

        String[] roles = this.getUserRoles(user.getUsername());
        UserBuilder userBuilder = User.withUsername(user.getUsername())
                .password(user.getPassword())
                .disabled(user.getEnable().equals(IS_DISABLED));
        if (roles != null) {
            userBuilder.roles(roles);
        }

        return userBuilder.build();
    }

    private String[] getUserRoles(String username) {
        List<SysUserRole> roles = sysUserRoleMapper.selectList(Wrappers.<SysUserRole>lambdaQuery()
                .eq(SysUserRole::getUsername, username)
        );
        if (CollectionUtils.isEmpty(roles)) {
            return null;
        }

# 五、密码存储

# 1.Spring Security 的密码加密

说到用户存储，也不得不提一下密码存储。一般来说我们需要对密码加密后进行存储，以防止 SQL 注入或查询数据库等问题导致用户密码泄露。

在 Spring Security 中，PasswordEncoder 接口用于密码加密、密码匹配。PasswordEncoder 接口方法如下：

  • encode：用于将未加密过的密码加密。
  • matches：用于原密码和加密密码匹配，即判断密码是否正确。
  • upgradeEncoding：该方法是一个默认方法，用于判断是否需要升级密码加密算法，默认会返回 false

# 2.配置密码加密

# 2.1 常用的密码编码器

在之前，我们一直使用的是 NoOpPasswordEncoder ，即不使用密码加密，这是非常不安全的。

在 Spring Security 中，提供了一系列的密码编码类，你可以查看 PasswordEncoder 接口的实现类来查询有哪些可用的密码编码类。常用的如下：

  • BCryptPasswordEncoder：使用BCrypt强哈希函数对密码进行加密和验证。它是一种单向哈希算法，每次生成的哈希值都是不同的，可以有效地防止密码被破解。使用时需要设置强度（strength）参数，指定计算哈希值的次数。
  • SCryptPasswordEncoder：使用SCrypt算法对密码进行加密和验证。它是一种基于密码的密钥派生函数，与BCrypt类似，但更加安全。
  • Pbkdf2PasswordEncoder：使用PBKDF2（Password-Based Key Derivation Function 2）算法对密码进行加密和验证。它是一种基于密码的密钥派生函数，通过多次迭代和随机化来增加密码的安全性。
  • ……

不过 Spring Security 官方默认使用 BCryptPasswordEncoder 。

# 3.使用 DelegatingPasswordEncoder 配置

# 3.1 DelegatingPasswordEncoder 介绍

DelegatingPasswordEncoder 类也实现了 PasswordEncoder 接口，不过它是一个代理类，其使用一个 Map 来存储不同前缀对应的 PasswordEncoder 实例。

  • 当对密码进行加密时，DelegatingPasswordEncoder 会根据默认配置的密码编码器来进行加密。
  • 当对密码进行验证时，DelegatingPasswordEncoder 会根据加密后的密码的前缀选择合适的 PasswordEncoder 来进行验证。

因此，DelegatingPasswordEncoder 可以兼容多种密码加密算法。并且，还支持直接升级密码加密算法：在用户登录时，如果密码匹配成功，则将用户的密码通过新的加密算法加密后（密码是用户在认证请求时传过来的），得到新算法加密的密码，然后将”新算法加密的密码“更新到数据库中，以此完成升级。

加密算法的具体逻辑，见源码 DaoAuthenticationProvider#createSuccessAuthentication。

# 3.2 配置使用

我们可以使用 PasswordEncoderFactories.createDelegatingPasswordEncoder() 来创建 DelegatingPasswordEncoder 实例。如下：

@Configuration
public class PasswordEncoderConfig {

    @Bean
    public PasswordEncoder passwordEncoder() {
       return PasswordEncoderFactories.createDelegatingPasswordEncoder();
    }

}

        PasswordEncoder noop = NoOpPasswordEncoder.getInstance();
        encoders.put("noop", noop);

        return new DelegatingPasswordEncoder("bcrypt", encoders);
