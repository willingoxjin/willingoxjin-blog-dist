09.Spring Security 的 Session 管理

Spring Security 的 Session 管理官方文档：Authentication Persistence and Session Management (opens new window)

# 一、Session 介绍

Session （会话）是我们在 Web 开发和面试中经常接触到的，下面我们来详细的了解一下 Session。

# 1.HTTP 协议

HTTP（Hypertext Transfer Protocol）是一种用于传输超文本的应用层协议。它是在 Web 应用中广泛使用的协议，用于客户端和服务器之间的通信。

HTTP 协议的特点如下：

  1. 简单性：HTTP 的设计简单明了，易于理解和实现。它使用简单的请求-响应模型，客户端发送请求，服务器返回响应。

  2. 无状态性：HTTP 是一种无状态协议，即服务器不会保存关于客户端的任何信息。每个请求都是独立的，服务器不会记住之前的请求。

  3. 可扩展性：HTTP 协议支持通过添加新的方法、头部字段和状态码来扩展功能。这使得它适应了不断发展的互联网需求。

  4. 基于请求-响应模型：客户端发送 HTTP 请求到服务器，服务器处理请求并返回 HTTP 响应。请求和响应都由起始行、头部字段和可选的消息体组成。

  5. 无连接：在每个请求和响应之间都需要建立一个新的连接。当客户端发送一个请求到服务器时，它会建立一个TCP连接，发送请求并等待服务器的响应。一旦服务器返回响应，连接就会关闭。下一个请求需要再次建立连接。

    • 无连接的特性意味着每个请求都需要额外的时间和资源来建立和关闭连接。这对于某些应用场景可能会带来一定的开销。为了解决这个问题，HTTP/1.1 引入了持久连接（Keep-Alive），允许在一个连接上发送多个请求和响应，从而减少了建立和关闭连接的开销。

  6. 支持多媒体内容：HTTP 支持传输各种类型的数据，包括文本、图像、音频、视频等。通过 Content-Type 头部字段，服务器和客户端可以识别和处理不同类型的数据。

  7. 使用 TCP/IP 协议：HTTP 使用 TCP/IP 协议作为传输层协议，通过建立可靠的连接来传输数据。

  8. 安全性：HTTP 本身不具备加密功能，因此传输的数据可能会被窃听或篡改。为了提供安全性，可以使用 HTTPS（HTTP Secure）协议，它在 HTTP 上添加了 SSL/TLS 加密层。

# 2.服务器如何识别用户？

由于 HTTP 是无状态的协议，这意味着每次客户端检索网页时，都要单独打开一个服务器连接，因此服务器不会存储先前客户端请求的任何信息。

服务端为了识别用户，通过 Session 来解决这个问题：服务端和客户端约定，在客户端请求时，会携带一个 id，如果客户端请求时，携带的 id 都相同，服务端就识别这些请求是同一个用户发出的，这样就达到了识别用户的目的。

为了让同一用户每个请求携带的 id 都相同，在用户第一次访问系统时，系统会为用户生成一个 sessionid，并将 sessionid 存储到 Cookie 中，这样每次请求时，就携带 Cookie ，服务端则从 Cookie 中读取 sessionid，以此识别是否为同一用户。

不过，有的用户会在浏览器中禁用 Cookie，这样的话，请求时通过 Cookie 携带 sessionid 来识别用户就无效了。所以，有的系统还支持 URL 重写的方式实现：如 http:/example.com;sessionid=ABCDEF。但是，URL 重写存在会话固定攻击的问题。

# 3.HttpSession

HttpSession 属于服务端的概念，服务端会在用户第一次访问系统时，创建一个 HttpSession 对象，并将其缓存起来。这个 HttpSession 对象会对应一个 sessionid，并将 sessionid 储存在 Cookie 中，即我们上面提到的 sessionid。

当用户再次访问系统时，就会携带 Cookie ，服务端则通过 Cookie 中的 sessionid 来查找对应的 HttpSession 对象，这就形成了“会话”。

当浏览器关闭后，并不会导致服务端的 HttpSession 失效。如果想让服务端的 HttpSession 失效，要么手动调用 HttpSession#invalidate() 方法，要么等到 Session 自动过期，要么重启服务端。

大概流程如下：

  • 用户第一次访问系统
  • 服务端未找到 sessionid，于是新建一个 HttpSession 缓存起来，生成对应的 sessionid，并在 HTTP 响应头中的 Set-Cookie 属性中设置 sessionid。
  • 用户的浏览器会读取 HTTP 响应头中的 Set-Cookie 属性，将 sessionid 存储到 Cookie 中。
  • 用户在后面的请求就会携带 Cookie。
  • 服务端则会读取 Cookie 中的 sessionid，通过 sessionid 读取到缓存的 HttpSession，就得知道用户的身份了。

为什么有的人会感觉浏览器关闭之后 Session 就失效了呢？

  • 因为默认情况下，浏览器关闭之后，保存在浏览器里边的 sessionId 就会被清除（默认情况不设置 Cookie 的过期时间，则 Expires 为 Session）。所以当浏览器再次访问服务端的时候，服务端会给浏览器重新分配一个 sessionId，这个 sessionId 和之前的 HttpSession 对应不上，所以用户就会感觉 Session 失效。

我们可以通过手动配置，让浏览器重启之后 sessionId 不丢失，但是这样会带来安全隐患，因此一般不建议修改此配置。

Set-Cookie 是HTTP响应头中的一个属性，用于在服务器端设置或修改Cookie。它的作用是在服务器端告知浏览器如何处理 Cookie。

常用的 Cookie 属性包括：

  • Expires：设置 Cookie 的过期时间。例如：Expires=Wed, 21 Oct 2022 07:28:00 GMT。
    • 如果不设置过期时间或最大存活时间，其默认会存储在浏览器的内存中，Expires 则为 Session ，当浏览器关闭时，就会自动清除 Cookie。
  • Max-Age：设置 Cookie 的最大存活时间（秒）。例如：Max-Age=3600 表示 Cookie 将在1小时后过期。
  • Domain：设置 Cookie 的作用域。例如：Domain=example.com 表示 Cookie 仅在 example.com 及其子域名下有效。
  • Path：设置 Cookie 的有效路径。例如：Path=/subdirectory 表示 Cookie 仅在 /subdirectory 目录下有效。
  • Secure：设置 Cookie 只能通过 HTTPS 安全连接发送。
  • HttpOnly：设置 Cookie 只能通过 HTTP 协议传输，禁止 JavaScript 访问，可以有效的防止 XSS 攻击。
  • SameSite：设置 Cookie 的同源策略，用于防止跨站点请求伪造（CSRF）攻击。可选配置如下：
    • None：表示 Cookie 将在所有跨站点请求中发送。
    • Secure：表示 Cookie 只能通过 HTTPS 安全连接发送。
    • Lax：表示 Cookie 会在跨站点的安全请求中发送。默认设置，如果未显式指定 SameSite 属性，则 Cookie 将被视为 SameSite=Lax。

# 二、防御会话固定攻击

# 1.什么是会话固定攻击？

# 1.1 URL 重写

上面，我们说到，对于浏览器不支持 Cookie 或用户禁用 Cookie 时，可以使用 URL 重写来携带 sessionId，以便服务器识别用户身份。下面我们来了解一下 URL 重写。

URL 重写（URL rewriting）是一种在 Web 服务器上修改传入请求 URL 的处理技术。它允许将用户请求的 URL 进行修改，以便更好地适应服务器的处理需求或提供更友好的 URL 结构。

URL 重写通常通过在服务器上配置规则来实现。这些规则可以基于正则表达式或其他模式匹配方式，将用户请求的 URL 映射到服务器上的不同资源或处理程序。重写规则可以修改 URL 的路径、查询参数、主机名等部分，甚至可以将动态 URL 重写为静态 URL。

URL 重写不需要往返服务器。重写的 URL 不会被返回客户端。例如用户访问的是 http://example.com/users ，但到了服务端被重写为 ``http://example.com/userList，但是用户在浏览器中访问的地址仍是http://example.com/users`，并且不会感知到 url 已经被重写了。

URL重写的一些常见应用包括：

  1. 美化 URL：将复杂的动态URL转换为更简洁、易读的静态URL，提高用户体验和SEO优化。例如，用户通过浏览器请求的 URL 为 http://example.com/products?id=1，那么这个 URL 在服务端可以被重写为 http://example.com/products/1，使 URL 转化为 restful 风格。
  2. 重定向：将旧的 URL 重定向到新的 URL，以确保用户访问的是最新的页面。
  3. 路由请求：根据 URL 的不同部分将请求路由到不同的处理程序或资源。
  4. 虚拟目录映射：将 URL 映射到服务器上的不同目录或文件路径。
  5. 传递会话标识：当用户禁用浏览器的 Cookie 时，可以使用 URL 重写来传递会话标识符（如 JSESSIONID）作为查询参数或路径参数。这样做可以绕过 Cookie 的限制，确保会话的正常运行。

# 1.2 会话固定攻击

我们再来回顾一下通过 URL 重写携带 sessionid 的流程：

  • 用户访问地址 http://www.example.com/products/item?id=123，此时，没有携带 sessionid，服务器就会创建一个新的 Session，并将该地址重定向到 http://www.example.com/products/item;sessionid=ABCDEF?id=123；
  • 此后，浏览器端每次请求都将在 URL 上携带 sessionid，服务端就根据 sessionid 来获取缓存的 HttpSession 对象，以此来识别用户身份。

在 URL 重写携带 sessionid 的过程中，如果用户 A 在自己的浏览器登录后，再把自己访问的链接发给用户 B，那用户 B 在自己的浏览器访问该链接，就会直接以用户 A 的身份直接访问系统了，这是非常不安全的。当然，我们也可以警告使用者不要将链接发给未授信的用户。

反过来，如果黑客在自己的浏览器中访问系统，不登录的情况下，系统也会在 URL 中携带 sessionid；然后，黑客就把这个携带 sessionid 的链接发给其他用户（如张三），当诱导用户张三登录成功后，黑客就可以直接通过这个 sessionid 直接通过用户张三的身份访问系统了。这是会话固定攻击（session fixation attack）了。

会话固定攻击不只是会使用 URL 重写的方式，还可以通过 XSS 攻击获取到 sessionid。当然，只要是通过获取已经登录用户的 sessionid 来攻击的方式，都属于会话固定攻击。

并且，Spring Security 还有其他机制来防止会话固定工具，主要包括以下方面：

  1. 使用 StrictHttpFirewall 拦截请求的 URL，如果 URL 中包含 ;、%3b、%3B （%3b 和 %3b 是 ; encoded 后的字符串） ，则会直接拒绝请求。
  2. 在用户认证成功后，通过响应头的 Set-Cookie 设置 Cookie 时，添加 HttpOnly 属性。以此避免通过 XSS 攻击来获取 Cookie 中的会话信息，进而防止会话固定攻击。
  3. 改变用户的认证成功后的 sessionid。
    • 如果认证成功后，我们修改用户的 sessionid，黑客提供的登录链接中的 sessionid 就是无效的了。
    • 利用 Spring Security 提供的防御会话固定攻击的策略即可实现认证成功后修改 sessionid 的功能。

对于上面的第三点：改变用户的认证成功后的 sessionid。Spring Security 中的 SessionFixationConfigurer 类中定义了会话固定防御的配置策略。如下：

  • newSession：在用户认证成功后，创建一个新的会话，旧会话将被废弃。这样会话 ID 将会更改。（Spring Security 相关属性仍将被复制）
  • migrateSession：启用会话迁移来防护会话固定攻击。在用户认证成功后，会生成一个新的会话，并将用户的会话数据从旧会话迁移到新会话中。（这是 Servlet 3.0 或更早版本容器中的默认设置）
  • changeSessionId：在用户认证成功后，生成一个新的会话ID，并将其分配给当前会话。（这是 Servlet 3.1 和更新版本容器中的默认设置）
  • none：禁用会话固定攻击防护。这是默认配置，会话ID不会在认证过程中更改。（不建议使用此策略）

当会话固定保护发生时，它会导致 SessionFixationProtectionEvent 事件在应用程序上下文中发布。如果使用的是 changeSessionId 的会话固定防御策略，此保护还会使所有的 jakarta.servlet.http.HttpSessionIdListener 被通知。

会话固定防御的配置策略的相关源码见：

  • 相关策略：

    • SessionFixationProtectionStrategy：其中的 migrateSessionAttributes 属性用于判断是否使用迁移会话还是新建 Session

    • ChangeSessionIdAuthenticationStrategy

    • NullAuthenticatedSessionStrategy

  • 应用的过滤器：SessionManagementFilter#sessionAuthenticationStrategy

# 3.代码示例

配置 SecurityFilterChain 对象：

@Configuration
public class SecurityConfig {
  
    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http,) throws Exception {
    		http.authorizeHttpRequests(auth -> auth
                    .requestMatchers("/hello").permitAll()
                    .requestMatchers("/user/**").hasRole("USER")
                    .anyRequest().authenticated()
    				)
            .formLogin(formLogin -> formLogin.permitAll())
            .csrf(AbstractHttpConfigurer::disable)
            .sessionManagement(sessionManagement -> sessionManagement
                 	// 防护会话固定攻击的策略
            			.sessionFixation(SessionFixationConfigurer::changeSessionId)
            );
 
    		return http.build();
    }
}

# 三、Session 过期

# 1.Session 过期

# 1.1 Session 过期介绍

当用户登录网站后，长时间未操作，当用户再访问网站时，此时服务器对 sessionid 校检，并未找到对应的 Session，此时浏览器提醒用户登录过期，这种情况就是 Session 过期了。

当然，用户手动退出登录，这也属于 Session 过期的一种特殊情况，相当于用户手动使 Session 过期。

# 1.2 配置 Session 过期时间

我们可以手动的配置 Session 的过期时间，在 Spring Boot 程序中，可以通过以下配置设置 Session 过期时间：

# servlet的session过期时间,默认是30m,最小设置为60s
server.servlet.session.timeout=60s
1
2

需要注意的是，Session 的过期时间最少设置为 1 分钟，即使设置小于 60 秒，在读取配置时，也会被修改为 1 分钟。具体见如下方法：

  • ServletWebServerFactoryCustomizer#customize 用于设置 ConfigurableServletWebServerFactory 的相关属性
  • TomcatServletWebServerFactory#getSessionTimeoutInMinutes 用于读取 AbstractServletWebServerFactory 的 session.timeout 配置。

# 1.3 Session 过期的处理策略

当 Session 过期时，Spring Security 提供了两种处理的策略：

  1. 识别到 Session 过期时，将重定向到指定的 URL。
  2. 自定义实现类，实现 InvalidSessionStrategy 接口。

下面，我们就一下看一下这两种策略的使用吧。

# 3.自定义 Session 失效的处理策略

下面，我们来自定义实现 InvalidSessionStrategy ，来处理 Session 过期。InvalidSessionStrategy 默认在 Spring Security 中有两个实现：

  • RequestedUrlRedirectInvalidSessionStrategy：当 SessionManagementFilter 检测到无效的请求会话时，执行重定向到原始请求 URL
  • SimpleRedirectInvalidSessionStrategy：当 SessionManagementFilter 检测到无效的请求会话时，执行重定向到固定的 URL

这两个实现都是用来重定向的。

下面，咱们来自定义一个返回 JSON 格式数据的 Session 过期策略。

首先，新建 ReturnJsonInvalidSessionStrategy 类，使其实现 InvalidSessionStrategy 接口，并重写 onInvalidSessionDetected 方法。代码如下：

public final class ReturnJsonInvalidSessionStrategy implements InvalidSessionStrategy {

    @Override
    public void onInvalidSessionDetected(HttpServletRequest request, HttpServletResponse response)
            throws IOException, ServletException {
        Response<Object> result = ResponseFactory.FACTORY.build(HttpStatus.UNAUTHORIZED.value())
                .setMsg("登录已过期，请重新登录");
        response.setContentType(MediaType.APPLICATION_JSON_UTF8_VALUE);
        response.getWriter().write(JsonUtils.toJsonStr(result));
    }

}

# 四、Session 并发控制

# 1.什么是 Session 并发控制？

Session 并发控制是一种用于管理用户会话（session）的技术，旨在限制同一用户在同一时间内只能有一个活动会话。它可以防止用户在多个浏览器标签或设备上同时执行敏感操作，从而提高安全性和数据的一致性。

即在同一个系统中，出于安全的目的，我们会要求只允许一个用户在一个终端上登录。

通过实施 Session 并发控制，可以确保用户的会话在时间和空间上是唯一且互斥的，从而防止潜在的安全风险和数据冲突。这在许多 Web 应用程序中都是重要的安全措施之一。

# 2.Session 并发控制的配置

# 2.1 并发控制方案

如果只允许同一个用户在一个终端上登录，我们对并发控制，可以有以下方案：

  1. 允许后面登录的踢掉前面登录的用户，如 QQ、微信等。
  2. 如果用户已经登录，则会禁止该账户在其他终端登录，如游戏账户就可以这样处理。
  3. 当然，还可以让后面的用户选择，是否踢掉已经登录的用户。

不过，有的系统是允许移动端、PC、网页同时登录的，对于这样的情况，可以在登录时根据登录平台做限制。

# 2.2 踢掉原有登录用户

如下配置，将设置最大登录人数为1，并且后面登录的会顶掉前面登录的 Session。

@Configuration
public class SecurityConfig {

    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http,) throws Exception {
    		http.authorizeHttpRequests(auth -> auth
                    .requestMatchers("/hello").permitAll()
                    .requestMatchers("/user/**").hasRole("USER")
                    .anyRequest().authenticated()
    				)
            .formLogin(formLogin -> formLogin.permitAll())
            .csrf(AbstractHttpConfigurer::disable)
            .sessionManagement(sessionManagement -> sessionManagement
                    // 防护会话固定攻击的策略
                    .sessionFixation(SessionFixationConfigurer::migrateSession)
                    .sessionConcurrency(sessionConcurrency -> sessionConcurrency
                            // 最大并发会话数,设置单个用户允许同时在线的最大会话数,如果没有额外配置,重新登录的会话会踢掉旧的会话.
                            // -1 则不限制
                            .maximumSessions(1)
                            // 当达到最大会话数时,是否保留已经登录的用户
                            .maxSessionsPreventsLogin(false)
                    )
            );

    		return http.build();
    }

}

# 3.Session 并发控制源码

在 Spring Security 中，会话的并发控制在 UsernamePasswordAuthenticationFilter 过滤器的父类 AbstractAuthenticationProcessingFilter 中进行处理的：

public abstract class AbstractAuthenticationProcessingFilter extends GenericFilterBean
		implements ApplicationEventPublisherAware, MessageSourceAware {

  // ...

	private SessionAuthenticationStrategy sessionStrategy = new NullAuthenticatedSessionStrategy();

  @Override
	public void doFilter(ServletRequest request, ServletResponse response, FilterChain chain)
			throws IOException, ServletException {
		doFilter((HttpServletRequest) request, (HttpServletResponse) response, chain);
	}

	private void doFilter(HttpServletRequest request, HttpServletResponse response, FilterChain chain)
			throws IOException, ServletException {
		if (!requiresAuthentication(request, response)) {
			chain.doFilter(request, response);
			return;
		}
		try {
			Authentication authenticationResult = attemptAuthentication(request, response);
			if (authenticationResult == null) {
				// return immediately as subclass has indicated that it hasn't completed
				return;
			}
      // 调用Session的代理处理器进行session的处理
			this.sessionStrategy.onAuthentication(authenticationResult, request, response);
			// Authentication success
			if (this.continueChainBeforeSuccessfulAuthentication) {
				chain.doFilter(request, response);
			}
			successfulAuthentication(request, response, chain, authenticationResult);
		}
		catch (InternalAuthenticationServiceException failed) {
			this.logger.error("An internal error occurred while trying to authenticate the user.", failed);
			unsuccessfulAuthentication(request, response, failed);
		}
		catch (AuthenticationException ex) {
			// Authentication failed
			unsuccessfulAuthentication(request, response, ex);
		}
	}

  // ...

}

而 AbstractAuthenticationProcessingFilter 过滤器中使用的 SessionAuthenticationStrategy 实现是 CompositeSessionAuthenticationStrategy 。

CompositeSessionAuthenticationStrategy 是一个 SessionAuthenticationStrategy 的代理实现类，其类中的 delegateStrategies 属性就是所有的代理对象，其 onAuthentication 方法会直接循环调用所有代理对象的 onAuthentication 方法。

CompositeSessionAuthenticationStrategy 的源码如下：

public class CompositeSessionAuthenticationStrategy implements SessionAuthenticationStrategy {

	private final Log logger = LogFactory.getLog(getClass());

	private final List<SessionAuthenticationStrategy> delegateStrategies;

	public CompositeSessionAuthenticationStrategy(List<SessionAuthenticationStrategy> delegateStrategies) {
		Assert.notEmpty(delegateStrategies, "delegateStrategies cannot be null or empty");
		for (SessionAuthenticationStrategy strategy : delegateStrategies) {
			Assert.notNull(strategy, () -> "delegateStrategies cannot contain null entires. Got " + delegateStrategies);
		}
		this.delegateStrategies = delegateStrategies;
	}

	@Override
	public void onAuthentication(Authentication authentication, HttpServletRequest request,
			HttpServletResponse response) throws SessionAuthenticationException {
    // 调用delegateStrategies中所有的SessionAuthenticationStrategy对象
		int currentPosition = 0;
		int size = this.delegateStrategies.size();
		for (SessionAuthenticationStrategy delegate : this.delegateStrategies) {
			if (this.logger.isTraceEnabled()) {
				this.logger.trace(LogMessage.format("Preparing session with %s (%d/%d)",
						delegate.getClass().getSimpleName(), ++currentPosition, size));
			}
      // 调用代理类的onAuthentication方法
			delegate.onAuthentication(authentication, request, response);
		}
	}

	@Override
	public String toString() {
		return getClass().getName() + " [delegateStrategies = " + this.delegateStrategies + "]";
	}

}

ConcurrentSessionControlAuthenticationStrategy 的源码如下：

public class ConcurrentSessionControlAuthenticationStrategy
		implements MessageSourceAware, SessionAuthenticationStrategy {

	protected MessageSourceAccessor messages = SpringSecurityMessageSource.getAccessor();

	private final SessionRegistry sessionRegistry;

	private boolean exceptionIfMaximumExceeded = false;

	private int maximumSessions = 1;

  // ...

	@Override
	public void onAuthentication(Authentication authentication, HttpServletRequest request,
			HttpServletResponse response) {
    // 获取允许session并发的最大数
		int allowedSessions = getMaximumSessionsForThisUser(authentication);
    // 若为-1,代表不限制并发数,直接放行
		if (allowedSessions == -1) {
			// We permit unlimited logins
			return;
		}
    // 获取当前用户对应的所有的Session
		List<SessionInformation> sessions = this.sessionRegistry.getAllSessions(authentication.getPrincipal(), false);
		int sessionCount = sessions.size();
    // 若未超过session最大并发数,直接放行
		if (sessionCount < allowedSessions) {
			// They haven't got too many login sessions running at present
			return;
		}
    // 若超过并发数,进行处理
		if (sessionCount == allowedSessions) {
			HttpSession session = request.getSession(false);
			if (session != null) {
				// 当前session是已经存在的session，直接放行
				for (SessionInformation si : sessions) {
					if (si.getSessionId().equals(session.getId())) {
						return;
					}
				}
			}
			// If the session is null, a new one will be created by the parent class,
			// exceeding the allowed number
		}
    
    // 超过并发数的处理
		allowableSessionsExceeded(sessions, allowedSessions, this.sessionRegistry);
	}
 
  // ...

  protected void allowableSessionsExceeded(List<SessionInformation> sessions, int allowableSessions,
			SessionRegistry registry) throws SessionAuthenticationException {
    // 若设置超过并发数时抛出异常（当达到最大会话数时,保留已经登录的用户）或session为空,直接抛出异常
		if (this.exceptionIfMaximumExceeded || (sessions == null)) {
			throw new SessionAuthenticationException(
					this.messages.getMessage("ConcurrentSessionControlAuthenticationStrategy.exceededAllowed",
							new Object[] { allowableSessions }, "Maximum sessions of {0} for this principal exceeded"));
		}
    // 若达到最大会话数时,顶出已经登录的用户
		// 按照最后请求时间排序,找出最长时间未请求的session，并将这些session标记为无效
		sessions.sort(Comparator.comparing(SessionInformation::getLastRequest));
		int maximumSessionsExceededBy = sessions.size() - allowableSessions + 1;
		List<SessionInformation> sessionsToBeExpired = sessions.subList(0, maximumSessionsExceededBy);
		for (SessionInformation session : sessionsToBeExpired) {
			session.expireNow();
		}
	}

}

# 五、分布式集群的 Session 处理

# 1.集群 Session 介绍

我们上面说到，服务端是通过 HTTP 请求中携带的 sessionid 来找到内存中存储的 Session，以此识别用户。

Session 存储在内存有如下缺点：

  • 当访问用户过多时，内存占用就会比较大，进而会增加服务器的压力

  • 一旦服务器重启，内存中的 Session 就会丢失。

以上缺点并不是致命性的问题，如果用户量少，单机部署就可以，但是用户量大，就要进行集群部署了，在集群环境下的 Session 就叫做集群 Session（集群会话）。

由于 Session 是存储在内存中的，如果是单机部署的服务，就不存在 Session 共享问题。但是在分布式或者集群项目中，不同机器之间的内存的数据没办法共享，就会造成识别用户异常，从而出现用户突然 Session 失效的问题。

比如，现在有三台服务器（服务01、服务02、服务03），每台服务器运行一个 Tomcat，用户请求通过 Nginx 代理到 Tomcat 集群中。

  • 如果用户的登录认证请求发送到服务01的 Tomcat，并且登录成功，此时服务01中的 Tomcat 的内存中会存储着用户的 Session。
  • 但是，后面的请求如何被发送到服务02的 Tomcat，此时 Tomcat 未在内存中找到对应的 Session，就会告知用户 Session 失效。
  • 这样，用户刚登陆，就失效了，就相当于一个 Bug 了。

# 2.分布式集群 Session 的处理方案

主流的解决方案有如下：

  1. Session 共享
  2. Session 复制（拷贝）
  3. 使用 Token 代替 Session
  4. 粘滞会话

# 2.1 Session 共享

Session 共享就是将 Session 从服务器中抽离出来，集中存储到独立的数据容器中。比较主流的使用方案就是将各个服务之间需要共享的 Session 数据，保存到一个公共的地方，大多数都使用 Redis 作为公共存储的容器（Redis 高并发下性能较好、可持久化、且支持过期策略等优点）。

其实，Session 共享的原理就是将 Session 从内存存储，迁移到了公共数据容器存储，以便于所有的服务都可以共享到整个分布式集群中的 Session。

Spring 官方提供了 Spring Session，用于 Spring Security 框架中对 Session 共享处理，使用 Spring Session 可以非常方便快捷的达到 Session 共享的目的。

不过，独立的数据库存储增加了网络交互，数据容器的读写性能、稳定性及网络 I/O 速度都成为性能的瓶颈。如果使用 Redis 等高性能的分布式存储，且服务器均在内网部署，IO 影响也是可以接受的。

# 2.2 Session 同步

Session 同步，也可以称为 Session 复制、Session 拷贝。就是在集群服务器之间同步复制 Session 数据，使得各个服务器实例之间会话状态一致性。即直接将所有的 Servlet 容器的 Session 数据互相拷贝。

可以使用 Terracotta 实现 Session 复制，官方网址：https://documentation.softwareag.com/terracotta/。

这种方式的缺点显而易见：

  • 效率低，只要集群中的某台机器的 Session 发生变化，就要拷贝到其他节点中。
  • 占用资源
    • 一旦服务器集群的数量很大，Session 的同步会占用网络带宽，效率也很低，也会存在延迟同步的问题。
    • 并且，用户并发量很大时，每台服务器的内存都要存储所有在线用户的 Session，内存占用过大。

所以，一般不建议使用 Session 同步的方案。

# 2.3 使用 Token 代替 Session

Token 是指访问资源的令牌凭据，用于检验请求的合法性，比较适合前后端分离的项目。

常用的 Token 是 JSON Web Token（JWT）认证授权机制，这是目前最流行的跨域认证解决方案。

使用 Token 代替 Session 方案的优点如下：

  • 可以使用数据库或内存存储 Token，如 Redis 。不过，如果使用 JWT 的方案，就无需存储 Token。
  • 可以避免 CSRF 攻击（可以不存储在 Cookie 中）。
  • 对于移动端来说更加友好。

# 2.4 粘滞会话

粘滞会话，也称 Session 保持。通常是采用 IP 哈希的负载均衡策略，将来自相同客户端的请求转发到相同的服务器上进行处理。即把某个用户在同一个 IP 下的请求总是打到同一台服务器中。

这种方式的优点就是简单易用，只需要在网关中配置 IP 哈希的负载均衡策略即可。

如在 Nginx 中，只需要在 upstream 中配置 ip_hash 即可：

upstream tomcat-cluster {
		ip_hash;
 		server 192.168.1.234:8080;
    server 192.168.1.235:8080;
    server 192.168.1.236:8080;
    keepalive 30;
}

server {
    listen      80;
    server_name  example.com;

    location / {
        proxy_pass  http://tomcat-cluster;
        proxy_http_version  1.1;
        proxy_set_header  Connection "";
    }
}

# 3.使用 Spring Session 实现集群 Session

Spring Session 可以使用多种数据库实现，如 Redis、JDBC、MongoDB 等，当然可以自己自定义实现。

Spring Session 使用 Spring 中的代理过滤器，将拦截所有的 Session 操作，自动的将数据同步到 Redis 中，或者自动的从 Redis 中读取数据。

Spring Session 也提供了很多 Spring Security 的实现，如 Remember-me 、Session 并发控制中 SessionRegistry 的实现。

不过 Spring Session 有多种实现方式（Redis、JDBC 等），这会导致获取所有已登录用户的方式可能会有所不同，因此，Spring Session 中并没有实现 SessionRegistry 接口中的 getAllPrincipals() 方法，

Spring Session 的 Redis 实现提供了 SessionRepository 的以下实现：

  • RedisSessionRepository：RedisSessionRepository 是一个基本的实现，它在 Redis 中存储会话数据而不需要任何额外的索引。
    • 它使用一个简单的键值结构来存储会话属性。每个会话被分配一个唯一的 sessionid，并且会话数据被存储在一个与该 sessionid 相关联的 Redis key 下。
    • 当需要检索会话时，存储库使用 sessionid 查询 Redis，以获取相关的会话数据。
    • 由于没有索引，因此根据属性或凭证（而不是 sessionid）查询会话可能效率很低。
  • RedisIndexedSessionRepository：RedisIndexedSessionRepository 是一个扩展实现，它为 Redis 中存储的会话提供了索引功能。
    • 它在 Redis 中引入了额外的数据结构，以基于属性或标准高效地查询会话。
    • 除了 RedisSessionRepository 使用的键值结构之外，它还维护了额外的索引以支持快速查找。例如，它可以基于用户ID或最后访问时间等会话属性创建索引。这些索引允许基于特定标准高效地查询会话，增强性能并启用高级会话管理特性。
    • 除此之外，RedisIndexedSessionRepository 还支持会话过期和删除。

这里，我们使用 @EnableRedisIndexedHttpSession 注解，就会自动的初始化 RedisIndexedSessionRepository 对象。

如果你不想使用 RedisIndexedSessionRepository 的实现，而想要使用 RedisSessionRepository，就可以使用 @EnableRedisHttpSession 注解来配置。

具体可见官方文档： Choosing Between RedisSessionRepository and RedisIndexedSessionRepository (opens new window)

# 3.4 Session 的配置

配置 SpringSessionBackedSessionRegistry 到 ConcurrencyControlConfigurer 中：

@Configuration
public class SecurityConfig {

  	@Autowired
    private SpringSessionBackedSessionRegistry<? extends Session> sessionRegistry;

    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
    		http.authorizeHttpRequests(auth -> auth
                    .requestMatchers("/hello").permitAll()
                    .requestMatchers("/user/**").hasRole("USER")
                    .anyRequest().authenticated()
    				)
            .formLogin(formLogin -> formLogin.permitAll())
            .csrf(AbstractHttpConfigurer::disable)
            .sessionManagement(sessionManagement -> sessionManagement
                    // 防护会话固定攻击的策略
                    .sessionFixation(SessionFixationConfigurer::migrateSession)
                    .sessionConcurrency(sessionConcurrency -> sessionConcurrency
                            // 最大并发会话数,设置单个用户允许同时在线的最大会话数,如果没有额外配置,重新登录的会话会踢掉旧的会话.
                            .maximumSessions(1)
                            // 当达到最大会话数时,是否保留已经登录的用户
                            .maxSessionsPreventsLogin(true)
                            .sessionRegistry(sessionRegistry)
                    )
            );
 
    		return http.build();
    }
}

这里报错是 SessionAuthenticationException 对象的反序列化异常，这很奇怪，为什么会对这个异常对象反序列化呢，看看 redis 中存储数据：发现确实在 spring:session:sessions:xxxxxx 的 sessionAttr:SPRING_SECURITY_LAST_EXCEPTION 字段中存储的就是 SessionAuthenticationException 的异常信息。

经过 debug，发现在 AbstractAuthenticationProcessingFilter#unsuccessfulAuthentication 方法中进行异常处理时，调用的是 SimpleUrlAuthenticationFailureHandler 的 onAuthenticationFailure 方法，在 onAuthenticationFailure 方法中有个 saveException 的方法，该方法将异常存储到了 Session 中，并且我们在序列化的配置中，并没有配置允许 SessionAuthenticationException 反序列化。

并且，在报错日志中也清楚的提示了我们。

相关 Issues：

  • https://github.com/spring-projects/spring-security/issues/2086

  • https://github.com/spring-projects/spring-security/issues/4370

将 RedisSessionSerializerConfig 中 objectMapper 方法的序列化配置修改为如下即可：

private ObjectMapper objectMapper() {
    ObjectMapper mapper = new ObjectMapper();
    // 注册SecurityJackson2Modules，用于处理Spring Security中的类型
    mapper.registerModules(SecurityJackson2Modules.getModules(this.loader));
    mapper.activateDefaultTyping(
            LaissezFaireSubTypeValidator.instance,
            ObjectMapper.DefaultTyping.NON_FINAL,
            JsonTypeInfo.As.PROPERTY
    );
    return mapper;
}

@JsonTypeInfo(use = JsonTypeInfo.Id.CLASS, include = JsonTypeInfo.As.PROPERTY)
@JsonIgnoreProperties(ignoreUnknown = true, value = { "cause", "stackTrace" })
class SessionAuthenticationExceptionMixin {

    /**
     * Constructor used by Jackson to create
     * {@link org.springframework.security.authentication.BadCredentialsException} object.
     * @param message the detail message
     */
    @JsonCreator
    SessionAuthenticationExceptionMixin(@JsonProperty("message") String message) {
    }

}


@Configuration(proxyBeanMethods = false)
public class RedisSessionSerializerConfig implements BeanClassLoaderAware {

    private ClassLoader loader;

    @Override
    public void setBeanClassLoader(ClassLoader classLoader) {
        this.loader = classLoader;
    }

    @Bean
    public RedisSerializer<Object> springSessionDefaultRedisSerializer() {
        return new GenericJackson2JsonRedisSerializer(objectMapper());
    }
  
    private ObjectMapper objectMapper() {
        ObjectMapper mapper = new ObjectMapper();
        // 注册SecurityJackson2Modules，用于处理Spring Security中的类型
        mapper.registerModules(SecurityJackson2Modules.getModules(this.loader));

        SimpleModule simpleModule = new SimpleModule() {
            @Override
            public void setupModule(SetupContext context) {
                context.setMixInAnnotations(SessionAuthenticationException.class, SessionAuthenticationExceptionMixin.class);
            }
        };
        mapper.registerModule(simpleModule);

        return mapper;
    }
  
}
