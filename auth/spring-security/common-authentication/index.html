03.Spring Security的常用认证

# 一、Spring Security 的认证

# 1.认证的概念

前面我们简单的解释了认证的宽泛概念，而在 Web 项目或后端服务中，认证就是用来判断系统中是否存在某用户，并判断该用户的身份是否合法的过程。

认证对应我们的业务实现，就是用户登录。认证的存在，主要是为了保护系统中的隐私数据与资源，只有合法的用户才可以访问系统中的资源

# 2.常见的认证方式

在 Spring Security 中，常见的认证方式可以按照 HTTP 层面和表单层面划分。

如下是常见的认证方式：

  1. HTTP 基本认证
  2. Form 表单认证
  3. HTTP 摘要认证

# 二、HTTP 基本认证

# 1.HTTP 基本认证步骤

在 Spring Security 4.x 版本中，默认的登录方式就是 HTTP 基本认证。

HTTP 基本认证是在 RFC2616 标准中定义的一种认证模式，它以一种很简单的方式与用户进行交互。HTTP 基本认证可以分为如下 4 个步骤：

  1. 首先，客户端发起一个未携带认证信息的请求；
  2. 然后，服务器端返回一个 401 Unauthorized 的响应信息，并在 key 为 WWW-Authentication 头部中说明认证形式：当进行 HTTP 基本认证时，WWW-Authentication 的内容为 Basic realm="被保护的页面"；
  3. 其次，浏览器（客户端）会收到这个 401 Unauthorized 响应信息，页面会弹出一个要求输入用户名密码的弹窗。当用户输入后，浏览器会将用户名和密码使用冒号进行拼接并用 Base64 编码，然后将其放入 key 为 Authorization 的请求头并发送给服务器（后续的每次请求都会携带 Authorization 请求头）；
  4. 最后，服务器端对客户端发来的信息进行解码得到用户名和密码，并对该信息进行校验判断是否正确。
    • 若校检正确，则给客户端返回响应内容。
    • 若校检不正确，则执行第二步。

# 2.HTTP 基本认证的缺点

HTTP 基本认证的方式并不安全，并不适合在 Web 项目中使用，但它是一些现代主流认证的基础，而且在 Spring Security OAuth 中，内部认证的默认方式就是用的 HTTP 基本认证。

HTTP 基本认证的缺点如下：

  • 无法实现 Remember-Me 功能：HTTP 基本认证是一种无状态的认证方式，与表单认证相比，HTTP 基本认证是一种基于 HTTP 层面的认证方式，无法携带 Session 信息，也就无法实现 Remember-Me 功能。

  • 用户名和密码在传递时仅做了一次简单的 Base64 编码，几乎等同于以明文传输，极易被进行密码窃听和重放攻击。所以在实际开发中，很少会使用这种认证方式来进行安全校验。

# 3.代码示例

我们配置修改 SecurityConfig 的配置如下：

@Configuration
public class SecurityConfig {

    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        // 配置基本认证方式
        http.authorizeRequests(auth -> auth.anyRequest().authenticated()) // 对任意请求都进行认证
                // 开启http basic认证
                .httpBasic();
        return http.build();
    }

}

# 4.退出登录

在成功认证之后，浏览器会把 Authorization 认证信息缓存一段时间，之后每次请求接口时都会自动带上。直到用户关闭浏览器才会销毁认证信息，也就是说我们无法在服务端进行有效的注销。

请求注销时，前端也可以手动在请求头配置一个错误的 Authorization。或者在浏览器的命令行执行 document.execCommand("ClearAuthenticationCache") 方法来清空认证信息，但该方式对部分浏览器无效（如谷歌）。

我们在学习或者调试的时候可以开启浏览器的无痕模式，以此避免缓存造成的问题。

# 三、HTTP 摘要认证

# 1.HTTP 摘要认证介绍

HTTP 摘要认证也是在 RFC2616 中定义的一种认证方式，它的出现是为了弥补 HTTP 基本认证存在的安全隐患，但该认证方式也并不是很安全。

HTTP 摘要认证使用随机数加 上 MD5 算法来对用户名、密码进行摘要编码，流程类似 HTTP 基本认证 ，但是更加复杂一些:

HTTP 摘要认证中，当客户端发起一个未经认证的请求时，服务器会返回一个 401 回应，并给客户端返回与验证相关的参数，期待客户端依据这些参数继续做出回应，从而完成整个验证过程。

# 2.HTTP 摘要认证步骤与参数

如下图为 HTTP 摘要认证的流程：

服务端给客户端返回验证相关的参数如下：

  • username：用户名。
  • password：用户密码。
  • realm：认证域，由服务器返回。
  • opaque：透传字符串，客户端应原样返回。
  • method：请求的方法。
  • nonce：由服务器生成的随机字符串，包含过期时间(默认过期时间 300s)和密钥。
  • nc：即 nonce-count,指请求的次数，用于计数，防止重放攻击。qop 被指定时，nc 也必须被指定。
  • cnonce：客户端发给服务器的随机字符串，qop 被指定时，cnonce 也必须被指定。
  • qop：保护级别，客户端根据此参数指定摘要算法。若取值为 auth,则只进行身份验证；若取值为 auth-int，则还需要校验内容完整性，默认的 qop 为 auth。
  • uri：请求的 uri。
  • response：客户端根据算法算出的摘要值，这个算法取决于 qop。
  • algorithm：摘要算法，目前仅支持 MD5。
  • entity-body：页面实体，非消息实体，仅在 auth-int 中支持。

通常服务器端返回的数据包括 realm、opaque、nonce、qop 等字段。如果客户端需要做出验证回应，就必须按照一定的算法得到一些新的数据并一起返回。在上面的参数中，对服务器而言，最重要的字段是 nonce；对客户端而言，最重要的字段是 response。

# 3.HTTP 摘要认证的缺点

HTTP 摘要认证与 HTTP 基本认证一样，都是基于 HTTP 层面的认证方式，都不能使用 Session 对象，因而也都不支持 Remember-Me 功能。

HTTP 摘要认证虽然解决了 HTTP 基本认证以明文传输密码的问题，但并未解决密码明文存储的问题，所以依然有安全隐患，所以在实际开发中，摘要认证的方式也不怎么使用。

# 4.2 配置 DigestAuthenticationEntryPoint

@Configuration
public class DigestAuthenticationConfig {

    // 配置认证入口端点，主要是设置认证参数信息
    @Bean
    public DigestAuthenticationEntryPoint digestAuthenticationEntryPoint(){
        DigestAuthenticationEntryPoint point = new DigestAuthenticationEntryPoint();
        point.setKey("Security");
        point.setRealmName("willingoxjin");
        point.setNonceValiditySeconds(500);
        return point;

}

DigestAuthenticationEntryPoint 是 Spring Security 中用于处理 HTTP 摘要认证失败的入口点。当用户尝试访问需要进行摘要认证的资源时，如果认证失败，DigestAuthenticationEntryPoint 就会被调用，它会返回一个 401 Unauthorized 的响应，并在响应头中添加一个 WWW-Authenticate 字段，告诉客户端需要进行摘要认证。

DigestAuthenticationEntryPoint 的主要作用是将 HTTP 摘要认证的具体实现细节隐藏在框架内部，使得开发者无需关心具体的认证实现细节，只需要关注认证失败时如何处理即可。在实际使用中，DigestAuthenticationEntryPoint 通常与其他 Spring Security 组件一起使用，如 DigestAuthenticationFilter、AuthenticationProvider 等。

需要注意的是，DigestAuthenticationEntryPoint 只是一个入口点，它并不参与具体的认证过程。具体的认证过程由 DigestAuthenticationFilter 和 AuthenticationProvider 等组件完成。

# 4.4 配置 SecurityConfig

首先，我们对接口添加配置，对 /admin/** 的接口设置只有 ADMIN 角色的用户才能访问，对 /user/** 的接口设置只有 USER 权限的用户才能访问。

其次，设置 DigestAuthenticationEntryPoint 和我们自定义的 HTTP 摘要过滤器（DigestAuthenticationFilter）。

@Configuration
public class SecurityConfig {

    @Autowired
    private DigestAuthenticationEntryPoint digestAuthenticationEntryPoint;

    @Autowired
    private UserDetailsService userDetailsService;

    private DigestAuthenticationFilter digestAuthenticationFilter(){
        DigestAuthenticationFilter filter = new DigestAuthenticationFilter();
        filter.setAuthenticationEntryPoint(digestAuthenticationEntryPoint);
        filter.setUserDetailsService(userDetailsService);
        return filter;
    }

    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        http.authorizeHttpRequests(auth -> 
                		auth.requestMatchers("/admin/**").hasRole("ADMIN")
                        .requestMatchers("/user/**").hasRole("USER")
                        .requestMatchers("/visitor/**").permitAll()
                        .anyRequest().authenticated()               
            )
            .csrf(AbstractHttpConfigurer::disable)
            // 当未认证时访问某些资源,则由该认证入口类来处理.
            .exceptionHandling(exceptionHandling ->
                    exceptionHandling.authenticationEntryPoint(digestAuthenticationEntryPoint)
            )
            // 添加自定义过滤器到过滤器链中
            .addFilter(digestAuthenticationFilter());
		}
}

  • .requestMatchers() 用于匹配指定需要进行安全性配置的 URL 路径；
  • "/admin/**" 表示匹配 "/admin/" 路径下的所有 URL；
  • .hasRole("ADMIN") 表示只有具有 "ADMIN" 角色的用户才可以访问这些 URL。
  • .permitAll() 表示允许未经身份验证的用户访问这些 URL。
  • .hasAnyRole("USER", "ADMIN") 表示只有具有 "USER" 或 "ADMIN" 角色的用户才可以访问这些 URL。

DigestAuthenticationFilter 是 Spring Security 中用于支持 HTTP 摘要认证的过滤器。HTTP 摘要认证是一种安全认证机制，它可以在不明文传输密码的情况下进行认证。DigestAuthenticationFilter 的作用是从 HTTP 请求中获取摘要认证信息，并将其交给 AuthenticationManager 进行认证。如果认证成功，它会向 SecurityContextHolder 中存储一个认证成功的 Authentication 对象，以便后续的授权操作使用。

如果我们没有手动配置 UserDetailsService ，则默认使用的是 InMemoryUserDetailsManager。并且还要配置 PasswordEncoder ，否则在验证密码时，客户端和服务端最后生成的摘要不一致，会造成认证失败。

# 1.表单认证介绍

在 Spring Boot Security 中，默认使用的就是表单认证的方式。我们可以在 SpringBootWebSecurityConfiguration.SecurityFilterChainConfiguration 类的 defaultSecurityFilterChain(HttpSecurity http) 方法中默认配置的就是表单认证，如下：

# 2.表单认证预置 url

  • GET /login：GET 请求 /login 时会跳转到这个页面。只要我们访问任意一个需要认证的请求时，都会跳转到这个登录界面。
  • POST /login：POST 请求 /login 时会触发这个接口，在登录页面点击登录时，默认的登录页面表单中的 action 属性配置的就是此接口。
  • /login?error：当用户名或密码错误时，会跳转到该页面。
  • /：登录成功后，默认跳转到该页面，如果存在 index.html 页面，则会重定向到 index.html 页面，当然这个页面要由我们自己实现。
  • /logout: 注销页面。
    • /login?logout: 注销成功后跳转到的页面。

# 3.代码示例

下面的代码配置使用表单认证的方式：

@Configuration
public class SecurityConfig {

    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        // 配置表单认证方式
        http.authorizeRequests(auth -> auth.anyRequest().authenticated())
                .csrf(AbstractHttpConfigurer::disable)
                .formLogin();
        return http.build();
    }

}

# 4.自定义表单认证页面

# 4.1 创建 SecurityConfig 配置类

@Configuration
public class SecurityConfig {

    /**
     * 该方法用来定义哪些请求需要忽略安全控制，哪些请求必须接受安全控制;
     * 可以在合适的时候清除SecurityContext以避免内存泄漏;
     * 可以用来定义请求防火墙和请求拒绝处理器;
     * 另外, 开启Spring Security Debug模式也是这里配置的
     */
    @Bean
    public WebSecurityCustomizer webSecurityCustomizer() {
        // js、css、images等静态资源无需认证
        return (web) -> web.ignoring().antMatchers("/js/**", "/css/**", "/images/**");
    }

    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        http.authorizeHttpRequests(auth -> auth.anyRequest().authenticated())
                // 设置自定义的登录页面地址, 并且该页面无需认证
                .formLogin(formLogin -> formLogin.loginPage("/myLogin.html").permitAll())
                // 需禁用CRSF防护功能, 否则登录不成功
                .csrf(AbstractHttpConfigurer::disable);
        return http.build();
    }

}

# 5.自定义表单的配置细化

如果我们想修改表单认证页面中请求的地址、请求参数的名称、定义认证失败时的错误处理页面、处理退出登录时的操作等配置。都是可以在 Spring Security 中进行自定义配置的。

# 5.1 SecurityConfig 配置

SecurityConfig 的配置如下：

@Configuration
public class SecurityConfig {

    @Bean
    public WebSecurityCustomizer webSecurityCustomizer() {
        return web -> web.ignoring().requestMatchers("/js/**", "/css/**", "/images/**", "/**/favicon.ico");
    }

    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        http.authorizeHttpRequests(auth -> auth.anyRequest().authenticated())
                // 设置自定义的登录页面地址, 并且该页面无需认证
                .formLogin(formLogin ->
                        // 设置自定义的登录页面地址
                        formLogin.loginPage("/myLogin.html").permitAll()
                        // 登录表单提交的接口地址, post请求, 登录验证由系统实现
                        .loginProcessingUrl("/login")
                        // 登录成功后，是否始终跳转到登录成功url。它默认为false
                        .defaultSuccessUrl("/index.html", false)
                        // 用户认证失败跳转接口
                        .failureUrl("/myLogin.html?error=ERROR%20Incorrect%20username%20or%20password")
                        // 登录时的用户参数名，默认username
                        .usernameParameter("username")
                        // 登录时的密码参数名，默认password
                        .passwordParameter("password")
                )
                .logout(logout ->
                        // 注销接口
                        logout.logoutUrl("/logout")
                        // 注销成功后跳转到的接口, 且无需认证
                        .logoutSuccessUrl("/myLogin.html").permitAll()
                        // 使session失效
                        .invalidateHttpSession(true)
                        // 清除认证信息
                        .clearAuthentication(true)
                        // 注销时,删除自定义的cookie
                        .deleteCookies("JSESSIONID"))
                // 需禁用CRSF防护功能, 否则登录不成功
                .csrf(AbstractHttpConfigurer::disable);
        return http.build();
    }

}

# 6.自定义页面源码分析

我们发现，自定义表单认证的代码很简单，那 Spring Security 内部是如何加载我们自定义的登录页面的呢？

实际上，相关的核心类有两个：WebSecurity 和 HttpSecurity，我们就一起来看看这个配置的加载过程。

# 6.1 WebSecurity

下图是我们请求服务时，WebSecurity 的执行流程：

在 webSecurityCustomizer() 方法会返回 WebSecurityCustomizer 对象，WebSecurityCustomizer 是一个函数式接口，接口中的方法为 void customize(WebSecurity web)。这个方法的唯一参数就是：WebSecurity 类。

在 WebSecurity 类中，有一个名为 securityFilterChainBuilders 的变量，该变量是一个集合（类型为 List<SecurityBuilder<? extends SecurityFilterChain>>），这个集合用于构建多个 SecurityFilterChain 过滤器链。

过滤器是 Servlet 中的知识，如果忘记或者概念模糊了，可以看下相关的教程：Web开发-使用Filter (opens new window)

当 WebSecurity 在执行时，会构建出一个名为 ”springSecurityFilterChain“ 的 Spring BeanFilterChainProxy 代理类，它的作用如下：

  • 定义哪些请求可以忽略安全控制，哪些请求必须接受安全控制；
  • 可以在合适的时候清除 SecurityContext 以避免内存泄漏；
  • 可以用来定义请求防火墙和请求拒绝处理器；
  • 也可以在这里开启 Spring Security 的 Debug 模式。

因为有上面这一系列的 Filter 过滤器，我们就可以利用 web.ignoring() 方法来配置想要忽略的静态资源 URL 地址，这样这些静态资源就可以不被拦截，从而可以被识别访问了。
