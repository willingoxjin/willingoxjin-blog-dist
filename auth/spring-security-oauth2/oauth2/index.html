01.OAuth2 介绍

# 一、为什么需要 OAuth2

以下内容来自阮一峰的 OAuth 2.0 的一个简单解释 (opens new window)

# 1.快递员问题

我住在一个大型的居民小区。

小区有门禁系统。

进入的时候需要输入密码。

我经常网购和外卖，每天都有快递员来送货。我必须找到一个办法，让快递员通过门禁系统，进入小区。

如果我把自己的密码，告诉快递员，他就拥有了与我同样的权限，这样好像不太合适。万一我想取消他进入小区的权力，也很麻烦，我自己的密码也得跟着改了，还得通知其他的快递员。

有没有一种办法，让快递员能够自由进入小区，又不必知道小区居民的密码，而且他的唯一权限就是送货，其他需要密码的场合，他都没有权限？

# 2.授权机制的设计

于是，我设计了一套授权机制。

第一步，门禁系统的密码输入器下面，增加一个按钮，叫做"获取授权"。快递员需要首先按这个按钮，去申请授权。

第二步，他按下按钮以后，屋主（也就是我）的手机就会跳出对话框：有人正在要求授权。系统还会显示该快递员的姓名、工号和所属的快递公司。

我确认请求属实，就点击按钮，告诉门禁系统，我同意给予他进入小区的授权。

第三步，门禁系统得到我的确认以后，向快递员显示一个进入小区的令牌（access token）。令牌就是类似密码的一串数字，只在短期内（比如七天）有效。

第四步，快递员向门禁系统输入令牌，进入小区。

有人可能会问，为什么不是远程为快递员开门，而要为他单独生成一个令牌？这是因为快递员可能每天都会来送货，第二天他还可以复用这个令牌。另外，有的小区有多重门禁，快递员可以使用同一个令牌通过它们。

# 3.互联网场景

我们把上面的例子搬到互联网，就是 OAuth 的设计了。

首先，居民小区就是储存用户数据的网络服务。比如，微信储存了我的好友信息，获取这些信息，就必须经过微信的"门禁系统"。

其次，快递员（或者说快递公司）就是第三方应用，想要穿过门禁系统，进入小区。

最后，我就是用户本人，同意授权第三方应用进入小区，获取我的数据。

简单说，OAuth 就是一种授权机制。数据的所有者告诉系统，同意授权第三方应用进入系统，获取这些数据。系统从而产生一个短期的进入令牌（token），用来代替密码，供第三方应用使用。

# 4.令牌与密码

令牌（token）与密码（password）的作用是一样的，都可以进入系统，但是有三点差异。

（1）令牌是短期的，到期会自动失效，用户自己无法修改。密码一般长期有效，用户不修改，就不会发生变化。

（2）令牌可以被数据所有者撤销，会立即失效。以上例而言，屋主可以随时取消快递员的令牌。密码一般不允许被他人撤销。

（3）令牌有权限范围（scope），比如只能进小区的二号门。对于网络服务来说，只读令牌就比读写令牌更安全。密码一般是完整权限。

上面这些设计，保证了令牌既可以让第三方应用获得权限，同时又随时可控，不会危及系统安全。这就是 OAuth 2.0 的优点。

注意，只要知道了令牌，就能进入系统。系统一般不会再次确认身份，所以令牌必须保密，泄漏令牌与泄漏密码的后果是一样的。 这也是为什么令牌的有效期，一般都设置得很短的原因。

# 二、OAuth2 协议

# 1. OAuth2.0 协议

RFC6749 标准 (opens new window)中定义了 OAuth2.0 的标准。

简单来说，OAuth（Open Authorization）就是一种授权机制。数据的所有者告诉系统，同意授权第三方应用进入系统，获取这些数据。系统从而产生一个短期的进入令牌（token），用来代替密码，供第三方应用使用。

所谓 OAuth2（即 OAuth2.0，以下我们都称作 OAuth2），表示这是 OAuth 协议第 2 版，但并不兼容之前的 OAuth1.0 协议，即现在已经完全废止了 OAuth1.0 协议。并且，我们现在进行开发时，采用的都是 OAuth2.0 协议。

OAuth2 在实际开发中很常见，使用场景如下：

  • 第三方应用登录：比如网站系统的微信、QQ、微博等第三方的授权登录；
  • 分布式或微服务系统中的授权：在开发分布式或微服务架构的系统时，后端业务拆分成若干服务，服务之间或客户端（APP、网页、小程序等）进行请求调用时，为了安全认证，可以使用 OAuth2.0 进行认证授权。

# 2.OAuth2.1 协议

OAuth 2.1 (opens new window) 是 OAuth 2.0 协议的一个更新版本，它在 OAuth 2.0 的基础上进行了一些改进和修订。下面是 OAuth 2.1 相对于 OAuth 2.0 的一些主要区别：

  1. 安全性增强：OAuth 2.1 强调了更严格的安全性要求，包括对客户端验证的强制要求、对重定向 URI 的验证和规范化等。这些改进旨在提高协议的安全性，减少潜在的攻击风险。

  2. 强制 PKCE：OAuth 2.1 强制要求使用 Proof Key for Code Exchange (PKCE)，这是一种用于增强授权码授权流程安全性的机制。PKCE 通过验证授权码的交换，防止了授权码被窃取或劫持的风险。

  3. 明确禁止密码授权模式：OAuth 2.1 明确禁止了使用密码授权模式（Resource Owner Password Credentials Grant），因为该模式存在着安全风险，容易导致密码泄露。

  4. 强调最小权限原则：OAuth 2.1 强调了最小权限原则，要求客户端只请求必要的权限范围，并且资源服务器应该在授权阶段明确告知用户所请求的权限范围。

  5. 更新了错误处理：OAuth 2.1 对错误处理进行了更新和改进，增加了一些新的错误状态码和错误描述，以提供更清晰和详细的错误信息。

OAuth 2.1 与 OAuth 2.0的区别 (opens new window)

# 3.OpenID Connect 1.0 协议

OpenID Connect 1.0 是一种建立在 OAuth 2.0 协议之上的身份验证协议，它为应用程序提供了一种安全的方式来验证用户的身份，并获取有关用户的基本信息。

OpenID Connect 1.0 在 OAuth 2.0 的基础上添加了身份验证功能，使得应用程序能够安全地验证用户的身份，并获取有关用户的基本信息。它提供了一种标准化的方式来实现跨平台的身份验证，简化了应用程序和服务提供商之间的集成过程。

由于 OpenID Connect 1.0 是建立在 OAuth 2.0 协议之上的身份验证协议，因此仍需要遵循 OAuth 2.0 的授权流程。

# 三、OAuth2 核心概念

# 1.核心概念

  1. 客户端（Client）与第三方应用：如果现在有一个 A 系统集成了微信登录，此时 A 系统相对于微信来说就是客户端， A 系统也是微信的第三方应用。
  2. 资源所有者（Resource Owner）：这里的资源可以理解为我们的接口、文件存储服务中的文件等各种数据。而资源所有者，就是登录的用户。
  3. 认证服务器（Authorization Server）：认证服务器负责验证资源所有者的身份，并颁发访问令牌给客户端。
  4. 资源服务器（Resource Server）：
    • 资源服务器存储受保护资源的服务器，根据访问令牌的有效性来控制对资源的访问。
    • 它与认证服务器，可以是同一台服务器，也可以是不同的服务器。
  5. 访问令牌（Access Token）：当客户端通过认证后，会返回一个令牌，客户端访问资源服务器中的受保护资源时，就会校检该令牌是否有权限访问。
  6. 授权许可（Authorization Grant）：资源所有者授权给客户端的凭证，用于获取访问令牌。

# 2.令牌

令牌用于验证客户端的身份和权限，确保客户端有权访问受保护资源。

令牌有如下特点：

  1. 令牌是短期的，具有一定的有效期限，到期会自动失效；
  2. 令牌可以被资源所有者撤销，会立即失效；
  3. 令牌有一定的授权范围（Scope），只能在授权范围内访问资源服务器。

令牌相关的概念：

  1. 令牌端点（Token Endpoint）：用于客户端请求访问令牌的终端。
  2. 范围（Scope）：用于定义访问令牌的权限范围，限制客户端对资源的访问

令牌的出现，既可以让第三方应用获得权限，同时又随时可控，不会危及系统安全。但是只要知道了令牌，就能进入系统，系统一般不会再次确认身份，所以令牌必须保密，泄漏令牌与泄漏密码的后果是一样的。所以，令牌的有效期一般都设置得很短。

我们来梳理一下上图中的认证流程：

  • A：客户端向授权服务器发起授权请求；

  • B：授权服务器询问用户是否允许客户端访问其受保护资源，用户同意授权后，授权服务器生成授权许可返回给客户端；

  • C：客户端收到授权许可后，使用授权许可向授权服务器请求访问令牌；

  • D：授权服务器验证授权许可的有效性，并向客户端颁发访问令牌；

  • E：客户端使用访问令牌向资源服务器请求访问受保护资源；

  • F：资源服务器验证访问令牌的有效性，并根据权限范围决定是否授权客户端访问资源。

结合 B 站使用微信登录的实际场景，如下：

  • A：当我们在 B 站 PC 网页端上点击微信登录时，就会跳转到微信的域名，并出现一个二维码，要求我们使用微信 App 扫码授权，
  • B：此时，我们使用微信手机 APP 扫码后就会出现“哔哩哔哩”申请使用我们的昵称、头像等信息的页面，我们点击同意后，微信授权服务器就会生成授权许可（对微信来说，是名称为 code 的参数），并携带 code 跳转回 B 站的页面（携带 code 参数跳转或重定向）；
  • C：B 站就会使用得到的授权许可（code）向微信的认证授权服务器请求访问令牌；
  • D：微信的认证授权服务器会验证授权许可（code），验证成功，则会返回一个 token；
  • E：后续，B 站就可以使用 token 向微信的资源服务器请求接口，如用户信息接口等；
  • F：微信的资源服务器收到请求时，就睡验证 token，若 token 有效且有权限，则会允许访问资源。

# 五、OAuth2.0 授权模式

OAuth2 协议一共支持 4 种不同的授权模式（Authorization Grant），即获得令牌的方式。

如下：

  1. 授权码模式
  2. 简化模式
  3. 密码模式
  4. 客户端模式

下面，我们就一起详细看看这几种模式。

# 1.授权码模式

授权码模式（Authorization Code)，指的是第三方应用先像授权服务器申请一个授权码，然后再根据该授权码获取访问令牌的方式。

这种方式是功能最完整、流程最严密的授权模式，也是开发时最常用，安全性最高的授权模式。我们常见的第三方平台登录功能基本都是使用授权码模式。

执行步骤：

  • A：用户请求客户端，客户端将用户导向认证服务器请求授权码页面；
  • B：若用户同意授权，认证服务器将用户导向客户端指定重定向的 URI，并且会附带上授权码；
  • C：客户端收到授权码，向认证服务器申请令牌；
  • D：认证服务器收到申请令牌请求，向客户端返回访问令牌（Access Token）。

OAuth 2.1 规范中的授权码模式的详细介绍见：Authorization Code Grant (opens new window)

# 2.简化模式

简化模式（Implicit Grant Type）不需要客户端服务器参与，直接在浏览器中向授权服务器申请令牌。比如有些 Web 应用只有前端静态资源，没有后端服务器（如静态博客系统），这时就只能将令牌储存在前端浏览器中。这种情况就可以使用简化模式。

简化模式的客户端一般是指用户浏览器，访问令牌通过重定向的方式传递到用户浏览器中，再通过浏览器的 JavaScript 代码来获取访问令牌。

执行步骤：

  • B：用户同意给客户端授权；
  • C：用户授权，认证服务器将用户导向客户端指定的"重定向URI"，并在 URI 的 Hash 部分包含了访问令牌；

  • G：浏览器将令牌保存到客户端。

简化模式的所有步骤都是在浏览器中完成，没有通过第三方应用程序的服务器，访问令牌直接暴露在浏览器端，令牌对访问者是可见的，且客户端不需要认证，所以可能会导致访问令牌被黑客获取。该模式适用于对安全性要求不高的应用，仅适用于需要临时访问的场景，并且令牌的有效期必须非常短，通常是在会话期间（Session）有效，当浏览器关闭后，令牌就会失效。

在 OAuth 2.0 认证的当前最佳实践 (opens new window)和OAuth2.1 中，不允许使用简化模式授权。

# 3.密码模式

密码模式（Resource Owner Password Credentials Grant）是用户直接把用户名密码告诉客户端，客户端使用说这些信息向授权服务器申请令牌。

密码模式需要用户对客户端高度信任，例如客户端应用和服务提供商就是同一家公司，如果公司系统自己做前后端分离登录就可以采用这种模式。并且，在 OAuth 2.0 认证的当前最佳实践 (opens new window)中和 OAuth2.1 中，不允许使用密码模式授权。

# 4.客户端模式

客户端模式是指客户端使用自己的名义，而不是用户的名义向"服务提供商"申请授权。

  • 严格来说，客户端模式并不能算作 OAuth2 协议要解决的问题的一种解决方案。在这种模式中，用户直接向客户端注册，客户端以自己的名义要求"服务提供商"提供服务，这个过程并不存在授权问题。
  • 客户端授权模式通常是由客户端提前向授权服务器申请应用公钥、私钥，并通过这些关键信息向授权服务器申请访问令牌，从而得到资源服务器提供的资源，比如常见的微信公众平台 (opens new window)、支付宝支付平台授权等。
  • 需要注意的是，这种方式给出的令牌，是针对第三方应用的，而不是针对用户的。

# 5.授权码扩展流程

PKCE（Proof Key for Code Exchange）是一种用于增强 OAuth 2.0 授权码授权流程的安全性机制，通常与授权码授权模式一起使用，它主要用于公共客户端中，以防止授权码泄露的风险。

这里讲到的的公共客户端是指不能安全保存客户端秘钥的客户端，如移动应用程序、单页应用（SPA，例如 React、Angular、Vue.js 等框架开发的应用）、桌面应用、 嵌入式设备等。

也可以理解为前后端分离的开发场景，这种场景可以使用抓包、反编译等手段获取到客户端秘钥（client_secret）。

在授权码授权流程中，如果授权码被窃取或劫持，攻击者可以使用该授权码获得访问令牌，从而访问受保护的资源。

而在授权码扩展流程中，客户端生成 code_verifier 和 code_challenge 与认证服务器进行交互，以生成的随机认证码进行身份认证。

# 6.设备授权码模式

设备授权码模式（Device Authorization Grant）主要会出现在凭证式授权类型中，它适用于那些没有浏览器或键盘的设备。无法通过用户输入凭据的方式进行授权。相比传统的授权码模式，设备授权码模式不需要用户输入用户名和密码。用户只需要输入一个设备授权码即可完成授权。

设备授权码模式中，设备会让用户在另一台设备上的浏览器中访问一个网页，以进行登录。 用户登录后，设备可以获取所需的访问令牌和刷新令牌。

  • 没有键盘的设备（如智能电视）
  • 无法安全输入密码的设备
  • 第三方设备（用户不想在该设备上输入密码）

# 7.刷新令牌

在获取“访问令牌（Access Token）”时会同步获取刷新令牌（Refresh Token），如果用户访问的时候，客户端的“访问令牌”已经过期，则需要使用“刷新令牌”申请一个新的访问令牌。

# 六、客户端认证方式

在 OAuth 2.0 和 OpenID Connect 1.0 协议中，客户端认证方法用于客户端向授权服务器证明其身份。

常见的客户端认证方法如下：

  1. client_secret_basic：使用 HTTP Basic 认证，在认证请求中，将 clientId 和 clientSecret 通过 : 号拼接，并通过 Base64 编码得到一个字符串。在请求时授权服务器时，将此编码字符串放到 Authorization 请求头中；
  2. client_secret_post：将 clientId 和 clientSecret 放到认证请求的请求体（表单数据）中；
  3. client_secret_jwt：
    • 客户端通过 HMAC 算法（对称加密），使用客户端秘钥（clientSecret）签名一个 JWT，并将该 JWT 发送作为认证凭据放到认证请求中；
    • 授权服务器也通过 clientSecret 校检 JWT 的签名；
  4. private_key_jwt：
    • 客户端通过 RSA 算法（非对称加密），使用私钥签名一个 JWT，并将该 JWT 作为认证凭据放到认证请求中。
    • 授权服务器则通过公钥（公钥是客户端提供的）校检 JWT 的签名；
  5. none：不提供客户端认证。当客户端是公共客户端时认证服务器不会对客户端进行验证，PKCE 流程要求客户端为公共客户端。
