01.CAS 单点登录介绍

# 一、单点登录

# 1.什么是单点登录

在公司的系统开发中，随着业务的逐渐增加，系统也会越来越多。每个系统都需要开发一套认证的功能，然后每个用户也要在每个系统中都进行注册，如果同时需要使用多个系统，那么就要登录很多次。

为了解决这个问题，就出现了单点登录：单点登录（Single Sign-On，简称 SSO）是一种身份验证和访问控制的解决方案，它允许用户使用一组凭据（例如用户名和密码）登录到多个应用程序或系统，而无需在每个应用程序中单独进行身份验证。

在 SSO 中，用户登录后，SSO 系统会为其生成一个安全的令牌，该令牌用于标识用户的身份。当用户尝试访问其他应用程序时，这些应用程序会向 SSO 系统发送令牌进行验证。如果令牌有效，用户将被自动登录到应用程序。

# 2.如何实现单点登录

# 2.1 相同顶级域名

如果我们的系统所使用的域名，都是同一个顶级域名，那我们只需要将 Cookie 存储到顶级域名，这样所有的子域名的服务都能获取到 Cookie。当用户请求服务时，均可以携带着这个共享的 Cookie 了，不过，这要求所有的服务的权限校检使用相同的框架和技术。

不过，这种共享 Cookie 的方式有很大的局限性：

  • 所有应用的顶级域名必须是同一个；
  • 所有系统使用的权限校检技术（或校检逻辑）要相同，否则 Cookie 中 Key 的名称就会不同，无法维持会话；
  • 共享 Cookie 的方式无法实现跨语言技术平台登录，比如无法在 Java、PHP、.Net 等之间共享 Cookie，除非自己开发一套权限框架，支持多语言；
  • Cookie 本身也不安全，会有 CSRF 攻击的风险。

# 2.2 顶级域名不同的单点登录

方案如下：

  • 基于令牌（Token）：用户登录后，认证服务器会颁发一个令牌给用户，该令牌包含用户的身份信息和权限。用户在访问其他应用程序时，将令牌发送给应用程序进行验证。这种方式可以使用 JSON Web Token（JWT）或其他类似的令牌机制。
  • 基于代理（Proxy）：在代理服务器中进行身份验证和授权，当用户访问其他应用程序时，代理服务器会将用户的身份信息传递给应用程序。这种方式可以通过反向代理或 API 网关来实现。
  • 基于标准协议：使用标准的身份验证和授权协议，如 CAS、SAML、OAuth、OpenID 等。这些协议提供了一套通用的规范和流程，可以实现跨应用程序的单点登录和授权。
  • 基于身份提供商（Identity Provider）：使用独立的身份提供商来处理用户的身份验证和授权。用户只需登录一次身份提供商，即可访问多个应用程序。常见的身份提供商包括Active Directory、Okta、Auth0等。

# 3.单点登录执行流程

# 3.1 单点登录流程

在单点登录系统中，有一个独立的认证中心。只有认证中心才能接受用户的认证凭据信息（如用户名、密码），而其他系统并不提供登录入口，只接受认证中心的间接授权，间接授权通过令牌实现。

  • 当用户登录时，会跳转到认证中心登录，认证中心如果通过了用户的认证凭据信息（如用户名、密码），则创建授权令牌。在接下来的跳转过程中，授权令牌会作为参数发送给各个子系统。

  • 子系统拿到授权令牌，即得到了授权，可以借此创建局部会话，局部会话的登录方式与单系统的登录方式相同。

单点登录流程如下图：

用户登录成功之后，会与 SSO 认证中心及各个子系统之间建立会话：

  • 用户与 SSO 认证中心建立的会话称为全局会话、
  • 用户与各个子系统建立的会话称为局部会话。
  • 局部会话建立之后，用户访问子系统受保护资源将不再通过 SSO 认证中心。

全局会话与局部会话的约束关系如下：

  1. 局部会话存在，全局会话一定存在；
  2. 全局会话存在，局部会话不一定存在；
  3. 全局会话销毁，局部会话必须销毁。

单点登录涉及到 SSO 认证中心与众多子系统，各子系统与 SSO 认证中心之间需要通信以交换令牌、校验令牌及发起注销请求。因而各子系统必须集成 SSO 客户端，SSO 认证中心则是 SSO 服务端，整个单点登录过程实质是 SSO 客户端与服务端通信的过程。

单点登录系统中，SSO 认证中心只负责认证，而授权由相应的子系统负责（因为用户在不同系统之间的权限有可能不一样）。

# 3.2 单点注销流程

当用户进行注销操作时，我们也需要经过 SSO 系统进行注销。即在一个子系统中注销后，所有子系统的会话都将被销毁，这个和单点登录相对应，叫做单点注销。

SSO 认证中心会一直监听全局会话的状态，一旦发现全局会话被销毁，监听器将通知所有注册系统执行注销操作。

# 二、CAS 认证服务系统

# 1.CAS 介绍

CAS（Central Authentication Service）全称叫做中央认证服务。这是由耶鲁大学发起的一个开源项目，目的是帮助 Web 应用系统构建一种可靠的单点登录解决方案，从目前企业实际项目来看，CAS 还是非常受欢迎的一种单点登录解决方案。

项目文档地址：https://apereo.github.io/cas/6.6.x/index.html (opens new window)

在 CAS 系统中，分为 CAS Server 与 CAS Client 两部分：

  • CAS Server 是认证服务，其对应是单点登录系统中负责验证的服务端，其主要用来校验用户的认证凭据（用户名、密码等），一般进行独立部署。
  • CAS Client 是我们单点登录系统中的子系统。

# 2.CAS 的支持

CAS Client 支持的平台有：

  • Apache httpd Server (mod_auth_cas module)
  • Java (Java CAS Client)
  • .NET (.NET CAS Client)
  • PHP (phpCAS)
  • Perl (PerlCAS)
  • Python (pycas)
  • Ruby (rubycas-client)

CAS 支持的通信协议有：

  • CAS (versions 1, 2, and 3)
  • SAML 1.1 and 2
  • OpenID Connect
  • OpenID
  • OAuth 2.0
  • WS Federation

CAS 支持的认证机制有：

  • JAAS
  • LDAP
  • RDBMS
  • SPNEGO
  • ...

# 3.CAS 核心概念

  • TGT（Ticket Grantfng Ticke）：这是用户登录后生成的票根，包含用户的认证身份、有效期等信息，存储于 CAS Server 中，类似于我们常见的服务器会话；
  • TGC（Ticket Granted Cookie）：TGC 以 Cookie 的形式保存在浏览器中，根据 TGC 可以帮助用户找到对应的 TGT，类似于 SessionID；
  • ST（Service Ticket）：这是 CAS Server 使用 TGT 签发的一张一次性票据，CAS Client 使用 ST 与 CAS Server 进行交互，以获取用户的认证状态。
    • 如果用户在 CAS Server 中已经登录，当用户访问其他的 CAS Client 服务时，如果发现没有 Cookie 或者 ST ，就会 302 重定向到 CAS Server 获取 ST，然后会携带着 ST 重定向回来，CAS Client 则通过 ST 去 CAS Server 上获取用户的登录状态。
