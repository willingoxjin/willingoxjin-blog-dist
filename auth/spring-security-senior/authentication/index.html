# 2.SecurityContextHolderFilter

可以看到，SecurityContextHolderFilter 过滤器的执行时机非常早，当请求到达该过滤时，执行逻辑如下：

  • 首先，会从 SecurityContextRepository 中把 SecurityContext 对象取出来；
  • 接下来，会把取出来的 SecurityContext 放入 SecurityContextHolder 的 ThreadLocal 中。
    • 如果 SecurityContextHolder 中使用的是 ThreadLocalSecurityContextHolderStrategy 的存储策略，即 SecurityContextHolder 使用 ThreadLocal 的存储策略，就会将 SecurityContext 存储到 ThreadLocal 中。不过一般默认都会使用 ThreadLocal 策略；
  • 在处理完请求后（所有拦截器都处理完成），SecurityContextHolderFilter 则将 SecurityContext 从 ThreadLocal 中清除。

SecurityContextHolderFilter 的源码如下：

public class SecurityContextHolderFilter extends GenericFilterBean {

	private static final String FILTER_APPLIED = SecurityContextHolderFilter.class.getName() + ".APPLIED";

	private final SecurityContextRepository securityContextRepository;

	private SecurityContextHolderStrategy securityContextHolderStrategy = SecurityContextHolder
			.getContextHolderStrategy();

	// ......
  
	private void doFilter(HttpServletRequest request, HttpServletResponse response, FilterChain chain)
			throws ServletException, IOException {
    // 防止一次请求中,重复执行该过滤器
		if (request.getAttribute(FILTER_APPLIED) != null) {
			chain.doFilter(request, response);
			return;
		}
		request.setAttribute(FILTER_APPLIED, Boolean.TRUE);
    // 从SecurityContextRepository中把SecurityContext对象取出来的方法
		Supplier<SecurityContext> deferredContext = this.securityContextRepository.loadDeferredContext(request);
		try {
      // 将SecurityContext设置到SecurityContextHolder中
			this.securityContextHolderStrategy.setDeferredContext(deferredContext);
      // 在过滤器链调用下一个过滤器
			chain.doFilter(request, response);
		}
		finally {
      // 在请求结束后清除SecurityContextHolder
			this.securityContextHolderStrategy.clearContext();
			request.removeAttribute(FILTER_APPLIED);
		}
	}

	// ......
}

# 1.1 UsernamePasswordAuthenticationFilter 源码

UsernamePasswordAuthenticationFilter 继承于 AbstractAuthenticationProcessingFilter。UsernamePasswordAuthenticationFilter 会拦截登录认证请求，并获取用户名、密码等参数，将这些参数组装成认证凭据 UsernamePasswordAuthenticationToken，然后通过 AuthenticationManager 去校检认证凭据。

源码如下：

public class UsernamePasswordAuthenticationFilter extends AbstractAuthenticationProcessingFilter {
	// 默认用户名的key
	public static final String SPRING_SECURITY_FORM_USERNAME_KEY = "username";
	// 默认密码的key
	public static final String SPRING_SECURITY_FORM_PASSWORD_KEY = "password";
	// 默认登录请求地址及方法（该过滤器匹配的HTTP请求）
	private static final AntPathRequestMatcher DEFAULT_ANT_PATH_REQUEST_MATCHER = new AntPathRequestMatcher("/login",
			"POST");
	// 可以通过对应的set方法修改用户名和密码的key
	private String usernameParameter = SPRING_SECURITY_FORM_USERNAME_KEY;
	private String passwordParameter = SPRING_SECURITY_FORM_PASSWORD_KEY;

  // 默认只支持 POST 请求
	private boolean postOnly = true;

  // 无参构造方法：设置默认的登录uri为`/login`请求方式是POST
	public UsernamePasswordAuthenticationFilter() {
		super(DEFAULT_ANT_PATH_REQUEST_MATCHER);
	}

  // 构造方法：设置默认的登录uri为`/login`请求方式是POST, 并传入一个AuthenticationManager
	public UsernamePasswordAuthenticationFilter(AuthenticationManager authenticationManager) {
		super(DEFAULT_ANT_PATH_REQUEST_MATCHER, authenticationManager);
	}

  // 实现其父类 AbstractAuthenticationProcessingFilter提供的方法,用于认证
	@Override
	public Authentication attemptAuthentication(HttpServletRequest request, HttpServletResponse response)
			throws AuthenticationException {
    // POST请求校检
		if (this.postOnly && !request.getMethod().equals("POST")) {
			throw new AuthenticationServiceException("Authentication method not supported: " + request.getMethod());
		}
    // 从HttpServletRequest对象中获取用户名、密码
		String username = obtainUsername(request);
		username = (username != null) ? username.trim() : "";
		String password = obtainPassword(request);
		password = (password != null) ? password : "";
    // 把用户名、密码组装认证Token对象中,这是就是一个通行证,但是此时的状态是不可信的,一旦通过认证就变为可信的
		UsernamePasswordAuthenticationToken authRequest = UsernamePasswordAuthenticationToken.unauthenticated(username,
				password);
		// Allow subclasses to set the "details" property
    // 会将HttpServletRequest中的一些细节,如: request.getRemoteAddr(),request.getSession存入的到Token中
		setDetails(request, authRequest);
    // 使用父类中的 AuthenticationManager对Token进行认证
		return this.getAuthenticationManager().authenticate(authRequest);
	}

	/**
	 * 获取密码
	 * 如果你想改变获取密码的方式要么在此处重写，要么通过自定义一个前置的过滤器保证能此处能获取到数据
	 */
	@Nullable
	protected String obtainPassword(HttpServletRequest request) {
		return request.getParameter(this.passwordParameter);
	}

	/**
	 * 获取用户名
	 * 如果你想改变获取密码的方式要么在此处重写，要么通过自定义一个前置的过滤器保证能此处能获取到数据
	 */
	@Nullable
	protected String obtainUsername(HttpServletRequest request) {
		return request.getParameter(this.usernameParameter);
	}

	/**
	 * 将HttpServletRequest中的一些细节,如: request.getRemoteAddr(),request.getSession存入的到Token中
	 */
	protected void setDetails(HttpServletRequest request, UsernamePasswordAuthenticationToken authRequest) {
		authRequest.setDetails(this.authenticationDetailsSource.buildDetails(request));
	}

	/**
	 * 设置用户名参数的key
	 */
	public void setUsernameParameter(String usernameParameter) {
		Assert.hasText(usernameParameter, "Username parameter must not be empty or null");
		this.usernameParameter = usernameParameter;
	}

	/**
	 * 设置密码参数的key
	 */
	public void setPasswordParameter(String passwordParameter) {
		Assert.hasText(passwordParameter, "Password parameter must not be empty or null");
		this.passwordParameter = passwordParameter;
	}

	/**
	 * 设置认证的请求方式是只支持POST请求
	 */
	public void setPostOnly(boolean postOnly) {
		this.postOnly = postOnly;
	}

	public final String getUsernameParameter() {
		return this.usernameParameter;
	}

	public final String getPasswordParameter() {
		return this.passwordParameter;
	}

}

# 1.2 我们基于 UsernamePasswordAuthenticationFilter 能做些什么？

我们可以自定义 UsernamePasswordAuthenticationFilter 的如下内容：

  • 定制我们的登录请求 URI 和请求方式。
  • 登录请求参数的格式定制化，比如可以使用 JSON 格式提交甚至几种并存。
  • 将用户名和密码封装入凭据 UsernamePasswordAuthenticationToken ，定制业务场景需要的特殊凭据。

# 4.AbstractAuthenticationProcessingFilter

我们上面看到，AbstractAuthenticationProcessingFilter 是 UsernamePasswordAuthenticationFilter 的基类，实现了认证过滤器的处理逻辑。

AbstractAuthenticationProcessingFilter 中的 doFilter() 方法及相关源码如下：

public abstract class AbstractAuthenticationProcessingFilter extends GenericFilterBean
		implements ApplicationEventPublisherAware, MessageSourceAware {
  
  // ......
  
  private AuthenticationManager authenticationManager;

  private SessionAuthenticationStrategy sessionStrategy = new NullAuthenticatedSessionStrategy();

  private AuthenticationSuccessHandler successHandler = new SavedRequestAwareAuthenticationSuccessHandler();

	private AuthenticationFailureHandler failureHandler = new SimpleUrlAuthenticationFailureHandler();
  
  // ......

  @Override
	public void doFilter(ServletRequest request, ServletResponse response, FilterChain chain)
			throws IOException, ServletException {
		doFilter((HttpServletRequest) request, (HttpServletResponse) response, chain);
	}

	private void doFilter(HttpServletRequest request, HttpServletResponse response, FilterChain chain)
			throws IOException, ServletException {
    // 通过请求的uri和方法来判断是否需要认证,比如默认的/login
		if (!requiresAuthentication(request, response)) {
			chain.doFilter(request, response);
			return;
		}
		try {
      // 接着就是执行子类的attemptAuthentication方法,来获取认证结果对象Authentication
      // 这个对象不能是空 否则直接return
			Authentication authenticationResult = attemptAuthentication(request, response);
			if (authenticationResult == null) {
				// return immediately as subclass has indicated that it hasn't completed
				return;
			}
      // 处理session策略,默认情况下sessionStrategy为NullAuthenticatedSessionStrategy,没有任何策略
			this.sessionStrategy.onAuthentication(authenticationResult, request, response);
			// Authentication success
      // 认证成功后继续执行其它的过滤器链
			if (this.continueChainBeforeSuccessfulAuthentication) {
				chain.doFilter(request, response);
			}
      // 认证成功,交给认证成功处理器AuthenticationSuccessHandler来处理
			successfulAuthentication(request, response, chain, authenticationResult);
		}
		catch (InternalAuthenticationServiceException failed) {
			this.logger.error("An internal error occurred while trying to authenticate the user.", failed);
      // 如果遇到异常 就会交给认证失败处理器AuthenticationFailureHandler来处理
			unsuccessfulAuthentication(request, response, failed);
		}
		catch (AuthenticationException ex) {
			// Authentication failed
      // 如果遇到异常 就会交给认证失败处理器AuthenticationFailureHandler来处理
			unsuccessfulAuthentication(request, response, ex);
		}
	}

  protected AuthenticationManager getAuthenticationManager() {
		return this.authenticationManager;
	}
  
  // ......
  
}

# 5.AnonymousAuthenticationFilter

AnonymousAuthenticationFilter 过滤器用于处理匿名用户的身份验证。

当应用中部分接口或资源无需认证和权限即可访问，且此时用户没有登录，此时 AnonymousAuthenticationFilter 就会创建一个临时的身份验证对象 AnonymousAuthenticationToken，并将其存储在 SecurityContextHolder 中。

AnonymousAuthenticationFilter 处理的具体逻辑见 AnonymousAuthenticationFilter#defaultWithAnonymous 方法。

# 6.ExceptionTranslationFilter

ExceptionTranslationFilter 过滤器用来捕获并处理两种异常： AuthenticationException 和 AccessDeniedException。

  • AuthenticationException 异常：若捕获该异常，则将会调用关联的 AuthenticationEntryPoint 类中的 commence() 方法进行处理。
    • 在处理之前，ExceptionTranslationFilter 会先使用 RequestCache 将当前的HttpServerletRequest 的信息保存起来，来保证用户认证登录成功后可以跳转到之前指定跳转到的界面。
  • AccessDeniedException 异常：若捕获该异常，则将根据当前用户是否已经登录认证做出不同的处理。
    • 如果未登录，将会调用关联的 AuthenticationEntryPoint 类的 commence() 方法来进行处理；
    • 如果已经登录，将会调用关联的 AccessDeniedHandler 类的 handle() 方法来进行处理。

# 7.AuthorizationFilter

AuthorizationFilter 在上面过滤器中排在最后面，其用于处理请求的授权逻辑。其主要功能是验证用户是否具有足够的权限来执行请求的操作。它通过检查用户的身份验证状态和访问请求的URL路径，来确定用户是否满足访问所需的权限。

当请求到达 AuthorizationFilter 过滤器时，它会执行以下操作：

  1. 检查授信凭证 Authentication
    • 首先，它会从 SecurityContextHolder 中获取当前用户的 Authentication 对象，该对象包含了用户的身份验证信息。
    • 然后，它会检查用户是否已经通过身份验证，即 Authentication 对象是否不为 null。若为 null ，则抛出 AuthenticationCredentialsNotFoundException 异常。
  2. 如果用户已经通过身份验证，则会检查 URL 访问资源的权限
    • 其会调用 AuthorizationManager 来决定用户是否有权限访问请求的URL路径。AuthorizationManager 是一个策略接口，它根据用户的身份和访问请求的上下文，决定是否授权用户访问。
    • 如果用户被授权访问请求的URL路径，AuthorizationFilter 将允许请求继续传递给下一个过滤器或请求处理器进行处理。
    • 如果用户未被授权访问请求的URL路径，AuthorizationFilter 将会抛出 AccessDeniedException 异常，表示用户没有足够的权限。

这里只是列举了一部分过滤器，其他过滤器请查看对应的源码。

# 2.AuthenticationManager 的初始化

# 2.1 初始化流程

在 HttpSecurityConfiguration#httpSecurity 方法用来进行 HttpSecurity 的配置和初始化处理。

在这些配置中就有构建 AuthenticationManagerBuilder 的步骤：

  1. 创建 AuthenticationManagerBuilder 对象，类型为 DefaultPasswordEncoderAuthenticationManagerBuilder；
  2. 通过 authenticationManager() 方法，获取 AuthenticationManager 对象，并将获取到的 AuthenticationManager 对象放入 AuthenticationManagerBuilder 中。
  3. 设置 AuthenticationEventPublisher。
  4. 最后，通过 HttpSecurity 的构造方法，将 AuthenticationManagerBuilder 对象放入 HttpSecurity 中。

HttpSecurityConfiguration 的 httpSecurity() 方法的源码 如下：

@Configuration(proxyBeanMethods = false)
class HttpSecurityConfiguration {
  
  private ObjectPostProcessor<Object> objectPostProcessor;

	private AuthenticationConfiguration authenticationConfiguration;
  
  // ......
  
  @Autowired
	void setObjectPostProcessor(ObjectPostProcessor<Object> objectPostProcessor) {
		this.objectPostProcessor = objectPostProcessor;
	}

	@Autowired
	void setAuthenticationConfiguration(AuthenticationConfiguration authenticationConfiguration) {
		this.authenticationConfiguration = authenticationConfiguration;
	}
  
  // ......
  
	@Bean(HTTPSECURITY_BEAN_NAME)
	@Scope("prototype")
	HttpSecurity httpSecurity() throws Exception {
    // 初始化LazyPasswordEncoder
		LazyPasswordEncoder passwordEncoder = new LazyPasswordEncoder(this.context);
    // 创建DefaultPasswordEncoderAuthenticationManagerBuilder
		AuthenticationManagerBuilder authenticationBuilder = new DefaultPasswordEncoderAuthenticationManagerBuilder(
				this.objectPostProcessor, passwordEncoder);
    // 设置AuthenticationManager
		authenticationBuilder.parentAuthenticationManager(authenticationManager());
		authenticationBuilder.authenticationEventPublisher(getAuthenticationEventPublisher());

在 Spring Security 中，ObjectPostProcessor 接口主要用于以下两个方面：

  1. 对象创建后的定制

    在 Spring Security 中，很多对象都是通过配置类来创建的，例如 HttpSecurity、WebSecurity、AuthenticationManagerBuilder 等。ObjectPostProcessor 接口允许在这些对象创建之后，对这些对象进行进一步的处理和定制，例如添加过滤器、设置认证方式等。

  2. 对象创建前的定制

    除了在对象创建后进行定制之外，ObjectPostProcessor 接口还可以在对象创建前进行定制。在这种情况下，ObjectPostProcessor 接口会作为对象创建工厂的一个回调函数，用于在对象创建之前进行一些前置处理，例如修改对象的构造参数、替换对象的实现类等。

通过 ObjectPostProcessor 接口，Spring Security 提供了一种高度灵活的方式，让开发者可以对各种对象进行精细化的定制和处理，从而实现更加细致和个性化的安全配置。

# 3.AuthenticationManager 的认证过程

AuthenticationManager 的实现 ProviderManager 管理着众多的 AuthenticationManager，每一个 ProviderManager 都支持特定类型的 Authentication。

在适配到对应类型的 Authentication 中：

  • 只要有一个 AuthenticationProvider 认证成功，那么就认为这个请求认证成功了。
    • 认证成功后，Authentication 就会变成授信凭据，并触发认证成功的事件。
  • 如果所有的 AuthenticationProvider 都没有认证成功则认为认证失败。
    • 认证失败后，就会抛出异常，并触发认证失败的事件。

AuthenticationManager 对特定的 Authentication 提供了特定的认证功能，我们可以借此实现多种认证并存的业务。比如，实现用户名密码认证、手机号验证码认证、邮箱验证码认证等等。

# 三、ProviderManager

AuthenticationManagerBuilder 类中的 performBuild() 方法的返回值为 ProviderManager 类型的对象，而 ProviderManager 则是 AuthenticationManager 的其中实现之一。

ProviderManager 的功能如下：

  1. 管理和调用 AuthenticationProvider。
    • ProviderManager 包含一个或多个 AuthenticationProvider。
    • 每个 AuthenticationProvider 都会尝试对用户进行身份验证。当一个 AuthenticationProvider 返回验证成功的 Authentication 对象时，ProviderManager 就会将该对象返回给调用方。否则，ProviderManager 会继续调用下一个 AuthenticationProvider 进行身份验证，直到身份验证成功或者所有的AuthenticationProvider都验证失败。
    • ProviderManager还支持缓存验证结果，以提高系统的性能。
  2. 在认证成功和失败时，通过 AuthenticationEventPublisher 发布事件。
    • AuthenticationEventPublisher 是 Spring Security 中的一个事件发布器，它的作用是在身份验证过程中发布相关的事件，以便其他组件可以监听这些事件并执行相应的操作。
    • 例如，当用户成功登录时，AuthenticationEventPublisher 可以发布一个 AuthenticationSuccessEvent 事件，其他组件可以监听该事件并执行一些操作，如记录日志、更新用户登录信息等。
    • 同样地，当身份验证失败时，AuthenticationEventPublisher 可以发布一个 AbstractAuthenticationFailureEvent 事件，其他组件也可以监听该事件并执行相应的操作，如记录日志、发送警报等。
    • AuthenticationEventPublisher 还可以自定义事件类型和事件处理程序，以满足特定应用程序的需求。

ProviderManager 中调用 authenticate() 方法认证的部分源码如下：

public class ProviderManager implements AuthenticationManager, MessageSourceAware, InitializingBean {
  
  private AuthenticationEventPublisher eventPublisher = new NullEventPublisher();

	private List<AuthenticationProvider> providers = Collections.emptyList();
  
  private AuthenticationManager parent;
  
  // ......
  
	@Override
  public Authentication authenticate(Authentication authentication) throws AuthenticationException {
      Class<? extends Authentication> toTest = authentication.getClass();
      AuthenticationException lastException = null; // 最后一次认证的异常
      AuthenticationException parentException = null; // 父AuthenticationManager认证的异常
      Authentication result = null; // 认证成功的授信凭证
      Authentication parentResult = null; // 父AuthenticationManager认证成功的授信凭证
      int currentPosition = 0;
      int size = this.providers.size();
      for (AuthenticationProvider provider : getProviders()) {
          // 匹配授信凭证对应的的AuthenticationProvider类型,未成功匹配直接跳过
          if (!provider.supports(toTest)) {
            	continue;
          }
          if (logger.isTraceEnabled()) {
              logger.trace(LogMessage.format("Authenticating request with %s (%d/%d)",
                  	provider.getClass().getSimpleName(), ++currentPosition, size));
        	}
          // 调用AuthenticationProvider的authenticate方法进行认证
          // 若授信成功,则复制原凭证中的details属性,并结束循环
          // 若授信失败,捕捉AccountStatusException和InternalAuthenticationServiceException异常,
          // 		并通过AuthenticationEventPublisher发布异常事件
        	try {
              result = provider.authenticate(authentication);
              if (result != null) {
                  copyDetails(authentication, result);
                  break;
              }
          }
          catch (AccountStatusException | InternalAuthenticationServiceException ex) {
              // 发布认证失败事件
              prepareException(ex, authentication);
              // SEC-546: Avoid polling additional providers if auth failure is due to
              // invalid account status
              throw ex;
          }
          catch (AuthenticationException ex) {
            	lastException = ex;
          }
      }
    	// for each end
    
      // 若认证失败,调用父AuthenticationManager进行认证
      if (result == null && this.parent != null) {
          try {
              parentResult = this.parent.authenticate(authentication);
              result = parentResult;
          }
          catch (ProviderNotFoundException ex) {
              // ignore as we will throw below if no other exception occurred prior to
              // calling parent and the parent
              // may throw ProviderNotFound even though a provider in the child already
              // handled the request
          }
          catch (AuthenticationException ex) {
              parentException = ex;
              lastException = ex;
          }
      }
      // 认证成功或父AuthenticationManager认证成功
      if (result != null) {
          if (this.eraseCredentialsAfterAuthentication && (result instanceof CredentialsContainer)) {
              // 认证完成。从身份验证中删除凭据和其他秘密数据
              ((CredentialsContainer) result).eraseCredentials();
          }
          // 如果父AuthenticationManager认证成功,则父AuthenticationManager中会发布一个AuthenticationSuccessEvent事件
          // 否则这里将发布一个AuthenticationSuccessEvent事件
          // 这样可以防止重复发布事件
          if (parentResult == null) {
            	this.eventPublisher.publishAuthenticationSuccess(result);
          }

        	return result;
      }

      // 父对象为空，或者没有进行身份验证(或抛出异常)。
      if (lastException == null) {
          lastException = new ProviderNotFoundException(this.messages.getMessage("ProviderManager.providerNotFound",
              		new Object[] { toTest.getName() }, "No AuthenticationProvider found for {0}"));
      }
      // 如果尝试了父AuthenticationManager并且失败了，则父AuthenticationManager中就会发布AbstractAuthenticationFailureEvent事件
      // 否则这里将发布一个AbstractAuthenticationFailureEvent事件
      // 这样可以防止重复发布事件
      if (parentException == null) {
        	prepareException(lastException, authentication);
      }
      throw lastException;
  }

	@SuppressWarnings("deprecation")
	private void prepareException(AuthenticationException ex, Authentication auth) {
		this.eventPublisher.publishAuthenticationFailure(ex, auth);
	}

@Override
public Authentication authenticate(Authentication authentication) throws AuthenticationException {
  Assert.isInstanceOf(UsernamePasswordAuthenticationToken.class, authentication,
      () -> this.messages.getMessage("AbstractUserDetailsAuthenticationProvider.onlySupports",
          "Only UsernamePasswordAuthenticationToken is supported"));
  // 获取请求中的用户名参数
  String username = determineUsername(authentication);
  // 判断是否使用了缓存
  boolean cacheWasUsed = true;
  // 先从缓存中加载用户信息
  UserDetails user = this.userCache.getUserFromCache(username);
  if (user == null) {
    // 缓存中为空,说明没有使用缓存
    cacheWasUsed = false;
    try {
      // 调用retrieveUser方法查询用户信息
      // retrieveUser方法是一个抽象方法,由子类DaoAuthenticationProvider实现
      user = retrieveUser(username, (UsernamePasswordAuthenticationToken) authentication);
    }
    catch (UsernameNotFoundException ex) {
      this.logger.debug("Failed to find user '" + username + "'");
      if (!this.hideUserNotFoundExceptions) {
        throw ex;
      }
      throw new BadCredentialsException(this.messages
          .getMessage("AbstractUserDetailsAuthenticationProvider.badCredentials", "Bad credentials"));
    }
    Assert.notNull(user, "retrieveUser returned null - a violation of the interface contract");
  }
  try {
    // 进行必要的认证前和额外认证的检查,如账户是否被禁用、是否被锁定、是否过期等
    this.preAuthenticationChecks.check(user);
    // 调用additionalAuthenticationChecks方法用于进行密码对比
    // additionalAuthenticationChecks是一个抽象方法,由子类DaoAuthenticationProvider实现
    additionalAuthenticationChecks(user, (UsernamePasswordAuthenticationToken) authentication);
  }
  catch (AuthenticationException ex) {
    if (!cacheWasUsed) {
      throw ex;
    }
    // 在发生异常时，尝试检查后再试一次
    cacheWasUsed = false;
    user = retrieveUser(username, (UsernamePasswordAuthenticationToken) authentication);
    this.preAuthenticationChecks.check(user);
    additionalAuthenticationChecks(user, (UsernamePasswordAuthenticationToken) authentication);
  }
  // 认证后的检查操作,检查密码是否过期
  this.postAuthenticationChecks.check(user);
  // 未缓存,则加到缓存中
  if (!cacheWasUsed) {
    this.userCache.putUserInCache(user);
  }
  Object principalToReturn = user;
  // 是否强制将Authentication中的principal属性设置为字符串
  if (this.forcePrincipalAsString) {
    principalToReturn = user.getUsername();
  }
  // 认证成功后,构建一个已经授信的Authentication对象,并返回
  return createSuccessAuthentication(principalToReturn, authentication, user);
}

AbstractUserDetailsAuthenticationProvider 的 authenticate 方法执行逻辑如下：

  1. 获取用户信息
    • 从 Authentication 提取出登录用户名；
    • 利用得到的 username，去缓存中查询是否有该用户；
    • 如果缓存中没有该用户，则去执行 retrieveUser 方法获取当前用户对象。（retrieveUser 方法由子类 DaoAuthenticationProvider 实现）。
  2. 前置校检
    • 执行 preAuthenticationChecks.check 方法，检验用户信息中的账户属性是否正常，例如账户是否被禁用、是否被锁定、是否过期等。
  3. 校检认证凭证
    • 执行additionalAuthenticationChecks 方法，进行密码比对。（additionalAuthenticationChecks 方法由子类 DaoAuthenticationProvider 实现）
  4. 后置校检
    • 调用 postAuthenticationChecks.check 方法，检查密码是否过期
  5. 缓存用户
    • 如果未进行缓存用户信息，则执行缓存操作。
    • 默认缓存使用 NullUserCache 对象。我们也可以自己手动设置 SpringCacheBasedUserCache 对象，SpringCacheBasedUserCache 对象中使用 org.springframework.cache.Cache 类型对象进行缓存（在实际开发中，我们可以自己实现 Cache 接口来自定义 Redis 或使用其他缓存中间件）。
  6. 是否强制将 Authentication 中的 principal 属性设置为字符串。
    • 如果 forcePrincipalAsString 属性为 true，则会对最后返回的 Authentication 的 principal 属性，设置为“用户的用户名”，而不是 UserDetails 对象。
  7. 构建 Authentication 对象并返回
    • 构建出一个新的且已认证的 UsernamePasswordAuthenticationToken 对象，并返回。

# 五、Authentication

# 1.Authentication 介绍

Authentication 我们经常提到，它是用来认证的，并且在内部维护了认证的信息和认证状态。

其实，Authentication 相当于一个容器，其中包含：

  • 用户身份（Principal）：要进行身份验证主体的身份，如果是用户名密码认证则 Principal 就是用户名。AuthenticationManager 实现通常会返回包含更丰富信息的身份验证作为应用程序使用的主体。许多身份验证提供程序将创建 UserDetails 对象作为主体。
  • 认证凭据（Credentials）：通常是一个密码，但可以是与身份验证管理器相关的任何内容。调用方应填充凭据。
  • 用户的权限（Authorities）：由身份验证管理器设置，表示已授予主体的权限。并且在身份认证成功后才会有效。
  • 其他信息（Details）：可以是 IP 地址、证书序列号等。

# 2.Authentication 的实现类

Authentication 接口有许多实现类，其中最常用的是 UsernamePasswordAuthenticationToken 和 AnonymousAuthenticationToken：

  • UsernamePasswordAuthenticationToken 表示使用用户名和密码进行身份验证的信息。
  • AnonymousAuthenticationToken 表示匿名用户的信息。

# 1.自定义登录方式

看了上面关于 Spring Security 的认证的源码，你心中大概有了认证的流程。如果让我们自定义登录，如手机验证码登录、邮箱验证码登录等等，我们其实只需要自定义相关的过滤器、认证处理器即可。

如，我们想要实现手机验证码登录，就可以自定义一下的类：

  • MobileCodeAuthenticationFilter：手机验证码认证过滤器，可以使其继承自 GenericFilterBean 接口或 继承自Spring Security 提供的的其他抽象过滤器类。
  • MobileCodeAuthenticationToken：手机验证码认证的 Authentication，可以将手机号和验证码放入其中，或添加其他自定义参数，如是否对未注册手机号自动注册登参数。
  • MobileCodeAuthenticationProvider：手机验证码认证的 Provide，其 supports 方法与 MobileCodeAuthenticationToken.class 相匹配。

配置示例：

@Configuration
public class AuthenticationManagerConfig {

    @Bean
    public AuthenticationManager authenticationManager(UserDetailsService userDetailsService) {
        MobileCodeAuthenticationProvider provider = new MobileCodeAuthenticationProvider();
        provider.setUserDetailsService(userDetailsService);
        provider.setPasswordEncoder(passwordEncoder);
        // 是否强制使用principal.toString()作为principal
        provider.setForcePrincipalAsString(true);
        // 是否隐藏用户不存在异常
        provider.setHideUserNotFoundExceptions(true);
        return new ProviderManager(provider);
    }

}

# 2.多种登录方式并存

在实际的开发中，我们往往需要集成多种登录方式，如用户名密码登录、手机号验证码登录、手机号一键登录、邮箱验证码登录、微信登第三方扫码或授权登录等等。所以多种登录方式并存其实是一个常见的需求。

# 2.1 定义多个过滤器

当然，我们上面已经说了自定义登录的思路，根据这个思路扩展一下：

  • 首先，我们定义不同的认证过滤器。
    • 在认证过滤器中，每个过滤器只拦截自己对应登录的 URL。比如手机验证码登录过滤器的 URL 为 /login/mobileCode，邮箱验证码登录过滤器的 URL 为 /login/emailCode。
    • 在认证过滤器中，都组装对应的 XxxxAuthenticationToken，并调用其对应的 AuthenticationManager。
  • 在 AuthenticationManager 中，调用对应的 XxxxAuthenticationProvider。
  • 在 AuthenticationProvider 中，根据登录方式指定对应的认证逻辑即可。

最后将认证过滤器都配置到 Spring Security 的过滤器链中，使他们在 UsernamePasswordAuthenticationFilter 的位置执行即可。

当然，我们也可以为不同的登录方式提供对应处理器： AuthenticationEntryPoint 、SavedRequestAwareAuthenticationSuccessHandler、ExceptionMappingAuthenticationFailureHandler、LogoutSuccessHandler。

# 2.2 定义代理类

当然，我们也可以定义一个代理过滤器，在代理过滤器中根据请求 URL 或者请求参数匹配对应的实际过滤器，原理其实都是一样的，不同的是匹配的方式不一样。
