# 一、GrantedAuthority

# 1.GrantedAuthority 接口

在之前的 Spring Security 的用户信息 文章中了解到，Spring Security 的用户认证过程中，UserDetailsService 接口用于查询用户的信息，该接口中的 loadUserByUsername 方法用于查询用户信息，该方法返回一个 UserDetails 类型的对象，UserDetails 对象中存储着用户相关的信息，其中就包含密码、权限等信息。

在认证成功后，会组装并返回一个已经认证的 Authentication 类型的对象，该对象会存储用户的认证信息、权限等数据。Authentication 接口中的 getAuthorities() 方法用来获取已经认证成功后用户的权限列表，其返回值的类型为一个 GrantedAuthority 类型的集合。

GrantedAuthority 是一个接口，代表了用户在系统中可以执行的特定操作或访问的资源

GrantedAuthority 接口源码如下：

public interface GrantedAuthority extends Serializable {
		String getAuthority();
}
1
2
3

GrantedAuthority 接口只有一个方法 getAuthority()，该方法返回一个字符串，表示授予给用户的权限的名称。权限名称通常是与用户角色或特定操作相关联的标识符。

GrantedAuthority 接口常用的实现类为 SimpleGrantedAuthority，此实现允许将任何用户指定的内容 String 转换为 GrantedAuthority。

# 2.角色前缀

默认情况下，基于角色的授权规则包含 ROLE_ 作为前缀。这意味着，如果有一个授权规则要求安全上下文具有“USER”角色，Spring Security 将默认查找返回 GrantedAuthority#getAuthority “ROLE_USER“的角色。

我们可以通过 GrantedAuthorityDefaults 来定义角色名称的前缀：

@Configuration
public class AuthorityConfig {

    /**
     * 使用static方法公开GrantedAuthorityDefaults,
     * 以确保Spring在初始化Spring Security的方法安全@Configuration类之前发布它
     * @return GrantedAuthorityDefaults
     */
    @Bean
    static GrantedAuthorityDefaults grantedAuthorityDefaults() {
        return new GrantedAuthorityDefaults("R_");
    }

}

# 二、基于 HTTP 请求的权限校检

# 1.基于 HTTP 请求的权限校检原理

请求授权组件的工作原理如下：

可以看到，Spring Security 的授权校检在 AuthorizationFilter 中。在默认情况下，该过滤器是 Spring Security 过滤器链中最后执行的一个过滤器。在这个过滤器中，会使用 AuthorizationManager 校检用户的权限。相关源码如下：

public class AuthorizationFilter extends GenericFilterBean {

  private final AuthorizationManager<HttpServletRequest> authorizationManager;

	private AuthorizationEventPublisher eventPublisher = AuthorizationFilter::noPublish;
  
  // ......
  
	@Override
	public void doFilter(ServletRequest servletRequest, ServletResponse servletResponse, FilterChain chain)
			throws ServletException, IOException {

		HttpServletRequest request = (HttpServletRequest) servletRequest;
		HttpServletResponse response = (HttpServletResponse) servletResponse;

		if (this.observeOncePerRequest && isApplied(request)) {
			chain.doFilter(request, response);
			return;
		}

		if (skipDispatch(request)) {
			chain.doFilter(request, response);
			return;
		}

		String alreadyFilteredAttributeName = getAlreadyFilteredAttributeName();
		request.setAttribute(alreadyFilteredAttributeName, Boolean.TRUE);
		try {
      // 校检权限
			AuthorizationDecision decision = this.authorizationManager.check(this::getAuthentication, request);
			this.eventPublisher.publishAuthorizationEvent(this::getAuthentication, request, decision);
			// 授权失败，抛出AccessDeniedException异常
      if (decision != null && !decision.isGranted()) {
				throw new AccessDeniedException("Access Denied");
			}
			chain.doFilter(request, response);
		}
		finally {
			request.removeAttribute(alreadyFilteredAttributeName);
		}
	}

}

AuthorizationManager 接口用于根据用户的身份信息和资源的权限要求，判断用户是否具有访问该资源的权限。

AuthorizationManager 的源码如下：

@FunctionalInterface
public interface AuthorizationManager<T> {
  	
  default void verify(Supplier<Authentication> authentication, T object) {
		AuthorizationDecision decision = check(authentication, object);
		if (decision != null && !decision.isGranted()) {
			throw new AccessDeniedException("Access Denied");
		}
	}
  
  @Nullable
	AuthorizationDecision check(Supplier<Authentication> authentication, T object);
  
}

AuthorizationManager 类用了 @FunctionalInterface 修饰，说明其是一个函数式接口。其定义的方法解释如下：

  • check 方法会根据用户的身份信息和资源的权限要求，判断用户是否具有访问该资源的权限。

  • verify 方法会调用 check ，并校检后的授权状态，若授权状态为 false ，则会抛出 AccessDeniedException 异常。

check 方法的返回一个 AuthorizationDecision 对象，该对象存储的 granted 属性用于标识是否授权成功。

AuthorizationDecision 的源码如下：

public class AuthorizationDecision {

	private final boolean granted;

	public AuthorizationDecision(boolean granted) {
		this.granted = granted;
	}

	public boolean isGranted() {
		return this.granted;
	}

	@Override
	public String toString() {
		return getClass().getSimpleName() + " [granted=" + this.granted + "]";
	}

}

# 2.AuthorizationManager 的实现

AuthorizationManager 的部分实现如下：

# 2.1 RequestMatcherDelegatingAuthorizationManager

AuthorizationManager 中也有一个代理类 RequestMatcherDelegatingAuthorizationManager ，该类的 check 方法中，会先匹配出与请求最合适的代理对象，然后调用代理对象的 check 方法校检当前请求是否有访问资源的权限。这样，AuthorizationManager 可以对一组实现进行轮询以做出授权决策。

RequestMatcherDelegatingAuthorizationManager 中还有一个静态内部类，名为 Builder，其用于构建 RequestMatcherDelegatingAuthorizationManager 中代理的 AuthorizationManager 集合。

# 2.2 AuthorityAuthorizationManager

AuthorizationManager 的实现类中，最常用的就是 AuthorityAuthorizationManager。它配置了一组所需的的权限，以便在当前的 Authentication 中查找。

如果 Authentication 包含访问资源所需要的权限，它将返回授权状态为 true 的 AuthorizationDecision。否则它将返回一个授权状态为 false 的 AuthorizationDecision。

# 2.3 AuthenticatedAuthorizationManager

还有一个常用的，就是 AuthenticatedAuthorizationManager ，它可用于区分匿名、完全认证的用户和 remeber-me 认证的用户。

许多网站允许 remebme-me 认证下进行某些有限的访问，但要求用户通过登录来确认其身份以获得完全访问。

# 2.4 自定义 AuthorizationManager

我们还可以自己实现 AuthorizationManager，并且可以将我们想要的任何访问控制逻辑放入其中。例如，可以创建一个可以查询开放策略代理或自己的授权数据库的实现。

自定义后，可以通过如下方式配置（CustomAuthorizationManager 则是我们自定义的实现）：

@Configuration
public class SecurityConfig {
    @Override
    protected void configure(HttpSecurity http) throws Exception {
        http
            .authorizeHttpRequests(auth -> auth
                 .requestMatchers("/hello").access(new CustomAuthorizationManager())
                	.anyRequest().authenticated()
            )
    }
}

# 三、基于方法的权限校检

现在有如下的 readCustomer 方法：

@Service
public class MyCustomerService {
    @PreAuthorize("hasAuthority('permission:read')")
    @PostAuthorize("returnObject.owner == authentication.name")
    public Customer readCustomer(String id) { ... }
}

Spring Security 基于方法的权限校检的逻辑，以上面的 readCustomer 方法为例，原理如下：

基于方法的权限校检是基于 Spring AOP（切面）实现的，流程如下：

  1. Spring AOP 调用 readCustomer 的代理方法，调用与 @PreAuthorization 注解对应切点匹配的AuthorizationManagerBeforeMethodInterceptor 拦截器；
  2. AuthorizationManagerBeforeMethodInterceptor 拦截器调用 PreAuthorizeAuthorizationManager#check 方法；
  3. PreAuthorizeAuthorizationManager 授权管理器使用 MethodSecurityExpressionHandler 解析注解中配置的 SpEL 表达式，并从包含 SupplierSupplier<Authentication> 和 MethodInvocation 的 MethodSecurityExpressionRoot 构建相应的 EvaluationContext；
  4. 拦截器使用 EvaluationContext 上下文对表达式求值；具体来说，它从 Supplier 读取 Authentication，并检查它是否在其权限集合中具有 read 权限；
  5. 如果权限校检通过，则 Spring AOP 继续调用该方法；
  6. 如果权限校检没有通过，拦截器会发布一个 AuthorizationDeniedEvent，并抛出一个 AccessDeniedException 异常，由 ExceptionTranslationFilter 捕获并返回 403 状态码到响应中。
  7. 方法返回后，Spring AOP 调用与 @PostAuthorize 注解对应切点匹配的 AuthorizationManagerAfterMethodInterceptor 拦截器，逻辑大致与上述的 AuthorizationManagerBeforeMethodInterceptor 相同；
  8. 如果校检通过（在这种情况下，返回值属于登录用户），处理继续正常进行；
  9. 如果校检没有通过，拦截器会发布一个 AuthorizationDeniedEvent，并抛出一个 AccessDeniedException，由 ExceptionTranslationFilter 捕获并返回 403 状态码到响应中（与上面相同）。

关于切面与拦截器的相关配置可见：

  • PrePostMethodSecurityConfiguration
  • SecuredMethodSecurityConfiguration
  • Jsr250MethodSecurityConfiguration

注解与对应的切面拦截器如下：

  • @PreAuthorize：
    • 拦截器：AuthorizationManagerBeforeMethodInterceptor#preAuthorize
    • 权限管理器：PreAuthorizeAuthorizationManager
  • @PostAuthorize：
    • 拦截器：AuthorizationManagerBeforeMethodInterceptor#postAuthorize
    • 权限管理器：PostAuthorizeAuthorizationManager
  • @PreFilter：
    • 拦截器：PreFilterAuthorizationMethodInterceptor
  • @PostFilter：
    • 拦截器：PostFilterAuthorizationMethodInterceptor
  • @Secured：
    • 拦截器：AuthorizationManagerBeforeMethodInterceptor#secured
    • 权限管理器：SecuredAuthorizationManager
  • JSR-250注解：
    • 拦截器：AuthorizationManagerBeforeMethodInterceptor#jsr250
    • 权限管理器：Jsr250AuthorizationManager
