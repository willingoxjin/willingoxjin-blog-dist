03.Spring Security 的过滤器

# 一、Spring Security 内置 Filter

Spring Security 内置了一些过滤器，我们就可以使用这些内置的过滤器，来解决我们的需求和问题。

下面我们就来了解一下内置的过滤器：

# 1.ChannelProcessingFilter

ChannelProcessingFilter 通常是用来过滤哪些请求必须用 https 协议，哪些请求必须用 http 协议， 哪些请求随便用哪个协议都行。它主要有两个属性:

  • ChannelDecisionManager：用来判断请求是否符合既定的协议规则。它维护了一个 ChannelProcessor 列表，这些 ChannelProcessor 是则根据对应的策略处理器进行执行。相关策略如下：

    • ANY_CHANNEL 策略 （任何通道都可以）
    • REQUIRES_SECURE_CHANNEL 策略 （只能通过 https 通道）
    • REQUIRES_INSECURE_CHANNEL 策略 （只能通过 http 通道）。

  • FilterInvocationSecurityMetadataSource：用来存储 URL 与 对应的 ANY_CHANNEL、REQUIRES_SECURE_CHANNEL、REQUIRES_INSECURE_CHANNEL 的映射关系。

ChannelProcessingFilter 可以通过 HttpScurity#requiresChannel() 等相关方法引入其配置对象 ChannelSecurityConfigurer 来进行配置。

# 2.ConcurrentSessionFilter

ConcurrentSessionFilter 主要用来判断 Session 是否过期以及更新最新的访问时间。其流程为：

  1. Session 检测，如果不存在直接放行去执行下一个过滤器。存在则进行下一步。
  2. 根据 sessionid 从 SessionRegistry 中获取 SessionInformation，从 SessionInformation 中获取 Session 并判断是否过期；
    • 没有过期则更新 SessionInformation 中的访问日期；
    • 如果过期，则执行 doLogout() 方法，这个方法会将 Session 无效，并将 SecurityContext 中 Authentication 存储的权限置空，同时在 SecurityContenxtHoloder 中清除 SecurityContext。然后查看是否有跳转的 expiredUrl，如果有就跳转，没有就输出提示信息。

ConcurrentSessionFilter 可以通过 SessionManagementConfigurer 来进行配置。

# 3.WebAsyncManagerIntegrationFilter

WebAsyncManagerIntegrationFilter 用于集成 SecurityContext 到 Spring 异步执行机制中的 WebAsyncManager。用来处理异步请求的安全上下文。具体逻辑为：

  1. 从请求属性上获取所绑定的 WebAsyncManager，如果尚未绑定，先做绑定。
  2. 从 asyncManager 中获取 key 为 CALLABLE_INTERCEPTOR_KEY 的安全上下文多线程处理器 SecurityContextCallableProcessingInterceptor。如果获取到的为 null，则新建一个 SecurityContextCallableProcessingInterceptor 并绑定 CALLABLE_INTERCEPTOR_KEY 注册到 asyncManager 中。

SecurityContextCallableProcessingInterceptor 实现了接口 CallableProcessingInterceptor。当它被应用于一次异步执行时，beforeConcurrentHandling() 方法会在调用者线程执行，该方法会相应地从当前线程获取 SecurityContext，然后被调用者线程中执行逻辑时，会使用这个 SecurityContext，从而实现安全上下文从调用者线程到被调用者线程的传输。

WebAsyncManagerIntegrationFilter 可以通过 WebSecurityConfigurerAdapter#getHttp() 方法添加到 HttpSecurity 中成为 DefaultSecurityFilterChain 的一个链节。

# 4.SecurityContextPersistenceFilter

SecurityContextPersistenceFilter 主要控制 SecurityContext 的在一次请求中的生命周期。

  • 请求来临时，创建安全上下文信息 SecurityContext；
  • 请求结束时清空 SecurityContextHolder 。

SecurityContextPersistenceFilter 可以通过 HttpScurity#securityContext() 及相关方法引入 其配置对象 来进行配置

不过该过滤器在新版中已废弃，由 SecurityContextHolderFilter 代替。

# 5.HeaderWriterFilter

HeaderWriterFilter 用来给 http 响应添加一些 Header。比如 X-Frame-Options、X-XSS-Protection、X-Content-Type-Options 等。

HeaderWriterFilter 可以通过 HttpScurity#headers() 来定制请求 Header 。

# 6.CorsFilter

CorsFilter 用于处理跨域资源共享（CORS）请求。它实现了跨域资源共享规范，为 Web 应用程序提供了跨域访问支持。

官方的解读：这是配置和 XML 命名空间 CORS 配置的替代方法， 仅对依赖于 spring-web 的应用程序有用（不适用于 spring-webmvc）或要求在 javax.servlet.Filter 级别进行 CORS 检查的安全约束链接。

CorsFilter 可以通过 HttpSecurity#cors() 来配置。

# 7.CsrfFilter

CsrfFilter 用于防止 csrf 攻击，前后端使用 JSON 交互需要注意的一个问题。

CsrfFilter 可以通过 HttpSecurity#csrf() 方法来开启或者关闭它。如果使用 jwt 等 token 技术时，是不需要这个的。

# 8.LogoutFilter

LogoutFilter 是处理注销的过滤器。

LogoutFilter 可以通过 HttpSecurity.logout() 来定制注销逻辑，非常有用。

# 9.OAuth2AuthorizationRequestRedirectFilter

OAuth2AuthorizationRequestRedirectFilter 需要依赖 Spring Sscurity OAuth2 相关的模块。

该过滤器是处理 OAuth2 请求首选重定向相关逻辑的。

# 10.Saml2WebSsoAuthenticationRequestFilter

Saml2WebSsoAuthenticationRequestFilter 需要依赖 Spring Security SAML 模块。

该过滤器主要用于基于 SMAL 的 SSO 单点登录请求认证。

SAML （Security Assertion Markup Language）称作安全断言标记语言。它是一个基于 XML 的标准，用于在不同的安全域（security domain）之间交换认证和授权数据。在 SAML 标准定义了身份提供者（identity provider）和服务提供者（service provider），这两者构成了前面所说的不同的安全域。 SAML 是 OASIS 组织安全服务技术委员会（Security Services Technical Committee）的产品。

SAML 是一个 XML 框架，也就是一组协议，可以用来传输安全声明。比如，两台远程机器之间要通讯，为了保证安全，我们可以采用加密等措施，也可以采用 SAML 来传输，传输的数据以 XML 形式，符合 SAML 规范，这样我们就可以不要求两台机器采用什么样的系统，只要求能理解 SAML 规范即可，显然比传统的方式更好。SAML 规范是一组 Schema 定义。

可以这么说，在 Web Service 领域，schema 就是规范，在 Java 领域，API 就是规范。

# 11.X509AuthenticationFilter

X509 认证过滤器。

X509AuthenticationFilter 可以通过 HttpSecurity#X509() 来启用和配置相关功能。

# 12.AbstractPreAuthenticatedProcessingFilter

AbstractPreAuthenticatedProcessingFilter 用来处理经过预先认证的身份验证请求的过滤器的基类，其中认证主体已经由外部系统进行了身份验证。 目的只是从传入请求中提取主体上的必要信息， 而不是对它们进行身份验证。

你可以继承该类，并进行具体实现，然后通过 HttpSecurity#addFilter 方法来添加个性化的 AbstractPreAuthenticatedProcessingFilter 。

# 13.CasAuthenticationFilter

CAS 单点登录认证过滤器。依赖 Spring Security CAS 模块。

# 14.OAuth2LoginAuthenticationFilter

依赖 spring-scurity-oauth2 相关的模块。

OAuth2LoginAuthenticationFilter 是 OAuth2 登录认证过滤器，用来处理通过 OAuth2 进行认证登录的逻辑。

# 15.Saml2WebSsoAuthenticationFilter

这个需要用到 Spring Security SAML 模块，

Saml2WebSsoAuthenticationFilter 是一个基于 SMAL 的 SSO 单点登录认证过滤器。

# 16.UsernamePasswordAuthenticationFilter

UsernamePasswordAuthenticationFilter 处理用户以及密码认证的核心过滤器。

认证请求提交的 username 和 password ，被封装成 token 进行一系列的认证，便是主要通过这个过滤器完成的，在表单认证的方法中，这是最最关键的过滤器。

UsernamePasswordAuthenticationFilter 可以通过 HttpSecurity#formLogin() 及相关方法引入其配置对象 FormLoginConfigurer 来进行配置。

# 17.OpenIDAuthenticationFilter

基于 OpenID 认证协议的认证过滤器。 你需要在依赖中依赖额外的相关模块才能启用它。

# 18.DefaultLoginPageGeneratingFilter

生成默认的登录页。默认 /login 。

# 19.DefaultLogoutPageGeneratingFilter

生成默认的退出页。默认 /logout 。

# 20.ConcurrentSessionFilter

参见 2.ConcurrentSessionFilter。 该过滤器可能会被多次执行。

# 21.DigestAuthenticationFilter

Digest 身份验证是 Web 应用程序中流行的可选的身份验证机制 。DigestAuthenticationFilter 能够处理 HTTP 头中显示的摘要式身份验证凭据。

可以通过 HttpSecurity#addFilter() 来启用和配置相关功能。

# 22.BasicAuthenticationFilter

和 Digest 身份验证一样，都是 Web 应用程序中流行的可选的身份验证机制 。BasicAuthenticationFilter 负责处理 HTTP 头中显示的基本身份验证凭据。

Spring Security 在 Spring Boot 环境中，会默认使用自动装配，启用 BasicAuthenticationFilter。

BasicAuthenticationFilter 可以通过 HttpSecurity#httpBasic() 及相关方法引入，使用配置对象 HttpBasicConfigurer 来进行配置。

# 23.RequestCacheAwareFilter

用于用户认证成功后，重新恢复因为登录被打断的请求。当匿名访问一个需要授权的资源时。会跳转到认证处理逻辑，此时请求被缓存。在认证逻辑处理完毕后，从缓存中获取最开始的资源请求进行再次请求。

RequestCacheAwareFilter 可以通过 HttpScurity#requestCache() 及相关方法引入其配置对象 RequestCacheConfigurer 来进行配置。

# 24.SecurityContextHolderAwareRequestFilter

用来实现 j2ee 中 一些接口方法。比如 getRemoteUser() 方法、isUserRole() 方法在使用 Spring Security 时其实就是通过这个过滤器来实现的。

SecurityContextHolderAwareRequestFilter 可以通过 HttpSecurity#servletApi() 及相关方法引入其配置对象 ServletApiConfigurer 来进行配置。

# 25.JaasApiIntegrationFilter

适用于 JAAS（Java 认证授权服务）。 如果 SecurityContextHolder 中拥有的 Authentication 是一个 JaasAuthenticationToken，那么该 JaasApiIntegrationFilter 将使用包含在 JaasAuthenticationToken 中的 Subject 继续执行 FilterChain。

# 26.RememberMeAuthenticationFilter

用来处理”记住我“功能的过滤器。当用户没有登录而直接访问资源时，会从 Cookie 里找出用户的信息, 如果 Spring Security 能够识别出用户提供的 remember me cookie，用户将不必填写用户名和密码，而是直接登录进入系统，该过滤器默认不开启。

可以通过 HttpSecurity#rememberMe() 及相关方法引入其配置对象 RememberMeConfigurer 来进行配置。

# 27.AnonymousAuthenticationFilter

匿名认证过滤器。 对于 Spring Security 来说，所有对资源的访问都是有 Authentication 的。对于无需登录就可以直接访问的资源，会授予其匿名用户身份。

可以通过 HttpSecurity.anonymous() 及相关方法引入其配置对象 AnonymousConfigurer 来进行配置。

# 28.SessionManagementFilter

Session 管理器过滤器，内部维护了一个 SessionAuthenticationStrategy 用于管理 Session。

可以通过 HttpSecurity.sessionManagement() 及相关方法引入其配置对象 SessionManagementConfigurer 来进行配置。

# 29.ExceptionTranslationFilter

ExceptionTranslationFilter 会捕获来自过滤器链的所有异常，并进行处理。但是只处理两类异常：AccessDeniedException 和AuthenticationException 异常，其他的异常会继续抛出。

如果捕获到的 AuthenticationException，那么将会使用其对应的 AuthenticationEntryPoint 的 commence() 方法处理。在处理之前，ExceptionTranslationFilter 先使用 RequestCache 将当前的 HttpServletRequest 的信息保存起来，方便用户登录成功后可以跳转到之前的页面。

可以自定义 AuthenticationException 的处理方法。需要实现 AuthenticationEntryPoint 接口，然后重写 commence() 方法。

# 30.FilterSecurityInterceptor

这个过滤器决定了访问特定路径应该具备的权限，访问的用户的角色，权限是什么?访问的路径需要什 么样的角色和权限?这些判断和处理都是由该类进行的。

如果要实现动态权限控制，就可以使用该类 。

# 31.SwitchUserFilter

SwitchUserFilter 用来做账户切换。默认的切换账号的 url 为 /login/impersonate，默认注销切换账号的 url 为 /logout/impersonate ，默认的账号参数为 username。

可以通过此类实现自定义的账户切换。

# 32.DisableEncodeUrlFilter

DisableEncodeUrlFilter 用于禁用 URL 编码。它可以防止 Spring Security 对 URL 进行编码，从而避免一些特殊字符被编码，导致 URL 访问失败的问题。

在默认情况下，Spring Security 会对 URL 进行编码，以保证 URL 的安全性。但是，在某些情况下，我们可能希望禁用 URL 编码，例如当我们使用自定义的 URL 路径时，或者当我们需要在 URL 中传递一些特殊字符时。

# 33.ForceEagerSessionCreationFilter

ForceEagerSessionCreationFilter 是 Spring Security 中的一个过滤器，它的作用是在用户登录之前强制创建一个 Session。当用户访问需要认证的资源时，Spring Security 会自动创建一个 Session。但是在某些情况下，需要在用户登录之前就创建一个 Session，例如在使用 CAS 单点登录时，需要在跳转到 CAS 服务器之前创建一个 Session，以便在跳转回来时将认证信息存储在 Session 中。

ForceEagerSessionCreationFilter 可以配置在 Spring Security 的过滤器链中，通过在 WebSecurityConfigurerAdapter 中添加以下代码进行配置：

protected void configure(HttpSecurity http) throws Exception {
    http.addFilterBefore(new ForceEagerSessionCreationFilter(), CasAuthenticationFilter.class);
}

# 34.SecurityContextHolderFilter

SecurityContextHolderFilter 过滤器是用来替换已经过时的 SecurityContextPersistenceFilter 过滤器的。

在 Spring Security 中，一旦用户成功登录并通过身份认证，Spring Security 会将用户的认证信息存储在一个称为 SecurityContext 的上下文中。SecurityContext 中包含了当前用户的身份认证信息，通常使用 Authentication 对象来表示。

当用户访问受保护的接口时，Spring Security 会通过一系列的过滤器链来处理请求。其中一个关键的过滤器是 SecurityContextHolderFilter，它负责在每个请求之间存储和恢复SecurityContext。

具体来说，SecurityContextHolderFilter 会在每个请求开始时从存储位置（如 HttpSession）中加载 SecurityContext，并将其设置到 SecurityContextHolder 中。这样，在后续的请求处理过程中，可以通过 SecurityContextHolder 获取当前用户的认证信息。

因此，当你成功登录后，Spring Security 会在 SecurityContext 中保存你的认证信息，并在每个请求中使用 SecurityContextHolderFilter 将其恢复，以便在后续的请求处理中识别你作为已认证的用户。这样，Spring Security 就知道你是已经认证的用户了。

# 35.AuthorizationFilter

AuthorizationFilter 是一种用于进行权限验证和授权的过滤器。它是 Spring Security 的核心组件之一，用于保护你的应用程序资源免受未经授权的访问。

AuthorizationFilter 的主要功能是根据预定义的权限规则，对请求进行权限验证。它通常在用户认证成功后，对用户的访问请求进行拦截，并检查用户是否具有足够的权限来执行该请求。

需要注意的是，AuthorizationFilter 只负责权限验证和授权，它并不处理用户的身份认证。用户身份认证通常由 UsernamePasswordAuthenticationFilter 或其他相关的组件来处理。

# 二、内置过滤器的排序规则

# 1.内置过滤器的排序

在 Spring Security 初始化核心过滤器时，HttpSecurity 会将 Spring Security 内置的一些过滤器进行注册。

org.springframework.security.config.annotation.web.builders.FilterOrderRegistration ，FilterOrderRegistration 提供了一些规则进行比较，然后根据比较的结果进行排序。

当前我使用的版本是 Spring Boot 2.7.11，Spring Security 5.7.8。之前老版本使用的是 FilterComparator 来对过滤器进行排序的。

由于不同版本的实现不同，其他版本大家可以自行查看初始化逻辑。

如下代码，HttpSecurity 中的 filterOrders 变量的作用就是用来获取过滤器执行的序号的。

public final class HttpSecurity extends AbstractConfiguredSecurityBuilder<DefaultSecurityFilterChain, HttpSecurity>
		implements SecurityBuilder<DefaultSecurityFilterChain>, HttpSecurityBuilder<HttpSecurity> {
  
	private FilterOrderRegistration filterOrders = new FilterOrderRegistration();

  // ......
  
}

      this.put(HeaderWriterFilter.class, order.next());

      this.put(LogoutFilter.class, order.next());

      this.filterToOrder.put("org.springframework.security.saml2.provider.service.servlet.filter.Saml2WebSsoAuthenticationRequestFilter", order.next());
      this.put(X509AuthenticationFilter.class, order.next());
      this.put(AbstractPreAuthenticatedProcessingFilter.class, order.next());
      this.filterToOrder.put("org.springframework.security.cas.web.CasAuthenticationFilter", order.next());

      this.filterToOrder.put("org.springframework.security.saml2.provider.service.servlet.filter.Saml2WebSsoAuthenticationFilter", order.next());
      this.put(UsernamePasswordAuthenticationFilter.class, order.next());
      order.next();
      this.filterToOrder.put("org.springframework.security.openid.OpenIDAuthenticationFilter", order.next());
      this.put(DefaultLoginPageGeneratingFilter.class, order.next());

      this.put(RequestCacheAwareFilter.class, order.next());
      this.put(SecurityContextHolderAwareRequestFilter.class, order.next());

      this.put(FilterSecurityInterceptor.class, order.next());
      this.put(AuthorizationFilter.class, order.next());
      this.put(SwitchUserFilter.class, order.next());
  }

}

# 2.手动注册过滤器的排序

而我们进行手动添加过滤器时，就会调用 HttpSecurity 的 addFilter() 、addFilterAfter() 、addFilterBefore() 方法。

如下代码，在 HttpSecurity 的 addFilter() 方法和 addFilterAtOffsetOf() 方法中，我们在手动添加过滤器时，就是通过调用 this.filterOrders.getOrder() 方法来获取序号的，filterOrders 变量的类型就是我们上面提到的 FilterOrderRegistration。

public final class HttpSecurity extends AbstractConfiguredSecurityBuilder<DefaultSecurityFilterChain, HttpSecurity>
		implements SecurityBuilder<DefaultSecurityFilterChain>, HttpSecurityBuilder<HttpSecurity> {

  // ......
  
	@Override
	public HttpSecurity addFilterAfter(Filter filter, Class<? extends Filter> afterFilter) {
		return addFilterAtOffsetOf(filter, 1, afterFilter);
	}

	@Override
	public HttpSecurity addFilterBefore(Filter filter, Class<? extends Filter> beforeFilter) {
		return addFilterAtOffsetOf(filter, -1, beforeFilter);
	}

	private HttpSecurity addFilterAtOffsetOf(Filter filter, int offset, Class<? extends Filter> registeredFilter) {
		// 根据registeredFilter的类型获取序号
    Integer registeredFilterOrder = this.filterOrders.getOrder(registeredFilter);
		if (registeredFilterOrder == null) {
			throw new IllegalArgumentException(
					"The Filter class " + registeredFilter.getName() + " does not have a registered order");
		}
    // offset=-1，则序号会比所有同类型的过滤器的序号都小，因此会在同类型的过滤器中优先执行
    // offset=1，则序号会比所有同类型的过滤器的序号都大，因此会在同类型的过滤器中最后执行
		int order = registeredFilterOrder + offset;
		this.filters.add(new OrderedFilter(filter, order));
    // 添加自定义的类型到filterOrders中
		this.filterOrders.put(filter.getClass(), order);
		return this;
	}

	@Override
	public HttpSecurity addFilter(Filter filter) {
		Integer order = this.filterOrders.getOrder(filter.getClass());
		if (order == null) {
			throw new IllegalArgumentException("The Filter class " + filter.getClass().getName()
					+ " does not have a registered order and cannot be added without a specified order. Consider using addFilterBefore or addFilterAfter instead.");
		}
		this.filters.add(new OrderedFilter(filter, order));
		return this;
	}

  // ......
}

# 三、Spring Security 的过滤器链

# 1.Spring Security 过滤器链

当客户端向后端服务发送请求时，Spring Security 会根据请求的 URI 的路径来确定该请求的过滤器链（Filter）以及最终的具体 Servlet 控制器（Controller）。

Spring Security 在 Servlet 的一个 Filter（FilterChainProxy）中，调用整个 Spring Security 的过滤器链。在 FilterChainProxy 中，实际内部代理着众多的 Spring Security Filter 。

# 2.Spring Security 过滤器链的初始化

所有 Filter 的初始化过程：

  • Filter 按照一定的顺序被 SecurityBuilder 的实现来组装成 SecurityFilterChain
  • 然后，通过 WebSecurity 将 SecurityFilterChain 注入到 FilterChainProxy 中，
  • 接着，在 WebSecurityConfiguration 中，FilterChainProxy 被注册到 Spring 容器中，Bean 名称为 springSecurityFilterChain。
  • 实际上，还有一个隐藏层：DelegatingFilterProxy 委托代理了 FilterChainProxy（Bean name=springSecurityFilterChain） ，将自己注册到 Servlet 的过滤器链中。

由于 Spring Security 的 Filter 是在 Spring Security 的内部创建的，并未交给 Spring IOC 管理，因此 Spring Security 的内置 Filter 对于 Spring IOC 容器来说都是不可见的。

Spring Security 允许多条过滤器链并行，Spring Security 的 FilterChainProxy 可以代理多条过滤器链并根据不同的 URI 匹配策略进行分发。但是每个请求每次只能被分发到一条过滤器链。如下图：

实际上，每条过滤器链就是一个 SecurityFilterChain。

# 1.Servlet Filter 体系

在 Servlet 体系中，客户端发起一个请求，会先经过 0 到 N 个 Filter ，最后才交给 Servlet 处理。

Filter 不但可以修改 HttpServletRequest 和 HttpServletResponse ，还可以让我们在请求响应的前后做一些事情，甚至可以终止过滤器链 FilterChain 的传递。

# 4.FilterChainProxy

一个请求需要经过多个安全过滤器的处理，例如身份验证过滤器、授权过滤器、异常处理过滤器等。FilterChainProxy 就是用来管理这些安全过滤器的。

FilterChainProxy 会根据配置的安全过滤器链，将请求分发给相应的过滤器进行处理。每个过滤器都会对请求进行特定的安全处理，例如身份验证、授权、异常处理等。当所有的过滤器都处理完毕后，FilterChainProxy 会将处理结果返回给调用方。

FilterChainProxy 还支持动态配置，可以根据不同的请求路径或请求方法，选择不同的安全过滤器链进行处理。这样可以实现更加灵活和精细的安全控制。

下图为 FilterChainProxy 的角色：

由于不同的请求会匹配到不同的 SecurityFilterChain，所以就需要通过 FilterChainProxy 筛选出符合要求的 SecurityFilterChain，然后链式的执行这些 Filter 。

FilterChainProxy 的源码中，则通过 getFilters 方法来匹配请求对应的 SecurityFilterChain。如下：

public List<Filter> getFilters(String url) {
    return getFilters(this.firewall.getFirewalledRequest(new FilterInvocation(url, "GET").getRequest()));
}

private List<Filter> getFilters(HttpServletRequest request) {
    int count = 0;
    for (SecurityFilterChain chain : this.filterChains) {
        if (logger.isTraceEnabled()) {
          logger.trace(LogMessage.format("Trying to match request against %s (%d/%d)", chain, ++count,
                this.filterChains.size()));
        }
        if (chain.matches(request)) {
            return chain.getFilters();
        }
    }
    return null;
}

在 FilterChainProxy 中，最终使用的是 VirtualFilterChain 来调用 Spring Security 的过滤器链的。VirtualFilterChain 是 FilterChainProxy 的一个内部类，其源码如下：

private static final class VirtualFilterChain implements FilterChain {

		private final FilterChain originalChain;

		private final List<Filter> additionalFilters;

		private final int size;

		private int currentPosition = 0;
 	
  	// code...
  
  	@Override
		public void doFilter(ServletRequest request, ServletResponse response) throws IOException, ServletException {
        // 如果当前的过滤器链执行完毕，则继续执行Servlet的过滤器链
      	if (this.currentPosition == this.size) {
            this.originalChain.doFilter(request, response);
            return;
        }
      	// 记录当前执行的offset
        this.currentPosition++;
      	// 获取到当前执行的过滤器链
        Filter nextFilter = this.additionalFilters.get(this.currentPosition - 1);
        if (logger.isTraceEnabled()) {
            String name = nextFilter.getClass().getSimpleName();
            logger.trace(LogMessage.format("Invoking %s (%d/%d)", name, this.currentPosition, this.size));
        }
      	// 执行filter，并将FilterChain传递下去，以便继续执行后面的filter
        nextFilter.doFilter(request, response, this);
		}

	}

}

# 5.SecurityFilterChain

业务中，经常会有不同 Ant Pattern 的请求对应不同的过滤器链的需求。比如，登录会去验证，然后返回登录结果；管理后台的接口走后台的安全逻辑；应用客户端的接口走客户端的安全逻辑。

于是，Spring Security 提供了一个 SecurityFilterChain 接口来满足被匹配 HttpServletRequest 走特定的过滤器链的需求。

# 6.Spring Security 的过滤器体系

最后，Spring Security 的过滤器体系就逐渐清晰，如下图：

总结：

  1. Servlet Filter：在 Servlet 的过滤器链中调用 DelegatingFilterProxy。
  2. DelegatingFilterProxy：在 DelegatingFilterProxy 中对 FilterChainProxy 注入依赖的 Bean 对象。
  3. FilterChainProxy：在 FilterChainProxy 中，通过调用 SecurityFilterChain 的 matches 方法匹配对应的 SecurityFilterChain。然后执行匹配到的 SecurityFilterChain 中的所有 过滤器。
  4. SecurityFilterChain：用于映射请求和过滤器集合的关系。

# 7.ExceptionTranslationFilter

ExceptionTranslationFilter 用来捕获由其他过滤器或处理器抛出的异常，并将其转换为Spring Security的异常体系中的异常，以便进行统一处理。

ExceptionTranslationFilter 会监听 Spring Security 的异常体系中的两个异常：AuthenticationException 和 AccessDeniedException。当这两个异常被抛出时，ExceptionTranslationFilter 会根据配置的异常处理器，将异常转换为相应的响应，例如跳转到登录页面或返回错误信息。
