01.物联网协议之 MQTT

# 一、MQTT 介绍

MQTT 官网 (opens new window)

MQTT 中文网 (opens new window)

MQTT（Message Queuing Telemetry Transport，消息队列遥测传输协议），是一种基于发布/订阅（publish/subscribe）模式的"轻量级"通讯协议，该协议构建于 TCP/IP 协议上。MQTT 以极少的代码和有限的带宽，为连接远程设备提供实时可靠的消息服务。作为一种低开销、低带宽占用的即时通讯协议，使其在物联网、小型设备、移动应用等方面有较广泛的应用。

MQTT 是一个基于客户端-服务器的消息发布/订阅传输协议，并且可以有效地将消息发送给一个或多个接收端。在很多情况下，包括受限的环境中。如：机器与机器（M2M）通信和物联网（IoT）。其在偶尔拨号的医疗设备、智能家居、水电表等物联网设备中已广泛使用。

# 2.特点

  • 轻量级，开放消息协议，简单易实现
  • 发布—订阅模式，一对多消息发布
  • 基于 TCP/IP 网络连接，提供有序，无损，双向连接。
  • 1 字节固定报头，2 字节心跳报文，最小化传输开销和协议交换，有效减少网络流量。
  • 消息 QoS 支持，可靠传输保证，针对物联网中多变的网络环境提供的多种服务质量等级
  • 适用于低功耗需求的优化设计
  • 支持在物联网应用数据安全

消息 QoS（Quality of Service）是指消息的服务质量，它是消息的发送者和接收者之间达成的一个协议，用于保证特定消息的交付级别。

# 3.应用场景

MQTT 协议出现的时间很久远，并且长期的使用和积淀使得 MQTT 的生态非常完善。

MQTT 的常见应用场景如下：

  • 物联网 M2M 通信，物联网大数据采集
  • Android 消息推送，Web 消息推送
  • 移动即时消息，例如 Facebook Messenger
  • 智能硬件、智能家具、智能电器
  • 车联网通信，电动车站桩采集
  • 智慧城市、远程医疗、远程教育
  • 电力、石油与能源等行业市场

# 4.MQTT 协议原则

MQTT 协议设计原则：

  1. 精简。不添加可有可无的功能；
  2. 发布/订阅（Pub/Sub）模式，方便消息在传感器与服务之间传递，将 Client/Server 解耦，不必预先知道对方的存在（ip/port），不必同时运行；
  3. 允许用户动态创建主题（不需要预先创建主题），零运维成本；
  4. 把传输量降到最低以提高传输效率；
  5. 把低带宽、高延迟、不稳定的网络等因素考虑在内；
  6. 支持连续的会话保持和控制（心跳）；
  7. 理解客户端计算能力可能很低；
  8. 提供服务质量（quality of service level. 即 QoS）管理;
  9. 不强求传输数据的类型与格式，保持灵活性(指的是应用层业务数据)。

# 二、MQTT 协议内容

# 1.角色

MQTT 协议中又三种角色：发布者（Publish）、代理服务（Broker）、订阅者（Subscribe）。其中，消息的发布者和订阅者都是客户端，消息代理是服务器，消息发布者可以同时是订阅者。

  • MQTT 客户端可以进行如下处理：
    1. 发布消息，若其他客户端订阅的主题与发布消息的主题一致，则其他客户端可以订阅到该小消息
    2. 订阅其它客户端发布的消息
    3. 退订或删除应用程序的消息
    4. 断开与服务器连接
  • MQTT服务器端又称为“消息代理”（Broker），它位于消息发布者和订阅者之间，它可以进行如下处理：
    1. 接受来自客户端的网络连接
    2. 接受客户端发布的消息
    3. 处理来自客户端的订阅和退订请求
    4. 向订阅的客户转发消息。

MQTT Broker 消息服务有很多开源产品 ，如：

  • Eclipse Mosquitto (opens new window)：Eclipse 开源，使用 C 语言实现的 MQTT Broker。
  • EMQ X (opens new window)：使用 Erlang 语言开发的 MQTT Broker，支持许多其他 IoT 协议比如 CoAP、LwM2M 等。
  • Mosca (opens new window)：使用 Node.JS 开发的 MQTT Broker，简单易用。
  • VerneMQ (opens new window)：使用 Erlang 开发的 MQTT Broker
  • ……

# 2.主题（Topic）

MQTT 协议传输的消息分为：主题（Topic）和负载（Payload）。

  • 主题（Topic）：可以理解为消息的类型，订阅者订阅主题后，就会收到该主题的消息内容（Payload）;
  • 负载（Payload）：可以理解为消息的内容，是指订阅者具体要使用的内容。

MQTT 将建立客户端到服务器的连接，使两者之间的一个有序的、无损的、基于字节流的双向传输的连接。当应用数据通过 MQTT 网络发送时，MQTT 会把与之相关的服务质量（QoS）和主题名（Topic）相关连

MQTT 是基于发布（Publish）—订阅（Subscribe） 模式来进行通信及数据交换的，与 HTTP 协议的请求（Request）—响应（Response）模式不同。

订阅者（Subscriber）会向消息服务器（Broker） 订阅一个 主题（Topic）。订阅者成功订阅后，消息服务器会将该主题下的消息转发给所有的订阅者。

主题（Topic）以 / 为分隔符以区分不同的层级。

  • 包含通配符 + 或 # 的主题又称为主题过滤器（Topic Filters）。例如：sensor/+/temperature、$SYS/broker/metrics/#
    • +： 表示通配一个层级，例如 a/+，匹配 a/x、a/y
    • #：表示通配多个层级，例如 a/#，匹配 a/x, a/b/c/d
  • 不含通配符的称为主题名（Topic Names）。例如：/chat/room/1、/$SYS/broker/metrics/packets/received

发布者（Publisher）只能向“主题名”发布消息，订阅者（Subscriber） 则可以通过订阅“主题过滤器”来通配多个主题名称。

# 3.MQTT 协议的数据包结构

MQTT 协议数据包包含固定头（Fixed Header）， 可变头（Variable Header）和消息载体（Payload）。

  • 固定头：存在于所有 MQTT 数据包中，表示数据包类型及数据包的分组类标识， 如连接，发布，订阅，心跳等。其中固定头是必须的，所有类型的 MQTT 协议中，都必须包含固定头。
  • 可变头：存在于部分 MQTT 数据包中，数据包类型决定了可变头是否存在及其具体内容。可变头部不是可选的意思，而是指这部分在有些协议类型中存在，在有些协议中不存在。
  • 消息载体：存在于部分 MQTT 数据包中，表示客户端收到的具体内容。 与可变头一样， 在有些协议类型中有消息内容，有些协议类型中没有消息内容。

# 3.1 固定头

MQTT 是一个轻量级的网络协议。由于物联网中的设备大多计算资源有限、网络带宽低，因此它在物联网系统中十分流行。

MQTT 的轻量级体现在两个方面：

  • 一个是 MQTT 消息使用二进制编码格式。
  • 另外一个是消息的具体交互流程设计非常简单。

MQTT 消息使用二进制编码格式，可以减少网络传输的数据量。MQTT 协议中的一个字节可以表示很多的内容，如下表为 MQTT 固定头的格式：

Bit 7 6 5 4 3 2 1 0
字节1 MQTT 数据包类型 不同类型 MQTT 数据包标识位
字节2 延续位 可变头部和消息体的字节个数

上面的固定头中，第一个字节分成了高 4 位（4～7）和低 4 位（0～3）：

  • 低 4 位是数据包标识位，其中的每一比特位又可以表示不同的含义；
  • 高 4 位是不同数据包类型的标识位。

第二个字节表示数据包头部和消息体的字节共个数，其中最高位（第7位）表示有没有第三字节存在，来和剩余的位数一起表示字节共个数。

如果有第三个字节，那它的最高位表示是否有第四个字节，来和第二个字节、第三个字节一起表示字节总个数。依此类推，还可能有第四个字节、第五个字节，不过这个表示可变头部和消息体的字节个数的部分，最多也只能到第五个字节，所以可以表示的最大数据包长度为 256 MB。

# 3.2 MQTT 报文类型

MQTT 协议中定义了一些方法（也称为动作），来于表示对确定资源所进行操作。这个资源可以代表预先存在的数据或动态生成数据，这取决于服务器的实现。通常来说，资源指服务器上的文件或输出。

主要方法如下：

信息类型 数值（二进制） 流方向 描述
CONNECT 0001 Client –> Broker Client 请求连接到 Broker
CONNACK 0010 Client <– Broker Broker 确认连接请求
PUBLISH 0011 Client <–> Broker 发布主题消息
PUBACK 0100 Client <–> Broker 主题消息的发送方收到的确认消息
PUBREC 0101 Client <–> Broker QoS2 中第一步发布确认
PUBREL 0110 Client <–> Broker QoS2 中第二步发布确认
PUBCOMP 0111 Client <–> Broker QoS2 中第三步发布确认
SUBSCRIBE 1000 Client –> Broker Client 请求订阅主题消息
SUBACK 1001 Client <– Broker Broker 确认订阅请求
UNSUBSCRIBE 1010 Client –> Broker Client 请求取消主题消息订阅
UNSUBACK 1011 Client <– Broker Broker 确认取消消息订阅
PINGREQ 1100 Client –> Broker Client 发送心跳（Client 发送 PING 请求）
PINGRESP 1101 Client <– Broker Broker 的心跳响应（Broker 回应 PING）
DISCONNECT 1110 Client –> Broker Client 中断连接请求

# 3.3 可变头

可变头的意思是可变化的消息头部。有些报文类型包含可变头部，有些报文则不包含。可变头部在固定头部和消息内容之间，其内容根据报文类型不同而不同。

常见的一种可变头：Packet Identifier（消息 ID）。一个消息 ID 包含2字节，高字节在前，低字节在后。包含 Packet Identifier 的协议类型有：PUBLISH(QoS > 0)、PUBACK、PUBREC、PUBREL、PUBCOMP、SUBSCRIBE、SUBACK、 UNSUBSCRIBE、UNSUBACK。

消息 ID 默认是从1开始并自增，如果一个消息 ID 被用完后，这个消息 ID 可以被重用。

  • 对于 PUBLISH（QoS 1） 来说，如果发送端接收到 PUBACK，那么这个消息 ID 就用完了。
  • 对于 PUBLISH（QoS 2），如果接收方收到 PUBCOMP， 那么这个消息ID就用完了。
  • 对于 SUBSCRIBE 和 UNSUBSCRIBE，消息 ID 使用完成的标记是发送方收到了对应的 SUBACK 和 UNSUBACK。

另外客户端和服务端的消息 ID 是独立分配的，客户端和服务端可以同时使用同一个消息 ID。

需要报文标识符的 MQTT 报文类型如下：

名字 值
CONNECT 不需要
CONNACK 不需要
PUBLISH 需要（如果QoS>0）
PUBACK 需要
PUBREC 需要
PUBREL 需要
PUBCOMP 需要
SUBSCRIBE 需要
SUBACK 需要
UNSUBSCRIBE 需要
UNSUBACK 需要
PINGREQ 不需要
PINGRESP 不需要
DISCONNECT 不需要
AUTH 不需要

# 3.4 消息体

有些报文类型是包含消息载体（Payload）的，如 PUBLISH 的 Payload 就是指消息内容（应用程序发布的消息内容）。而 CONNECT 的 Payload 则包含 Client Identifier，Will Topic，Will Message，Username，Password 等信息。

包含 Payload 的报文类型如下：

MQTT 控制报文 有效载荷
CONNECT 需要
CONNACK 不需要
PUBLISH 可选
PUBACK 不需要
PUBREC 不需要
PUBREL 不需要
PUBCOMP 不需要
SUBSCRIBE 需要
SUBACK 需要
UNSUBSCRIBE 需要
UNSUBACK 需要
PINGREQ 不需要
PINGRESP 不需要
DISCONNECT 不需要
AUTH 不需要

# 4.低功耗特性

MQTT 协议的低功耗体现在能耗和通信次数上。

当 Client 和 Broker 的连接中断时，MQTT 协议为了使得双方及时了解，重新建立连接，保证主题消息的可靠性传输，提供了 Keepalive 机制。

传统的 Keepalive 机制的原理：Client 和 Broker 都基于 Keepalive 确定的时间长度，来判断一段时间内是否有消息在双方之间传输，如果超出这个时间长度，双方没有收到新的数据包，那么就判定连接断开。

需要注意的是，Keepalive 要求一段时间内必须有数据包传输，但实际情况是，Client 和 Broker 不可能时时刻刻都在传输主题消息。

  • MQTT 协议为了解决这个问题，定义了 PINGREQ 和 PINGRESP 这两种消息类型。这两种消息类型都没有可变头部和消息体，只有 2 个字节大小。Client 和 Broker 通过分别发送 PINGREQ 和 PINGRESP 消息以满足 Keepalive 机制的要求。
  • 并且，在 Keepalive 时间长度内，Client 和 Broker 之间有数据传输，就不会额外的发送 PINGREQ 和 PINGRESP 消息了。

在 MQTT 5.0 中，新增了重复主题特性。Client 在重复发送一个主题的消息时，可以从第二次开始，将主题名长度设置为 0，这样 Broker 会自动按照上次的主题来处理消息。这种情况对传感器设备来说十分常见，所以这个特性在工作中很有实际意义。

# 5.可靠通信—QoS

物联网设备经常遇到网络环境不稳定的问题。为了解决不稳定的通信条件，MQTT 协议设计了 3 种不同的 QoS（Quality of Service 服务质量）级别。

3 种服务质量的级别如下：

  1. QoS 0，表示消息最多收到一次，即消息可能丢失，但是不会重复。
  2. QoS 1，表示消息至少收到一次，即消息保证送达，但是可能重复。
  3. QoS 2，表示消息只会收到一次，即消息有且只有一次。

上图中的 QoS 2 为了保证有且只有一次的可靠传输，流程相对更加复杂。

正常情况下，QoS 2 有 PUBLISH、PUBREC、PUBREL 和 PUBCOMP 4 次交互。

不正常的情况下，发送方需要重复发送消息。比如：

  • 一段时间内没有收到 PUBREC消息，就需要再次发送 PUBLISH 消息。不过要注意，这时要把消息中的 “重复”标识设置为 1，以便接收方能正确处理。
  • 同样地，如果没有收到 PUBCOMP 消息，发送方就需要再次发送 PUBREL 消息。

# 6.安全传输

MQTT 提供了“用户名/密码”的机制，在建立连接过程中，它可以通过判断用户名和密码的正确性，来筛选有效连接请求。

另外，MQTT 协议支持 SSL/TLS 加密通信方式。使用 SSL/TLS 加密通信方式后，MQTT 协议将转换为 MQTTS 协议。并且，我们在连接的使用时候，还需要将 mqtt://test.mosquitto.org:1883 修改为 mqtts://test.mosquitto.org:8883。并且在连接时使用客户端证书。

使用如下命令可订阅 Broker 服务 test.mosquitto.org:1883 主题为 /willingoxjin/iot 的消息：

mosquitto_sub -h test.mosquitto.org -p 1883 -t "/willingoxjin/iot" -v
1

参数说明：

  • -h 要连接到的 MQTT Broker 服务器。默认为 localhost。
  • -p 要连接的网口。对于普通 MQTT 默认为 1883，对于基于 TLS 的 MQTT 默认为 8883。
  • -t 要订阅的MQTT主题。可以重复多次。
  • -v 详细地打印已发布的消息。

使用如下命令，可向 Broker 服务 test.mosquitto.org:1883 主题为 /willingoxjin/iot 发布消息：

mosquitto_pub -h test.mosquitto.org -p 1883 -t "/willingoxjin/iot" -m "Hello,mqtt"
1

参数说明：

  • -m 发布的消息内容

若想要使用 mqtts 协议，需要下载证书。证书链接地址：https://test.mosquitto.org/ssl/mosquitto.org.crt (opens new window)

并使用 --cafile 指定客户端证书文件。如下：

mosquitto_sub -h test.mosquitto.org -p 8883 -t "/willingoxjin/iot" -v --cafile ~/mosquitto.org.crt

mosquitto_pub -h test.mosquitto.org -p 8883 -t "/willingoxjin/iot" -m "Hello,mqtt" -cafile ~/mosquitto.org.crt
